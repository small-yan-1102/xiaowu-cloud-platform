## 三、管理规范

### 3.1 首次 Clone

```bash
# 进入目标系统的 code 目录
cd systems/AMS/code

# clone 仓库
git clone git@xxx/ams-server.git ams-server
git clone git@xxx/ams-web.git ams-web
```

### 3.2 迭代后更新

每个迭代测试启动前，执行以下操作保持代码最新：

```bash
# 方式一：手动更新指定仓库
cd systems/AMS/code/ams-server
git pull origin main

# 方式二：使用批量更新脚本（推荐）
# 在项目根目录执行：
powershell -File scripts/update-all-repos.ps1
```

### 3.3 切换迭代分支

测试特定迭代的功能时，可以切换到对应分支：

```bash
cd systems/AMS/code/ams-server
git fetch origin
git checkout feature/video-takedown    # 切换到迭代功能分支
# 测试完成后切回主分支
git checkout main
```

### 3.4 代码用途说明

代码仓库的主要用途（只读参考，不修改代码）：

| 用途 | 说明 | 示例 |
|------|------|------|
| 功能梳理 | 阅读路由/模型/配置梳理系统功能 | 阅读 router/ 目录了解 API 清单 |
| PRD 审阅辅助 | 对照代码验证 PRD 中的技术可行性 | 确认某字段是否已有数据库定义 |
| 测试用例设计 | 参考代码中的校验逻辑设计边界用例 | 阅读校验函数了解实际边界值 |
| 变更影响分析 | 通过 git diff 了解迭代代码变更范围 | `git diff main..feature/xxx --stat` |
| 接口分析 | 阅读接口定义辅助接口测试 | 查看 controller 层了解参数和返回 |

## 四、与知识库的协同

代码仓库与 `knowledge/` 知识文档的协同更新流程：

```
迭代启动
    │
    ├── 1. git pull 更新代码
    │
    ├── 2. 阅读代码变更
    │       └── git diff main..feature/xxx
    │
    ├── 3. 更新 knowledge/功能清单.md
    │       └── 新增/修改功能模块记录
    │
    └── 4. 更新 knowledge/changelog.md
            └── 追加本次迭代变更记录
```

## 五、注意事项

1. **只读使用**：代码仓库仅用于阅读参考，不在此目录进行代码修改和提交
2. **不追踪 code/**：如果本项目目录使用 git 管理，应在 `.gitignore` 中排除所有 `code/` 子目录下的仓库
3. **磁盘空间**：大型仓库可使用 `git clone --depth 1` 进行浅克隆，减少磁盘占用
4. **敏感信息**：代码中可能包含配置文件，注意不要将包含敏感信息的配置文件分享到外部
