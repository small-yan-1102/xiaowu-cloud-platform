# distribution-server 权限逻辑分析

> 分析范围：`distribution-server`（后端）+ `distribution-server-web`（前端）
> 最后更新：2026-03-18

---

## 1. 系统概述

distribution-server 是面向分销商团队的资源管理系统，包含两大一级菜单：

| 菜单 | 路径 | 子菜单 |
|------|------|--------|
| 资源管理 | `resourceManagement` | 海外频道、作品管理、作品登记 |
| 系统设置 | `systemSetting` | 员工管理 |

---

## 2. 菜单树（含路由编码）

```
资源管理（resourceManagement）
├── 海外频道（ytChannel-list）       /resourceManagement/ytChannel/List
├── 作品管理（works-management-list） /resourceManagement/works-management/List
└── 作品登记（work-registration-list）/resourceManagement/work-registration/List

系统设置（systemSetting）
└── 员工管理（employeeManagement-list）/systemSetting/employeeManagement/List
```

## 3. 权限体系总览

系统采用 **三级权限模型**：菜单权限 → 页面（路由）权限 → 按钮权限。

### 3.1 权限数据来源

- 前端在路由守卫（`router/index.js`）中登录后调用接口 `POST /appApi/user/userAuthorityList`，后端返回：
  - `data.page`：页面/菜单权限列表（含 `code`、`type`）
  - `data.btn`：按钮权限列表（含 `code`）
- 权限数据存入 Vuex Store（`auth` 模块），分别存储：
  - `buttonAuth`：按钮权限 code 数组
  - `routerAuth`：路由权限列表
  - `menus`：动态菜单列表

### 3.2 菜单权限过滤逻辑

```
userAuthorityList 返回 page 列表
        ↓
handlePermissionMenu(menuList, codeList)   ← routerUtils.js
        ↓
对 routerList.js 中 menuList 进行过滤：
  子菜单的 code/route/path 与 page 列表中任一 code 匹配，则保留
        ↓
store.dispatch('auth/saveMenu', newMenu)
        ↓
侧边导航栏仅渲染有权限的菜单项
```

### 3.3 路由权限过滤逻辑

```
handlePermissionData(routes, codeList)     ← routerUtils.js
        ↓
同上机制过滤 routerList.js 中 routes
        ↓
extractWithoutChildren 取出叶子路由
        ↓
router.addRoute(item) 动态注册路由
        ↓
无权限路由直接 404，无法通过 URL 直接访问
```

### 3.4 按钮权限控制

前端提供两种方式控制按钮级权限：

#### 方式一：自定义指令 `v-auth`

```javascript
// directives/modules/anth.js
// 原理：DOM inserted 时检查 buttonAuth 中是否包含该按钮 code，不包含则直接 el.remove()
v-auth="['ytChannel-allocate-btn']"         // 单个或多个 code，any 匹配即显示
```

#### 方式二：工具函数 `can()`

```javascript
// utils/useAccess.js
// 原理：遍历 buttonAuth，检查是否包含指定 code，返回 true/false
can(['ytChannel-bind-btn'])                  // 用于 JS 逻辑判断（如 Tab 显隐）
canLike(['prefix'])                          // 模糊匹配，buttonAuth[i].includes(roleList[j])
```

## 4. 各菜单权限详细分析

### 4.1 海外频道（ytChannel）

**菜单访问权限**：`ytChannel-list`（由后端 userAuthorityList 下发）

**Tab 页签权限**（按钮级）：

| Tab 名称 | 按钮 code | 权限行为 |
|---------|-----------|---------|
| 运营中 | `ytChannel-bind-btn` | `can()` 判断，无权限则 Tab 不显示 |
| 已解约 | `ytChannel-unBind-btn` | `can()` 判断，无权限则 Tab 不显示 |

**操作按钮权限**（按钮级）：

| 按钮 | 按钮 code | 位置 | 控制方式 |
|------|-----------|------|---------|
| 分配（运营人员） | `ytChannel-allocate-btn` | 列表行操作列 | `v-auth` 指令，无权限则按钮 DOM 被移除 |

**初始化 Tab 逻辑**（权限决定默认 Tab）：

```javascript
// onMounted 中
if (can(['ytChannel-bind-btn'])) {
  activeIn.value = 'bindCount'        // 默认显示运营中
} else if (can(['ytChannel-unBind-btn'])) {
  activeIn.value = 'terminateCount'   // 无运营中权限则默认已解约
}
```

**后端接口权限**（`@NeedLogin`）：

| 接口 | 方法 | URL | 鉴权 |
|------|------|-----|------|
| 海外频道分页列表 | POST | `/appApi/targetChannel/pageList` | `@NeedLogin` |
| Tab 统计 | POST | `/appApi/targetChannel/tabStatistic` | `@NeedLogin` |
| 分配运营人员 | POST | `/appApi/targetChannel/allocateOperator` | **无 `@NeedLogin`**（⚠️ 鉴权缺失） |
| 运营人员下拉 | POST | `/appApi/team/teamMemberDictList` | `@NeedLogin` |

**分配运营人员弹窗**（AddEdit.vue）：
- 弹窗打开时读取当前行的 `operatorId` 预填入运营人员字段作为默认值
- 运营人员下拉内容来自 `teamMemberDictList`（`includeDisable: false`，`includeAdmin: true`），经过数据权限过滤
- 运营人员字段必填

## 5. 列表数据权限（行级数据隔离）

系统通过 `管理范围（authType）` 字段实现 **行级数据隔离**，决定每个用户能看到哪些数据行。

### 5.1 authType 枚举值

| authType 值 | 含义 | 数据过滤行为 |
|------------|------|-----------|
| `1` | 本部门及下级（department） | **代码实际行为：仅可见自己的数据**（scopeList 不展开部门，与 authType=3 逻辑相同，仅能看自己一人的数据） |
| `2` | 全部员工（ALL） | **不添加任何 operatorId 过滤**，可见全部数据 |
| `3` | 按员工选择（SCOPE） | 仅可见自己 + scopeList 中指定员工的数据 |

> ⚠️ **注意**：`authType=1`（本部门及下级）在当前代码实现中，`managedUserList` 进入的是与 authType=3 相同的逻辑分支——只添加自己（`operatorIds.add(userId)`），不进行部门树展开。若期望实现「本部门及下级」的树形展开，代码尚未支持。

> 代码来源：`AuthType.java`（枚举）；`MemberAuthClientImpl.managedUserList()`（核心实现）

### 5.2 核心数据权限方法：`managedUserList(teamId, userId)`

```java
// MemberAuthClientImpl.java
public List<String> managedUserList(String teamId, String userId) {
    MemberAuth memberAuth = queryByTeamUserId(teamId, userId);
    if (memberAuth == null) return []  // 无权限配置 → 返回空，查询不到任何数据

    if (AuthType.ALL == memberAuth.getAuthType()) {
        return []  // authType=2（全部）→ 不限制，返回空列表（不加 IN 条件）
    }
    // authType=1 或 3：先加自己
    operatorIds.add(userId)
    if (AuthType.SCOPE == memberAuth.getAuthType()) {
        // authType=3：再加指定人员
        operatorIds.addAll(scopeList中的managedUserId)
    }
    return operatorIds  // SQL 中 WHERE operatorId IN (operatorIds)
}
```

### 5.3 各菜单数据权限应用路径

#### 海外频道（ytChannel）

```
managedUserList(teamId, currentUserId)
        ↓
operatorIds（可能为空=不限制，或含指定人员列表）
        ↓
TargetChannel 表 WHERE teamId=? AND operatorId IN (operatorIds)  ← 仅当 operatorIds 非空时加此条件
        ↓
仅展示当前用户管理范围内的海外频道行
```

**Tab 统计**（运营中/已解约数量）同样经过 `managedUserList` 过滤，Tab 上的数字只反映当前用户可见范围。

#### 作品管理（works-management）

**作品服务包下拉（compositionServicePackageList）** 也应用数据权限：

```
managedUserList(teamId, currentUserId) → operatorIds
        ↓
作品服务包表 WHERE operatorId IN (operatorIds)  ← 仅当非空时加此条件
        ↓
仅展示当前用户管理范围内作品的服务包下拉选项
```

**列表数据权限**：

```
managedUserList(teamId, currentUserId)
        ↓
operatorIds 合并前端传入的 operatorIdList（搜索框筛选人员）
        ↓
Composition 表 WHERE teamId=? AND operatorId IN (合并后 operatorIds)
        ↓
仅展示分配给当前用户管理范围内运营人员的作品行
```

**待分配提示数量**（「N条作品待分配」）额外限制：`authType ≠ '2'`（全部员工）时才显示，即：
- authType=1 或 3 → 能看到待分配提示数量
- authType=2（全部员工）→ 不显示待分配提示

#### 作品登记（work-registration）

数据权限采用**双重过滤**机制（作品 + 频道）：

```
Step 1: managedUserList(teamId, currentUserId) → operatorIds
        ↓
Step 2: authChannelIdList(teamId, operatorIds)
        = 查 TargetChannel 表，取当前用户管理范围内的 channelId 列表
        ↓
Step 3: if authChannelIdList 为空 → 直接返回空页（无权看任何视频）
        ↓
Step 4: 若有频道名称搜索条件 → 再与 AMS 返回的 channelIds 取交集
        ↓
Step 5: VideoComposition 表 WHERE channelId IN (authChannelIdList)
        ↓
仅展示当前用户管理范围内频道下的视频行
```

**导入登记时的数据权限校验**（`batchHandleData`）：
- 导入的每一行视频 ID，后端同样通过 `authChannelIdList` 校验其所属频道是否在当前用户权限范围内
- 不在权限范围内的视频行 → 报错「频道ID不存在或无权限」，拦截导入

#### 员工管理（employeeManagement）

- 员工列表接口 `queryTeamMember` **无** `@NeedLogin` 且无 `managedUserList` 过滤
- 所有有菜单权限的用户可查看本团队全量员工列表（按 `teamId` 隔离）

#### 运营人员下拉选项（teamMemberDictList）

```
managedUserList(teamId, currentUserId) → operatorIds
        ↓
团队成员列表中 filter：operatorIds 为空则不过滤；非空则仅保留 userId 在 operatorIds 中的成员
        ↓
运营人员下拉仅展示当前用户管理范围内的人员
        ↓
同时受 includeAdmin（是否含超管）、includeDisable（是否含停用）参数控制
```

> 影响范围：海外频道「分配运营人员」弹窗 / 作品管理「批量分配」弹窗 / 作品管理搜索框「运营人员」筛选项等所有使用该接口的下拉选择器。

#### 批量导入分配运营人员（CompositionAllocateImportListener）

导入文件中填写的运营人员姓名，后端会校验是否在当前用户的管理范围内：

```
managedUserList(teamId, currentUserId) → userIds（当前用户可管理的 userId 列表）
        ↓
从作品表查询匹配姓名 + 在 userIds 范围内的运营人员
        ↓
超出管理范围的运营人员姓名 → 标记 isPermission=N → 该行导入拦截（无权限指定此人）
```

### 5.4 数据权限生效时机汇总

| 场景 | 过滤字段 | 过滤方式 | authType=2（全部）时 |
|------|---------|---------|-----------------|
| 海外频道 - 列表 | `operatorId` | IN 条件 | 不加过滤，可见全部 |
| 海外频道 - Tab 统计 | `operatorId` | IN 条件 | 不加过滤，全量统计 |
| 作品管理 - 列表 | `operatorId` | IN 条件（与前端筛选合并） | 不加过滤，可见全部 |
| 作品管理 - Tab 统计 | `operatorId` | IN 条件 | 不加过滤，全量统计 |
| 作品管理 - 待分配提示 | `authType` | 字段判断 | 不显示待分配提示 |
| 作品登记 - 列表 | `channelId`（经频道权限转换） | IN 条件 | 不加过滤，可见全部 |
| 作品登记 - 导入校验 | `channelId` | IN 条件（逐行校验） | 不加过滤，全部可导入 |
| 员工管理 - 列表 | 无 | 仅 teamId 隔离 | — |
| 运营人员下拉 | `userId` | 成员列表 filter | 不过滤，展示全部成员 |
| 批量导入分配-运营人员校验 | `userId`（按姓名匹配后校验） | isPermission 标记 | 不过滤，全部人员可指定 |

## 6. 后端权限机制汇总

### 6.1 登录鉴权（`@NeedLogin` 注解）

- **实现类**：`LoginAspect.java`（AOP 环绕切面）
- **触发条件**：方法标注 `@NeedLogin` 注解
- **校验流程**：
  1. 从请求头 `accessToken` 取 token
  2. `JwtUtils.parseToken(token)` 解析，失败则返回未登录错误
  3. 调用 `offlineClient.userRemove(userId)` 检查用户是否被下线（Redis）
  4. 通过后将 `userId`、`name`、`teamId`、`memberId` 写入 `ApplicationInternalContext`（ThreadLocal）

### 6.2 接口限流（`@RateLimit` 注解）

- **实现类**：`GlobalInterceptor.java`（拦截器）
- **触发条件**：方法标注 `@RateLimit` 注解
- **参数**：`window`（时间窗口秒数）+ `count`（最大请求数）+ `key`（限流 key 前缀）
- **限流 key 构成**：`key + ip + methodName + requestParams`（基于 Redis 计数）
- **当前使用接口**：`createOrEditUser`（3秒内重复提交拦截）、`addOrEditTeam`

### 6.3 语言透传（`RequestHeaderInterceptor`）

- 从请求头 `language` 字段读取语言设置，写入 `ApplicationInternalContext`
- 用于接口返回内容的国际化

## 7. 权限数据流完整路径

```
用户登录（邮箱/手机号）
        ↓
POST /appApi/user/emailAndPwdLogin → 返回 accessToken
        ↓
存入 localStorage（accessToken、userId、teamId）
        ↓
路由守卫 beforeEach 首次触发
        ↓
POST /appApi/user/userAuthorityList → 返回 {page: [...], btn: [...]}
        ↓
┌─── page 列表 ─────────────────────────────────────┐
│  handlePermissionData → 动态注册路由               │
│  handlePermissionMenu → store.dispatch saveMenu   │
│  侧边导航栏渲染有权限的菜单项                        │
└───────────────────────────────────────────────────┘
┌─── btn 列表 ──────────────────────────────────────┐
│  btn.map(item => item.code) → btnCodes            │
│  store.dispatch('auth/saveButtonAuth', btnCodes)  │
│  v-auth 指令 / can() 函数读取 buttonAuth 做控制     │
└───────────────────────────────────────────────────┘
```
