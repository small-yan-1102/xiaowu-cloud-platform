**状态流转路径**（来源：`OverdueSettlementImportHandler.buildOverdueEntity()` + `VideoCompositionOverdueServiceImpl.batchSplit()`）：

```
                    ┌─────────────────────┐
   导入时：          │    未登记 (0)        │ ← 导入初始状态（videoComposition 不存在 或 relatedType ≠ OVERDUE）
   relatedType      └──────────┬──────────┘
   ≠ OVERDUE                   │
                               │ 触发方式 A：分销商在剧老板登记
                               │ → 剧老板调用 synVideoComposition(jsonStr)
                               │ → VideoCompositionServiceImpl 处理登记
                               │ → syncOverdueSettlementStatus() 查 overdue 表 status=0 的记录
                               │ → 更新 status=1 + 同步字段（pipeline/子集/CP/套餐/语种/登记时间）
                               │
   导入时：          ┌──────────▼──────────┐
   relatedType      │ 逾期登记未拆分 (1)    │ ← 触发方式 B：导入时已登记（relatedType=OVERDUE）直接写 status=1
   = OVERDUE        └──────────┬──────────┘
                               │
                               │ 字段同步分支（status=1 时再次收到登记同步）：
                               │ → syncOverdueSettlementStatus() 只同步字段，status 保持 1 不变
                               │
                               │ 财务批量拆分（batchSplit）
                               │ @Transactional
                    ┌──────────▼──────────┐
                    │     已拆分 (2)       │ ← 记录操作人 + 操作时间
                    └─────────────────────┘
```

> **status 0→1 触发链路**（来源：`VideoCompositionServiceImpl.java:246-250,396-521`）：
> 1. 剧老板系统调用 `synVideoComposition(jsonStr)` 同步视频登记数据
> 2. `VideoCompositionServiceImpl` 处理登记数据（新增/更新 `video_composition` 表）
> 3. 处理完成后调用 `syncOverdueSettlementStatus(allProcessed)` ——筛选 relatedType=OVERDUE 的记录
> 4. 查询 `video_composition_overdue` 表中 status=0（未登记）或 status=1（已登记未拆分）的记录
> 5. **status=0 → 更新为 1** + 同步字段（pipelineId、signChannelId、signChannelName、cpName、servicePackageCode、langCode、publishedDate、teamId、teamName、registrationTime）
> 6. **status=1 → 只同步字段，status 不变**（支持登记信息变更场景）

### 3.5 后端字典转换逻辑

来源：`VideoCompositionOverdueServiceImpl.enrichVoFields()` L80-115

**批量转换模式**（非逐行查询）：
1. 从当前页所有记录中收集所有 distinct 的 `servicePackageCode` 和 `langCode`
2. 批量查询 `dict_data` 表：`WHERE dict_type = :type AND dict_value IN (:codes) AND status = '0'`
3. 构建 code→label 映射 Map
4. 遍历记录，逐行设置 `servicePackageName` 和 `langName`
5. **字典查不到时 fallback**：`getOrDefault(code, code)` — 返回 code 原值，不报错

| 字典类型 | dict_type 值 | DB 存储字段 | 转换后 VO 字段 |
|---------|-------------|-----------|---------------|
| 运作套餐 | `compositionServicePackage` | service_package_code | servicePackageName |
| 运作语种 | `compositionLanguage` | lang_code | langName |

### 4.2 异步调度框架

来源：`VideoCompositionOverdueController.java` → `ExcelImportTaskServiceImpl.java` → `OverdueSettlementImportHandler.java`

**架构模式**：Controller 接收请求 → 通用导入服务创建任务 → **线程池异步执行** → 策略模式分发到具体 Handler

```
前端 ImportDialog（确认按钮）
    │ POST /import/async（ossKey + receiptedMonth）
    ▼
Controller.asyncImport()
    │ excelImportTaskService.asyncImportByOssKey()
    ▼
ExcelImportTaskServiceImpl：
    │ ① validateOssImportParams()        — 参数校验
    │ ② checkImportingTask()             — 任务互斥检查
    │ ③ createImportTaskByOssKey()       — 创建 task 记录（status=0，operator_id=当前用户）
    │ ④ save(task) → 返回 taskId 给前端  — 前端收到成功响应后触发 ImportStatus.fetchStatus()
    │    注：前端轮询通过 getImportStatus() 按 importType+operatorId 查最新任务，不是按 taskId 轮询
    │ ⑤ executeImportAsync(task)         — 提交到线程池
    ▼
ExecutorHolder.getFixedExecutorService().submit(() → {
    │ ⑥ 从 OSS 下载文件 → InputStream
    │ ⑦ handler.handle(inputStream, task) — 策略模式分发
    │    └→ OverdueSettlementImportHandler（本模块处理器）
    │ ⑧ updateTaskSuccess/Failed()       — 更新任务状态
    │ ⑨ generateFailFile()               — 生成失败文件（如有）
})
```

**任务互斥**（来源：`ExcelImportTaskServiceImpl.checkImportingTask()` L134-158）：

| 场景 | 处理 |
|------|------|
| 同类型 + 同月份存在 status=0 的任务，且创建 ≤ 30 分钟 | 抛 BizException："当前存在导入中的任务，请稍后再试" |
| 同类型 + 同月份存在 status=0 的任务，且创建 > 30 分钟 | 自动将该任务标记为失败（"导入超时，任务自动终止"），允许新导入 |
| 无正在导入的任务 | 允许导入 |

**任务状态流转**（来源：`updateTaskSuccess()` L228-251 + `updateTaskFailed()` L256-261）：

```
task.status = 0（导入中）
    │
    ├── Handler 正常返回 ImportResult
    │   ├── successCount > 0（全部或部分成功）
    │   │   → status = 1（已导入）
    │   │   → 如有 failRows → 生成失败文件写入 failFileKey
    │   │
    │   └── failCount = totalCount（全部校验失败，无数据入库）
    │       → status = 2（导入失败）
    │       → failReason = "所有数据均校验失败，无数据入库"
    │       → 生成失败文件写入 failFileKey
    │
    └── Handler 抛异常
        → status = 2（导入失败）
        → failReason = 异常信息（截断至 500 字符）
```

**失败文件生成**（来源：`generateFailFile()` L266-300）：

1. 从 OSS 下载导入模板（`逾期结算导入模板.xlsx`）
2. 在表头行第 2 列追加"失败原因"
3. 逐行写入：视频ID（第 1 列）+ 失败原因（第 2 列）
4. 上传到 OSS 路径 `{COMPOSITION_OVERDUE}/错误/逾期结算导入错误信息.xls`
5. 返回 OSS fileKey → 写入 `task.failFileKey`

**服务重启保护**（来源：`@PostConstruct cleanupStaleImportingTasks()` L52-65）：
服务启动时将所有残留的 status=0 任务批量更新为 status=2，failReason="服务重启，任务自动终止"。

### 4.3 Handler 业务处理

来源：`OverdueSettlementImportHandler.java`

**导入流程**：
1. **Listener 层解析** Excel（EasyExcel）：校验表头（第1列须为 `*视频ID`）→ 逐行读取（超 20000 行中止）→ 空行跳过 → 自动 trim
2. **前置校验**：importList 为空 → 直接返回；去重后有效 videoIds 为空 → 全部标记失败
3. 查 `yt_month_channel_revenue_source`（**month = 到账月份 - 1 个月**，如到账月份=2026-03 则查 2026-02）获取视频级收益（来源：`OverdueSettlementImportHandler.java:90`）
4. 查 `video_composition` 获取登记信息
5. 查 `yt_month_cms_end`（按 channelId + receiptedAt LIKE 到账月份）获取频道+cms 维度总收益（含调差，作为占比计算分母）
6. 检查重复导入（按 receipted_month + cms + video_id 判重）
7. **逐行校验**并构建逾期结算记录（5 项校验，短路执行，详见 §4.4）
8. **隐式处理**：cmsEnd 不存在时收益置零；同一 videoId 多 cms 维度分别生成记录
9. 批量保存成功记录（batchSize=500）（来源：`OverdueSettlementImportHandler.java:163`）

**收益计算公式**（来源：`buildOverdueEntity()` → `calculateRevenue()`）：

```
总收益 = (cmsEnd.revenue + adjustedAmount1) × vRevenueRatio
美国收益 = (cmsEnd.usRevenue + adjustedAmount2) × vUsRevenueRatio
新加坡收益 = adjustedAmount3 × vSgRevenueRatio
```
精度：2位小数，HALF_UP。

**`vRevenueRatio` 字段说明**（来源：`YtMonthChannelRevenueSource.java:84-86` + `YTMonthChannelRevenueTaskServiceImpl.java:296`）：
- **含义**：视频级收益占比 = 单个视频收益 / 频道+cms 下全部视频总收益
- **精度**：8 位小数，HALF_UP
- **计算时机**：由月度收益汇总定时任务（`YTMonthChannelRevenueTaskServiceImpl`）预先计算，写入 `yt_month_channel_revenue_source.v_revenue_ratio` 字段
- **非实时计算**：导入时直接读取此字段，不重新计算
- **三个占比字段**：`vRevenueRatio`（总收益占比）、`vUsRevenueRatio`（美国收益占比）、`vSgRevenueRatio`（新加坡收益占比）

**状态设置**（来源：`buildOverdueEntity()` L303-307）：
- videoComposition 存在且 relatedType = OVERDUE → status = 1（逾期登记未拆分）
- 否则 → status = 0（未登记）

**导入限制**：最大行数 20000 | 文件格式 .xlsx/.xls | 文件大小 ≤ 5MB

**完整流程**：
1. **校验**：勾选 ID 非空；查询记录存在；筛选 status=1 的有效记录（来源：`validateAndGetRecords()`）

3. **分组**：按【**month + channelId + cms**】分组（来源：`groupByDimension()`）

4. **构建参数**：

   - 查找冲销表父行：条件为合集频道（channelType=2）或单开已转型（channelType=0 且 transformDate >= month）（来源：`findParentReport()`）
   - 校验冲销表已到账（receivedStatus ≠ 0）
   - 校验 pipeline_id 唯一性（`checkPipelineIdExist`）
5. **执行拆分**：调用 `ytReversalReportService.batchSplitSubsetRevenue()`
6. **更新状态**：status 改为 2（已拆分），记录操作人和操作时间

### 5.3 子集生成决策矩阵（pipelineId + sourceChannelId）{#决策矩阵}

> **适用范围**：本节描述的决策矩阵来自 **YT 核算自动计算流程**（`YtCalculateAutoVideoService.java`），即冲销报表/预估报表生成阶段。逾期批量拆分的下游行为（§5.4）与此矩阵密切相关——逾期拆分产生的子集收益会从合集的「无归属收益」中扣除。

#### 5.3.1 相关常量

#### 5.3.2 pipelineId 赋值来源

来源：`YTMonthChannelRevenueTaskServiceImpl.java:255-260`

月度收益汇总定时任务中，为每个视频设置 pipelineId：

| 场景 | pipelineId 赋值 | 说明 |
|------|----------------|------|
| pipelineMap 中存在且视频非逾期 | `pipelineMap.get(videoId)` | 正常通道 ID |

#### 5.3.3 sourceChannelId 赋值来源

来源：`YtCalculateAutoVideoService.setAmsSignChannelInfo()` L214-234

通过 AMS 接口批量查询 `pipelineId → signChannelId` 映射：

| 场景 | sourceChannelId | sourceChannelName | 附加行为 |
|------|----------------|-------------------|---------|
| AMS 查到对应子集 | `amsPipeLineSignChannelDTO.getSignChannelId()`（正常值） | AMS 返回的子集名称 | — |

#### 5.3.4 决策矩阵

来源：`YtCalculateAutoVideoService.shouldSkipSubsetGeneration()` L1112-1122 + `isUnattributedRevenue()` L1135-1145

以下矩阵决定每条子集数据在**核算自动生成冲销报表/预估报表**时的处理方式：

| pipelineId | sourceChannelId | 是否生成子集 | 是否累计到无归属收益 | 判定逻辑说明 |
|---|---|---|---|---|
| `"unattributed"` | `-1` | **否**（跳过） | **是**（累计） | 无归属通道，不生成子集行，收益计入合集 `unattributedRevenue` |
| `"-1"` | `-1` | **是**（生成） | **否**（不累计） | 原创/运营选择，仍生成子集行（子集名=`"原创/无需结算(运营选择)"`） |
| 其他 | `-1`（AMS 查不到） | **否**（跳过） | **是**（累计） | 有通道但 AMS 查不到子集，不生成子集行，收益计入无归属 |
| 其他 | 正常值 | **是**（生成） | **否**（不累计） | 正常子集，生成子集行并参与结算 |

**判定优先级**（短路执行，来源：`shouldSkipSubsetGeneration()` 方法）：
1. `pipelineId = "unattributed"` → 直接跳过（不论 sourceChannelId 为何值）
2. `pipelineId = "-1"` → 直接生成（不论 sourceChannelId 为何值）
3. 其余情况 → 根据 `sourceChannelId == -1` 判断

#### 5.3.5 误差抹平逻辑

来源：`YtCalculateAutoVideoService.java` L858-895（预估报表）、L1048-1085（冲销报表）

**核算自动计算**中，拆分子集收益后执行误差抹平：

1. **可拆分金额** = 频道总金额（CMS 收益 + 调差） − 无归属金额之和
2. **误差** = 可拆分金额 − 有归属子集金额之和
3. 将误差**调整到收益最大的有归属子集**上（找 `max(cmsRevenue)` 的子集，加上误差值）
4. 分别对总收益、美国收益、新加坡收益独立执行误差抹平

> 仅对**有归属子集**（`shouldSkip=false` 的子集）执行误差抹平，无归属收益不参与。

#### 5.3.6 合集无归属收益字段

来源：`YtCalculateAutoVideoService.java` L897-900（预估）、L1087-1090（冲销）

核算自动计算完成后，将无归属收益之和写入合集（父行）的三个字段：

| 字段 | 说明 |
|------|------|
| `unattributed_revenue` | 无归属总收益 |
| `unattributed_us_revenue` | 无归属美国收益 |
| `unattributed_sg_revenue` | 无归属新加坡收益 |

> 这些字段是逾期批量拆分后 `adjustUnattributedRevenueAfterSplit` 的扣减基础（见 §5.4）。

### 5.4 拆分执行与无归属收益调整 {#拆分执行}

逾期批量拆分经过 §5.2 的参数构建后，调用 `ytReversalReportService.batchSplitSubsetRevenue(splitQueries)` 执行实际拆分。

来源：`YtReversalReportServiceImpl.batchSplitSubsetRevenue()` L233-258

**完整执行步骤**（在 `@Transactional` 事务中）：

```
① need2AddList(splitSubsetQueryList)       — 组装新增子集记录
   │ 批量取汇率（按到账月份）
   │ 批量取联邦税率（按结算月份）
   │ 批量查 CRM 合约（按 signChannelId + "YouTube" 平台）
   │ 遍历每个 SplitSubsetQuery：
   │   ├ 设置子集基础信息（channelType=SUBSET, parentId, cms, 汇率等）
   │   ├ 合约匹配 → 成功：填充合约信息、分成比、税率、计算可分配收益
   │   │                失败：填充报错原因（无合约原因）
   │   └ 计算可分配收益 = CMS收益 - 联邦税额 - 新加坡税额
   ↓
② handleNoSettlement(need2AddList)          — 不结算名单过滤
   ↓
③ saveBatch(need2AddList)                   — 批量保存子集记录到 yt_reversal_report
   ↓
④ 更新被拆分的父行（合集/单开转型）状态：
   │ channelSplitStatus = SPLIT
   │ settlementCreatedStatus = NONEEDCREATE（不需要生成结算单）
   ↓
⑤ adjustUnattributedRevenueAfterSplit()     — 扣减合集无归属收益
```

#### 5.4.1 无归属收益调整（拆分后扣减）

来源：`YtReversalReportServiceImpl.adjustUnattributedRevenueAfterSplit()` L264-287 + `updateParentUnattributedRevenue()` L322-353

**计算公式**：

```
拆分后合集无归属总收益 = 原 unattributedRevenue − 新增子集 revenue 之和
拆分后合集无归属美国收益 = 原 unattributedUsRevenue − 新增子集 usRevenue 之和
拆分后合集无归属新加坡收益 = 原 unattributedSgRevenue − 新增子集 sgRevenue 之和
```

> **注意**：只更新合集（父行）的 `unattributed_revenue` / `unattributed_us_revenue` / `unattributed_sg_revenue` 三个字段，不再更新「无归属子集」行的收益（来源：代码注释 L284 "只更新合集的无归属收益字段（不再更新无归属子集的收益）"）。

#### 5.4.2 无归属收益调整（删除子集后加回）

来源：`YtReversalReportServiceImpl.deleteSubsetById()` L442-470 + `adjustUnattributedRevenueAfterDelete()` L476-488

删除子集时执行反向操作：

```
删除后合集无归属总收益 = 原 unattributedRevenue + 被删除子集的 revenue
删除后合集无归属美国收益 = 原 unattributedUsRevenue + 被删除子集的 usRevenue
删除后合集无归属新加坡收益 = 原 unattributedSgRevenue + 被删除子集的 sgRevenue
```

删除子集的前置校验：
- 记录必须存在
- channelType 必须为 SUBSET（子集类型）
- settlementCreatedStatus 不能为 CREATED（已生成结算单的不允许删除）

删除后若该合集下不再存在任何子集 → 将合集 channelSplitStatus 改回 UNSPLIT（未拆）。

### 7.3 处理流程（synVideoComposition）

```
① 解析 MQ 消息 → List<VideoComposition>
    │
② 批量查 youtube_channels → channelId→channelName 映射
    │
③ 批量调 AMS 接口 → amsCompositionId→(memberId,cpName) 映射
    │
④ 批量查 video_composition 已有记录 → videoId→existRecord 映射
    │
⑤ 分类处理：
    │  ├── 已存在 → buildUpdateEntity() → updateList
    │  └── 不存在 → buildInsertEntity() → insertList
    │
⑥ 批量新增 saveBatch(insertList, 500)
⑦ 批量更新 updateBatchById(updateList, 500)
    │
⑧ syncOverdueSettlementStatus(allProcessed)  ← 触发逾期状态同步
    │ 筛选 relatedType = OVERDUE
    │ 查 overdue 表 status IN (0,1)
    │ ├── status=0 → 更新为 1 + 同步全部字段
    │ └── status=1 → 只同步字段，status 不变
```

**事务**：整个方法在 `@Transactional(rollbackFor = Exception.class)` 中执行（L154）。

**关键校验**：
- videoId 为空 → 跳过（L206-208）
- videoCompositions 字段为空 → 直接返回（L161-164）

| 调用方 | 方法 | 说明 |
|--------|------|------|
| `youtube_channels` 表 | `getChannelNameMap(channelIds)` | 批量获取频道名称 |

### 12.4 各场景查询条件汇总

**导入场景**（§4，`OverdueSettlementImportHandler`）：

| 步骤 | 查询表 | WHERE 条件 | 返回结构 |
|------|--------|-----------|---------|
| 查视频收益 | `yt_month_channel_revenue_source` | `target_video_id IN (:videoIds) AND month = :revenueMonth`（revenueMonth = 到账月份-1） | `Map<videoId, List<Source>>` |
| 查登记信息 | `video_composition` | `video_id IN (:videoIds) AND deleted = 0` | `Map<videoId, Composition>` |
| 查频道总收益 | `yt_month_cms_end` | `channel_id IN (:channelIds) AND receipted_at LIKE ':receiptedMonth%'` | `Map<channelId:cms, CmsEnd>` |
| 查重复导入 | `video_composition_overdue` | `receipted_month = :month AND video_id IN (:videoIds) AND deleted = 0` | `Set<receiptedMonth:cms:videoId>` |

**批量拆分场景**（§5，`VideoCompositionOverdueServiceImpl.batchSplit()`）：

| 步骤 | 查询表 | WHERE 条件 | 说明 |
|------|--------|-----------|------|
| 校验记录 | `video_composition_overdue` | `id IN (:ids)` + 筛选 `status = 1` | 只取逾期登记未拆分的记录 |

| 查冲销表父行 | `yt_reversal_report` | `month = :m AND channel_id = :ch AND cms = :cms AND (channel_type = 2 OR (channel_type = 0 AND transform_date IS NOT NULL AND transform_date >= :monthStart))` LIMIT 1 | 合集 OR 单开已转型 |

**列表查询场景**（§3，`VideoCompositionOverdueMapper.xml`）：

| 条件字段 | 匹配方式 | 动态条件 |
|---------|---------|---------|

| month | 精确 `=` | 可选 |
| video_id | 精确 `=` | 可选 |
| channel_id | 精确 `=` | 可选 |
| channel_name | 模糊 `LIKE %x%` | 可选 |

| sign_channel_name | 模糊 `LIKE %x%` | 可选 |
| team_name | 模糊 `LIKE %x%` | 可选 |
| status | 精确 `=` | 必传（前端 Tab 决定） |
| cms | 精确 `=` | 可选 |
| deleted | 精确 `= 0` | 固定 |

**状态同步场景**（§7，`VideoCompositionServiceImpl.syncOverdueSettlementStatus()`）：

| 查询表 | WHERE 条件 | 说明 |
|--------|-----------|------|
| `video_composition_overdue` | `video_id IN (:videoIds) AND status IN (0, 1) AND deleted = 0` | 查 status=0 或 1 的记录做字段同步 |
