# 测试知识库规范

> **版本**：v1.0
> **更新日期**：2026-04-13
> **用途**：定义 `knowledge/` 目录的结构、各类文件的格式、命名规则和维护时机，供测试工程师跨迭代积累和复用测试经验。

---

## 目录位置

知识库位于**项目工作区根目录**下的 `knowledge/`，与 `input/`、`review/`、`test_suites/` 同级：

```
{项目工作区}/
├── input/              ← PRD、技术文档（迭代输入）
├── review/             ← 审阅报告、测试点、用例草稿（迭代中间产物）
├── test_suites/        ← 最终测试用例套件
├── test_reports/       ← 执行报告
└── knowledge/          ← 跨迭代知识积累（本文档规范）
    ├── constraints/
    ├── risk-profiles/
    ├── defect-patterns/
    └── case-samples/
```

---

## 四类知识文件

### 1. 业务约束（`constraints/`）

**用途**：记录模块级业务规则，跨迭代复用，避免每次重新梳理。  
**加载时机**：`test-case-design` Phase 1 Mode A 自动加载，与当前迭代业务约束文档合并（当前迭代优先）。  
**维护者**：测试工程师  
**更新时机**：模块业务规则发生变更时手动更新

**文件命名**：`knowledge/constraints/{模块名}.md`

**格式模板**：

```markdown
# {模块名} 业务约束

> 最后更新：YYYY-MM-DD（迭代：{迭代编号}）

## 核心业务规则

- {规则描述，格式："当...时，系统应..."}
- 

## 字段约束

| 字段名 | 类型 | 约束条件 | 备注 |
|-------|------|---------|------|
| | | | |

## 状态机

（如有，描述状态流转）

## 权限规则

（如有，描述角色权限边界）

## 已知业务边界

（容易出错的业务边界，每次迭代遇到新情况时追加）

- 
```

### 2. 风险档案（`risk-profiles/`）

**用途**：记录模块历史高风险维度，`test-point-extraction` 加载后在对应维度加深覆盖。  
**加载时机**：`test-point-extraction` Phase 1 自动加载。  
**维护者**：测试工程师  
**更新时机**：每轮迭代结束后，将本次出现的新高风险维度追加进来

**文件命名**：`knowledge/risk-profiles/{模块名}.md`

**格式模板**：

```markdown
# {模块名} 风险档案

> 最后更新：YYYY-MM-DD（迭代：{迭代编号}）

## 历史高风险维度

| 维度 | 风险描述 | 首次出现迭代 | 处置状态 |
|------|---------|-----------|---------|
| 权限控制（AU） | 角色边界检查历史漏测较多 | 迭代-003 | 持续关注 |
| 状态机（SM） | 并发操作导致状态异常 | 迭代-005 | 已修复，仍需回归 |
| 边界值（BV） | 金额字段精度问题 | 迭代-007 | 持续关注 |

## 历史漏测场景

（曾经漏测、后来发现的场景，每次迭代结束后追加）

- {YYYY-MM-DD} 迭代-{N}：{场景描述}
- 
```

### 3. 缺陷模式（`defect-patterns/`）

**用途**：按缺陷类型和维度归类历史缺陷，`test-point-extraction` 加载后将高频缺陷对应的测试点优先级提升。  
**加载时机**：`test-point-extraction` Phase 1 自动加载。  
**维护者**：测试工程师（每轮迭代后更新，未来可由 AI 辅助整理）  
**更新时机**：`test-report` 完成后，将本次迭代缺陷归类追加

**文件命名**：`knowledge/defect-patterns/{模块名}.md`

**格式模板**：

```markdown
# {模块名} 缺陷模式

> 最后更新：YYYY-MM-DD（迭代：{迭代编号}）

## 缺陷统计（按维度）

| 维度 | 缺陷数（历史累计） | 最近一次出现 | 严重程度 |
|------|-----------------|-----------|---------|
| 正向流程（PF） | 2 | 迭代-006 | P1 |
| 权限控制（AU） | 5 | 迭代-008 | P0 |

## 高频缺陷详情

### {缺陷类型标题}

- **频次**：N 次（迭代-{N1}、迭代-{N2}...）
- **场景**：{触发条件描述}
- **根因**：{根本原因，如已知}
- **建议测试点覆盖**：{应重点覆盖的测试场景}

---
```

### 4. 用例样本（`case-samples/`）

**用途**：存放经评审认可的高质量用例，供 `test-case-design` 参考格式和覆盖深度，不直接复制。  
**加载时机**：`test-case-design` Phase 1 Mode A 自动加载。  
**维护者**：测试工程师  
**更新时机**：`test-case-review` 评审通过后，从用例集中挑选质量较高的用例手动放入

**文件命名**：`knowledge/case-samples/{模块名}/{用例编号}-{简短描述}.md`

**格式**：直接保存原始用例 Markdown 内容，无需额外包装。

**挑选标准**：
- 覆盖了容易遗漏的边界或异常场景
- 前置条件写法规范（含 `[验证:]` 标签、`[数据准备:]` 策略）
- 预期结果精确（L1/L2/L3 标注清晰，无禁用词）

## 文件命名规则

| 规则 | 说明 |
|------|------|
| 模块名与 PRD 保持一致 | 如 PRD 中模块叫"企业认证"，文件名就用 `企业认证.md` |
| 允许使用中文 | 本地文件系统和 git 均支持中文路径 |
| 大小写敏感 | 保持一致，避免混用 |
| `case-samples/` 子目录 | 每个模块一个子目录，目录名与模块名一致 |

---

## 各 Skill 与知识库的关系

| Skill | 读取 | 写入 |
|-------|------|------|
| `test-point-extraction` | `risk-profiles/`、`defect-patterns/` | 不写入（由测试工程师手动维护） |
| `test-case-design` Mode A | `constraints/`、`case-samples/` | 不写入（样本由测试工程师手动挑选） |
| `test-case-design` Mode B | `constraints/` | 不写入 |

> **当前阶段**：所有写入均由测试工程师手动完成。后续规划（第二步）：`test-report` 完成后由 AI 辅助整理缺陷模式并写入 `defect-patterns/`。

---

## 知识库维护节奏

| 时机 | 操作 |
|------|------|
| 迭代启动前 | 检查 `constraints/` 是否需要更新（业务规则有无变化） |
| `test-case-review` 评审通过后 | 从用例集中挑选优质样本放入 `case-samples/` |
| `test-report` 完成后 | 将本次缺陷归类追加到 `defect-patterns/`；将新高风险维度追加到 `risk-profiles/` |

---

## 初始化说明

知识库在团队首次使用时不存在，各 Skill 会静默跳过知识加载，不影响正常流程。建议在完成 **3 轮以上迭代** 后开始积累，优先创建 `risk-profiles/` 和 `defect-patterns/`（因为这两类信息最直接影响后续迭代的测试点覆盖质量）。
