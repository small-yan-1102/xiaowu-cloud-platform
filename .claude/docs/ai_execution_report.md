# AI 执行报告规则

> **版本**: v1.2 | **定位**: 执行结果记录格式、截图规范、汇总报告模板、执行顺序策略（AI直接执行版）

## 1. 单条用例执行结果格式

### 1.1 结果记录模板

每条用例执行完成后，按以下格式记录结果：

```markdown
### [用例编号] [用例名称]

**执行结果**: 通过 / 失败 / 阻塞 / 跳过  
**执行时间**: YYYY-MM-DD HH:mm:ss  
**执行耗时**: X 分 X 秒  
**执行环境**: [环境URL]

**步骤执行明细**:

| 步骤 | 操作 | 预期结果 | 实际结果 | 验证级别 | 状态 | 截图 |
|------|------|---------|---------|---------|------|------|
| 1 | 操作描述 | 预期描述 | 实际看到的内容 | L1 | 通过/失败 | CP1_XXX_step01_pass_时间戳.png |
| 2 | 操作描述 | 预期描述 | 实际看到的内容 | L1 | 通过/失败 | - |
| 3 | 操作描述 | 预期描述 | 实际看到的内容 | L2 | 通过/警告 | - |

**失败详情**（仅失败时填写）:
- 失败步骤: 步骤X
- 期望: [预期结果描述]
- 实际: [实际看到的内容]
- 截图: [失败前状态截图, 失败时截图]（至少 2 张）
- 失败类型: Bug / 用例写错 / 环境问题 / AI误判（AI 初判，人工确认）

**备注**: [重试情况、环境异常、人工介入说明等]
```

### 1.2 执行结果状态定义

| 状态 | 含义 | 触发条件 |
|------|------|---------|
| **通过** | 所有 L1 验证通过 | 全部 L1 强验证通过（L2 警告不影响） |
| **失败** | 存在 L1 验证不通过 | 任一 L1 强验证未通过 |
| **阻塞** | 无法执行 | 环境问题、前置条件不满足、依赖用例失败 |
| **跳过** | 主动跳过 | 依赖的前序用例失败导致跳过 |

### 1.3 警告记录

L2 软验证不通过时，不影响用例结果判定，但必须记录警告：

```markdown
**L2 警告**:
- [步骤X] 预期: XXX，实际: YYY（不影响用例判定）
```

## 2. 截图规范（双证据体系）

> 截图的详细执行规范（调用时序、开销控制等）见 `ai_execution_strategy.md` 第 7 节。  
> 本节定义截图的场景分级、命名格式和存储规则。

### 2.1 三级检查点模型

截图按**证据价值**分为三级，在覆盖度和开销之间取得平衡。

#### CP1 必截（MUST）— 缺失则报告视为不完整

| # | 场景 | 截图时机 | 证据目的 |
|---|------|---------|---------|
| 1 | **用例起始状态** | 执行第一个操作步骤之前 | 证明前置条件满足、页面初始状态正确 |
| 2 | **用例最终状态** | 最后一步验证完成之后 | 人工可审核最终结果是否与报告一致 |
| 3 | **L1 断言失败** | L1 验证不通过时立即截图 | 保留失败现场 |
| 4 | **关键状态变更** | toast 出现、弹窗确认、列表增减等 | 记录核心业务操作结果 |
| 5 | **异常/错误现场** | 意外弹窗、白屏、网络错误等 | 记录非预期现象 |

#### CP2 条件截（SHOULD）— 满足条件时截图

| # | 场景 | 触发条件 |
|---|------|---------|
| 1 | **重试操作** | 进入重试流程时（重试前后各一张） |
| 2 | **L3 观察性验证** | 遇到 L3 验证点时（截图代替人工主观判断） |
| 3 | **数据准备操作** | 执行非测试步骤的数据准备时 |

#### CP3 可选截（MAY）— AI 自主判断有审计价值时截图

### 2.2 截图命名规范

```
{CP级别}_{用例编号}_{步骤标识}_{状态}_{时间戳}.png

时间戳格式: YYYYMMDDHHmmss
```

| 字段 | 可选值 | 说明 |
|------|-------|------|
| CP级别 | `CP1` / `CP2` / `CP3` | 检查点强制级别 |
| 用例编号 | `JQB-TEAM-001` 等 | 用例唯一标识 |
| 步骤标识 | `init` / `step01` / `final` / `dataprep` | 截图时刻标识 |
| 状态 | `start` / `pass` / `fail` / `retry1` / `error` | 截图时的判定状态 |

**示例**：
```
CP1_JQB-TEAM-001_init_start_20260325100000.png       ← 用例起始状态
CP1_JQB-TEAM-001_step03_pass_20260325100030.png       ← 关键状态变更
CP1_JQB-TEAM-001_final_pass_20260325100045.png        ← 用例最终状态
CP1_JQB-TEAM-034_step06_fail_20260325103000.png       ← L1断言失败
CP2_JQB-TEAM-034_dataprep_pass_20260325102800.png     ← 数据准备
CP2_JQB-TEAM-012_step01_retry1_20260325101500.png     ← 重试第1次
```

### 2.3 截图存储

- **统一存储路径**：`test_reports/screenshots/YYYY-MM-DD/`（与执行报告同级目录树）
- **每日独立子目录**：按执行日期分目录
- **证据索引文件**：每日截图目录下生成 `evidence_index.md`，记录截图清单和统计

### 2.4 证据索引文件

每日截图目录下自动生成 `evidence_index.md`，格式如下：

```markdown
# 证据索引 - YYYY-MM-DD

**关联报告**: test_report_YYYYMMDD_XXX.md  
**截图总数**: N 张（CP1: X, CP2: Y, CP3: Z）

| 文件名 | 检查点 | 用例编号 | 步骤 | 状态 | 时间 |
|--------|-------|---------|------|------|------|
| CP1_JQB-TEAM-001_init_start_xxx.png | CP1 | JQB-TEAM-001 | init | start | HH:mm:ss |
| ... | ... | ... | ... | ... | ... |
```

## 3. 测试套件执行顺序

### 3.1 套件级执行策略

```
┌──────────────────┐
│ 1. 冒烟测试 (P0)  │  先执行，验证系统基本可用
└────────┬─────────┘
         ↓  全部通过？
    ┌────┴────┐
    │ 是      │ 否 → 终止执行，报告冒烟失败，阻塞部署
    ↓         └───────────────────────────────────┐
┌──────────────────┐                               │
│ 2. E2E 测试 (P1) │  冒烟通过后执行                  │
└────────┬─────────┘                               │
         ↓                                         │
┌──────────────────┐                               │
│ 3. 异常测试 (P2) │  主流程测试通过后执行              │
└────────┬─────────┘                               │
         ↓                                         │
┌──────────────────┐                               │
│ 4. 边界测试 (P3) │  最后执行                        │
└────────┬─────────┘                               │
         ↓                                         ↓
┌──────────────────────────────────────────────────┐
│                  生成汇总报告                       │
└──────────────────────────────────────────────────┘
```

### 3.2 模块内执行顺序

同一模块内的用例按以下顺序执行：

1. **前置依赖用例**（如登录）
2. **创建类操作**（如新增记录）
3. **查询/展示类操作**（如列表查看）
4. **修改类操作**（如编辑记录）
5. **删除类操作**（如删除记录）

### 3.3 依赖链处理

| 场景 | 处理方式 |
|------|---------|
| 用例 A 是用例 B 的前置依赖，A 失败 | B 标记为"跳过"，原因记录"前置用例 A 失败" |
| 登录用例失败 | 该模块所有后续用例标记为"阻塞" |
| 冒烟套件中任一用例失败 | 报告冒烟失败，根据策略决定是否继续 E2E |

## 4. 汇总报告模板

### 4.1 报告结构

每次测试执行完成后，生成以下汇总报告：

```markdown
# 测试执行报告

**执行日期**: YYYY-MM-DD  
**执行时间**: HH:mm ~ HH:mm  
**总耗时**: X 分钟  
**测试环境**: [环境URL]  
**执行人**: AI 自动执行

---

## 执行概览

| 指标 | 数值 |
|------|------|
| 用例总数 | N |
| 通过 | N（XX%） |
| 失败 | N（XX%） |
| 阻塞 | N |
| 跳过 | N |
| L2 警告数 | N |

## 证据摘要

| 检查点级别 | 应截数 | 实截数 | 覆盖率 |
|-----------|-------|-------|-------|
| CP1 必截   | N     | N     | XX%   |
| CP2 条件截 | N     | N     | XX%   |
| CP3 可选截 | -     | N     | -     |

**CP1 覆盖率**: XX% ✅/⚠️ (要求 ≥ 95%)  
**截图存储路径**: screenshots/YYYY-MM-DD/

### CP1 缺失告警

（无缺失 / 列出缺失的 CP1 截图及原因）

## 冒烟测试结果

| 编号 | 用例名称 | 结果 | 耗时 | 备注 |
|------|---------|------|------|------|
| SMOKE-XXX-001 | 用例名称 | 通过/失败 | Xm Xs | |

**冒烟测试结论**: 通过 / 失败（阻塞部署）

## E2E 测试结果

| 编号 | 用例名称 | 结果 | 耗时 | 备注 |
|------|---------|------|------|------|
| PRJ-XXX-001 | 用例名称 | 通过/失败 | Xm Xs | |

## 失败用例详情

### [失败用例编号] [用例名称]
- **失败步骤**: 步骤X - [操作描述]
- **期望**: [预期结果]
- **实际**: [实际结果]
- **截图**: [截图链接]
- **可能原因**: [AI 分析的失败原因]
- **失败类型**: Bug / 用例写错 / 环境问题 / AI误判（AI 初判，人工确认）

## L2 警告汇总

| 用例编号 | 步骤 | 警告内容 |
|---------|------|---------|
| PRJ-XXX-001 | 步骤X | 预期XXX，实际YYY |

## 环境问题记录

| 时间 | 问题描述 | 处理方式 | 影响用例 |
|------|---------|---------|---------|
| HH:mm | 问题描述 | 刷新/重试/终止 | 用例编号 |

---

## 审计清单

执行完成后，由 AI 逐项自查后输出：

- [ ] CP1 截图完整度 ≥ 95%
- [ ] 每条失败用例附有 ≥ 2 张截图（失败前 + 失败时）
- [ ] 证据索引文件 `evidence_index.md` 已生成
- [ ] 报告中的截图路径可访问（相对路径正确）
- [ ] 截图命名符合 `{CP级别}_{用例编号}_{步骤}_{状态}_{时间戳}.png` 规范
- [ ] 证据摘要章节已填写，CP1 覆盖率数值准确

---

**报告生成时间**: YYYY-MM-DD HH:mm:ss
```

### 4.2 报告存储

- 文件命名：`test_report_YYYYMMDD_HHmmss.md`
- 存储位置：项目根目录的 `test_reports/` 文件夹
- 截图目录关联：报告中引用的截图路径指向 `screenshots/YYYY-MM-DD/` 目录

## 5. 结果同步

### 5.1 与 Excel 测试表的同步

测试执行完成后，如需将结果同步到 Excel 测试用例表：

| 同步字段 | Excel 列 | 填写内容 |
|---------|---------|---------|
| 执行结果 | 测试结果列 | 通过 / 失败 / 阻塞 / 跳过 |
| 执行日期 | 执行日期列 | YYYY-MM-DD |
| 失败原因 | 备注列 | 失败步骤 + 实际结果描述 |

同步操作可使用 `test-result-sync` skill 自动完成。

### 5.2 缺陷关联

测试失败时，报告中应包含足够信息以便创建缺陷：

- 失败用例编号和名称
- 重现步骤（即用例步骤的 1~N 步）
- 实际结果 vs 预期结果
- 截图证据
- 环境信息

## 6. 线上回归报告模板

> **适用场景**：D5 发版后在生产环境执行线上回归验证，使用本模板生成专用报告。

## 附录：报告字段速查

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| 用例编号 | 必填 | SMOKE-/PRJ- 格式 |
| 执行结果 | 必填 | 通过/失败/阻塞/跳过 |
| 执行时间 | 必填 | 精确到秒 |
| 执行耗时 | 必填 | 精确到秒 |
| 步骤明细 | 必填 | 每步的实际结果（含截图列） |
| 失败详情 | 失败时必填 | 期望 vs 实际 |
| 截图 | CP1 检查点必填 | 按三级检查点模型执行；CP1 缺失则报告不完整 |
| 证据摘要 | 必填 | CP1/CP2/CP3 统计与覆盖率 |
| 审计清单 | 必填 | 执行完成后 AI 自查项 |
| L2 警告 | 有警告时必填 | 软验证不通过记录 |
| 环境问题 | 有问题时必填 | 环境异常记录 |
| 备注 | 选填 | 重试、人工介入等补充信息 |
