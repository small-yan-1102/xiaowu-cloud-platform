## 1. 浏览器操作规范

### 1.1 元素定位优先级

AI 定位页面元素时，按以下优先级依次尝试：

| 优先级 | 定位方式 | 说明 | 示例 |
|-------|---------|------|------|
| 1 | **可见文本** | 按钮、链接、标签等的显示文字 | 点击「提交认证」按钮 |
| 2 | **aria-label / role** | 无障碍属性标识 | 具有 role="dialog" 的弹窗 |
| 3 | **placeholder / title** | 输入框的占位文字或提示 | placeholder 为"请输入公司名称"的输入框 |
| 4 | **相对位置描述** | 基于上下文的位置关系 | "企业认证"标题下方的第一个输入框 |

**禁止**：在用例中使用 CSS 选择器、XPath、data-testid 等技术定位符——AI 直接执行模式不依赖 DOM 结构。

### 1.2 操作类型规范

| 操作类型 | 规范要求 | 注意事项 |
|---------|---------|---------|
| **点击** | 确保元素可见且可交互后再点击 | 被遮挡的元素需先滚动到可视区域 |
| **文本输入** | 先清空输入框已有内容，再输入新值 | 处理有默认值的输入框 |
| **下拉选择** | 先点击下拉框展开，再选择目标选项 | 等待下拉列表完全展开 |
| **文件上传** | 通过文件选择对话框上传 | 确认上传成功提示出现 |
| **拖拽** | 明确起点和终点元素 | 等待拖拽动效完成 |
| **滚动** | 目标元素不在可视区域时先滚动 | 长列表懒加载时需等待数据加载 |

### 1.3 键盘操作

| 场景 | 推荐操作 |
|------|---------|
| 表单字段间切换 | Tab 键 |
| 提交表单 | Enter 键（当焦点在提交按钮上时） |
| 关闭弹窗 | Escape 键（作为点击关闭按钮的备选） |
| 搜索触发 | Enter 键（在搜索框输入后） |

## 2. 等待策略

### 2.1 核心原则

**永远不要使用固定时间等待（如"等待3秒"）**，应基于页面状态信号判断操作时机。

### 2.2 页面状态等待信号

| 等待信号 | 适用场景 | 说明 |
|---------|---------|------|
| **加载指示器消失** | 页面加载、数据请求 | 等待 loading spinner / 进度条消失 |
| **目标元素出现** | 页面跳转、弹窗打开 | 等待预期元素在页面中变为可见 |
| **文本内容变化** | 状态更新、数据刷新 | 等待特定文本出现或变化 |
| **URL 变化** | 页面导航 | 等待浏览器地址栏 URL 符合预期 |
| **网络请求完成** | AJAX 数据加载 | 等待无新的网络请求发出 |

### 2.3 超时设置

| 操作类型 | 默认超时 | 说明 |
|---------|---------|------|
| 元素出现/可见 | **10 秒** | 超过则判断为元素未出现 |
| 页面导航完成 | **15 秒** | 包含页面资源加载 |
| 文件上传 | **30 秒** | 大文件上传需要更多时间 |
| 数据加载（列表/表格） | **15 秒** | 大数据量查询可能较慢 |

超时后不立即标记失败，先执行一次**页面刷新重试**（见第3节）。

### 2.4 SPA 路由切换

单页应用（SPA）路由切换不会触发传统页面加载，AI 需通过以下信号判断导航完成：

1. URL hash 或路径变化
2. 页面主要内容区域更新
3. 导航菜单高亮状态切换
4. 面包屑导航内容更新

## 3. 错误恢复与重试策略

### 3.1 重试决策矩阵

| 错误类型 | 处理策略 | 最大重试次数 | 说明 |
|---------|---------|------------|------|
| **元素未找到** | 滚动页面 → 等待 → 重试点击 | 2 次 | 可能被折叠或懒加载 |
| **页面加载超时** | 刷新页面 → 重新执行当前步骤 | 1 次 | 网络波动导致 |
| **意外弹窗遮挡** | 关闭弹窗 → 继续当前操作 | 1 次 | Cookie 提示、广告、引导弹窗等 |
| **Session 过期** | 重新登录 → 导航回当前页面 → 重试 | 1 次 | 长时间执行导致 |
| **页面白屏/崩溃** | 刷新页面 → 检查是否恢复 | 1 次 | 不恢复则标记为环境问题 |
| **网络请求失败** | 等待 5 秒 → 刷新页面 → 重试操作 | 1 次 | 后端临时不可用 |
| **验证码/人工介入** | 暂停并通知用户 | 不重试 | AI 无法自动完成 |

### 3.2 重试原则

- **幂等性检查**：重试前确认之前的操作是否已经生效（如表单是否已提交成功），避免重复提交
- **状态回退**：如果重试需要回到之前的页面状态，先导航回到正确的页面
- **重试上限**：单个步骤最多重试 2 次，超过则标记该步骤失败并继续执行后续步骤
- **整体熔断**：同一用例中累计 3 个步骤重试失败，终止该用例执行，标记为环境不稳定

### 3.3 意外弹窗处理

执行过程中遇到非预期弹窗时的处理优先级：

1. **Cookie 同意弹窗**：点击"接受"或"同意"按钮
2. **系统通知/推送弹窗**：点击关闭按钮或空白区域关闭
3. **首次登录引导弹窗**：点击"跳过"或"我知道了"
4. **浏览器权限弹窗**（通知、位置等）：拒绝或关闭
5. **无法识别的弹窗**：截图记录 → 尝试按 Escape 关闭 → 如仍存在，标记为阻塞问题

### 3.4 自愈边界规则

> **核心原则**：环境层问题可以自愈重试，业务层失败必须如实报告。禁止通过自愈机制掩盖真正的业务 Bug。

#### 3.4.1 问题分层定义

| 层级 | 定义 | 典型现象 | AI 可否自愈 |
|------|------|---------|-----------|
| **环境层** | 与被测业务逻辑无关的基础设施/环境问题 | 页面加载超时、Session 过期、网络抖动、弹窗遮挡、元素懒加载未完成 | ✅ 可自愈 |
| **业务层** | 被测功能的实际行为与预期不符 | 预期状态变更未发生、金额计算错误、权限验证失效、流程走不通 | ❌ 禁止自愈 |

#### 3.4.2 自愈决策树

```
验证步骤失败
│
├── 失败原因是否可归类为环境层？
│   │
│   ├── 是（元素未找到/页面超时/Session过期/弹窗遮挡）
│   │   └── 执行自愈策略（见 §3.1）→ 重试后仍失败？
│   │       ├── 否 → 继续执行
│   │       └── 是 → 标记为"环境问题"，不计入业务失败
│   │
│   └── 否（预期结果与实际不符 / 业务校验失败 / 数据状态异常）
│       └── **禁止自愈**，直接标记为"业务失败"
│
└── 记录失败详情（期望 vs 实际）
```

#### 3.4.3 环境层 vs 业务层判断规则

| 判断信号 | 归类 | 说明 |
|---------|------|------|
| 元素在页面中不存在 | 环境层（可能）/ 业务层（可能） | 需进一步判断：页面是否正常加载？如正常加载但元素不存在，则为业务层 |
| 页面显示 Loading 超时 | 环境层 | 网络或服务端响应慢 |
| 页面显示错误码（500/502/504） | 环境层 | 后端服务问题 |
| toast 提示"操作成功"但数据未变更 | 业务层 | 业务逻辑 Bug |
| 按钮点击无响应（JS 报错） | 环境层 | 前端代码问题，需开发修复 |
| 预期"审核通过"但状态仍为"待审核" | 业务层 | 状态机未按预期流转 |
| 金额显示 1000，预期 1100 | 业务层 | 计算逻辑错误 |

#### 3.4.4 自愈禁区（绝对不可自愈的场景）

| 场景 | 原因 |
|------|------|
| L1 强验证预期结果与实际不符 | 核心业务断言失败是最重要的测试反馈 |
| 确定性断言（`[断言: API]` / `[断言: DB-Query]`）失败 | API/DB 返回值是确定的，不存在"看错"的可能 |
| 权限验证失败（能访问不应访问的资源） | 安全问题，必须报告 |
| 数据写入后查询不到 | 数据一致性问题 |
| 操作后状态未按预期变更 | 业务逻辑问题 |

## 4. 测试环境准备

### 4.1 环境配置信息

AI 执行测试前，必须从用例文档或环境配置中获取以下信息：

| 配置项 | 说明 | 获取方式 |
|-------|------|---------|
| **测试环境 URL** | 被测系统地址 | 用例文档"前置条件"或独立配置文件 |
| **测试账号** | 登录用户名和密码 | 用例文档"测试数据"表格 |
| **浏览器设置** | 默认使用 Chromium | browser-use MCP 默认配置 |

### 4.2 浏览器初始状态

每次测试套件执行前，确保浏览器处于干净状态：

| 初始化项 | 操作 |
|---------|------|
| 窗口大小 | 设置为 1920×1080（桌面标准分辨率） |
| 缓存和 Cookie | 根据用例需要决定是否清除 |
| 语言设置 | 与被测系统语言一致（默认中文） |

### 4.3 登录态管理

| 策略 | 适用场景 | 说明 |
|------|---------|------|
| **每次重新登录** | 用例间状态隔离要求高时 | 保证独立性，但耗时较长 |
| **复用登录态** | 同模块连续用例执行时 | 效率高，但需确认 Session 有效 |
| **多账号切换** | 权限验证类用例 | 需要先登出再登入不同账号 |

**默认策略**：同一测试套件内复用登录态，每个套件开始时重新登录。执行过程中 Session 失效则自动重新登录（见 3.1 节）。

### 4.4 测试数据准备原则

- **不依赖特定的数据库数据**：用例中的测试数据应可在页面上直接创建
- **唯一性数据**：需要唯一值的字段（如公司名称），加入时间戳后缀避免冲突
- **清理策略**：创建的测试数据在用例执行完成后，如有删除入口则清理；否则标注在执行报告中

### 4.5 部署预检（冒烟前置 / 线上回归前置）

目标环境部署完成后、执行冒烟测试或线上回归之前，AI 自动执行以下预检。预检不通过则不启动后续测试，避免浪费执行时间。

| # | 预检项 | 验证方式 | 失败处理 |
|---|-------|---------|---------|
| 1 | **目标环境可访问** | 访问目标环境 URL，页面返回非错误状态 | 终止执行，报告"环境不可用" |
| 2 | **登录功能正常** | 使用测试账号执行登录操作，验证登录成功 | 终止执行，报告"登录失败" |
| 3 | **核心页面可加载** | 登录后访问本次变更涉及的主要页面，页面无白屏、无 JS 报错 | 终止执行，报告"页面加载异常"并截图 |

**预检结果判定**：

| 结果 | 条件 | 后续动作 |
|------|------|---------|
| ✅ 预检通过 | 3 项全部通过 | 进入冒烟测试或线上回归 |
| ❌ 预检失败 | 任一项不通过 | 终止执行，输出预检报告（含失败截图），通知测试工程师排查环境问题 |

**预检报告格式**：

```markdown
## 部署预检结果

**检查时间**: YYYY-MM-DD HH:mm:ss
**目标环境**: [环境URL]
**环境类型**: 测试环境 / 生产环境
**预检结论**: 通过 / 失败

| 预检项 | 状态 | 说明 |
|-------|------|------|
| 环境可访问 | ✅/❌ | |
| 登录功能 | ✅/❌ | |
| 核心页面加载 | ✅/❌ | [异常截图] |
```

## 5. AI 判断校准

### 5.1 文本匹配规则

> **关联**: `tech_doc_review.md` U-8（文案差异标注）。当前端实际渲染文案与 PRD/原型存在差异时（空格、标点、全角/半角等），技术文档审阅会要求开发在文档中标注实际渲染值。AI 执行测试时应以技术文档标注的实际渲染值为准进行匹配。

| 匹配类型 | 规则 | 示例 |
|---------|------|------|
| **精确匹配** | 预期结果明确指定完整文本时 | 预期"企业认证已通过" → 必须看到完整的这句话 |
| **包含匹配** | 预期结果使用"包含""出现"等描述时 | 预期"出现成功提示" → 页面任意位置出现含"成功"的提示即可 |
| **忽略空格** | 对比文本时忽略前后空格和多余空白 | " 已通过 " 等同于 "已通过" |
| **大小写敏感** | 中文内容不涉及；英文内容默认大小写敏感 | "Success" ≠ "success"（除非用例特别说明） |

### 5.2 元素可见性判断

| 判断标准 | 说明 |
|---------|------|
| **可见** | 元素在页面 DOM 中存在，且在当前视口内可见（不被其他元素完全遮挡） |
| **不可见** | 元素不在 DOM 中，或 display:none，或完全被遮挡 |
| **部分可见** | 视为可见——只要能看到元素的主要内容（如按钮文字） |

### 5.3 数值比较容差

| 场景 | 容差规则 |
|------|---------|
| 精确金额（如积分余额） | 必须精确匹配，无容差 |
| 百分比显示 | 允许 ±1% 的显示误差 |
| 时间/日期显示 | 允许与当前时间 ±1 分钟误差 |
| 列表数量 | 必须精确匹配（如"共10条"） |

### 5.4 异步操作判断

某些操作结果不会立即反映在页面上：

| 场景 | 处理方式 |
|------|---------|
| 提交后需后台处理 | 等待处理状态变更，最长等待页面提示或 15 秒 |
| 数据最终一致性 | 刷新页面后重新检查，最多刷新 2 次 |
| 通知/消息延迟 | 等待消息提示出现，最长等待 10 秒 |

### 5.5 间歇性失败(Flaky)处理

当某个验证步骤失败时：

1. **首次失败**：调用 `browser_take_screenshot` 记录失败现场（CP1） → 等待 2 秒 → 重新检查一次
2. **二次失败**：调用 `browser_take_screenshot` 记录二次失败现场（CP2） → 刷新页面 → 重新检查一次
3. **三次失败**：调用 `browser_take_screenshot` 记录最终失败现场（CP1） → 确认标记为失败

每次重试截图文件名使用 `retry1` / `retry2` 后缀区分。在执行报告中标注该步骤经历了重试，便于后续分析是否为 Flaky 问题。

### 5.6 确定性断言执行规则

> **定位**：规范 `[断言: API]` 和 `[断言: DB-Query]` 类型预期结果的执行方式。消费 `ai_test_case_design.md` §4.6 定义的断言格式。

#### 5.6.1 断言类型识别与执行

| 断言类型 | 标识 | 执行方式 | 失败处理 |
|---------|------|---------|---------|
| **视觉断言** | `[断言: 视觉]` 或无标识 | 通过 `browser_snapshot` 观察页面内容 | 按 §5.5 Flaky 处理流程 |
| **API 断言** | `[断言: API]` | 解析接口地址/参数/预期响应，执行 HTTP 调用 | **禁止重试**，失败即报告 |
| **DB 断言** | `[断言: DB-Query]` | 解析 SQL 语句/预期结果，执行只读查询 | **禁止重试**，失败即报告 |

#### 5.6.2 API 断言执行流程

```
1. 解析预期结果中的 API 断言块
   - 接口地址（必填）
   - 请求参数（选填，支持 {引用:变量名}）
   - 预期响应（必填）
   - 比对字段（必填）

2. 构造 HTTP 请求
   - 使用当前浏览器的登录态（Cookie/Token）
   - GET 请求默认超时 10 秒

3. 执行请求并解析响应

4. 按比对字段规则对比
   - 精确匹配：实际值 === 预期值
   - 包含匹配：实际值 contains 预期值
   - 数值比较：支持 >, <, >=, <=, =

5. 结果判定
   - 匹配 → 断言通过
   - 不匹配 → 断言失败，记录期望值和实际值
   - 请求失败 → 标记为"API 调用失败"，记录错误详情
```

#### 5.6.3 DB 断言执行流程

```
1. 解析预期结果中的 DB 断言块
   - SQL 语句（必填，仅支持 SELECT）
   - 预期结果（必填）
   - 比对规则（必填）

2. 安全检查
   - 确认为 SELECT 语句（禁止 INSERT/UPDATE/DELETE）
   - 确认有 WHERE 条件（禁止全表扫描）
   - 确认有 LIMIT 限制（禁止无限结果集）

3. 执行查询（通过配置的数据库连接）

4. 结果比对
   - 按比对规则对比查询结果与预期结果

5. 结果判定
   - 匹配 → 断言通过
   - 不匹配 → 断言失败，记录期望值和实际值
   - 查询失败 → 标记为"DB 查询失败"，记录错误详情
```

#### 5.6.4 混合断言执行顺序

当一条预期结果包含多种断言类型时，按以下顺序执行：

1. **视觉断言优先**：先完成 UI 层面的验证
2. **API 断言次之**：验证接口返回的精确值
3. **DB 断言最后**：验证数据库层的最终状态

**原因**：视觉断言验证用户可见的反馈（如成功提示），API/DB 断言验证底层数据的准确性。先确认 UI 反馈正常，再验证数据正确性。

#### 5.6.5 确定性断言失败报告格式

```markdown
### 断言失败详情

**断言类型**: [断言: API]
**接口地址**: /api/credit/balance
**预期响应**: { "balance": 1100 }
**实际响应**: { "balance": 1000 }
**比对字段**: balance
**失败原因**: 实际值 1000 ≠ 预期值 1100
**业务判定**: 积分充值未到账，疑似充值逻辑 Bug
```

## 6. 执行流程总览

### 6.1 单条用例执行流程

```
┌──────────────────┐
│ 1. 读取用例        │ 解析编号、前置条件、步骤、预期结果
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 2. 检查前置        │ 确认登录态、页面位置、数据准备
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 3. 数据就绪验证    │ 按 [验证: ...] 标签确认数据存在且状态正确
│                    │ → 无标签 → 跳过，直接进入步骤 4
│                    │ → 验证通过 → 继续
│                    │ → 数据缺失 → 自愈（1次）→ 成功则继续 / 失败则阻塞
│                    │ → 状态异常 → 标记阻塞
│                    │ （详见 §6.3）
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 4. CP1 起始截图    │ browser_take_screenshot → 记录初始页面状态
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 5. 逐步执行        │ 操作 → 等待稳定 → browser_snapshot 验证
│                    │ → [关键状态变更时] browser_take_screenshot (CP1)
│                    │ → [L1失败时] browser_take_screenshot (CP1)
└────────┬─────────┘
         ↓  （操作失败？→ 执行重试策略 → 重试成功？→ 继续 / 标记失败）
┌──────────────────┐
│ 6. CP1 最终截图    │ browser_take_screenshot → 记录最终页面状态
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 7. 记录结果        │ 通过/失败/警告 + 截图文件名 + 耗时
└────────┬─────────┘
         ↓
┌──────────────────┐
│ 8. 状态恢复        │ 为下一条用例准备干净的初始状态
└──────────────────┘
```

> **注意**：步骤 4 和步骤 6 的 CP1 截图是 **MUST**（必须），缺失则该用例的报告视为不完整。详见第 7 节。

### 6.2 异常中断处理

| 中断类型 | 处理 |
|---------|------|
| 单步骤失败（非阻塞） | 记录失败，继续执行后续步骤 |
| L1 验证失败 | 记录失败，**终止当前用例**，进入下一用例 |
| 连续 3 步重试失败 | 标记环境不稳定，终止当前用例 |
| 人工介入点 | 暂停执行，通知用户 |

### 6.3 数据就绪验证规则

> **定位**：补充 §6.1 步骤 3 的详细执行规则。仅当用例前置条件包含 `[验证: ...]` 标签时执行。
> **验证特征编写规范**参见 `ai_test_case_design.md` 附录 E.3.5。

#### 6.3.1 验证执行规则

| 数据准备策略 | 验证方式 | 通过判定 |
|------------|---------|---------|
| `[数据准备: UI自建]` | 导航到验证特征指定页面，搜索/查看数据 | 数据可见且关键字段值匹配 |
| `[数据准备: API]` | 优先调用 GET 接口查询，备选 UI 页面验证 | 响应包含预期数据 |
| `[数据准备: 依赖用例]` | 检查依赖用例执行状态 + UI/API 验证数据存在 | 依赖用例已通过且数据可见 |
| `[数据准备: 环境预置]` | 导航到验证特征指定页面，确认数据存在 | 数据可见 |
| 无 `[验证:]` 标签 | **跳过验证**，直接进入步骤 4 | — |

**验证超时**：单条前置条件的验证操作超过 10 秒未完成，视为验证失败。

#### 6.3.2 自愈策略

| 验证结果 | 自愈动作 | 限制 |
|---------|---------|------|
| **数据不存在** | UI自建 → 执行前置条件中的创建步骤；API → 重新调用创建接口 | 最多 1 次 |
| **数据状态不符** | 不自愈，标记阻塞 | — |

**不可自愈的策略**：
- `[数据准备: 环境预置]` → AI 无法重建环境预置数据，直接标记阻塞
- `[数据准备: 依赖用例]` → 需要重新执行前序用例，直接标记阻塞
- 自愈成功后须再次验证确认数据已就绪

#### 6.3.3 失败处理

| 场景 | 用例状态 | 后续处理 |
|------|---------|---------|
| 验证通过 | 继续执行 | 进入步骤 4（CP1 起始截图） |
| 验证失败 + 自愈成功 | 继续执行 | 报告中标注"数据就绪验证: 自愈成功" |
| 验证失败 + 自愈失败 | **阻塞** | 记录阻塞原因"数据就绪验证失败: {原因}"，进入下一用例 |
| 验证失败 + 不可自愈 | **阻塞** | 记录阻塞原因，进入下一用例 |
| 验证超时（>10 秒） | **阻塞** | 记录超时原因，进入下一用例 |

## 7. 截图检查点执行规范（双证据体系）

### 7.1 核心原则

AI 直接执行测试时，**必须（MUST）** 同时产生两种证据：

| 证据类型 | 工具 | 用途 | 性质 |
|---------|------|------|------|
| **文本证据** | `browser_snapshot` | 精确文本断言（通过/失败判定依据） | 机器可比较 |
| **视觉证据** | `browser_take_screenshot` | 人可审计的页面截图 | 人可审核 |

- **判定依据仍然是 `browser_snapshot` 文本**——精确、可编程比较
- **截图不参与通过/失败判定**，仅作为审计附件供人工交叉验证
- **缺少 CP1 级别截图的用例报告视为不完整**，不得作为有效测试证据

### 7.2 三级检查点模型

#### CP1 必截（MUST）— 缺失则报告不完整

| # | 场景 | 截图时机 | 证据目的 |
|---|------|---------|---------|
| 1 | **用例起始状态** | 执行第一个操作步骤之前 | 证明前置条件满足、页面初始状态正确 |
| 2 | **用例最终状态** | 最后一步验证完成之后 | 人工可审核最终结果是否与报告一致 |
| 3 | **L1 断言失败** | L1 验证不通过时立即截图 | 保留失败现场 |
| 4 | **关键状态变更** | toast 出现、弹窗确认、列表增减等 | 记录核心业务操作结果 |
| 5 | **异常/错误现场** | 意外弹窗、白屏、网络错误等 | 记录非预期现象 |

#### CP2 条件截（SHOULD）— 满足条件时截图

| # | 场景 | 触发条件 |
|---|------|---------|
| 1 | **重试操作** | 进入重试流程时（重试前后各一张） |
| 2 | **L3 观察性验证** | 遇到 L3 验证点时（截图代替人工主观判断） |
| 3 | **数据准备操作** | 执行非测试步骤的数据准备时 |
| 4 | **数据就绪自愈** | 数据就绪验证失败触发自愈时（自愈前后各一张） |

#### CP3 可选截（MAY）— AI 自主判断

AI 认为某个中间状态有审计价值时，可主动截图。不做强制要求。

### 7.3 截图调用时序

每条用例的截图调用嵌入执行流程中（对应第 6.1 节流程图）：

```
用例开始
  │
  ├─ 导航到目标页面 + 等待加载完成
  ├─ 【CP1 必截】browser_take_screenshot → {用例}_init_start_{时间戳}.png
  │
  ├─ 步骤 1: 操作
  │    ├─ browser_snapshot → 文本断言
  │    └─ [若为关键状态变更] 【CP1 必截】browser_take_screenshot
  │
  ├─ 步骤 2: 操作
  │    ├─ browser_snapshot → 文本断言
  │    └─ [若 L1 失败] 【CP1 必截】browser_take_screenshot
  │
  ├─ ... (后续步骤)
  │
  ├─ 【CP1 必截】browser_take_screenshot → {用例}_final_{结果}_{时间戳}.png
  │
  └─ 记录结果 + 截图文件名列表
```

### 7.4 截图命名规范

```
{CP级别}_{用例编号}_{步骤标识}_{状态}_{时间戳}.png

时间戳格式: YYYYMMDDHHmmss
```

| 字段 | 可选值 | 说明 |
|------|-------|------|
| CP级别 | `CP1` / `CP2` / `CP3` | 检查点强制级别 |
| 用例编号 | `JQB-TEAM-001` 等 | 用例唯一标识 |
| 步骤标识 | `init` / `step01` / `final` / `dataprep` | 截图时刻标识 |
| 状态 | `start` / `pass` / `fail` / `retry1` / `error` | 截图时的判定状态 |

**示例**：
```
CP1_JQB-TEAM-001_init_start_20260325100000.png       ← 用例起始状态
CP1_JQB-TEAM-001_step03_pass_20260325100030.png       ← 关键状态变更(toast)
CP1_JQB-TEAM-001_final_pass_20260325100045.png        ← 用例最终状态
CP1_JQB-TEAM-034_step06_fail_20260325103000.png       ← L1断言失败
CP2_JQB-TEAM-034_dataprep_pass_20260325102800.png     ← 数据准备
CP2_JQB-TEAM-012_step01_retry1_20260325101500.png     ← 重试第1次
```

### 7.5 截图存储

- **统一存储路径**：`report/screenshots/YYYY-MM-DD/`（与执行报告同在迭代 `report/` 目录下）
- **每日独立子目录**：按执行日期分目录
- **证据索引文件**：每日截图目录下生成 `evidence_index.md`，记录截图清单和统计

### 7.6 开销控制

| 指标 | 估算值 |
|------|-------|
| 每条用例 CP1 截图数 | 2-4 张（起始 + 最终 + 0-2 张状态变更） |
| 10 条用例总截图数 | 30-40 张 |
| 单张截图耗时 | 1-2 秒 |
| 额外总耗时 | < 60 秒（< 3% 执行时间） |
| 单张截图大小 | 200-500 KB |
| 单次执行总存储 | 10-20 MB |

### 7.7 截图失败处理

- CP1 截图调用失败（如路径错误、权限不足）时：**记录告警到报告，但不终止用例执行**
- 在报告的「证据摘要」章节中标注缺失的截图
- CP1 覆盖率低于 95% 时，报告中必须标注告警

## 8. 生产环境执行约束

> **适用场景**：D5 发版后在生产环境执行线上回归验证。生产环境与测试环境的核心差异是**真实用户和数据正在使用中**，任何非预期操作都可能影响线上业务。

### 8.1 五项基本原则

| # | 原则 | 说明 | 违反后果 |
|---|------|------|---------|
| 1 | **只读优先** | 默认只执行查询、展示、导航操作；禁止 CUD 业务数据、资金操作、配置变更 | 可能影响线上用户数据 |
| 2 | **幂等安全** | 必须执行写操作时需同时满足：操作可幂等 + 用例包含回滚步骤 + 使用测试标记数据 | 产生不可回滚的脏数据 |
| 3 | **数据隔离** | 使用专用生产测试账号；测试数据加 `_AITEST_` 标记，与真实业务数据隔离 | 测试数据混入业务数据 |
| 4 | **时间窗口** | 默认在业务低峰期执行；可通过用例的 `[时间约束]` 标签覆盖 | 影响高峰期系统性能 |
| 5 | **操作审计** | 每步操作记录审计日志；所有截图升级为 CP1 必截（无 CP2/CP3 降级） | 无法追溯操作影响 |

### 8.2 操作决策矩阵

| 操作类型 | 生产环境允许 | 前置条件 | 示例 |
|---------|------------|---------|------|
| 页面导航 / 查看 / 搜索 | **允许** | 使用测试账号 | 浏览订单列表、查看详情页 |
| 表单填写（不提交） | **允许** | 仅验证渲染和交互 | 填写表单验证字段校验提示 |
| 创建测试数据 | **有条件允许** | 幂等 + 回滚步骤 + `_AITEST_` 标记 | 创建名为"测试公司_AITEST_{时间戳}"的记录 |
| 修改已有数据 | **禁止** | — | — |
| 删除已有数据 | **禁止** | — | — |
| 资金 / 交易操作 | **绝对禁止** | — | 充值、扣费、退款、转账 |
| 系统配置变更 | **绝对禁止** | — | 开关、权限策略、全局配置 |

### 8.3 生产环境熔断机制

在生产环境执行时，出现以下任一情况**立即停止**所有后续用例执行：

| # | 熔断触发条件 | 处理动作 |
|---|------------|---------|
| 1 | 任何非预期的数据修改 | 立即停止 → 截图现场 → 通知负责人 |
| 2 | 页面出现 5xx 服务器错误 | 立即停止 → 截图现场 → 通知负责人 |
| 3 | 操作影响了非测试账号数据 | 立即停止 → 截图现场 → 通知负责人 |
| 4 | 系统响应明显变慢（页面加载 >10s） | 立即停止 → 截图现场 → 等待低峰期重试 |
| 5 | 连续 2 条用例执行失败 | 暂停执行 → 截图现场 → 通知负责人评估是否继续 |

**熔断后处理**：
1. 记录熔断原因和当前执行进度
2. 检查已执行的写操作是否需要回滚
3. 执行所有待回滚步骤
4. 生成中断报告（引用 `ai_execution_report.md` §6）

### 8.4 生产测试账号规范

| 要求 | 说明 |
|------|------|
| **专用账号** | 生产环境必须使用专用的测试账号，禁止使用真实用户账号 |
| **最小权限** | 测试账号仅具有验证所需的最小权限，禁止管理员/超级权限 |
| **可识别** | 账号名称包含 `test` 或 `aitest` 标识，便于审计时区分 |
| **数据标记** | 测试账号创建的所有数据必须包含 `_AITEST_` 标记 |

## 9. D5 线上回归执行流程

> **前提**：D4 测试全部通过 + 代码已部署至生产环境。  
> **引用**：用例设计参考 `ai_test_case_design.md` G.3.5（线上回归用例适配规则）；执行约束参考本文档 §8。

### 9.1 执行流程总览

```
前提检查: D4 报告全部通过 + 确认生产部署完成
    │
    ▼
阶段 1: 生产环境预检（复用 §4.5，切换为生产环境 URL）
    │
    ├── 失败 → 终止线上回归，通知运维排查
    │
    ▼
阶段 2: 线上冒烟验证
    │   执行 [ENV: PROD] 或 [ENV: BOTH] 的 P0 冒烟用例
    │
    ├── 失败 → 触发回滚通知，终止线上回归
    │
    ▼
阶段 3: 已有功能回归
    │   执行 [ENV: PROD] 或 [ENV: BOTH] 的回归用例（G.3.4 策略选取）
    │
    ▼
阶段 4: 新功能线上验证
    │   执行 PROD-{MODULE}-{SEQ} 线上验证用例
    │
    ▼
阶段 5: 生成线上回归报告
        引用 ai_execution_report.md §6 模板
```

### 9.2 各阶段详细说明

#### 阶段 1：生产环境预检

- 复用 §4.5 部署预检流程，将目标环境切换为生产环境 URL
- 预检报告中 `**环境类型**` 填写"生产环境"
- 预检失败时**不启动**任何线上回归用例

#### 阶段 2：线上冒烟验证

- 从冒烟用例中筛选 `[ENV: PROD]` 或 `[ENV: BOTH]` 标记的 P0 用例
- 严格遵守 §8 生产环境执行约束
- **冒烟失败判定**：任一 P0 冒烟用例 L1 验证失败 → 立即终止 → 触发回滚通知
- 冒烟通过后方可进入阶段 3

#### 阶段 3：已有功能回归

- 按 `ai_test_case_design.md` G.3.4 策略选取回归用例
- 仅执行标记为 `[ENV: PROD]` 或 `[ENV: BOTH]` 的用例
- 标记为 `[ENV: TEST]` 的用例跳过，记录在报告的"仅测试环境用例备注清单"中
- 涉及写操作的用例执行完毕后**立即执行回滚步骤**

#### 阶段 4：新功能线上验证

- 执行 `PROD-{MODULE}-{SEQ}` 编号的线上验证专用用例
- 重点验证新功能在生产环境的展示和交互是否正确
- 以只读验证为主，写操作验证需满足 §8.2 条件

#### 阶段 5：生成线上回归报告

- 按 `ai_execution_report.md` §6 模板生成报告
- 报告必须包含操作审计清单
- 确认所有 PROD 写操作已回滚、测试数据已清理

### 9.3 线上回归执行配置

执行线上回归时，AI 需要以下配置信息：

| 配置项 | 说明 | 来源 |
|-------|------|------|
| **生产环境 URL** | 生产系统访问地址 | 用户提供 |
| **生产测试账号** | 专用测试账号和密码 | 用户提供 |
| **关联发版版本** | 本次发版的版本号 | 用户提供 |
| **D4 测试报告** | D4 阶段的执行报告路径 | 自动关联 |
| **回归用例集** | suite_prod_regression.md 路径 | test-case-design Mode C 产出 |

## 10. 临时文件管理

> **定位**：规范 AI 执行测试过程中产生的临时文件的存放、命名和清理，避免污染项目目录结构。

### 10.1 临时文件定义

| 文件类型 | 典型示例 | 产生场景 |
|---------|---------|---------|
| 调试脚本 | `db_check.py`、`db_verify.py` | 执行过程中临时编写的数据校验脚本 |
| 调试截图 | `debug_*.png`、`screenshot_*.png` | 非正式检查点的调试截图 |
| 调试日志 | `debug_*.log`、`trace_*.txt` | 临时输出的日志片段 |
| 临时配置 | `temp_*.json`、`temp_*.yaml` | 临时生成的配置或数据文件 |
| 录屏文件 | `*.mp4`、`*.webm` | 浏览器录屏（非正式报告附件） |

### 10.2 存放位置

```
iterations/{迭代名}/testcase/_temp/    ← 迭代专属临时文件统一存放目录
```

- 所有临时文件**必须**放在当前迭代的 `testcase/_temp/` 目录下
- **禁止**将临时文件放在项目根目录、`scripts/`、`testcase/` 根目录或其他非 `_temp/` 位置
- 正式截图（检查点截图）仍按 §7 规范存放在 `report/screenshots/YYYY-MM-DD/`

### 10.3 清理时机

| 清理时机 | 执行者 | 操作 |
|---------|-------|------|
| 单次会话结束 | AI 自检 | 检查 `_temp/` 目录，删除本次会话产生的临时文件 |
| 测试报告生成完成 | AI 自动 | 确认报告已生成后，清空 `_temp/` 目录 |
| 迭代归档时 | 人工确认 | 确认 `_temp/` 为空或仅保留有价值的辅助文件 |

### 10.4 AI 执行约束

1. **脚本存放**：临时编写的数据查询/校验脚本放 `testcase/_temp/`，不放 `testcase/data/` 或 `testcase/scripts/`
2. **截图路由**：调试截图放 `testcase/_temp/`，正式检查点截图放 `report/screenshots/YYYY-MM-DD/`
3. **会话结束自检**：每次测试会话结束前，AI 应主动检查并清理 `_temp/` 中的无用文件
4. **禁止行为**：
   - 禁止在项目根目录创建临时文件
   - 禁止在 `scripts/` 目录创建迭代专属的数据查询脚本
   - 禁止将调试截图与正式截图混放在 `report/screenshots/` 中
