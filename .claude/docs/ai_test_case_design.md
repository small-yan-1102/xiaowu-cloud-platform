# AI测试用例设计规则

> **版本**: v3.6 | **定位**: 测试用例设计原则、场景覆盖、验证判断、数据管理（AI直接执行版）

## 1. 测试用例设计原则

### 1.1 完整性原则
- 测试用例应覆盖所有功能需求和业务场景
- 包含正常场景、异常场景、边界场景
- 覆盖不同用户角色和权限场景
- 考虑多组测试数据验证

### 1.2 独立性原则
- 每个测试用例**优先**独立执行，不依赖其他用例的执行结果
- 当独立性无法满足时（如 CRUD 链、跨模块数据依赖），必须通过 `**依赖用例**` 字段显式声明依赖关系（详见附录 G.2）
- 测试用例之间不应隐式共享状态——所有数据传递必须通过依赖声明和数据准备策略标签显式表达
- 每个测试用例应包含独立的前置条件描述（详见附录 E）

### 1.3 可维护性原则
- 使用清晰的命名规范，用例名称应描述测试场景
- 添加详细的用例描述，说明测试目的、前置条件、测试步骤、预期结果
- 公共前置条件提取为可复用的前置步骤

### 1.4 可读性原则
- 测试用例应清晰易懂，像文档一样描述测试场景
- 使用有意义的测试数据（不用 "aaa"/"123" 等无意义数据）
- 预期结果必须具体明确，便于 AI 执行时判断通过/失败
- 遵循"准备-执行-验证"三段式结构

## 3. 测试场景覆盖要求

### 3.1 功能测试场景
- **正常流程**: 验证功能在正常输入下的表现
- **异常流程**: 验证错误处理和异常情况
- **边界条件**: 验证边界值和极限情况
- **业务规则**: 验证业务逻辑和计算规则

### 3.2 必须覆盖的场景类型

1. **正常场景测试**
   - 标准输入和操作流程
   - 典型用户使用场景

2. **异常场景测试**
   - 无效输入（空值、格式错误）
   - 越权操作
   - 网络异常、超时
   - 系统错误和异常处理

3. **边界值测试**
   - 最小值、最大值
   - 临界值（如最大长度-1、最大长度、最大长度+1）
   - 空值、零值测试

4. **数据验证测试**
   - 必填字段验证
   - 数据格式验证（邮箱、手机号、日期等）
   - 数据长度验证
   - 特殊字符处理

5. **权限和安全测试**
   - 未登录访问
   - 不同角色权限验证
   - 数据隔离验证

6. **兼容性测试**
   - 不同浏览器
   - 不同设备和分辨率

## 4. 验证判断规范

### 4.1 验证原则
- 每个测试步骤应包含明确的预期结果
- AI 执行时必须对预期结果做出明确的通过/失败判断
- 验证失败时应提供清晰的失败描述（看到了什么 vs 期望什么）
- 避免过多验证点集中在一个用例，必要时拆分

### 4.2 验证层级

| 级别 | 名称 | 适用场景 | 断言类型 | AI 执行要求 |
|---|---|---|---|---|
| **L1 强验证** | 必须通过 | 核心业务结果（登录成功/认证完成/操作生效） | `[断言: 视觉]` / `[断言: API]` / `[断言: DB-Query]` | 不通过则用例失败 |
| **L2 软验证** | 建议通过 | 数据一致性、UI 布局、次要展示字段 | `[断言: 视觉]` | 不通过则记录警告，不阻断 |
| **L3 观察性** | 记录即可 | 性能感受、动效、样式等主观判断项 | `[断言: 视觉]` | 截图记录，不做判断 |

**断言类型说明**：
| 类型 | 标识 | 验证方式 | 适用场景 |
|------|------|---------|---------|
| **视觉断言** | `[断言: 视觉]` | AI 通过页面快照/截图观察 | 文案、按钮状态、布局、toast 提示等 UI 可见内容 |
| **API 断言** | `[断言: API]` | AI 调用 GET 接口获取精确值 | 金额、积分、状态码、计数等需要精确值比对的场景 |
| **DB 断言** | `[断言: DB-Query]` | AI 执行只读 SQL 查询 | 多表关联数据、UI 不可见的后台状态、批量数据一致性 |

> **核心原则**：让 AI 做擅长的事（UI 流程验证用视觉断言），用确定性方法兜底 AI 不擅长的事（精确值比对用 API/DB 断言）。

### 4.3 操作-验证配对原则

**每一个操作后，必须紧跟对应的状态验证。**

```
操作：点击「提交认证」按钮  →  验证：出现"提交成功"提示（L1）
操作：填写公司名称          →  验证：输入框显示已填内容（L1）
操作：点击缩略图            →  验证：视频播放器弹出（L1）
```

禁止"做了操作不检查结果"的用例设计。

### 4.4 反向验证要求

每个涉及表单提交或数据操作的用例，成功验证后**必须追加错误反向检查**：

- 正向验证：确认操作成功（如出现成功提示）
- 反向验证：确认没有出现错误信息（如没有错误 toast）

### 4.5 检索结果数据一致性验证

检索/筛选类用例不仅要验证"没有报错"，还要验证**结果数据是否符合筛选条件**：

**4.5.1 内容一致性**（每条结果是否匹配筛选条件）：

- 日期范围搜索 → 验证结果中所有记录日期在范围内
- 关键词搜索 → 验证结果中所有记录包含关键词
- 状态筛选 → 验证结果中所有记录状态匹配

**4.5.2 数量完整性**（结果总数是否符合预期）：

检索/筛选类用例除验证内容一致性外，还应验证结果数量的完整性，确保筛选没有遗漏或多余：

| 前置条件数据精度 | 预期结果写法 | 示例 |
|----------------|------------|------|
| **精确数据准备**：前置条件明确了匹配数据的精确数量 | 验证分页控件总条数 = 已知数量 | 前置条件准备了 3 条匹配数据 → 预期"分页控件显示共 3 条"（L1） |
| **半控制**：前置条件未明确数量，但可通过对比验证 | 搜索前记录总数，搜索后对比变化 | 搜索前 `{记录为:搜索前总数}` → 搜索后"总条数 > 0 且 < `{引用:搜索前总数}`"（L1） |
| **不控制**（应避免） | 无法验证数量完整性 | 仅写"分页控件显示过滤后的总条数" — **验证形同虚设，应改进前置条件** |

**推荐做法**：搜索/筛选类用例的前置条件应明确匹配数据的**精确数量**（见附录 E.3），使预期结果可以写出确定的总条数。当无法精确控制数据时，至少应使用对比验证模式（增加搜索前记录步骤）。

### 4.6 确定性断言编写规范

> **目的**：规范 `[断言: API]` 和 `[断言: DB-Query]` 的使用场景、编写格式和约束条件，确保 AI 能正确执行确定性验证。

#### 4.6.1 什么时候使用确定性断言

| 场景 | 推荐断言类型 | 原因 |
|------|-------------|------|
| 验证 toast 提示文案 | `[断言: 视觉]` | 文案可见，视觉验证即可 |
| 验证按钮禁用状态 | `[断言: 视觉]` | 状态可见，视觉验证即可 |
| 验证金额计算结果 | `[断言: API]` / `[断言: DB-Query]` | 精确数值比对，视觉可能有格式干扰 |
| 验证积分变动 | `[断言: API]` | 需要前后值比对，API 更可靠 |
| 验证状态机流转 | `[断言: API]` | 后台状态可能与 UI 显示不同步 |
| 验证多表数据一致性 | `[断言: DB-Query]` | UI 可能只展示部分字段 |
| 验证批量操作结果 | `[断言: DB-Query]` | 逐条验证效率低，SQL 聚合更高效 |

**选择原则**：
- **默认使用 `[断言: 视觉]`**：大多数 UI 验证场景适用
- **涉及精确数值（金额/积分/计数）时升级为 `[断言: API]`**：避免格式化显示干扰（如 "¥1,234.56" vs "1234.56"）
- **涉及 UI 不可见数据或多表关联时升级为 `[断言: DB-Query]`**：如后台状态、审计日志、关联表计数

#### 4.6.2 API 断言编写格式

```markdown
**预期结果**：
3. [断言: API] GET /api/credit/balance 返回 balance = 1000（L1）
   接口地址：/api/credit/balance
   请求参数：{ "user_id": "{引用:用户ID}" }
   预期响应：{ "balance": 1000 }
   比对字段：balance（精确匹配）
```

**必要字段**：
| 字段 | 说明 | 是否必填 |
|------|------|---------|
| 接口地址 | 相对路径，如 `/api/credit/balance` | 必填 |
| 请求参数 | JSON 格式，支持 `{引用:变量名}` 引用前序步骤数据 | 选填 |
| 预期响应 | 预期的响应体（完整或部分字段） | 必填 |
| 比对字段 | 需要精确比对的字段名 | 必填 |

**认证说明**：API 调用默认使用当前登录态的 Cookie/Token，无需额外配置。

#### 4.6.3 DB 断言编写格式

```markdown
**预期结果**：
5. [断言: DB-Query] 积分明细表新增一条记录（L1）
   SQL：SELECT amount, type FROM credit_log WHERE user_id = '{引用:用户ID}' ORDER BY created_at DESC LIMIT 1
   预期结果：amount = 100, type = 'RECHARGE'
   比对规则：精确匹配
```

**必要字段**：
| 字段 | 说明 | 是否必填 |
|------|------|---------|
| SQL | 只读 SELECT 语句，支持 `{引用:变量名}` | 必填 |
| 预期结果 | 预期的查询结果（字段 = 值） | 必填 |
| 比对规则 | 精确匹配 / 包含 / 大于等于 等 | 必填 |

**约束条件**：
- **只允许 SELECT 语句**：禁止 INSERT/UPDATE/DELETE
- **必须限制结果集**：使用 WHERE 条件 + LIMIT
- **禁止全表扫描**：必须有索引字段作为查询条件

#### 4.6.4 混合断言示例

一条用例可以同时包含多种断言类型：

```markdown
**预期结果**：
1. [断言: 视觉] 页面显示"充值成功"toast 提示（L1）
2. [断言: 视觉] 积分余额区域显示新余额（L2）
3. [断言: API] GET /api/credit/balance 返回 balance = 1100（L1）
   接口地址：/api/credit/balance
   预期响应：{ "balance": 1100 }
   比对字段：balance（精确匹配）
4. [断言: DB-Query] 积分流水表新增充值记录（L1）
   SQL：SELECT amount, type FROM credit_log WHERE user_id = 'u001' ORDER BY id DESC LIMIT 1
   预期结果：amount = 100, type = 'RECHARGE'
   比对规则：精确匹配
```

**执行顺序**：视觉断言 → API 断言 → DB 断言（按预期结果编号顺序）

## 5. 测试数据管理

### 5.1 测试数据原则
- 使用独立的测试数据，不依赖生产数据
- 测试数据应可重复使用
- 敏感数据应脱敏处理
- 测试用例中必须明确列出所需测试数据

### 5.2 数据驱动测试

对于需要多组数据验证的场景（如登录、表单校验），在用例中以表格形式列出所有数据组合：

| 场景 | 用户名 | 密码 | 预期结果 |
|------|--------|------|---------|
| 正常登录 | valid_user | valid_pass | 登录成功，跳转首页 |
| 无效用户 | invalid_user | valid_pass | 提示用户不存在 |
| 错误密码 | valid_user | wrong_pass | 提示密码错误 |
| 空凭证 | (空) | (空) | 提示请输入 |

## 6. 测试分层策略

遵循测试金字塔原则，从下到上分为五层：

| 层级 | 测试类型 | 说明 | 用例优先级映射 |
|------|---------|------|---------------|
| L1 | 单元测试 | 验证最小可测试单元 | - |
| L2 | API/契约测试 | 验证接口契约、响应格式 | P3（边界测试） |
| L3 | 集成测试 | 验证模块间交互、数据一致性 | P2（异常测试） |
| L4 | E2E测试 | 端到端用户流程验证 | P1（主流程测试） |
| L5 | 冒烟测试 | 核心功能快速验证 | P0（冒烟测试） |

**AI 直接执行主要覆盖 L4（E2E）和 L5（冒烟）层级**，即通过浏览器操作验证完整用户流程和核心功能。

**各层在实际项目中的实践**：
- **L1 单元测试**：工具函数（如积分计算、日期格式化）、Vue组件逻辑
- **L2 API测试**：企业认证接口、积分充值接口、团队成员管理接口
- **L3 集成测试**：认证通过后积分赠送、策略变更后日志记录
- **L4 E2E测试**：完整认证流程、充值申请流程、团队管理流程
- **L5 冒烟测试**：登录→认证→核心功能访问的快速验证

## 7. AI 执行测试时的注意事项

### 7.1 执行前必须确认的信息
- 被测功能的详细需求和业务规则
- 测试环境 URL 和账号信息
- 需要覆盖的测试场景优先级
- **需求基线版本**（如 PRD V5.1）

### 7.2 AI 执行测试的步骤

1. **读取测试用例文档**
   - 识别用例编号、优先级、前置条件
   - 理解测试步骤和预期结果

2. **准备测试环境**
   - 打开浏览器，导航到目标页面
   - 确认前置条件满足（如已登录）

3. **逐步执行操作**
   - 按用例步骤操作浏览器（点击、填写、选择等）
   - 每步操作后等待页面响应稳定

4. **验证预期结果**
   - L1 强验证：必须明确判断通过/失败
   - L2 软验证：记录实际结果，标注是否符合预期
   - L3 观察性：截图记录

5. **记录执行结果**
   - 通过/失败判定
   - 失败原因描述（实际看到什么 vs 期望什么）
   - 关键步骤截图

6. **处理人工介入点**
   - 遇到验证码、人脸识别等无法自动完成的步骤时，暂停并通知用户

### 7.3 AI 执行时的判断原则

| 场景 | AI 应如何判断 |
|------|-------------|
| 页面出现"成功"提示 | 截图确认，标记步骤通过 |
| 页面出现错误 toast | 截图记录错误内容，标记步骤失败 |
| 页面无响应/加载超时 | 等待合理时间后截图，标记为环境问题 |
| 预期元素未出现 | 截图当前页面状态，标记步骤失败并描述差异 |
| 数据与预期不一致 | 记录实际值和预期值，标记步骤失败 |
| 遇到验证码等人工介入 | 暂停执行，通知用户介入 |

- [ ] **标注 `[数据准备: 静态数据池]` 的用例是否引用数据池文档和数据键**（见附录 E.7）

## 附录 B：人工介入标识规范

### B.1 定义

**人工介入（Manual Intervention Required）**：AI 无法独立完成该用例的全部步骤，需要人工在某一环节操作配合。

### B.2 人工介入分级（三级分级体系）

> **设计原则**：不是所有"人工介入"场景都需要真的暂停等人。部分场景可以通过 Mock/绕过/工具辅助解决，只有真正必须人工操作的才需要暂停。

| 级别 | 标识 | 含义 | AI 执行策略 | 适用场景 |
|------|------|------|------------|---------|
| **Tier-1** | `[人工介入: Tier-1 可Mock]` | 可通过 Mock 或测试环境配置绕过 | AI 使用预配置的 Mock 值/绕过机制，不暂停 | 图形验证码（测试环境固定值）、短信验证码（测试环境万能码）、第三方回调（Mock 服务） |
| **Tier-2** | `[人工介入: Tier-2 必须人工]` | 必须真人操作，无法 Mock | AI 暂停执行，通知用户介入，等待用户确认后继续 | 人脸识别、指纹解锁、硬件设备操作（U盾、NFC）、真实第三方 OAuth 授权 |
| **Tier-3** | `[人工介入: Tier-3 专用工具]` | 需要专用工具/脚本辅助 | AI 调用指定工具完成，不暂停 | 批量数据导入（使用数据导入脚本）、复杂计算验证（使用计算器脚本）、外部系统状态触发（使用 API 触发脚本） |

#### B.2.1 分级判断决策树

```
该步骤是否需要人工操作？
│
├── 否 → 不标注人工介入
│
└── 是 → 是否可通过测试环境配置/Mock 绕过？
        │
        ├── 是 → Tier-1（标注 Mock 方式）
        │        示例：[人工介入: Tier-1 可Mock（验证码固定值: 1234）]
        │
        └── 否 → 是否有专用工具/脚本可完成？
                │
                ├── 是 → Tier-3（标注工具路径）
                │        示例：[人工介入: Tier-3 专用工具（脚本: scripts/import_data.py）]
                │
                └── 否 → Tier-2（必须真人操作）
                         示例：[人工介入: Tier-2 必须人工（原因: 需刷脸认证）]
```

#### B.2.2 各级别典型场景

| 场景 | 旧分类 | 新分级 | Mock/工具说明 |
|------|--------|--------|-------------|
| 图形/滑动验证码 | 需人工 | **Tier-1** | 测试环境验证码固定值或万能码 |
| 短信/邮件验证码 | 需人工 | **Tier-1** | 测试环境万能验证码（如 123456） |
| 第三方 OAuth 弹窗 | 需人工 | **Tier-1** / **Tier-2** | Mock OAuth 服务 → Tier-1；真实授权 → Tier-2 |
| 人脸识别 / 生物特征 | 需人工 | **Tier-2** | 无法 Mock，必须真人 |
| 硬件依赖操作（U盾、NFC） | 需人工 | **Tier-2** | 无法 Mock，必须真人 |
| 人工判断视觉结果 | 需人工 | **Tier-2** | 主观判断，AI 无法替代 |
| 外部系统手动触发 | 需人工 | **Tier-1** / **Tier-3** | 有 API → Tier-3 工具调用；无 API → Tier-1 提前准备 |
| 批量数据导入 | - | **Tier-3** | 使用数据导入脚本 |
| 复杂业务计算验证 | - | **Tier-3** | 使用计算器脚本 |

### B.3 标注格式

**Tier-1 可 Mock**：
```markdown
**是否需要人工介入**：[人工介入: Tier-1 可Mock]
Mock 配置：验证码固定值 = 1234；万能短信码 = 123456
```

**Tier-2 必须人工**：
```markdown
**是否需要人工介入**：[人工介入: Tier-2 必须人工]
介入原因：用户需完成人脸识别认证
介入时机：执行步骤 5 后，AI 将暂停等待确认
```

**Tier-3 专用工具**：
```markdown
**是否需要人工介入**：[人工介入: Tier-3 专用工具]
工具路径：scripts/trigger_callback.py
调用参数：--order_id={引用:订单ID} --event=payment_success
```

**无需人工介入**：
```markdown
**是否需要人工介入**：否
```

## 附录 C：预期结果分级与验证强度规范

### C.1 预期结果必须分为三个强度级别

| 级别 | 名称 | 适用场景 | AI 执行要求 |
|---|---|---|---|
| **L1 强验证** | 必须通过 | 核心业务结果（登录成功/认证完成/操作生效） | 不通过 → 用例失败 |
| **L2 软验证** | 建议通过 | 数据一致性、UI 布局、次要展示字段 | 不通过 → 记录警告，继续执行 |
| **L3 观察性** | 记录即可 | 性能感受、动效、样式等主观判断项 | 仅截图记录 |

### C.2 操作-验证配对原则

每一个 UI 操作后，必须紧跟对应的状态验证：

```
操作：点击「上传」按钮      →  验证：上传对话框已弹出（L1）
操作：选择文件              →  验证：文件名显示在输入框（L1）
操作：点击「确认」          →  验证：成功提示弹出（L1）
                           验证：列表新增一条记录（L1）
                           验证：记录数据与输入一致（L2）
操作：点击缩略图            →  验证：视频播放器已弹出（L1）
```

### C.3 反向验证要求

每个涉及表单提交或数据操作的用例，成功验证后必须追加错误反向检查：

- 正向验证：确认操作成功（如看到"提交成功"提示）
- 反向验证：确认没有出现错误信息（如页面无错误 toast、无报错弹窗）

### C.4 验证覆盖度自查表

编写或评审测试用例时，对照以下清单逐项确认：

```
☐  每个测试步骤的预期结果，都有明确的验证描述
☐  没有"操作了但不检查结果"的步骤
☐  涉及表单提交的用例，已添加错误反向验证
☐  预期结果描述足够具体，AI 能明确判断通过/失败
☐  验证点数量 ≥ 预期结果条数
```

## 附录 D：预期结果反模式清单

> **目的**：AI 直接执行模式下，模糊的预期结果会导致 AI 无法准确判断 pass/fail，是误判的主要来源。本附录列出高频模糊预期词汇及其改写示范，设计和审查用例时**必须对照检查**。

### D.1 禁用模糊词及改写示范

| # | 模糊预期（禁止使用） | 问题说明 | 具体预期（正确写法示例） |
|---|-------------------|---------|----------------------|
| 1 | 页面正常显示 | AI 无法定义"正常"标准 | 页面标题显示"企业认证"，表单包含"公司名称""统一社会信用代码"两个输入框（L1） |
| 2 | 数据正确 / 数据正常 | AI 不知道"正确"的数据长什么样 | 列表第一行显示公司名称"XX有限公司"，状态列显示"审核中"（L1） |
| 3 | 提交成功 | 缺少成功的具体表征 | 页面顶部出现绿色 toast 提示"提交成功"（L1），且 3 秒后自动消失（L2） |
| 4 | 无异常 / 无报错 | AI 无法穷举"无异常"的范围 | 页面无错误弹窗、无红色错误提示文字、浏览器控制台无 Error 级别日志（L1） |
| 5 | 加载成功 / 加载完成 | AI 不知道"完成"的判定条件 | 页面骨架屏消失，列表显示至少 1 条数据记录，底部分页器显示总条数（L1） |
| 6 | 正确跳转 | AI 不知道目标页面是什么 | 浏览器地址栏 URL 变为 `/dashboard`，页面标题显示"工作台"（L1） |
| 7 | 样式正确 / 显示正常 | 纯主观判断，AI 无法执行 | 按钮文字为"提交"，按钮可点击（非 disabled 状态）（L1） |
| 8 | 功能正常 | 最大的模糊词，毫无判断依据 | 点击"导出"按钮后，浏览器触发文件下载，文件名包含当前日期（L1） |
| 9 | 符合预期 / 与预期一致 | 循环定义，预期结果不能引用"预期" | 输入框中的值为"张三"，与测试数据表中第 1 行"姓名"字段一致（L1） |
| 10 | 弹窗弹出 / 弹窗消失 | 缺少弹窗内容的具体描述 | 弹出确认对话框，标题为"确认删除"，内容包含"此操作不可恢复"，底部有"取消"和"确认"两个按钮（L1） |

### D.2 判断规则

编写预期结果时，逐条对照以下规则：

1. **可观测性**：预期结果必须描述 AI 能通过页面快照（accessibility snapshot）或截图**直接观测到**的内容（文本、元素、状态、URL）
2. **无歧义性**：同一条预期结果，不同人（或不同 AI 会话）读到后应得出相同的 pass/fail 判断
3. **自包含性**：预期结果本身包含判断所需的全部信息，不依赖读者的"常识"或"经验"去补全
4. **禁止循环引用**：预期结果中不得使用"符合预期""与预期一致""正确""正常"等自我引用词汇

### D.3 快速自检公式

编写完预期结果后，用以下公式快速自检：

```
能否回答：AI 在页面上看到【什么元素/文本/状态】就判定通过？
```

- 能明确回答 → 预期结果合格
- 回答不出或需要"看情况" → 预期结果模糊，必须改写

## 附录 E：前置条件与数据准备编写规范

> **目的**：AI 直接执行模式下，模糊的前置条件和未明确的数据准备是用例执行卡住的首要原因。本附录规范前置条件和测试数据的编写方式，确保 AI 能自主判断"条件是否满足"并知道"如何达成条件"。
>
> **与 `ai_execution_strategy.md` 第 4 节的关系**：第 4 节规范 AI 执行时的环境操作行为（如何登录、何时清缓存）；本附录规范用例编写时如何把前置条件和数据写清楚。一个管"怎么写"，一个管"怎么做"。

### E.1 前置条件分类与编写规则

每条前置条件必须标注**类型**和 **AI 达成方式**，让 AI 知道该自己准备还是跳过。

| 类型 | 说明 | AI 是否可自主达成 | 编写要求 |
|------|------|-----------------|---------|
| **登录态** | 以某角色身份登录系统 | 可以 | 必须写明账号、密码（或引用测试数据表）和登录页 URL |
| **页面定位** | 处于某个特定页面 | 可以 | 必须写明页面导航路径或直接 URL |
| **数据存在** | 系统中已存在某条数据 | 视情况 | 标注数据来源策略（见 E.3） |
| **系统配置** | 某个开关/配置已开启 | 通常不可以 | 标注"需运维/管理员预配置"，并说明如何确认配置已生效 |
| **外部依赖** | 第三方服务可用、回调地址已配 | 不可以 | 标注"外部依赖"，并说明不可用时的替代方案或跳过策略 |

### E.2 前置条件反模式清单

| # | 模糊前置条件（禁止使用） | 问题说明 | 具体前置条件（正确写法示例） |
|---|----------------------|---------|--------------------------|
| 1 | 已登录系统 | 未指定账号和角色 | 使用管理员账号 admin/Test1234 登录管理端（URL: https://test.example.com/admin/login） |
| 2 | 已有测试数据 | AI 不知道什么数据、在哪里 | 系统中已存在一条企业认证记录：公司名称="测试公司_{时间戳}"，状态="待审核"（通过「认证管理→新增认证」页面创建） |
| 3 | 进入XX页面 | 未给出导航路径 | 导航到「系统管理 → 团队管理」页面（URL: /admin/team-manage） |
| 4 | 环境正常 | AI 无法判断"正常" | 测试环境 https://test.example.com 首页可访问，登录功能正常（能成功登录并看到工作台页面） |
| 5 | 数据库中已有XX记录 | AI 无法操作数据库 | 通过管理端「数据管理→新增」页面创建一条记录：名称="测试记录"，类型="普通" |
| 6 | 完成XX操作 | 未说明具体步骤 | 已完成企业认证提交：进入「认证中心」→ 填写公司名称"测试公司" → 上传营业执照 test_license.png → 点击「提交」→ 看到"提交成功"提示 |

### E.3 数据准备策略

测试用例中涉及"系统中需要已存在某条数据"时，必须标注数据来源策略：

| 策略 | 标识 | 适用场景 | 编写要求 | 验证特征要求 |
|------|------|---------|---------|-------------|
| **UI 自建** | `[数据准备: UI自建]` | 数据可通过被测系统界面创建 | 在前置条件中写出完整的创建步骤（页面路径 + 填写字段 + 提交操作） | 写明验证页面路径 + 搜索条件 + 预期可见的关键字段值 |
| **前序用例产出** | `[数据准备: 依赖用例 PRJ-XXX-NNN]` | 数据由另一条用例创建 | 标注依赖的用例编号，并说明该用例必须先执行且通过 | 写明依赖用例编号 + 数据验证页面路径 |
| **环境预置** | `[数据准备: 环境预置]` | 初始化脚本或运维预配置的数据 | 标注数据的识别特征（如固定 ID、固定名称），以便 AI 在页面上定位 | 写明数据固定识别特征 + 确认页面路径 |
| **API 注入** | `[数据准备: API]` | 通过接口直接创建（适用于复杂数据、批量数据、UI 创建步骤过多的场景） | 引用技术系分文档路径，标注接口地址、请求方法、关键参数、认证方式；AI 可自主调用（详见 `test-data-preparation` skill） | 写明 GET 验证接口或 UI 页面路径 + 预期关键字段值 |
| **静态数据池** | `[数据准备: 静态数据池]` | 高频复用的稳定测试数据（如标准账号、固定配置、基础字典数据） | 引用数据池文档路径，标注数据池名称和数据键；执行前通过数据池管理确保数据可用 | 写明数据池文档路径 + 数据键 + 预期状态；详见 E.7 |

**策略选择指南**：
- **优先 `UI 自建`**：数据量少、创建流程简单（≤5 步），最自然且可验证
- **优先 `API 注入`**：数据结构复杂、批量创建、UI 创建步骤过多（>5 步），需提供技术系分文档
- **`前序用例产出`**：存在 CRUD 链或上下游数据依赖时使用
- **`环境预置`**：基础字典数据、系统配置等不变数据
- **`静态数据池`**：跨用例/跨套件复用的稳定数据，减少重复准备成本；详见 E.7

**搜索/筛选类用例的数据准备要求**：

涉及搜索、筛选、条件查询的用例，前置条件中的数据准备应明确**匹配数据的精确数量**，以便预期结果可以验证数量完整性（见 §4.5.2）：

```
正确：[数据存在][数据准备: API] 系统中已存在 3 条作品名称含"测试"的任务单
       （通过 POST /api/takedown/create 创建，引用技术文档：xxx）
不足：[数据存在] 系统中存在包含关键词的任务单（AI 无法验证结果数量是否完整）
```

#### E.3.5 验证特征编写规范

**定义**：验证特征是一行描述，告诉 AI 在执行用例步骤之前如何确认数据仍然存在且状态正确。

**标签格式**：`[验证: {验证描述}]`，写在数据描述之后。

**三种验证方式**：

| 验证方式 | 格式 | 示例 |
|---------|------|------|
| **页面验证** | `[验证: 「{页面}」搜索"{关键词}"可见, {字段}="{值}"]` | `[验证: 「团队管理」搜索"测试成员"可见, 角色="运营"]` |
| **API 验证** | `[验证: GET {endpoint} 返回 {条件}]` | `[验证: GET /company/comp_001 返回 status=active]` |
| **存在性验证** | `[验证: 「{页面}」包含"{固定标识}"]` | `[验证: 「系统配置」包含"默认角色"]` |

**编写原则**：
- 验证特征应 ≤ 1 行，简洁明了
- 必须包含可观测的判定条件（页面可见 / API 有返回 / 字段值匹配）
- 无 `[验证:]` 标签的前置条件，执行时跳过数据就绪验证（向后兼容）

**示例**：

```
（正确）
3. [数据存在][数据准备: UI自建] 系统中已存在一个团队成员：
   - 创建路径：点击「添加成员」→ 填写姓名"测试成员_{时间戳}" → 点击「确认」
   [验证: 「团队管理」搜索"测试成员_{时间戳}"可见, 角色="运营"]

（不足）
3. [数据存在][数据准备: UI自建] 系统中已存在一个团队成员
   （无验证特征 — AI 执行时无法确认数据是否就绪）
```

### E.4 数据清理规则

> **核心目标**：确保测试用例可**重复执行** — 第 N+1 次执行时，不会因为第 N 次执行残留的数据而失败。

#### E.4.1 清理策略选择

| 场景 | 清理策略 | 编写要求 |
|------|---------|---------|
| 被测系统有删除功能 | **UI 清理**：用例末尾增加清理步骤 | 在用例最后添加「清理步骤」区块，写明删除操作路径和成功标志 |
| 被测系统有删除 API | **API 清理**：通过接口删除 | 在 `test-data-preparation` skill 的数据清理指引中标注清理接口 |
| 被测系统无删除功能 | **唯一标识规避**：每次用新数据 | 数据名称加入时间戳后缀（如"测试公司_{yyyyMMddHHmmss}"），每次执行自动生成新值 |
| 数据清理可能影响其他用例 | **不清理，仅标注** | 在备注中标注"本用例产生的测试数据未清理"，在执行报告中汇总 |

#### E.4.2 清理时机

| 时机 | 说明 | 适用场景 |
|------|------|---------|
| **用例后清理** | 每条用例执行完毕后立即清理该用例创建的数据 | 数据仅本用例使用，不被其他用例依赖 |
| **批次后清理** | 整批用例全部执行完毕后统一清理 | 数据被多条用例共享（如依赖用例产出的数据） |
| **执行前清理** | 执行用例前先检测并清除上次残留数据 | 上次执行中断、清理步骤未执行到的场景 |

**默认策略**：`用例后清理` + `执行前清理`（双保险）。

#### E.4.3 执行前残留数据检测

为确保可重复执行，用例执行前 AI 应执行残留数据检测：

```
执行前检测流程：
1. 检查前置条件中涉及的唯一性数据是否已存在
   - 如：在列表页搜索"测试公司_{本次时间戳}"
   - 或：调用 GET/LIST 接口查询
2. 如果数据已存在：
   a. 有删除功能 → 先删除残留数据，再执行用例
   b. 无删除功能 → 更换时间戳后缀，使用新数据执行
3. 如果数据不存在 → 正常执行用例
```

**用例编写要求**：对需要唯一性的数据字段，必须使用动态标记 `{时间戳}` 而非硬编码值：
- 正确：`公司名称 = "测试公司_{时间戳}"`
- 错误：`公司名称 = "测试公司_20260326"`（第二次执行会因重名失败）

#### E.4.4 依赖链的清理顺序

当数据之间存在依赖关系时（如外键、父子关系），清理必须按**依赖关系的逆序**执行：

```
创建顺序：企业(A) → 部门(B, 依赖A) → 成员(C, 依赖B)
清理顺序：成员(C) → 部门(B) → 企业(A)    ← 反向

禁止：直接删除企业(A) — 会因部门(B)的外键约束而失败
```

**编写要求**：
- 有依赖链的清理步骤必须按逆序编写
- 在清理步骤中标注依赖关系：`先删除成员（因为成员依赖部门）→ 再删除部门 → 最后删除企业`

#### E.4.5 跨系统数据清理

跨系统用例（附录 G）产生的数据需要在**每个系统**中分别清理：

```
用例：在系统A创建订单 → 在系统B审核订单
清理：
1. [SYS-B] 删除/撤回系统B中的审核记录（如有删除功能）
2. [SYS-A] 删除系统A中的订单记录
```

**编写要求**：
- 跨系统清理步骤必须标注 `[SYS-X]` 系统标识
- 清理顺序：下游系统先清理，上游系统后清理（与数据流向相反）

#### E.4.6 清理失败处理

| 清理失败原因 | 处理方式 |
|------------|---------|
| 数据被其他业务锁定 | 记录"清理失败"，标注原因，不阻断后续用例执行 |
| 删除接口返回错误 | 重试一次；仍失败则记录错误信息，继续执行 |
| 无权限删除 | 标注"需人工清理"，在执行报告中汇总 |
| 系统无删除入口 | 使用唯一标识规避策略（E.4.1），无需清理 |

**原则**：清理失败**不阻断**测试执行，但必须记录在执行报告中。

#### E.4.7 清理步骤编写模板

```markdown
**清理步骤**（用例执行完毕后执行）：
1. [清理] 导航到「XXX管理」列表页
2. [清理] 搜索本用例创建的数据（关键词："{测试数据中的唯一标识}"）
3. [清理] 点击该记录的「删除」按钮
4. [清理] 在确认弹窗中点击「确认删除」
5. [清理] 验证列表中已无该记录（L1）

**清理失败处理**：如删除失败，在备注中记录"数据残留：{数据描述}"，不阻断后续用例
```

### E.6 快速自检公式

编写完前置条件后，逐条对照以下问题：

```
1. AI 读到这条前置条件后，能否在不问任何人的情况下自主达成？
   - 能 → 合格
   - 不能 → 是否标注了"需要人工介入"或"环境预置"？
     - 标注了 → 合格
     - 没标注 → 不合格，必须补充达成方式

2. 如果前置条件要求"数据已存在"，AI 是否知道：
   - 数据长什么样？（具体字段和值）
   - 数据在哪个页面能看到？（页面路径）
   - 数据如何创建？（操作步骤或标注为环境预置）
   - 三个问题都能答 → 合格
   - 有一个答不出 → 不合格，必须补充

3. 如果前置条件要求"数据已存在"，AI 是否知道如何在执行前验证数据仍然就绪？
   - 是否标注了 [验证: ...] 标签？
   - 标签中是否包含具体的页面路径/API 和预期字段值？
   - 两个问题都能答 → 合格
   - 有一个答不出 → 不合格，必须补充
```

### E.7 静态数据池规范

> **目的**：定义静态数据池的结构、使用场景和管理规则，为高频复用的稳定测试数据提供统一管理方案，减少用例间的重复数据准备。

#### E.7.1 什么是静态数据池

**静态数据池**是一组预定义、稳定存在、可跨用例/跨套件复用的测试数据集合。与动态创建的测试数据不同，静态数据池中的数据：
- **预先存在**：在测试执行前已准备就绪
- **相对稳定**：不会因单次用例执行而被删除或修改
- **可重复使用**：多条用例可引用同一数据
- **有恢复机制**：被修改后可通过脚本重置

#### E.7.2 适用场景

| 适用 | 不适用 |
|------|--------|
| 登录账号（各角色标准测试账号） | 一次性消耗的数据（如优惠券码） |
| 基础配置数据（角色、权限、字典） | 状态会被用例改变且无法重置的数据 |
| 固定查询条件的数据（搜索/筛选测试用） | 需要唯一性的数据（如每次不同的公司名） |
| 跨模块依赖的公共数据（如公共客户、公共订单） | 临时性测试数据 |

#### E.7.3 数据池文档结构

静态数据池以 Markdown 文档形式管理，存放于 `test_data/pools/` 目录：

```markdown

> **适用环境**: TEST / UAT
> **维护人**: {测试工程师姓名}
> **更新日期**: {YYYY-MM-DD}

## 数据清单

| 数据键 | 描述 | 字段值 | 状态 | 验证方式 |
|--------|------|--------|------|---------|

| `operator_account` | 运营角色账号 | 账号: operator01, 密码: Test1234 | 活跃 | 登录成功可见运营工作台 |
| `test_company_001` | 标准测试企业 | ID: comp_001, 名称: "静态测试公司", 状态: 已认证 | 活跃 | 「企业管理」搜索"静态测试公司"可见 |
```

#### E.7.5 静态数据池 vs 其他数据准备策略

| 维度 | 静态数据池 | UI 自建 | API 注入 | 环境预置 |
|------|-----------|---------|---------|---------|
| 数据生命周期 | 长期存在 | 用例执行时创建 | 用例执行时创建 | 环境初始化时创建 |
| 复用性 | 高（跨用例/跨套件） | 低（单用例） | 中（可批量） | 高（但不可变） |
| 可修改性 | 有限（可重置） | 高 | 高 | 无 |
| 适用数据量 | 小批量高频数据 | 单条 | 批量 | 基础配置 |
| 维护成本 | 中（需维护数据池文档） | 低 | 低 | 低 |

#### E.7.6 静态数据池管理规则

| 规则 | 说明 |
|------|------|
| **命名规范** | 数据键使用 snake_case，如 `admin_account`、`test_company_001` |
| **版本控制** | 数据池文档纳入版本管理，变更需走评审 |
| **环境隔离** | 不同环境（TEST/UAT/PROD）使用不同数据池或不同数据 |
| **状态标记** | 每条数据标注状态（活跃/禁用/待重置），禁用数据不可引用 |
| **重置机制** | 每个数据池必须配套重置脚本，确保数据可恢复 |
| **定期巡检** | 每周执行一次数据池健康检查，验证数据是否仍有效 |

## 附录 F：测试步骤编写规范

> **目的**：规范测试步骤的粒度、动作词、元素描述方式、动态内容处理和条件分支写法，确保 AI 读到每一步都能准确执行，不需要猜测或补全。

### F.1 步骤粒度标准

**核心原则：一步 = 一个 UI 动作 + 对应的验证。**

| 粒度 | 示例 | 是否合适 |
|------|------|---------|
| **过粗** | "填写表单并提交" | 不合适 — AI 不知道要填哪些字段，用什么值 |
| **合适** | "在「公司名称」输入框中输入"测试公司"" | 合适 — 一个动作，目标明确 |
| **过细** | "将光标移动到输入框 → 单击左键 → 键入第一个字符'测'" | 不合适 — 噪声过多，AI 理解即可 |

**粒度判断规则**：
1. 每一步 AI 只需要执行**一个浏览器操作**（点击、输入、选择、上传等）
2. 涉及多个字段的表单，**每个字段单独一步**（不要合并为"填写表单"）
3. 连续的简单操作（如"勾选复选框A → 勾选复选框B"）可合并为一步，前提是它们逻辑紧密且验证点相同

### F.2 标准动作词表

AI 解析测试步骤时，通过动作词识别操作类型。用例编写必须使用以下标准动作词：

| 标准动作词 | 对应浏览器操作 | 禁用的模糊词 | 示例 |
|-----------|-------------|-------------|------|
| **点击** | click | 按下、单击、触碰、选择按钮 | 点击「提交」按钮 |

### F.4 动态内容预期结果处理

当预期结果涉及动态变化的值（时间戳、自增 ID、系统生成编号等）时，不能写死具体值。

#### F.4.1 动态值标记语法

| 标记 | 含义 | 示例 |
|------|------|------|
| `{当前日期}` | 执行当天的日期（YYYY-MM-DD） | 创建时间显示为 `{当前日期}`（L1） |
| `{当前时间}` | 执行时刻（HH:mm，±1分钟容差） | 操作时间显示为 `{当前日期} {当前时间}`（L2） |
| `{时间戳}` | 引用测试数据中使用的时间戳后缀 | 公司名称显示为"测试公司_{时间戳}"（L1） |
| `{非空}` | 值不为空即可，不关心具体内容 | 订单编号为 `{非空}`（L1） |
| `{格式:PATTERN}` | 值需符合指定格式 | 订单号 `{格式:ORD-\d{8}-\d{3}}`（L1） |
| `{与测试数据.字段名一致}` | 值需与测试数据表中指定字段一致 | 显示的公司名称 `{与测试数据.公司名称一致}`（L1） |
| `{包含:关键词}` | 值包含指定关键词即可 | 提示信息 `{包含:成功}`（L1） |

#### F.4.2 匹配模式选择指南

| 场景 | 推荐匹配模式 | 示例 |
|------|-------------|------|
| 系统生成的编号/ID | `{非空}` 或 `{格式:PATTERN}` | 工单号 `{格式:WO-\d{6}}` |
| 创建/更新时间 | `{当前日期}` + 容差 | 创建时间为 `{当前日期}`（±1天容差） |
| 由用例输入的值 | `{与测试数据.字段名一致}` | 姓名显示为 `{与测试数据.姓名一致}` |
| 提示消息 | `{包含:关键词}` 或精确匹配 | toast 提示 `{包含:提交成功}` |
| 已知固定值 | 精确匹配 | 状态显示为"待审核" |

### F.5 条件分支步骤

测试执行中可能遇到"不确定是否会出现"的场景，需要在用例中表达条件分支。

#### F.5.1 条件步骤语法

```markdown
**测试步骤**：
1. 导航到登录页面
2. [条件] 如果出现「Cookie 使用同意」弹窗 → 点击「同意」按钮
3. 在「用户名」输入框中输入"admin"
4. 在「密码」输入框中输入"Test1234"
5. 点击「登录」按钮
6. [条件] 如果出现「首次登录引导」弹窗 → 点击「跳过」
```

#### F.5.2 条件步骤规则

| 规则 | 说明 |
|------|------|
| 使用 `[条件]` 前缀标记 | AI 识别到 `[条件]` 时，先判断条件是否成立，成立则执行，不成立则跳过 |
| 条件步骤**不参与** pass/fail 判定 | 条件未触发（如弹窗未出现）不算失败 |
| 条件步骤**不影响**步骤编号连续性 | 跳过的条件步骤在报告中标记为"跳过（条件未触发）" |
| 避免过多条件步骤 | 同一用例中条件步骤不超过 3 个，过多说明用例拆分不够 |

#### F.5.3 条件步骤 vs 意外弹窗处理的界限

| 场景 | 归属 | 说明 |
|------|------|------|
| 可预见的弹窗（Cookie同意、首次引导） | 用例中用 `[条件]` 步骤处理 | 属于已知场景，写入用例 |
| 不可预见的弹窗（广告、系统通知） | `ai_execution_strategy.md` §3.3 自动处理 | 属于环境噪声，不写入用例 |

## 附录 G：跨系统测试与用例组织规范

> **目的**：规范涉及多个系统的测试用例编写方式，以及用例间依赖关系声明和套件组织方法，确保 AI 能正确处理跨系统验证和有序执行。

### G.1 跨系统测试编写规范

#### G.1.1 系统注册表

涉及多系统测试时，用例文档头部必须声明所有涉及的系统：

```markdown
## 系统注册表

| 系统标识 | 系统名称 | 测试环境 URL | 登录账号 | 登录密码 | 说明 |
|---------|---------|-------------|---------|---------|------|
| SYS-A | 管理后台 | https://admin-test.example.com | admin | Test1234 | 内部管理系统 |
| SYS-B | 客户端 | https://client-test.example.com | user01 | Test5678 | C端用户系统 |
| SYS-C | 第三方平台 | https://partner-test.example.com | partner | Test9012 | 合作方系统 |
```

#### G.1.2 跨系统步骤编写

每个操作步骤必须标注**目标系统**，让 AI 知道要切换到哪个系统：

```markdown
**测试步骤**：
1. [SYS-A] 导航到「订单管理」页面
2. [SYS-A] 点击「新建订单」按钮
3. [SYS-A] 在「客户名称」输入框中输入"测试客户"
4. [SYS-A] 点击「提交」按钮
5. [SYS-A] 验证：出现"订单提交成功"提示，订单号为 {非空}（L1）
6. [SYS-B] 导航到「我的订单」页面
7. [SYS-B] 验证：订单列表中出现一条新订单，客户名称为"测试客户"，状态为"待确认"（L1）
8. [SYS-B] 点击该订单的「确认」按钮
9. [SYS-A] 刷新「订单管理」页面
10. [SYS-A] 验证：该订单状态变为"已确认"（L1）
```

#### G.1.3 系统切换规则

| 规则 | 说明 |
|------|------|
| 切换系统时 AI 必须在**新标签页**打开目标系统 | 保持原系统页面状态不丢失 |
| 每次切换到新系统时，先确认登录态 | 如果未登录，使用系统注册表中的凭证登录 |
| 系统切换后增加稳定等待 | 等待目标系统页面完全加载后再执行操作 |
| 跨系统验证允许数据同步延迟 | 默认等待 5 秒后验证；首次验证失败则刷新页面重试一次 |

#### G.1.4 跨系统数据一致性验证

跨系统测试中，核心验证目标是**数据在系统间的一致性**：

```markdown
**预期结果**：
5. [SYS-A] 订单状态显示"已提交"，订单号为 {记录为:订单号}（L1）
7. [SYS-B] 订单列表中出现订单号为 {引用:订单号} 的记录，状态为"待确认"（L1）
```

**变量传递语法**：
| 标记 | 含义 | 示例 |
|------|------|------|
| `{记录为:变量名}` | 记录当前步骤的某个值，供后续步骤引用 | 订单号为 `{记录为:订单号}`（保存该值） |
| `{引用:变量名}` | 引用前面步骤记录的值 | 显示订单号 `{引用:订单号}`（与前面记录的一致） |

### G.2 用例间依赖关系声明

#### G.2.1 依赖声明语法

当用例之间存在数据依赖（无法通过独立的前置条件准备）时，必须显式声明：

```markdown
### PRJ-CERT-003 编辑已提交的企业认证信息

**优先级**：P1
**用例类型**：功能测试
**所属模块**：企业认证模块
**依赖用例**：PRJ-CERT-001（需先执行且通过，产出数据：一条状态为"待审核"的认证记录）

**前置条件**：
1. [登录态] 使用账号 admin/Test1234 登录管理端
2. [数据存在][数据准备: 依赖用例 PRJ-CERT-001] 系统中存在一条认证记录（由 PRJ-CERT-001 创建）
```

#### G.2.2 依赖关系规则

| 规则 | 说明 |
|------|------|
| **显式声明** | 依赖关系必须在用例头部 `**依赖用例**` 字段声明，不能隐含 |
| **单向依赖** | 禁止循环依赖（A 依赖 B，B 又依赖 A） |
| **最小依赖** | 每条用例最多依赖 2 条用例；超过说明用例拆分不合理 |
| **依赖失败处理** | 依赖的用例失败 → 本用例标记为"跳过（依赖未满足: PRJ-XXX-NNN 失败）"，而非"失败" |
| **依赖数据传递** | 依赖用例产出的数据（如记录 ID、名称）通过 `{引用:变量名}` 传递到后续用例 |

#### G.2.3 依赖关系可视化

同模块内有依赖关系的用例，在模块文档头部用依赖图表达：

```markdown
## 用例依赖关系

PRJ-CERT-001 创建认证
  └→ PRJ-CERT-003 编辑认证
  └→ PRJ-CERT-004 撤回认证
       └→ PRJ-CERT-005 重新提交认证
PRJ-CERT-002 提交无效信用代码（独立，无依赖）
```

#### G.3.4 回归测试套件选取策略

当需求变更或 Bug 修复后，从完整套件中选取回归用例的规则：

| 选取范围 | 说明 |
|---------|------|
| **直接关联用例** | 变更功能的所有用例（必选） |
| **上游依赖用例** | 依赖该功能数据的用例（必选） |

#### G.3.5 线上回归用例适配规则

> **目的**：发版后需要在生产环境执行回归验证，但生产环境有数据安全和业务影响约束。本节规范如何将已有测试用例适配为可在生产环境安全执行的线上回归用例。

##### G.3.5.1 执行环境标签

每条测试用例必须标注执行环境标签，标明该用例可在哪个环境执行：

| 标签 | 含义 | 适用场景 |
|------|------|---------|
| `[ENV: TEST]` | 仅限测试环境 | 涉及不可逆写操作、大量数据修改、资金/交易操作 |
| `[ENV: PROD]` | 可在生产环境执行 | 纯查询/展示验证，或满足安全适配条件的写操作 |
| `[ENV: BOTH]` | 测试和生产环境均可 | 无 CUD 操作的只读验证用例 |

**标签位置**：写在用例模板的 `**用例类型**` 字段之后：

```markdown
**用例类型**：功能测试
**执行环境**：[ENV: PROD]
```

##### G.3.5.2 安全适配决策树

为已有用例标注 ENV 标签时，按以下决策树逐条判断：

```
用例是否包含 CUD（Create/Update/Delete）操作？
│
├── 否（纯查询/展示/导航）
│   └── 标记 [ENV: BOTH]
│
└── 是
    ├── CUD 操作是否同时满足以下三个条件？
    │   ① 操作可幂等（重复执行结果一致）
    │   ② 用例包含回滚/清理步骤
    │   ③ 使用测试标记数据（如 _AITEST_ 后缀）
    │
    ├── 三个条件全部满足
    │   └── 标记 [ENV: PROD]，用例末尾追加回滚步骤
    │
    └── 任一条件不满足
        └── 标记 [ENV: TEST]
```

##### G.3.5.3 禁止标记 [ENV: PROD] 的场景

以下场景**绝对禁止**在生产环境执行，无论是否满足安全适配条件：

| # | 禁止场景 | 原因 |
|---|---------|------|
| 1 | 修改/删除真实用户数据 | 影响线上用户 |
| 2 | 资金操作（充值、扣费、退款、转账） | 财务风险 |
| 3 | 不可逆状态变更（审核通过、合同签署、订单完结） | 无法回滚 |
| 4 | 系统配置变更（开关、权限、策略） | 影响所有用户 |
| 5 | 批量数据操作（批量导入、批量删除） | 影响范围不可控 |

##### G.3.5.5 PROD 用例编号规范

线上验证专用用例使用独立编号体系，与标准用例编号区分：

```
PROD-{MODULE}-{SEQ}

示例:
- PROD-CERT-001 验证企业认证页面信息展示正确
- PROD-TEAM-001 验证团队成员列表加载正常
- PROD-CREDIT-001 验证积分余额显示正确
```

**编号规则**：
- 前缀固定为 `PROD`，表示该用例专为生产环境线上验证设计
- `{MODULE}` 复用标准用例的模块前缀（见附录 A 模块前缀对照表）
- `{SEQ}` 在 PROD 命名空间内独立编号，从 001 开始
