# 技术文档测试审阅规则

> **版本**: v2.3 | **定位**: 定义审阅检查标准和判定规则，供 tech-doc-review skill 执行审阅时引用

## 0. 核心定位

### 0.1 审阅目标

技术文档测试审阅服务于一个核心目标：

> **确保 AI 能基于技术文档正确编写测试用例并成功执行，同时不遗漏技术方案中"AI 自身无法发现"的风险。**

这个目标分解为两个层面：

1. **AI 可测试性审阅**（主体）：技术文档是否为 AI 编写用例和执行测试提供了充分、无歧义的信息？
2. **风险场景识别**（补充）：技术方案中是否存在 AI 测试无法自动发现、需要人工风险意识补充的问题？

### 0.2 设计原则

当测试用例编写和执行均由 AI 完成时，审阅的一切维度都应服务于 AI 测试链路：

| 传统测试思维的价值 | 在 AI 全程测试中的定位 | 处理方式 |
|------------------|---------------------|---------|
| **分析思维**（风险识别、边界意识） | 仍然必要 — AI 不擅长"想到没写的场景" | 保留，产出转化为 AI 测试场景 |
| **需求一致性校验** | 仍然必要 — 技术设计偏离 PRD 会导致 AI 按错误逻辑设计用例 | 保留，作为审阅前置步骤 |
| **独立的安全/性能审查** | 不必要 — AI 浏览器测试无法验证后端安全、数据库索引等 | 精简为风险登记 |

### 0.3 与 PRD 审阅的关系

PRD 审阅关注"需求是否完整、逻辑是否自洽"；技术文档审阅关注"技术方案是否正确实现需求、是否可被 AI 测试验证"。

## 1. 需求一致性与风险场景检查标准（维度代码：C）

> **产出原则**：本维度发现的每一个问题，最终产出必须是以下之一：
> - 阻塞项 → 要求开发修正技术文档
> - 风险场景（AI标准覆盖）→ 仅标记确认
> - 风险场景（AI标准不覆盖）→ 产出补充 AI 测试场景建议

### 1.1 需求-技术一致性检查项

| 检查项编号 | 检查项 | 审查要点 | 严重级别 |
|-----------|-------|---------|---------|
| C-1 | 功能点覆盖 | PRD 中的每个功能点是否在技术文档中有对应设计 | B（阻塞） |
| C-2 | 业务规则一致 | 技术文档中的业务逻辑是否与 PRD 定义一致（排序规则、状态流转、计算公式等） | B（阻塞） |
| C-3 | 状态机一致 | 技术文档中的状态流转图是否与 PRD 状态机完全匹配 | B（阻塞） |
| C-4 | 枚举值一致 | 下拉选项、状态值等枚举是否与 PRD 定义的完整列表一致 | S（补充） |
| C-5 | 文案一致 | 技术文档中的提示文案、按钮名称是否与 PRD/原型一致 | S（补充） |
| C-6 | 超范围设计 | 技术文档是否引入了 PRD 未提及的额外逻辑或功能 | Q（疑问） |

### 1.2 风险场景检查项与产出分类

> **关键区分**：审阅阶段对 6 类风险**全部检查**，但产出方式根据 `ai_test_case_design.md` 是否已有标准覆盖而不同。

| 检查项编号 | 检查项 | 审查要点 | AI标准覆盖状态 | 产出方式 |
|-----------|-------|---------|--------------|---------|
| C-7 | 并发竞态 | 多用户同时操作同一数据时的问题 | **未覆盖** | 产出补充场景 |
| C-8 | 异步失败可观测性 | 异步操作失败时页面是否有可观测反馈 | **视PRD而定** | 按判定规则 |
| C-9 | 操作不可逆风险 | 删除、状态变更等不可逆操作的确认机制 | **已覆盖**（§3.2 #1） | 仅标记确认 |
| C-10 | 数据依赖完整性 | 操作 A 的数据被操作 B 依赖时，修改/删除是否导致异常 | **未覆盖** | 产出补充场景 |
| C-11 | 跨系统数据不一致 | 上游系统数据变更后，下游系统展示是否不一致 | **视PRD而定** | 按判定规则 |
| C-12 | 权限边界 | 无权限用户尝试操作时的页面表现 | **已覆盖**（§3.2 #5） | 仅标记确认 |

### 1.3 去重判定规则

```
对每个识别出的风险场景，按以下决策树判定产出方式：

┌─ 该风险是否属于 ai_test_case_design.md 标准覆盖范围？
│
├─ 是（C-9, C-12）
│   └─ 产出方式：仅标记确认
│      格式："确认：{风险类型}已在技术文档中定义，AI用例设计阶段按 §X.X 自动覆盖"
│      ⚠ 不产出补充场景建议
│
├─ 视PRD而定（C-8, C-11）
│   ├─ PRD 中有明确定义该场景 → 同"已覆盖"，仅标记确认
│   └─ PRD 中未定义该场景 → 同"未覆盖"，产出补充场景
│
└─ 否（C-7, C-10）
    └─ 产出方式：产出补充AI测试场景建议
       必须包含：风险描述、AI可测性评估、建议的验证策略
```

### 1.4 AI 不可测风险登记

以下风险 AI 浏览器测试**无法验证**，在审阅报告中登记即可，不展开为检查清单：

| 风险类型 | 为什么 AI 不可测 | 处理方式 |
|---------|----------------|---------|
| 后端安全（SQL注入、XSS防护） | 需要专业安全扫描工具 | 登记到"需专项测试"清单 |
| 数据库索引/查询性能 | 需要性能测试工具 | 登记到"需专项测试"清单 |
| 消息队列积压/消费速率 | 需要监控平台和压测工具 | 登记到"需专项测试"清单 |
| 数据库事务完整性 | 非 UI 可观测 | 登记到"需专项测试"清单 |

## 2. AI 可测试性审阅维度

> **核心原则**：检查技术文档对 AI 测试的信息支撑度——AI 能否基于此文档正确编写测试用例并成功执行。

### 2.1 UI 元素文案精确性（维度代码：U）

> **关联**：`ai_test_case_design.md` 附录 F.3（元素描述规范）、附录 D（预期结果反模式）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| U-1 | 按钮文案 | 每个操作按钮的精确文案是否定义 |
| U-2 | Tab/标签文案 | 页面标签的精确名称是否列出 |
| U-3 | Toast提示文案 | 每个操作的成功/失败提示文案是否精确到具体用词 |
| U-4 | 弹窗文案 | 确认弹窗的标题、正文内容、按钮文字是否完整定义 |
| U-5 | 空状态文案 | 列表无数据/搜索无结果时的展示文案是否定义 |
| U-6 | 错误提示文案 | 表单校验失败、接口报错时的错误提示文案是否逐一定义 |
| U-7 | 标签/状态展示 | 状态值在页面上的展示形态（文本 + 颜色 + 位置）是否明确 |
| U-8 | 文案差异标注 | 前端实现的文案若与 PRD/原型存在差异（空格、标点、全角/半角等），需在技术文档中明确标注实际渲染值 |

**快速检查**：AI 写用例时能否写出 `点击「XXX」按钮` 和 `toast 提示{包含:XXX}`？

**U-8 补充说明**：AI 直接执行测试时通过视觉/语义定位元素，文案精确性直接影响执行成功率。若 PRD 写「确认」但前端实际渲染「确 认」（含空格），AI 将无法定位该按钮。

### 2.2 表单字段约束完整性（维度代码：F）

> **关联**：`ai_test_case_design.md` §3.2（必须覆盖的场景类型 - 边界值/数据验证）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| F-1 | 字段类型 | 每个表单字段的输入类型是否明确（文本/数字/日期/下拉/上传等） |
| F-2 | 必填/选填 | 每个字段的必填性是否标注 |
| F-3 | 长度约束 | 文本字段的最小/最大字符数是否定义 |
| F-4 | 字符计数规则 | 中文/英文/特殊字符的计数方式是否明确 |
| F-5 | 格式约束 | 特定格式要求是否定义（如日期格式、编号格式、正则规则） |
| F-6 | 取值范围 | 数值字段的取值范围、日期字段的可选范围是否定义 |
| F-7 | 下拉选项枚举 | 下拉框的完整选项列表是否列出 |
| F-8 | 字段联动规则 | 字段之间的联动逻辑是否描述清楚 |
| F-9 | 默认值 | 字段是否有默认值，默认值是什么 |

**快速检查**：对每个字段能否回答 `类型？必填？最小长度？最大长度？格式？取值范围？默认值？`

### 2.3 状态流转的可观测性（维度代码：O）

> **关联**：`ai_test_case_design.md` 附录 D（预期结果反模式）、附录 C（验证层级）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| O-1 | 状态展示形态 | 每个状态值在页面上的展示形态是否明确定义（文本 + 颜色 + 图标） |
| O-2 | 状态变更可观测 | 操作后状态变更通过页面哪个元素可以观测到 |
| O-3 | 异步完成标识 | 异步流程完成后页面上的可观测变化是什么 |
| O-4 | 中间态表现 | loading、处理中等中间态的 UI 表现是否有描述 |
| O-5 | 刷新后状态保持 | 页面刷新/返回后状态是否保持的规则 |

**快速检查**：AI 能否写出具体的预期结果（如 `状态列显示"创建中"橙色标签`），而非模糊的 `页面正常显示`？

### 2.4 测试数据准备支撑（维度代码：D）

> **关联**：`ai_test_case_design.md` 附录 E.3（数据准备策略）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| D-1 | CRUD API 定义 | 创建/查询/更新/删除的 API 接口是否完整定义 |
| D-2 | 中间状态构造 | 是否有方法直接构造中间状态数据，而非必须走完整流程 |
| D-3 | 认证方式 | API 调用所需的认证方式是否明确 |
| D-4 | 数据清理接口 | 是否有删除/清理测试数据的 API 接口 |
| D-5 | 批量数据构造 | 需要多条数据的场景是否有批量创建方式 |
| D-6 | 唯一性约束 | 有唯一性约束的字段的生成规则，是否支持可重复创建 |

**快速检查**：用例能否标注 `[数据准备: API]` 高效构造数据，还是只能 `[数据准备: UI自建]` 走多步 UI 流程？

### 2.5 异步流程的等待策略信息（维度代码：A）

> **关联**：`ai_test_case_design.md` §7.2（AI 执行测试的步骤）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| A-1 | 预期处理时长 | 异步操作的预期完成时间是否明确 |
| A-2 | 完成标识 | 异步完成后页面上哪个元素发生变化，AI 如何判断已完成 |
| A-3 | 状态轮询接口 | 是否有 API 可查询异步任务状态 |
| A-4 | 超时阈值 | 超过多长时间可判定为异步失败，超时后页面 UI 表现 |
| A-5 | 重试机制 | 异步失败后是否自动重试，最多重试几次 |

### 2.6 data-testid 覆盖度（维度代码：T）【可选】

> **定位说明**：在 AI 直接执行模式下（非自动化脚本），AI 通过视觉/语义定位元素，data-testid 非必须。此维度降级为**可选维度**，仅在需要生成自动化脚本时作为必审维度。
>
> **关联**：`ai_test_case_design.md` 附录 F.3（元素描述规范）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| T-1 | 交互元素覆盖 | 所有需要交互的元素是否都有 data-testid |
| T-2 | 动态元素覆盖 | 列表中每行的操作按钮等动态生成元素是否有 data-testid 方案 |
| T-3 | 命名规范一致 | data-testid 命名是否遵循统一规范 |
| T-4 | 埋点状态 | data-testid 的开发埋点进度是否有跟踪 |

### 2.7 排序/筛选规则的精确性（维度代码：R）

> **关联**：`ai_test_case_design.md` §4.5（检索结果数据一致性验证）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| R-1 | 排序维度完整 | 多维排序的每一级排序字段和方向是否完整列出 |
| R-2 | 优先级枚举值 | 排序优先级涉及的枚举值是否有完整映射表 |
| R-3 | 同级排序规则 | 排序值相同时的次级排序规则是否明确 |
| R-4 | 筛选逻辑 | 搜索条件之间是 AND 还是 OR 关系、模糊/精确匹配方式是否明确 |
| R-5 | 分页规格 | 默认每页条数、可选每页条数、总数统计口径是否定义 |

### 2.8 跨系统测试的可执行性（维度代码：K）

> **关联**：`ai_test_case_design.md` 附录 G（跨系统测试规范）

| 检查项编号 | 检查项 | 审查要点 |
|-----------|-------|---------|
| K-1 | 系统清单 | 所有涉及的系统名称和测试环境 URL 是否列出 |
| K-2 | 数据同步时效 | 系统间数据同步的延迟时间是否明确 |
| K-3 | 数据映射关系 | 上游系统的数据字段如何映射到下游系统的字段 |
| K-4 | 同步触发条件 | 数据同步是自动触发还是需要手动操作 |

## 3. 问题分级标准

### 3.1 严重级别定义

| 级别 | 代码 | 含义 | 对 AI 测试链路的影响 | 处理要求 |
|------|------|------|-------------------|---------|
| **阻塞** | B | 技术文档存在设计缺陷或重大遗漏 | AI 用例设计被阻塞 | 必须在测试设计前解决 |
| **补充** | S | 技术文档信息不完整 | AI 用例质量降低 | 建议补充，可并行处理 |
| **疑问** | Q | 技术文档描述不清晰或存在歧义 | AI 用例可能偏差 | 需要开发回复确认 |
| **风险** | R | 技术方案存在潜在风险场景 | 需要识别并分类处理 | 按 §1.3 去重判定规则处理 |
| **建议** | A | 技术文档改进建议 | 影响 AI 执行效率 | 可选改进 |

### 3.2 各维度典型问题级别参考

| 维度 | 典型阻塞（B） | 典型补充（S） | 典型风险-确认（R） | 典型风险-补充（R） |
|------|-------------|-------------|------------------|------------------|
| C | 状态流转与PRD矛盾 | 枚举值未列完整 | 删除确认弹窗 | 并发审核竞态 |
| U | 核心按钮文案缺失 | Toast提示文案未精确 | - | - |
| F | 字段无任何约束定义 | 字符计数规则缺失 | - | - |
| O | 状态变更无可观测标识 | 中间态UI表现缺失 | - | 异步超时验证 |
| D | 无任何API接口定义 | 中间状态构造方法缺失 | - | - |
| A | 异步操作无超时定义 | 状态轮询接口未提供 | - | 重试耗尽验证 |

## 附录 B：技术文档类型识别指南

AI 在审阅前需先识别技术文档类型，以确定审阅维度的选择。

| 文档特征 | 判定为 | 必审维度 | 重点维度 | 可选维度 |
|---------|--------|---------|---------|---------|
| 数据库表结构、ER图、SQL DDL | 数据库设计文档 | C | F | - |
| API 路径、请求/响应 JSON | 接口定义文档 | C, D | F, A, R | - |
| 页面布局、组件交互、前端路由 | 前端用户故事/开发规格 | C, U, O | F, R | T |
| data-testid 映射表 | data-testid 映射文档 | U | - | T |
| 系统架构图、状态机、时序图 | 后端技术设计文档 | C | D, A, K | - |
| 包含以上多种内容 | 综合技术方案 | 全部维度 | 根据内容侧重 | T |
