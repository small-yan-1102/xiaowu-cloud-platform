# 冒烟测试专项设计规则

> **版本**: v2.0（AI直接执行版）  
> **创建日期**: 2026-03-11  
> **更新日期**: 2026-03-24  
> 本规则是 `ai_test_case_design.md` 第 6 节（测试分层 L5）的专项细化，  
> 用于指导冒烟测试用例的选取、设计与维护。  
> 适用模式：AI 读取冒烟测试用例 → 直接操作浏览器执行 → 自主判断结果

---

## 1. 冒烟测试定义与定位

### 1.1 定义

冒烟测试（Smoke Test）是对系统核心功能的**快速健康检查**，目的是在每次部署或构建后，以最短时间验证系统是否处于**基本可用**状态。

### 1.2 在测试金字塔中的位置

```
        /\
       /  \  L5 冒烟测试 ← 本规则覆盖范围
      /----\
     / L4   \    E2E测试
    /--------\
   /   L3     \  集成测试
  /------------\
 /     L2       \ API测试
/----------------\
/       L1         \ 单元测试
/--------------------\
```

| 属性 | 冒烟测试（L5） | E2E测试（L4） |
|------|---------------|--------------|
| 目标 | 验证系统基本可用，核心链路畅通 | 验证完整用户流程的正确性 |
| 覆盖范围 | 仅 Happy Path，核心价值链 | 包含正常/异常/边界场景 |
| 单条用例时长 | **< 2 分钟** | 可达 5-10 分钟 |
| 套件总时长 | **< 5 分钟** | 可达 30-60 分钟 |
| 执行频率 | 每次部署前 | 每日 / 每周 |
| 失败影响 | **阻塞部署** | 记录缺陷，不阻塞 |
| 优先级映射 | P0 | P1 |

### 1.3 核心约束

| 约束 | 值 | 说明 |
|------|----|------|
| 单条用例执行时间 | **< 2 分钟** | 超过 2 分钟需拆分或降级为 E2E |
| 冒烟套件总执行时间 | **< 5 分钟** | 超过 5 分钟需裁剪用例数量 |
| 用例数量上限 | **每模块 3-5 条** | 宁缺勿滥，只保留核心链路 |
| 验证密度 | **L1 强验证为主** | 冒烟测试不做细节校验 |

## 2. 冒烟用例选取标准

### 2.1 必须纳入冒烟的场景（Include）

| 类别 | 判定标准 | 示例 |
|------|---------|-----------|
| **核心价值链** | 系统最核心的业务流程，缺失则系统无使用价值 | 登录 → 企业认证 → 核心功能访问 |
| **资金/交易操作** | 涉及金额变动、充值、支付等不可逆操作 | 积分充值申请 → 审核通过 → 积分到账 |
| **认证/状态变更** | 用户身份或系统状态的关键转换 | 未认证 → 提交认证 → 认证通过 |
| **跨模块数据流** | 模块 A 的操作直接影响模块 B 的数据 | 认证通过 → 积分赠送 → 功能菜单解锁 |
| **用户准入流程** | 新用户从注册到可正常使用的完整路径 | 注册 → 登录 → 首次引导 → 认证 |

### 2.2 不应纳入冒烟的场景（Exclude）

| 类别 | 原因 | 应归入的层级 |
|------|------|------------|
| 单字段格式校验 | 粒度过细，不影响系统可用性 | E2E / P1 |
| 边界值测试 | 属于质量深度验证 | E2E / P2 |
| UI 样式/布局验证 | 不影响功能可用性 | E2E / P2 |
| 性能基准测试 | 执行时间长，不适合快速验证 | 独立性能测试 |
| 需要人工介入的流程 | 无法快速自动完成 | 标注人工介入 |
| 异常/容错场景 | 冒烟只走 Happy Path | E2E / P2 |
| 权限矩阵验证 | 组合场景多，耗时长 | E2E / P1 |

### 2.3 选取决策树

```
该功能是否属于系统核心价值链？
│
├─ 否 → 不纳入冒烟
│
└─ 是 → 该功能不可用时，系统是否完全不可使用？
      │
      ├─ 是 → 必须纳入冒烟（P0）
      │
      └─ 否 → 该功能涉及资金/状态不可逆变更？
            │
            ├─ 是 → 纳入冒烟（P0）
            │
            └─ 否 → 该功能是跨模块数据流的入口？
                  │
                  ├─ 是 → 纳入冒烟（P0）
                  │
                  └─ 否 → 降级为 E2E（P1）
```

## 3. 冒烟用例编号规范

### 3.1 编号格式

```
SMOKE-{MODULE}-{SEQ}     模块级冒烟用例
SMOKE-E2E-{SEQ}          跨模块端到端冒烟用例
```

| 字段 | 说明 | 示例 |
|------|------|------|
| `SMOKE` | 固定前缀，标识冒烟测试 | - |
| `MODULE` | 模块缩写，与 PRJ 模块前缀一致 | CERT / TEAM / CREDIT |
| `E2E` | 跨模块用例专用标识 | SMOKE-E2E-001 |
| `SEQ` | 三位数字序号 | 001, 002, 003 |

### 3.2 冒烟用例编号示例

| 编号 | 用例名称 | 所属模块 |
|------|---------|---------|
| SMOKE-CERT-001 | 登录后完成企业认证全流程 | 企业认证 |
| SMOKE-CERT-002 | 认证通过后验证状态变更与菜单解锁 | 企业认证 |
| SMOKE-CREDIT-001 | 积分充值申请提交 | 积分中心 |
| SMOKE-TEAM-001 | 团队成员邀请与角色分配 | 团队管理 |
| SMOKE-E2E-001 | 注册→认证→积分赠送→功能访问 | 跨模块 |

### 3.3 与 PRJ 编号的关系

- 冒烟用例使用独立的 `SMOKE-` 前缀，**不占用** `PRJ-` 编号序列
- 一个 `SMOKE-` 用例可以对应一个或多个 `PRJ-` 用例的核心步骤
- 在冒烟用例的 **备注** 字段中标注关联的 `PRJ-` 编号

## 4. 冒烟用例设计原则

### 4.1 Happy Path Only（仅正常路径）

冒烟测试**只走正常流程**，不测试异常分支：

- ✅ 冒烟：登录 → 填写正确表单 → 提交 → 验证成功
- ❌ 不属于冒烟：输入无效信用代码 → 提交 → 验证报错

### 4.2 状态变更优于 UI 细节

冒烟测试关注**业务状态是否正确变更**，而非 UI 展示细节：

| 验证目标 | 冒烟测试（推荐） | E2E测试（详细） |
|---------|----------------|----------------|
| 认证结果 | 侧边栏状态文字变为"企业认证已通过" | 状态文字 + 颜色 + 图标 + 详情弹窗内容 |
| 充值结果 | 积分余额数值变化 | 余额 + 明细记录 + 交易流水号 + 时间戳 |
| 登录结果 | 跳转到首页 + 用户名显示 | URL + 用户名 + 头像 + 菜单权限 |

### 4.3 步骤可合并原则

冒烟用例中，**连续的同类操作可合并描述**，减少步骤数：

- ✅ 冒烟："填写企业认证表单全部字段（公司名称、信用代码、法人姓名、身份证号、法人手机）"
- ❌ 过细："步骤3填写公司名称 → 步骤4验证已填 → 步骤5填写信用代码 → 步骤6验证已填 → ..."

### 4.4 容错处理

AI 执行冒烟测试时应具备适度的容错能力，应对环境不稳定因素：

| 策略 | 说明 | 示例 |
|------|------|------|
| 弹窗自动关闭 | 首次登录的引导弹窗等 | 遇到意外弹窗时关闭后继续 |
| 页面刷新重试 | 状态未及时同步时刷新页面 | 刷新后重新检查状态 |
| 等待页面稳定 | 操作后等待页面加载完成 | 等待加载指示器消失 |

## 5. 冒烟用例验证层级

### 5.1 验证分布要求

冒烟测试的验证应以 **L1 强验证为主**，辅以少量 L2 软验证：

| 验证级别 | 在冒烟测试中的占比 | 说明 |
|---------|-------------------|------|
| **L1 强验证** | **60-70%** | 核心业务结果，不通过即用例失败 |
| **L2 软验证** | **20-30%** | 辅助验证，不通过记录警告但不阻断 |
| **L3 观察性** | **≤ 10%** | 仅截图记录，不做判断 |

### 5.2 各级别验证示例

| 验证级别 | 示例 | AI 判断方式 |
|---------|------|-----------|
| **L1 强验证** | 提交认证后侧边栏显示"企业认证已通过" | 必须看到该文字，否则标记失败 |
| **L1 强验证** | 登录后跳转到首页 | 必须确认页面 URL 或首页标识，否则标记失败 |
| **L2 软验证** | 认证后积分中心菜单可点击 | 检查菜单状态，不可点击则记录警告 |
| **L3 观察性** | 认证详情弹窗内容 | 截图记录即可 |

### 5.3 冒烟测试禁止的验证方式

| 禁止方式 | 原因 | 替代方案 |
|---------|------|---------|
| 跳过未出现的预期元素 | 导致假通过 | 明确标记为失败 |
| 不等待直接判断 | 页面未加载完就判断 | 等待合理时间后再判断 |
| 纯 UI 样式验证 | 不属于冒烟范畴 | 移至 E2E 用例 |
| 超过 5 个字段的逐字段校验 | 粒度过细 | 合并为状态验证 |

## 6. 冒烟失败阻塞策略

### 6.1 阻塞模式定义

| 模式 | 触发条件 | 动作 |
|------|---------|------|
| **Blocking（阻塞）** | 任一冒烟用例失败 | 阻止部署，通知相关人员 |
| **Reporting（报告）** | 冒烟全部通过 | 生成报告，继续部署 |

### 6.2 失败报告规范

冒烟测试失败时，报告内容应包含：

| 字段 | 说明 | 示例 |
|------|------|------|
| 失败用例编号 | SMOKE 编号 | SMOKE-CERT-001 |
| 失败步骤 | 具体哪一步失败 | 步骤22: 验证认证状态变更 |
| 错误信息 | 实际看到什么 vs 期望什么 | 期望"企业认证已通过"，实际未找到 |
| 环境信息 | 执行环境 | staging / 浏览器 chromium |
| 截图 | 失败时的页面截图 | 附件 |

## 7. 冒烟用例维护规则

### 7.1 新增模块同步

**强制规则**：新增业务模块时，必须同步评估是否需要新增冒烟用例。

| 触发事件 | 必须执行的动作 |
|---------|--------------|
| 新增业务模块 | 评估核心链路，新增 SMOKE-{MODULE}-001 |
| 核心流程变更 | 同步更新对应冒烟用例的步骤和预期结果 |
| 页面改版 | 更新用例中涉及的操作描述 |

### 7.2 定期审查机制

| 审查周期 | 审查内容 |
|---------|---------|
| **每月** | 检查冒烟用例是否与当前业务流程一致 |
| **每季度** | 全面评估冒烟套件：用例增减、执行时间 |
| **版本发布前** | 确认冒烟用例覆盖新版本核心功能 |

### 7.3 执行顺序建议

冒烟测试应按**业务依赖关系**排序执行：

```
1. 登录冒烟（前置依赖）
2. 企业认证冒烟（依赖登录）
3. 积分中心冒烟（依赖认证）
4. 团队管理冒烟（依赖认证）
5. 跨模块 E2E 冒烟（依赖以上全部）
```

## 8. 冒烟用例 Markdown 模板

### 8.1 标准模板

```markdown
### SMOKE-{MODULE}-{SEQ} {用例名称}

**优先级**: P0  
**用例类型**: 冒烟测试  
**所属模块**: {模块名称}  
**关联需求**: PRD V5.1 第X.X节  
**关联PRJ用例**: PRJ-{MODULE}-001, PRJ-{MODULE}-002, ...  
**预计执行时间**: < 2 分钟

**前置条件**:
- 条件1
- 条件2

**测试数据**:
| 字段 | 值 | 说明 |
|------|-----|------|
| 字段1 | 值1 | 说明1 |

**测试步骤与预期结果**:

| 阶段 | 步骤 | 操作 | 预期结果 | 验证级别 |
|------|------|------|---------|---------|
| 阶段一：XXX | 1 | 操作描述 | 期望结果 | L1 |
| | 2 | 操作描述 | 期望结果 | L1 |
| 阶段二：XXX | 3 | 操作描述 | 期望结果 | L1 |
| | 4 | 操作描述 | 期望结果 | L2 |

**预期结果汇总**:
- L1 强验证:
  1. 结果1
  2. 结果2
- L2 软验证:
  1. 结果1
- L3 观察性:
  1. 截图记录项

**是否需要人工介入**: 否  
**备注**: 关联 PRJ 用例编号
```

### 8.2 模板使用说明

| 字段 | 填写要求 |
|------|---------|
| 用例编号 | 必须使用 `SMOKE-` 前缀 |
| 优先级 | 固定为 **P0** |
| 用例类型 | 固定为 **冒烟测试** |
| 关联PRJ用例 | 列出该冒烟用例对应的详细 PRJ 用例编号 |
| 预计执行时间 | 必须 < 2 分钟 |
| 验证级别 | 每个预期结果必须标注 L1/L2/L3 |
| 阶段划分 | 按业务流程阶段分组 |

## 附录：冒烟测试 Quick Reference

### 速查：该用例是否属于冒烟？

```
1. 属于核心价值链？          → 否 → 不纳入
2. 只走 Happy Path？         → 否 → 不纳入（需拆分正常路径部分）
3. 单条 < 2 分钟？           → 否 → 不纳入（需精简步骤）
4. 涉及状态变更或资金操作？    → 是 → 纳入
5. 验证以 L1 强验证为主？     → 否 → 调整验证层级
```

### 速查：冒烟测试设计检查清单

```
☐  编号格式为 SMOKE-{MODULE}-{SEQ}
☐  优先级标记为 P0
☐  仅包含 Happy Path，无异常分支
☐  验证以 L1 强验证为主（60-70%）
☐  单条用例执行时间 < 2 分钟
☐  包含反向错误检查（确认无错误提示）
☐  容错策略考虑完善（弹窗/刷新/等待）
☐  关联的 PRJ 用例编号已标注
☐  步骤按阶段合理划分
```
