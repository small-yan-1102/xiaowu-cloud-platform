# 技术优化 AI 测试规则体系 - 使用说明

> 本文档说明规则体系的用途、适用场景、文件构成和使用方法。

---

## 这套规则解决什么问题

开发提交技术优化项目时，通常只有一份技术设计文档，没有产品需求文档。测试工程师面临三个困难：

1. **不知道测什么** — 设计文档写的是"怎么改"，没有明确"改了之后行为变成什么样"
2. **不知道怎么借助 AI** — AI 需要结构化的输入才能生成有效的测试辅助产物
3. **没有统一标准** — 每个项目靠个人经验判断，质量不稳定

这套规则定义了一个标准化流程：从开发交付物的准入检查，到测试工程师的产出物规范，再到 AI 直接执行测试的工作模式，形成完整闭环。

---

## 适用场景

| 适用 | 不适用 |
|------|--------|
| 开发发起的技术优化/重构 | 有完整需求文档的功能开发 |
| 无产品需求文档 | 纯代码重构、无行为变更 |
| 涉及数据模型、流程、定时任务、性能等变更 | 紧急线上 hotfix |

## 文件构成

```
.claude/
├── rules/                                        ← 规则、模板、说明
│   ├── tech-optimization-ai-testing-rules.md     ← 规则（本体）
│   ├── tech-optimization-ai-testing-guide.md     ← 使用说明（本文档）
│   └── templates/                                ← 可复制的模板
│       ├── tech-optimization-ai-test-skill-template.md   ← AI Skill 模板
│       └── tech-optimization-test-cases-template.md      ← 测试用例模板
│
└── skills/                                       ← Skill 实例 + 项目文件
    ├── {项目名}-test-ai-skill.md                  ← 项目 Skill 实例（从模板实例化）
    ├── {项目名}-testing-supplement.md              ← 测试补充文档
    ├── {项目名}-test-cases.md                      ← 测试用例
    └── {项目名}/                                   ← 或按目录组织项目文件
```

### 各文件的角色

| 文件 | 角色 | 使用者 | 一句话说明 |
|------|------|--------|-----------|
| **rules（规则）** | 制度 | 测试工程师 + 开发 | 定义"谁提供什么、达到什么标准才能开始测试" |
| **skill-template（Skill 模板）** | 工具说明书 | AI | 定义"AI 在测试中能做什么、怎么做、输出什么格式" |
| **test-cases-template（用例模板）** | 工作表格 | 测试工程师 | 定义"测试用例按什么结构组织、每条用例包含什么字段" |

三者的关系：

```
规则 定义了"需要哪些输入"
  → 开发按规则提供 设计文档 + 变更确认表 + 补充文档 + 源代码
     → AI 自动克隆仓库到本地，按 用例模板 编写用例
        → AI 读取文档 + 源码，按 Skill 模板 执行测试
```

## 使用方法

### 第一步：收到开发的技术设计文档

拿到设计文档后，按**规则第四节（准入检查清单）**逐项检查：

```
必须项（缺任何一项则打回）:
  □ 技术设计文档已提交，包含全部必含章节
  □ 变更确认表已提交，开发已确认，且逐条覆盖设计文档影响分析中的全部条目
  □ 测试补充文档 P0 项已全部填写（验证手段已按变更确认表逐项提供）
  □ 源代码已提交至 Codeup 代码仓库，仓库克隆地址和功能分支名称已提供
  □ 部署清单已提交（服务名称 + 版本号，如有部署顺序或配置变更需一并说明）
  □ 测试环境已按部署清单完成部署，基本功能可访问
  □ 数据库变更脚本已提供（如涉及）
  □ 新增的异步任务/定时任务已在测试环境可触发（如涉及）
```

通常第一次检查大部分项目都不达标。此时进入第二步。

### 第二步：生成项目专属的测试补充文档

根据**规则第二节第③项（测试补充文档）**的章节要求，结合该项目的设计文档内容，生成一份项目专属的补充文档模板：

- 将规则中定义的通用章节，映射为该项目的具体问题
- 用 `<!-- -->` 占位标记开发需要填写的内容
- 已能从设计文档中确认的信息直接预填
- "适用条件"不满足的章节不列入（如无迁移则不要求"存量数据现状"）

将补充文档模板发给开发填写。

### 第三步：准入复检 + 源码准备

开发填写完补充文档 + 提交代码至 Codeup 后，再次按准入检查清单复检。全部通过后，AI 自动完成源码准备：

1. 根据开发提供的仓库克隆地址，将应用仓库克隆到本地工作目录（如已克隆则 `git fetch` 更新）
2. 切换到开发提供的功能分支（如 `feature/xxx`）
3. 执行 `git diff master...feature/xxx` 获取本次改造的完整变更范围

```bash
# 首次克隆（AI 自动执行）
git clone {Codeup仓库克隆地址} {本地目录}
cd {本地目录}
git checkout {功能分支名}

# 已有仓库更新（AI 自动执行）
cd {本地目录}
git fetch origin
git checkout {功能分支名}
git pull origin {功能分支名}

# 获取变更范围（AI 自动执行）
git diff master...{功能分支名} --stat          # 变更文件列表
git diff master...{功能分支名}                  # 完整变更内容
```

源码准备完成后，进入测试阶段。

### 第四步：编写测试用例

复制 `tech-optimization-test-cases-template.md`，替换占位符，按项目实际情况：

- 删除不涉及的阶段（如无迁移删除阶段一）
- 根据变更确认表的"预期变更"项填写阶段二（核心流程验证）
- 根据变更确认表的"不变"项填写阶段四（回归测试）
- 评估是否需要阶段五（并发）和阶段六（性能）

### 第五步：实例化 AI Skill

复制 `tech-optimization-ai-test-skill-template.md`，替换占位符：

- `{项目名称}` → 实际项目名
- `{设计文档}` `{补充文档}` `{测试用例}` → 实际文件名
- `{应用仓库本地路径}` → 第三步克隆的本地仓库路径
- 删除不需要的工作模式（如无迁移删除"迁移风险评估"模式）

### 第六步：AI 直接执行测试

通过 Skill 定义的指令与 AI 交互：

| 测试阶段 | 常用 AI 指令 |
|---------|-------------|
| 准备测试数据 | `构造数据 TC-XXX-01` |
| 执行后验证结果 | `验证SQL TC-XXX-01` |
| 结果不符时排查 | `分析问题 执行XX后状态不对` |
| 验证代码变更范围 | `验证变更` |
| 发现遗漏场景 | `补充用例 XX边界场景` |
| 复杂计算验证 | `算 需求量 参数A=5 参数B=3` |
| 迁移前检查 | `迁移评估` |

---

## 规则维护

- 随着项目实践积累，如发现规则中有遗漏的通用章节或不合理的要求，更新 `tech-optimization-ai-testing-rules.md`
- 模板文件跟随规则同步更新
- 项目专属文件（补充文档、用例、Skill）不反向影响通用规则
