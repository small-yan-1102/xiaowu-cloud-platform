# 信息提取与验证规则

> Reviewer 模式资源。Step 5 识别接口信息前加载，作为判断信息是否可信的唯一依据。

---

## 核心原则

**只记录在 Bug 描述、错误日志、截图中明确存在的信息。**

| ✅ 允许 | ❌ 禁止 |
|--------|--------|
| 在原文中能找到对应文本 | "应该是"、"可能是" |
| 每条信息标注具体来源 | "推断是"、"我觉得" |
| 无法确认时标注"未找到" | 基于功能名推断接口路径 |
| 用户补充时验证来源 | 基于模块名推断微服务 |

---

## 接口信息识别优先级

### P1 — 直接接口路径（直接找到，100% 确认）

满足以下任一条件即可记录：

1. Bug 文字中明确写出的 URL
   ```
   示例："调用 /api/v1/orders/create 接口时出错"
   ```

2. 错误日志中显示的请求 URL
   ```
   示例："POST /api/v1/videos/take-down 400 Bad Request"
   ```

3. 截图中清晰可见的请求信息（网络请求面板 / 接口调试截图）

4. 代码片段或 curl 命令中的 URL
   ```
   示例：curl -X POST /order-service/api/v1/orders/create
   ```

**验证方式**：在原始 Bug 描述中能找到对应文字 / 截图中能看清楚。

---

### P2 — HTTP 方法和请求信息（在描述中明确见到时才记录）

- HTTP 方法：GET / POST / PUT / DELETE / PATCH（在错误日志或截图请求中看到）
- 完整请求示例：`POST /api/v1/users/register Content-Type: application/json`
- 请求参数：Bug 描述中明确列出的参数名或字段名
- 响应状态码：如 400、500 等

---

### P3 — 微服务名称（来自确定来源时才记录）

| ✅ 可记录 | ❌ 不可记录 |
|---------|-----------|
| Bug 标题/描述中明确提到服务名 | 基于 API 路径中的前缀推断 |
| 错误日志中显示的服务标识 | 基于功能模块名猜测 |
| 代码包路径中的服务名（如 `com.xw.order.service`）| |

---

### P4 — 数据库信息（来自确定来源时才记录）

| ✅ 可记录 | ❌ 不可记录 |
|---------|-----------|
| Bug 描述中明确提到的表名 | 基于功能推断可能涉及的表 |
| SQL 错误日志中的表名 | |
| 代码注释/文档中明确提到的表 | |

---

### P5 — 代码位置（来自确定来源时才记录）

| ✅ 可记录 | ❌ 不可记录 |
|---------|-----------|
| 错误堆栈跟踪中的类名和行号 | 基于功能名猜测类名 |
| Bug 描述中明确提到的文件路径 | |
| 代码片段中的类和方法名 | |

---

## 用户补充信息验证规则

用户补充信息时，必须严格判断：

```
信息是否直接来自 Bug 描述、日志或截图中的真实内容？
   ├─ 是 → 接受，标注来源为"用户补充（来自 Bug 描述第X行）"
   └─ 否 → 拒绝，回复：
            "请提供来自 Bug 描述或截图中的真实内容。
             例如：'我在错误日志中看到：POST /api/v1/xxx'
             不接受推断或猜测的信息。"
```

**接受的示例**：
- ✅ "我在 Bug 描述中看到：POST /api/v1/orders/create"
- ✅ "错误堆栈显示：OrderService.java 第 245 行"
- ✅ "SQL 日志中有：Error in table 'orders'"

**拒绝的示例**：
- ❌ "应该是 /api/v1/xxx"
- ❌ "可能涉及 order-service"
- ❌ "推断是在 OrderService.java"
- ❌ "我觉得调用的是创建订单接口"

---

## 识别结果输出格式

每条识别到的信息必须包含：

```
| 属性     | 值         | 来源                    | 确定性       |
|---------|-----------|------------------------|------------|
| API 路径 | /api/v1/xx | Bug 错误日志第 5 行       | ✅ 直接找到  |
| HTTP方法 | POST       | Bug 错误日志中明确写出    | ✅ 直接找到  |
| 类名     | OrderSvc   | Bug 描述中的堆栈跟踪      | ✅ 直接找到  |
```

无法确认的信息统一输出：

```
### 无法确认的信息

以下信息在 Bug 描述中未找到明确证据，不予列出：
- 微服务名称（Bug 描述中未提及）
- 数据库表（无 SQL 日志或相关描述）
```
