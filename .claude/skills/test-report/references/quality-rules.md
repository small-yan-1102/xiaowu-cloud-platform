# 测试质量判定规则（共享权威源）

> **版本**：v1.0
> **引用方**：`test-report` / `release-gate` / `submission-gate` 等所有涉及质量判定的 skill
> **原则**：**唯一事实来源** — 任何一方的判定阈值/公式/边界改动都在本文件发生，不在各 skill 内部重复定义。
> **避免漂移**：`verify_descriptions.py` 会扫各 skill 的判定段，若与本文件不一致 → 报警。

---

## 一、整体质量结论判定

### 阈值表

| 结论 | P0 用例 | Open Fatal 缺陷 | Open Critical 缺陷 | **总通过率** |
|---|---|---|---|---|
| 🟢 **质量良好，可发布** | 全部通过 | 0 | 0 | **≥90%** |
| 🟡 **基本达标，有已知风险** | 全部通过 | 0 | >0 或 | 80-89% |
| 🔴 **未达标，建议修复重测** | 有失败 | >0 | — | <80% |

### 判定规则

- 🟢 **所有条件必须全部满足**
- 🔴 **任一条件触发即判定**
- 🟡 **介于两者之间的情况**（P0 全通过 + Open Fatal = 0，但 Open Critical > 0 或 总通过率 80-89%）

### 判定基准：**总通过率**

> 总通过率 = **总通过数** ÷ (**总执行数** - 暂缓 - 阻塞)
>
> 使用"总通过率"而非"AI 通过率"，因为业务视角关心**所有通过的用例**，不区分执行方式。
> AI / 人工 通过率作为**分析辅助指标**单独展示，不作为门禁。

---

## 二、三元通过率定义

| 指标 | 公式 | 用途 |
|---|---|---|
| **AI 通过率** | AI 通过数 ÷ (AI 执行数 - AI 暂缓 - AI 阻塞) | 反映自动化能力 |
| **人工通过率** | 人工通过数 ÷ (人工执行数 - 人工暂缓 - 人工阻塞) | 反映人工覆盖 |
| **总通过率** ← 门禁用 | 总通过数 ÷ (总执行数 - 暂缓 - 阻塞) | **业务真值** |

### 执行方式判定

- **AI 执行**：用例标注 `[执行方式: AI]`，且有 AI 回写的 `✅/❌/🚫/⏭ AI 日期` 标记
- **人工执行**：用例标注 `[执行方式: 人工]` 或 AI 未触达（由人类勾选）

### 特殊处理：P0 人工执行

若**人工执行清单中含 P0 用例**，整体结论后追加 ⚠️ 提示：
```
> ⚠️ 含 {N} 条 P0 人工执行用例，需测试工程师确认已人工验证
```
原因：P0 是核心质量门，不能因"人工自认通过"就放行。

## 三、P0 / P0+P1 通过率

| 指标 | 公式 | 用途 |
|---|---|---|
| P0 通过率 | P0 通过 ÷ (P0 执行 - 暂缓 - 阻塞) | 核心质量门 |
| P0+P1 通过率 | (P0+P1 通过) ÷ (P0+P1 执行 - 暂缓 - 阻塞) | 重要功能综合 |

**P0 通过率 100%** 是 🟢 结论的**硬性前提**。

## 四、模块风险等级

| 通过率 | 风险等级 | 颜色 |
|---|---|---|
| ≥95% | 🟢 低 | 绿 |
| 90-94% | 🟡 中 | 黄 |
| <90% | 🔴 高 | 红 |

**应用场景**：模块 × 优先级透视表的"风险"列；各模块汇总表的"风险等级"列。

## 五、需求覆盖状态判定

> 仅当 `review/test-points.md` 存在时适用。

| 条件 | 覆盖状态 | 标识 |
|---|---|---|
| 测试点有关联用例 AND 全部通过 | 已覆盖 | 🟢 |
| 测试点有关联用例 AND 存在失败 | 覆盖但失败 | 🔴 |
| 测试点有关联用例 AND 全部阻塞/暂缓/跳过 | 覆盖未验证 | 🟡 |
| 测试点无关联用例 | 未关联用例（覆盖缺口）| ⚪ |

### 关键指标

- **P0 测试点覆盖率** = 有关联用例的 P0 测试点 ÷ P0 测试点总数
- **覆盖缺口** = ⚪ 状态测试点数量，建议由 `test-case-design` 补充用例

## 六、缺陷指标

| 指标 | 公式 | 说明 |
|---|---|---|
| **缺陷密度** | 总缺陷数 ÷ 已执行用例数 | 低密度（<5%）= 质量良好 |
| **严重缺陷率** | (Fatal + Critical) ÷ 总缺陷数 | 关注深度问题占比 |
| **缺陷修复率** | 已关闭缺陷 ÷ 总缺陷数 | 反映修复进度 |

### 严重等级定义（与缺陷系统对齐）

| 等级 | 定义 |
|---|---|
| **Fatal** | 系统不可用 / 数据损坏 / 安全漏洞 |
| **Critical** | 核心功能失效 / 无法完成业务流程 |
| **Major** | 非核心功能失效 / 体验严重影响 |
| **Minor** | 小缺陷 / 文案/样式问题 |
| **环境阻塞** | 非代码缺陷（环境/数据/依赖）|

---

## 七、风险项识别规则

| 类别 | 触发条件 | 建议处理 |
|---|---|---|
| 🔴 高风险 | P0 失败用例 / 开放 Fatal/Critical 缺陷 | 修复后立即补测 / 发布前必须解决 |
| 🟡 中风险 | 通过率 < 90% 的模块 / 大量 L2 警告 | 观察趋势，不阻断 |
| 🔵 需复测 | 阻塞用例（环境/数据原因）| 补充测试数据后重执行 |

## 八、后续建议（按结论分支）

### 🟢 质量良好
```
→ 可执行 /release-gate 发布门禁检查
```

### 🟡 基本达标
```
→ 建议先确认开放 Critical 缺陷的影响范围
→ 与产品/开发确认是否可带缺陷发布
→ 如确认可发布，执行 /release-gate；如需修复，修复后重新生成测试报告
```

### 🔴 未达标
```
→ 修复高风险项（P0 失败 / Fatal/Critical 缺陷）
→ 重新执行相关测试 (/test-execution / /api-test-execution)
→ 重新执行 /bug-sync 更新缺陷状态
→ 重新生成测试报告 (/test-report)
```

## 九、共享约定（给其他 skill）

### release-gate / submission-gate 的引用方式

在其 SKILL.md 或 OVERRIDES.md 里明确写：

```markdown
## 判定规则来源

本 skill 的质量判定沿用 `.claude/skills/test-report/references/quality-rules.md`（共享权威源），**禁止**在本 skill 内独立定义阈值以避免漂移。
```

### 版本升级约定

- 本文件版本升级（v1.0 → v1.1）必须：
  1. 在文件顶部 `版本：vX.X` 更新
  2. 评估所有引用方是否兼容（查询 `grep -r quality-rules.md .claude/skills/`）
  3. 若有不兼容改动，同步修改所有引用方的 skill 或 OVERRIDES
- **禁止静默改阈值**（会破坏长期趋势分析）
