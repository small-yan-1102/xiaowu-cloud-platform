> **版本**: v1.0

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载源报告
- [ ] Phase 2: 数据聚合与质量分析
- [ ] Phase 3: 输出测试报告
```

---

### Phase 0: 定向收集

**目标**：确认报告来源文件。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **执行报告**：
   - UI 执行报告（默认：`test_reports/test_report_*.md`，取最新一份）
   - 接口执行报告（默认：`test_reports/api_report_*.md`，取最新一份）
   - 可同时指定多份
2. **缺陷清单**（默认：`bug_reports/bug-report-*.md`，取最新一份）
3. **版本信息**（可选）：发布版本号 / 迭代名称 / 测试周期？

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答或表示"按默认来" → 使用默认路径，进入 Phase 1

---

### Phase 1: 加载源报告

**目标**：读取所有来源文件，提取报告所需的原始数据。

**步骤**：

1. **检查报告模板**：
   - 检查 `assets/report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「报告模板文件缺失」

2. **读取 UI 执行报告**（如有）：
   - 提取：用例总数 / 通过 / 失败 / 阻塞 / 跳过 / **⏭️ 人工执行** / L2 警告数
   - 提取：各模块执行结果明细（模块名 / 用例数 / 通过数）
   - 提取：P0 / P1 / P2 / P3 各优先级通过数（人工执行用例单独列出，不计入通过率分母）
   - 提取：失败用例列表（编号 / 名称 / 失败原因摘要）
   - 提取：人工执行用例清单（编号 / 名称，提示测试工程师确认已人工验证）
   - 提取：执行时间 / 总耗时 / 测试环境

3. **读取接口执行报告**（如有）：
   - 提取字段同上，补充：各接口通过率

4. **读取缺陷清单**（如有）：
   - 提取缺陷总数、按严重等级分布
   - 提取环境阻塞项数量

5. **自动查询云效当前缺陷状态**：
   - 检查 `tools/yunxiao/config.yaml` 是否存在且配置完整（需包含 `yunxiao.personal_access_token`、`yunxiao.organization_id`、`projex.project_id`、`projex.workitem_type_id`）
   - 配置完整 → 运行以下命令获取实时缺陷状态：
     ```bash
     python tools/yunxiao/scripts/query_defects.py --json
     # 如需限定当前迭代，使用：
     python tools/yunxiao/scripts/query_defects.py --json --sprint-id <sprint_id>
     ```
   - 解析 JSON 输出，提取各严重等级的 `open` 和 `closed` 数量作为**权威缺陷状态**
   - 配置不完整或命令执行失败 → 降级策略：使用缺陷清单中的数量（仅含已登记缺陷，注明「可能不含最新修复进展」）

6. **合并数据**：
   - 将 UI 执行报告和接口执行报告的数据合并（按模块对齐，用例编号去重）
   - 使用步骤 5 获取的缺陷状态（云效实时数据优先，降级时使用缺陷清单）

7. **输出数据摘要**：已读取 N 条执行结果，缺陷状态来源（云效实时 / 缺陷清单），准备进入分析

> ⛔ **DO NOT** 进入 Phase 2，直到所有源报告均已读取完毕。

---

### Phase 2: 数据聚合与质量分析

**目标**：计算关键质量指标，给出整体质量结论。

**计算指标**：

#### 执行覆盖统计

| 指标 | 计算公式 |
|------|---------|
| 总通过率 | 通过数 ÷ AI执行数（不含跳过、不含人工执行）× 100% |
| P0 通过率 | P0 通过数 ÷ P0 AI执行数 × 100% |
| P0+P1 通过率 | (P0+P1 通过数) ÷ (P0+P1 AI执行数) × 100% |
| 各模块通过率 | 各模块通过数 ÷ 各模块 AI执行数 × 100% |

> **人工执行用例处理**：`[执行方式: 人工]` 用例不计入通过率分母，在报告中单独列为"人工执行清单"，注明「以下用例需测试工程师确认已完成人工验证」。若人工执行清单中包含 P0 用例，在质量结论中标注⚠️提示。

#### 缺陷密度

| 指标 | 计算公式 |
|------|---------|
| 缺陷密度 | 总缺陷数 ÷ 执行用例总数 × 100% |
| 严重缺陷率 | (Fatal + Critical 数) ÷ 总缺陷数 × 100% |
| 缺陷修复率 | 已关闭缺陷数 ÷ 总缺陷数 × 100% |

#### 整体质量结论判定

| 结论 | 判定条件 |
|------|---------|
| 🟢 质量良好，可进入发布流程 | P0 全部通过 + 开放 Fatal = 0 + 开放 Critical = 0 + P0+P1 通过率 ≥ 90% + **人工执行清单中无未验证的 P0 用例** |
| 🟡 质量基本达标，存在已知风险 | P0 全部通过 + 开放 Fatal = 0 + (开放 Critical > 0 或 P0+P1 通过率 80%~89% 或 **人工执行清单中有未验证的 P0 用例**) |
| 🔴 质量未达标，建议修复后重新测试 | P0 有失败 / 开放 Fatal > 0 / P0+P1 通过率 < 80% |

> **注意**：质量结论是综合评估，最终发布决策由 `release-gate` 执行。

#### 风险项识别

- P0 失败用例 → 高风险
- 开放 Fatal/Critical 缺陷 → 高风险
- 通过率低于 90% 的模块 → 中风险
- 大量 L2 警告 → 低风险（提示关注）
- 阻塞用例（环境/数据原因）→ 注明需复测

> ⛔ **DO NOT** 进入 Phase 3，直到所有指标均已计算且质量结论已确定。

---

### Phase 3: 输出测试报告

**目标**：按 `assets/report-template.md` 中的模板，生成完整的测试报告文件。

**步骤**：

1. **读取报告模板**：加载 `assets/report-template.md`

2. **确定输出路径**：`test_reports/test-report-YYYYMMDD.md`
   - 同日期已有报告 → 追加序号：`test-report-YYYYMMDD-2.md`

3. **填写报告各章节**（参照模板结构）：
   - 报告头部（版本 / 周期 / 环境 / 生成时间）
   - 测试范围与覆盖（用例数 / 执行数 / 覆盖率）
   - 执行结果汇总（各模块通过率表格 + 优先级分布）
   - 失败用例列表（编号 / 名称 / 失败原因）
   - 缺陷汇总（按严重等级 / 开放 vs 已关闭）
   - 整体质量结论（🟢 / 🟡 / 🔴 + 说明）
   - 风险项与建议
   - 后续建议（推荐操作：进入 release-gate / 修复后重测）

4. **向用户展示**：
   - 报告文件路径
   - 质量结论（🟢/🟡/🔴）
   - 关键数字（总通过率 / P0+P1 通过率 / 开放严重缺陷数）
   - 风险项摘要（如有）

## 验收标准

### 数据准确性

- [ ] 总通过率、P0+P1 通过率计算正确（分母不含跳过和人工执行用例，仅含 AI 执行用例）
- [ ] 人工执行用例已单独列出，未混入 AI 执行统计
- [ ] 人工执行清单中含 P0 用例时，质量结论中已标注 ⚠️
- [ ] 缺陷统计优先使用 Phase 1 从云效自动查询的实时数据（降级时使用缺陷清单）
- [ ] 各模块通过率已逐模块计算并列出

### 报告完整性

- [ ] 报告已写入 `test_reports/test-report-YYYYMMDD.md`
- [ ] 包含执行结果汇总、缺陷汇总、质量结论、风险项四个核心章节
- [ ] 质量结论标注了 🟢 / 🟡 / 🔴 及判定理由
- [ ] 有失败用例时，失败列表已列出

### 下游衔接

- [ ] 报告末尾给出推荐的后续操作（进入 release-gate / 修复后重测）

## 与其他 Skill 的衔接

| 上游 Skill | 提供给本 Skill | 说明 |
|-----------|--------------|------|
| `test-execution` | UI 执行报告 | P0/P1/P2/P3 各优先级执行结果 |
| `api-test-execution` | 接口执行报告 | 各接口执行结果 |
| `bug-sync` | 缺陷清单 | 缺陷分布参考（当前状态由 query_defects.py 自动查询）|
| `tools/yunxiao/scripts/query_defects.py` | 实时缺陷状态 JSON | 自动查询云效 Projex 各严重等级开放/已关闭数 |

| 下游 Skill | 触发条件 | 说明 |
|-----------|---------|------|
| `release-gate` | 质量结论为 🟢 或 🟡 时 | 执行二元发布判定 |
| 重新测试 | 质量结论为 🔴 时 | 修复后重走执行 → bug-sync → test-report |
