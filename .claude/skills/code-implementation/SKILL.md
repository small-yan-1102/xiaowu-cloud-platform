# 接口联调实施计划技能

> **设计模式**：Inversion（收集范围）→ Pipeline（收集上下文 → 规则预读 → 逐字段对照 → 生成计划），差异必须逐一确认

**⚠️ 核心约束：本阶段专注于接口联调计划，请勿生成页面代码。**

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集（Inversion）
- [ ] Phase 1: 收集需求上下文
- [ ] Phase 2: 规则预读
- [ ] Phase 3: Mock 数据与接口文档对照分析
- [ ] Phase 4: 生成联调计划
```

### Phase 0: 范围收集（Inversion 模式）

**⚠️ 硬门控：DO NOT proceed until scope is confirmed.**

使用 AskUserQuestion 收集：
- API 文档路径
- 对应的 mock 数据路径
- 相关页面代码路径
- 版本号

---

### Phase 1: 收集需求上下文

- 读取 API 设计文档
- 读取现有 mock 数据定义
- 读取相关页面代码（service 文件和组件）

---

### Phase 2: 规则预读

按优先级加载以下规范文件（存在则加载，不存在则跳过）：

1. **接口规范**：`{迭代目录}/input/tech/` 下的接口定义文档（Swagger/OpenAPI/手写接口文档）
2. **前端编码规范**：项目根目录或 `src/` 下的 `.eslintrc` / `tsconfig.json`（推断命名风格和类型约束）
3. **请求封装层**：`src/api/` 或 `src/services/` 下的现有 request 封装（推断拦截器、baseURL、响应结构）

> 找不到规范文件时不阻塞，在 Phase 3 对照分析中标注「未找到项目规范，按接口文档为准」。

---

### Phase 3: Mock 数据与接口文档对照分析

**⚠️ 关键步骤：逐字段对照，发现差异立即暂停询问**

1. **枚举值对照**：对比 mock 枚举与接口文档的枚举定义
2. **接口参数对照**：对比请求参数的字段名、类型、是否必填
3. **接口返回对照**：对比 mock 数据结构与接口文档返回结构
4. **差异处理**：
   - **发现任何差异** → 立即暂停，向用户确认
   - **接口文档缺失** → 立即暂停，询问用户
   - **接口定义有疑问** → 立即暂停，向用户澄清

**⚠️ 硬门控：DO NOT proceed to Phase 4 until all differences are confirmed.**

**差异对照输出格式**：

```markdown
## 接口对照：{接口名称}

### 请求参数对照
| 字段 | Mock 定义 | 接口文档 | 状态 |
|------|----------|---------|------|
| id | `taskId` | `task_id` | ⚠️ 不一致 |

### 返回结构对照
| 字段 | Mock 定义 | 接口文档 | 状态 |
|------|----------|---------|------|

### 枚举对照
| 枚举名 | Mock 定义 | 接口文档 | 状态 |
|--------|----------|---------|------|
```

---

### Phase 4: 生成联调计划

创建联调实施文档，将接口联调拆解为具体的 Quest：

每个 Quest 包含：
- 接口名称和用途
- Mock 数据 vs 接口文档的对照表
- 需要修改的文件路径
- 具体修改内容（字段名、枚举值、请求/响应结构）

---

## 核心约束

- **严禁跳过对照**：必须逐字段对比 mock 数据与接口文档
- **严禁自行定义**：不确定的字段、枚举值必须暂停询问
- **严禁字段映射**：mock 数据必须与接口文档完全一致，禁止前端做字段转换映射
- **接口文档优先**：以接口文档为准，mock 数据必须同步修改
