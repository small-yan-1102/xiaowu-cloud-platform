# 提测门禁技能

> **版本**: v1.0
> **更新日期**: 2026-04-08
> **适用模式**: AI 读取技术文档审阅汇总报告 → 核验各模块判定结果 → 输出提测门禁结论（通过 / 不通过）

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载审阅汇总报告
- [ ] Phase 2: 执行门禁检查
- [ ] Phase 3: 输出门禁结论报告
```

---

### Phase 0: 定向收集

**目标**：确认审阅汇总报告位置和开发自测状态，避免测试执行资源浪费在低质量提测上。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **审阅汇总报告路径**：技术文档审阅汇总报告位于哪里（默认：`review/tech-review-summary.md`）？
2. **检查范围**：检查全部模块，还是指定某个模块？（默认：全部模块）
3. **开发自测证明**：开发是否已按冒烟套件（`test_suites/suite_smoke.md`）逐条执行并提交结果？
   - 执行结果须包含：每条用例的通过/失败标注 + 截图（Happy Path 和 EX 用例均须截图）
   - 若有失败用例：须说明是已知 Bug 还是环境问题，及影响范围
   
   > 未按冒烟套件执行 / 无截图 / 纯口头声明 → 门禁直接不通过。

4. **单元测试覆盖率**：
   - 是否有 CI 流水线覆盖率报告链接，或覆盖率报告截图？
   - 要求：**分支覆盖率（Branch Coverage）≥ 60%**

5. **Code Review 状态**：
   - 本次提测对应的 MR/PR 链接是什么？
   - 是否已有 ≥ 1 人 Approve？

6. **API 契约测试状态**：
   - CI 流水线是否已集成 Swagger/OpenAPI 契约测试（Contract Testing）？
   - 若已集成：提供 CI 报告链接或截图，确认接口入参/出参/状态码与文档一致
   - 若未集成：说明原因（如团队尚未引入、纯前端项目等）

7. **测试环境就绪状态**：
   - 开发提测的分支名称是什么？是否已合并到 test 分支？（由测试部门自行部署，此信息用于确认部署依据）
   - 测试部门是否已完成本次版本的测试环境部署？（测试自行确认）
   - 测试环境 URL 是什么？（用于 AI 执行登录预检）
   - 测试账号是什么？

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户回答或表示"按默认来" → 使用默认路径，检查全部模块，进入 Phase 1

---

### Phase 1: 加载审阅汇总报告

**目标**：读取审阅汇总报告，确认其可用且包含必要结构。

**步骤**：

1. **检查门禁报告模板**：
   - 检查 `assets/gate-report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「门禁报告模板文件缺失」

2. **读取审阅汇总报告**：
   - 加载用户指定路径（默认 `review/tech-review-summary.md`）
   - 不存在 → **Fatal**：终止执行，提示「审阅汇总报告不存在，请先执行 tech-doc-review」

3. **确认报告结构**：
   - 确认报告含有「§〇 审阅完成度」章节及「模块级用例生成条件判定」表格
   - 不含 → **Fatal**：终止执行，提示「审阅汇总报告结构不完整，无法执行门禁检查」

4. **提取检查数据**：从「模块级用例生成条件判定」表格提取：
   - 各模块名称及英文标识
   - 各模块的**模块判定**（`🟢 / 🟡 / 🔴`）
   - 各模块的判定理由

> ⛔ **DO NOT** 进入 Phase 2，直到报告加载成功且结构完整。

---

### Phase 2: 执行门禁检查

**目标**：对每个模块逐项核验，确定整体门禁结论。

**检查项优先级说明**：

| 优先级 | 含义 | 不通过时的处理 | 涉及检查 |
|-------|------|-------------|---------|
| **P0** | 无任何例外情形，必须通过 | 门禁直接不通过，必须修复后重新触发 | 检查 1/2/3/4/6/8 |
| **P1** | 有明确例外情形，允许带条件通过 | 在报告中标注风险说明，由测试负责人确认后可放行 | 检查 5/7/9 |

---

#### 检查 1【P0】：是否所有模块均已完成审阅

- 扫描「§〇 审阅完成度」中`审阅状态`列
- 发现 `⏳ 待审阅` 的文档类型 → 记录为**待审阅项**（贡献 FAIL）
- 发现 `➖ 不适用` → 跳过（合理缺失，不贡献 FAIL）
- 全部为 `✅ 已完成` 或 `➖ 不适用` → 检查 1 通过

#### 检查 2【P0】：是否存在 🔴 模块 / 🟡 模块降级说明

- 扫描「模块级用例生成条件判定」表格的**模块判定**列
- 发现任一模块为 `🔴 建议等待` → 记录该模块和判定理由，贡献 FAIL
- 发现 `🟡 技术文档不完整` 的模块 → **不贡献 FAIL**，但必须在报告中单独列出「🟡 模块覆盖范围说明」：
  - 该模块本次**仅接口层可覆盖**，UI E2E 用例覆盖不完整
  - 测试执行时该模块应以 `api-test-execution` 为主，`test-execution` 覆盖受限
  - 风险由测试负责人知悉并签字确认（在报告中体现）
- 全部模块为 `🟢` 或 `🟡` → 检查 2 通过

#### 检查 3【P0】：开发自测状态

开发自测以**冒烟套件（`test_suites/suite_smoke.md`）为执行依据**，逐条执行并提交结果截图。

- 根据 Phase 0 用户提供的自测执行结果执行：

  | 情况 | 检查方式 | 结论 |
  |------|---------|------|
  | 提供了冒烟套件执行结果 + 每条截图 | 确认：① Happy Path 用例全部执行 ② EX 用例（关键异常核心）全部执行 ③ 失败用例有说明（已知Bug/环境问题） | 三项均满足 → 通过；缺少任一项 → **FAIL**：标注缺失项，要求补充 |
  | 提供了执行结果但缺少 EX 用例截图 | — | **FAIL**：关键异常核心未验证，要求补充 EX 用例执行截图 |
  | 未按冒烟套件执行（自制清单代替）| — | **FAIL**：须以冒烟套件为执行依据，不接受开发自制的临时清单 |
  | 仅口头声明（"我测过了"）| — | **直接 FAIL**：口头声明不接受 |
  | **未提供任何证明** | — | **直接 FAIL**：提示开发按冒烟套件执行并提交结果后再触发门禁 |

  > **前提**：冒烟套件须已由测试团队通过 `test-case-design` Mode D 生成（含 EX 用例）。若套件不存在，检查 3 自动视为「无法执行」，门禁结论直接不通过（同检查 6 套件存在性联动）。

#### 检查 4【P0】：测试环境就绪

根据 Phase 0 用户提供的环境信息执行：

**Step 4a — 部署确认**

测试环境由**测试部门自行部署**，触发门禁时需确认以下信息：

| 检查内容 | 检查方式 | 失败条件 |
|---------|---------|---------|
| 开发分支 + 是否合并到 test 分支 | 记录开发提供的分支信息 | 开发未提供分支信息 → **FAIL（高亮警告）**：部署依据不明，建议确认后再触发门禁 |
| 测试部门是否已完成本版本部署 | 由测试人员触发门禁时自行确认 | 明确表示尚未部署 → **FAIL**：环境未就绪 |
| **未提供任何说明** | — | **FAIL（高亮警告）**：环境状态未知，冒烟验证可能因环境原因失败 |

**Step 4b — 登录预检（AI 执行）**

使用 Phase 0 提供的测试环境 URL 和账号，执行以下预检：

| # | 预检项 | 验证方式 | 失败处理 |
|---|-------|---------|---------|
| 1 | 测试环境可访问 | 访问 URL，页面返回非错误状态 | FAIL：环境不可达 |
| 2 | 登录功能正常 | 使用测试账号登录，验证登录成功 | FAIL：登录失败 |

> 登录预检失败 → 检查 4 不通过（记录错误截图供排查）
> 测试环境 URL 未提供 → 跳过登录预检，在报告中标注「未执行环境预检，建议冒烟验证前人工确认」

#### 检查 5【P1】：PRD 门禁结论核验

- 检查 `review/prd-gate-result.md` 是否存在：
  - **存在且结论为「通过」** → 检查 5 通过
  - **存在但结论为「不通过」** → 记录阻塞原因，贡献 FAIL（PRD 质量问题未解决即进入测试，用例可靠性存疑）
  - **不存在** → ⚠️ 警告（非阻塞）：本次提测未经 prd-gate 检验，在报告中标注「PRD 门禁未执行，建议确认 PRD 质量是否满足测试设计要求」；不贡献 FAIL

#### 检查 6【P0】：用例套件存在性

- 扫描 `test_suites/` 目录，检查：

  | 检查项 | 通过条件 | 失败条件 |
  |-------|---------|---------|
  | 冒烟套件（`test_suites/suite_smoke.md`） | 文件存在 | 不存在 → **FAIL**：冒烟执行无法启动 |
  | 模块用例套件（`test_suites/suite_*.md`） | 至少一个模块套件存在 | 全部不存在 → **FAIL**：无用例可执行 |

- `suite_smoke.md` 不存在 → 记录为阻塞项，贡献 FAIL
- 模块套件存在但部分模块缺失 → ⚠️ 警告（非阻塞）：在报告中列出缺失套件的模块，标注「该模块本次执行覆盖不完整」
- 全部套件均存在 → 检查 6 通过

#### 检查 7【P1】：单元测试覆盖率

- 根据 Phase 0 用户提供的覆盖率信息执行：

  | 情况 | 检查方式 | 结论 |
  |------|---------|------|
  | 提供了 CI 流水线链接或覆盖率报告截图 | 读取或记录分支覆盖率数值 | 分支覆盖率 ≥ 60% → 通过；< 60% → **FAIL**：标注实际值和差距 |
  | 未提供任何覆盖率证明 | — | **直接 FAIL**：要求提供 CI 报告链接或截图后再触发门禁 |
  | 项目明确不含服务端业务逻辑（如纯配置/纯前端静态页） | 记录说明，由测试负责人确认 | ⚠️ 警告（非阻塞）：在报告中标注「本次提测不适用单元测试要求，已由测试负责人确认」 |

> **覆盖率类型说明**：要求使用**分支覆盖（Branch Coverage）**，而非行覆盖（Line Coverage）。分支覆盖能识别 if/else 漏测路径，约束力更强。

#### 检查 8【P0】：Code Review 状态

- 根据 Phase 0 用户提供的 MR/PR 信息执行：

  | 情况 | 检查方式 | 结论 |
  |------|---------|------|
  | 提供了 MR/PR 链接且已有 ≥ 1 人 Approve | 记录链接和 Approve 人员 | 通过 |
  | 提供了 MR/PR 链接但无 Approve | 记录链接 | **FAIL**：Code Review 未通过，要求至少 1 人 Approve 后再提测 |
  | 未提供 MR/PR 链接 | — | **直接 FAIL**：无法确认 Code Review 状态，要求提供 MR/PR 链接 |

#### 检查 9【P1】：API 契约测试

- 根据 Phase 0 用户提供的 API 契约测试状态执行：

  | 情况 | 检查方式 | 结论 |
  |------|---------|------|
  | CI 已集成契约测试 + 提供报告链接/截图 + 全部通过 | 记录链接，确认结论为通过 | 通过 |
  | CI 已集成契约测试 + 提供报告链接/截图 + 存在失败项 | 记录失败接口清单 | **FAIL**：接口与文档不一致，要求修复后重新提测 |
  | CI 已集成但未提供报告 | — | **FAIL**：已有契约测试但未提供结果，要求补充 |
  | CI 尚未集成契约测试（团队未引入） | 记录说明 | ⚠️ 警告（非阻塞）：标注「API 契约测试未集成，接口与文档一致性风险由测试执行阶段承担」；建议团队纳入 CI 流水线 |
  | 纯前端/纯配置项目，无接口层 | 记录说明 | ⚠️ 不适用，非阻塞 |

> **背景**：API 契约测试通过 Swagger/OpenAPI 文档自动校验接口入参、出参、状态码是否与文档一致，在提测前拦截接口文档失真问题，避免测试执行阶段因接口变更导致大量失败。

**门禁结论判定规则**：

| 条件 | 优先级 | 结论 |
|------|-------|------|
| 检查 1～9 全部通过（含警告项） | — | **通过** |
| 检查 1 不通过（存在未完成审阅）| P0 | **不通过** |
| 检查 2 不通过（存在 🔴 模块）| P0 | **不通过** |
| 检查 3 不通过（未提供书面自测清单 / 口头声明 / 无截图录屏 / 内容不完整）| P0 | **不通过** |
| 检查 4 不通过（环境未部署 / 登录预检失败）| P0 | **不通过** |
| 检查 5 不通过（prd-gate 结论为「不通过」）| P1 | **不通过** |
| 检查 6 不通过（冒烟套件缺失 / 无任何模块套件）| P0 | **不通过** |
| 检查 7 不通过（分支覆盖率 < 60% / 未提供覆盖率证明）| P1 | **不通过** |
| 检查 8 不通过（MR/PR 无 Approve / 未提供链接）| P0 | **不通过** |
| 检查 9 不通过（已集成契约测试但存在失败项 / 已集成但未提供报告）| P1 | **不通过** |
| 多项不通过 | — | **不通过**（列出全部阻塞项）|

> ⛔ **DO NOT** 进入 Phase 3，直到所有检查项均已完成且结论已确定。

### Phase 3: 输出门禁结论报告

**目标**：按 `assets/gate-report-template.md` 中的模板，生成提测门禁结论报告。

**步骤**：

1. **读取门禁报告模板**：加载 `assets/gate-report-template.md`

2. **确定输出路径**：`review/submission-gate-result.md`
   - 文件已存在 → **覆盖写入**（本次门禁检查为最新结果）

3. **填写报告内容**：
   - 门禁结论（**通过** / **不通过**）—— 置于报告首行，醒目展示
   - 开发自测状态表（检查 3 的结果：清单内容 + 截图/录屏存在性）
   - 单元测试覆盖率（检查 7 的结果：实际分支覆盖率 vs 60% 阈值，或不适用说明）
   - Code Review 状态（检查 8 的结果：MR/PR 链接 + Approve 人员）
   - API 契约测试状态（检查 9 的结果：已集成且通过 / 已集成但失败 / 未集成⚠️ / 不适用）
   - 测试环境就绪状态（检查 4 的结果：部署确认 + 登录预检结论）
   - PRD 门禁状态（检查 5 的结果：通过 / 不通过 / 未执行⚠️）
   - 用例套件就绪状态（检查 6 的结果：各套件文件存在性清单）
   - 各模块状态汇总表（含 🟡 模块的覆盖范围说明）
   - 阻塞项列表（仅不通过时）：每项标注检查编号、具体原因、建议解决方向
   - 警告项列表（不阻塞但需关注）：prd-gate 未执行警告、部分模块套件缺失警告、单元测试不适用确认
   - 后续步骤建议

4. **向用户展示**：
   - 直接输出门禁结论（通过/不通过）
   - 通过 → 提示进入 test-data-preparation，随后执行 test-execution / api-test-execution
   - 不通过 → 列出阻塞项，提示解决方向

## 门禁结论后的后续步骤

### 通过时

用例设计阶段应已在开发期间并行完成（`test-case-design` / `api-test-case-design`），门禁通过后直接进入执行准备：

| 步骤 | Skill | 说明 |
|------|-------|------|
| 1 | `api-test-execution` | 先执行接口用例，验证后端接口层正确性 |
| 2 | `test-data-preparation` | 接口确认可用后，通过 API 调用创建 UI 测试所需数据 |
| 3 | `test-execution` | 使用已准备好的数据执行 UI E2E 测试用例 |

> **执行顺序说明**：接口先行，原因有二：① UI 测试底层依赖接口，接口失败时 UI 失败原因不明确；② test-data-preparation 本身通过 API 创建数据，接口验证通过后再准备数据可避免因接口问题导致数据准备失败。

> **注意**：若用例尚未设计完成，应在触发门禁前补齐；门禁通过不再回退到设计阶段。

### 不通过时

门禁报告中列出每个阻塞项及其解决建议，开发团队解决后**重新执行 tech-doc-review**，再次触发提测门禁。

## 验收标准

### 结论准确性

- [ ] 存在 🔴 模块时，结论必须为**不通过**
- [ ] 存在 ⏳ 未完成审阅的文档时，结论必须为**不通过**
- [ ] 开发未提供冒烟套件执行结果时，结论必须为**不通过**
- [ ] 开发仅口头声明（"我测过了"）时，结论必须为**不通过**
- [ ] EX 用例（关键异常核心）未执行或无截图时，结论必须为**不通过**
- [ ] 开发以自制清单代替冒烟套件执行时，结论必须为**不通过**
- [ ] 单元测试分支覆盖率 < 60% 时，结论必须为**不通过**
- [ ] 未提供单元测试覆盖率证明（CI链接/截图）时，结论必须为**不通过**
- [ ] MR/PR 无 Approve 或未提供链接时，结论必须为**不通过**
- [ ] 单元测试不适用时，以警告（非阻塞）标注并注明测试负责人已确认，不错误判定为 FAIL
- [ ] API 契约测试已集成且存在失败项时，结论必须为**不通过**
- [ ] API 契约测试已集成但未提供报告时，结论必须为**不通过**
- [ ] API 契约测试未集成时，以警告（非阻塞）标注，不错误判定为 FAIL
- [ ] 测试环境未部署或登录预检失败时，结论必须为**不通过**
- [ ] prd-gate 结论为「不通过」时，结论必须为**不通过**
- [ ] 冒烟套件（`suite_smoke.md`）不存在时，结论必须为**不通过**
- [ ] 所有模块套件均不存在时，结论必须为**不通过**
- [ ] 🟡 模块已在报告中单独列出覆盖范围说明，而非混入汇总表静默通过
- [ ] prd-gate 结论文件不存在时，以警告（非阻塞）标注，不错误判定为 FAIL
- [ ] 部分模块套件缺失时，以警告（非阻塞）标注缺失模块，不错误判定为 FAIL
- [ ] 所有检查通过（含警告项）时，结论为**通过**
- [ ] 门禁结论不出现除「通过」「不通过」以外的第三种状态
