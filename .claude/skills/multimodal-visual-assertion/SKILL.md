# 多模态视觉断言技能

> **版本**：v1.0
> **更新日期**：2026-04-13
> **定位**：test-execution 的补充执行能力，处理 DOM 文本验证覆盖不到的视觉内容

---

## 适用场景 vs 不适用场景

| 适用 | 不适用 |
|------|--------|
| 图表（折线图/柱状图/饼图）数据趋势或比例验证 | 纯文字内容 → 使用 `[断言: 视觉]`（DOM 文本更准确） |
| Canvas 渲染内容（地图、流程图、白板） | 精确数值 → 使用 `[断言: API]` |
| 图片是否正常显示（非损坏、非占位符） | 数据库状态 → 使用 `[断言: DB-Query]` |
| PDF / 文档预览内容 | 列表条数、金额等可结构化数据 |
| 富文本排版、图文混排布局 | |
| 图标/徽章/状态颜色（无对应文本） | |

---

## 用例中的标注方式

测试用例预期结果使用 `[断言: 视觉-AI]` 标注，并附带自然语言描述的验证目标：

```markdown
预期结果：
- [断言: 视觉-AI] 折线图显示近 7 天数据呈上升趋势，最后一个数据点高于第一个数据点
- [断言: 视觉-AI] 页面右上角头像区域显示用户头像，非默认灰色占位图
- [断言: 视觉-AI] 导出的图表缩略图与页面展示内容一致（颜色、图例、大致形状）
```

**描述要求**：
- 描述**可观测的视觉特征**，避免主观审美判断
- 允许模糊匹配（"呈上升趋势"）；需要精确验证时使用 `[断言: API]`
- 每条 `[断言: 视觉-AI]` 只验证一个视觉维度

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集
- [ ] Phase 1: 截图采集
- [ ] Phase 2: AI 视觉分析
- [ ] Phase 3: 输出断言结果
```

---

### Phase 0: 定向收集

**独立调用时**，向用户确认：

1. **验证目标**：截图路径（已有截图）或"当前页面"（现场截图）？
2. **断言描述**：需要验证什么视觉内容？
3. **验证区域**（可选）：全页面还是特定区域（如"图表区域"、"右上角"）？

**由 test-execution 调用时**，从用例的 `[断言: 视觉-AI]` 标注中提取输入，跳过用户问答。

---

### Phase 1: 截图采集

1. **使用已有截图**：若 test-execution 在执行步骤中已产生 CP1/CP2 截图，直接使用对应截图路径
2. **现场截图**：若无可用截图，调用 `browser_take_screenshot` 采集当前页面

**区域截图**（当断言描述指定了特定区域时）：
- 优先通过元素定位截取目标区域（`browser_take_screenshot` 的 element 参数）
- 无法定位目标元素时，截取全页面，在 Phase 2 分析时限定观察范围

---

### Phase 2: AI 视觉分析

将截图与断言描述一并提交 AI 视觉模型分析，按以下结构输出判定：

**分析维度**（根据断言描述自适应选择）：

| 验证类型 | 关注点 |
|---------|-------|
| 图表趋势 | 数据点走向、最高/最低点相对位置、大体比例 |
| 图片展示 | 是否加载成功（非破损图标、非默认占位符）、主体内容可辨认 |
| 颜色/状态 | 目标区域颜色符合预期（绿色=成功、红色=错误等） |
| 布局合规 | 元素位置关系（A 在 B 左侧、C 位于页面右上角） |
| 内容一致性 | 两个区域的视觉内容大体相同（缩略图与原图） |

**置信度规则**：

| 置信度 | 含义 | 处理方式 |
|-------|------|---------|
| 高（≥ 85%） | AI 视觉判断明确 | 直接输出通过/失败 |
| 中（60-84%） | 特征存在但不确定 | 输出结果并标注「⚠️ 视觉置信度中等，建议人工确认」 |
| 低（< 60%） | 无法从截图得出结论 | 输出「🔵 视觉无法判定」，建议调整断言方式或人工核实 |

**禁止推断规则**：
- 不得根据其他步骤结果推断视觉内容（必须从截图本身判断）
- 不得对精确数值做出判定（如图表 Y 轴具体数字）

---

### Phase 3: 输出断言结果

**结果状态**：

| 状态 | 判定条件 | 对 test-execution 的影响 |
|------|---------|----------------------|
| ✅ 视觉通过 | 置信度高，特征符合断言描述 | 同 L1/L2 通过 |
| ⚠️ 视觉通过（待确认）| 置信度中等 | 同 L2 通过，执行报告中标注「需人工确认」|
| ❌ 视觉失败 | 置信度高，特征明显不符 | 同 L1 失败，触发截图存档 |
| 🔵 视觉无法判定 | 置信度低 / 截图质量不足 | 记录为「视觉断言未完成」，不计入通过/失败统计 |

**输出格式**（嵌入 test-execution 执行记录中）：

```
[断言: 视觉-AI] {断言描述}
结果：{✅/❌/⚠️/🔵} {结论一句话}
置信度：{高/中/低}
分析依据：{AI 观察到的视觉特征描述}
截图：{截图路径}
```

**截图存档**：
- 通过：截图归入 CP1/CP2 常规存档
- 失败：截图归入 CP1 必存（文件名追加 `_visual_fail` 后缀）
- 无法判定：截图归入 CP3 存档，文件名追加 `_visual_unclear` 后缀

---

## 验收标准

- [ ] 每条 `[断言: 视觉-AI]` 均有对应的截图和分析结果
- [ ] 置信度中等时已标注「需人工确认」
- [ ] 视觉无法判定时未强行输出通过/失败，记录为「未完成」
- [ ] 精确数值断言未通过本 skill 处理（已转交 `[断言: API]`）
- [ ] 所有截图已按命名规范存档

---

## 与 test-execution 的衔接

**引入方式**：在 test-execution Phase 2 执行用例时，遇到 `[断言: 视觉-AI]` 标注的预期结果，加载本 skill 处理该断言，结果回填至当前用例的执行记录。

**未引入时的处理**：若 test-execution 未加载本 skill，遇到 `[断言: 视觉-AI]` 则记录为「⏭️ 跳过（multimodal-visual-assertion 未启用）」，不影响用例其余步骤的执行。

---

## 局限性

| 局限 | 说明 |
|------|------|
| 不支持精确数值验证 | 图表具体数值需通过 `[断言: API]` 获取 |
| 依赖截图质量 | 分辨率过低、遮挡、动画未完成时置信度下降 |
| 不支持动态内容实时验证 | 仅验证截图时刻的静态画面 |
| 颜色判断受显示器色域影响 | 精确颜色值校验建议用 CSS/DOM 属性查询 |
