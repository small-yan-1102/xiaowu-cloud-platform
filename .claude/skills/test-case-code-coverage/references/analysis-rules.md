# 覆盖分析规则

> 本文件由 test-case-code-coverage skill 在 Phase 1 开始前加载。
> 修改此文件可调整解析粒度和映射策略，无需修改 SKILL.md。

---

## 一、代码解析规则

### 1.1 各语言分支识别模式

#### Java / Kotlin
| 构造 | 解析为覆盖目标 |
|------|--------------|
| `if (condition)` | [B] true 分支 + false 分支各一个 |
| `else if (condition)` | [B] 独立分支 |
| `switch (var) { case X: }` | [B] 每个 case 一个，含 default |
| `condition ? a : b` | [B] true 分支 + false 分支 |
| `catch (ExceptionType e)` | [E] 异常类型作为覆盖目标 |
| `throw new XxxException` | [E] 抛出点作为覆盖目标 |
| `return xxx` / `return null` | [R] 不同 return 路径各一个 |
| `for` / `while` / `forEach` | [L] 空集合 + 非空集合 |
| `Optional.orElse` / `orElseThrow` | [B] present + empty 各一个 |
| `@GetMapping` / `@PostMapping` 等 | [F] API 入口（高优先级） |

#### Python
| 构造 | 解析为覆盖目标 |
|------|--------------|
| `if condition:` | [B] True + False 分支 |
| `elif condition:` | [B] 独立分支 |
| `except ExceptionType:` | [E] 异常类型 |
| `raise XxxError` | [E] 抛出点 |
| `for x in collection:` | [L] 空 + 非空 |
| `return value` | [R] 不同返回路径 |
| `def func_name` | [F] 函数入口 |

#### TypeScript / JavaScript
| 构造 | 解析为覆盖目标 |
|------|--------------|
| `if (condition)` | [B] true + false |
| `switch (var)` | [B] 每个 case + default |
| `condition ? a : b` | [B] |
| `catch (error)` | [E] |
| `throw new Error` | [E] |
| `return` | [R] 不同路径 |
| 函数声明 / 箭头函数（export） | [F] |
| `?.` 可选链 | [B] null/undefined + 有值 各一 |
| `??` 空值合并 | [B] null + 非null 各一 |

#### Go
| 构造 | 解析为覆盖目标 |
|------|--------------|
| `if condition {` | [B] true + false |
| `switch {` / `case condition:` | [B] 每个 case |
| `if err != nil {` | [E] 错误路径（高优先级） |
| `return val, err` | [R] 不同错误/成功路径 |
| `func FuncName(` | [F] 函数入口 |
| `for range collection` | [L] 空 + 非空 |

---

### 1.2 优先级规则

解析时，不同类型的覆盖目标重要性不同：

| 类型 | 优先级 | 说明 |
|------|-------|------|
| [F] API 入口 / 公共方法 | P0 | 必须有用例覆盖 |
| [E] 异常处理路径 | P1 | 涉及错误场景，高价值 |
| [B] 业务条件分支 | P1 | 核心业务逻辑 |
| [R] 返回值分支 | P2 | 可通过分支覆盖推断 |
| [L] 循环边界 | P2 | 空集合边界较重要 |
| [B] 防御性 null 检查 | P3 | 框架层通常已保证 |

**分析时默认只报告 P0-P2 的盲区**，P3 盲区仅在用户要求"详细分析"时展示。

---

### 1.3 跳过规则（不生成覆盖目标）

以下代码构造**不生成覆盖目标**：

- `log.debug()` / `log.info()` 等日志调用
- `@Override` 的 getter/setter 方法
- 构造函数（除非有明确的业务初始化逻辑）
- 私有工具方法（仅被单一公有方法调用的内部辅助逻辑）
- 注释中的示例代码
- 测试代码本身（`@Test`、`test_`前缀函数等）

---

### 1.4 业务语义提取

对每个覆盖目标，必须将技术条件翻译为**业务语言**，用于后续的语义匹配：

| 技术描述 | 业务语义翻译示例 |
|---------|--------------|
| `if (balance >= amount)` | 余额是否足够 |
| `if (user == null)` | 用户是否存在 |
| `catch (DuplicateKeyException)` | 数据重复/唯一性约束冲突 |
| `if (list.isEmpty())` | 列表为空的情况 |
| `if (status == PENDING)` | 状态为待处理时 |
| `if (!hasPermission(role))` | 无权限时 |

## 二、映射规则

### 2.1 置信度判定标准

#### HIGH（高置信）—— 直接映射

满足以下**任意一条**：

- 用例明确描述了与覆盖目标完全对应的输入条件（如"余额为0"对应`balance < amount`的 false 分支）
- 用例预期结果中出现了覆盖目标对应的响应（如"返回余额不足错误"对应 InsufficientBalanceException）
- 用例名称或描述中直接点出了目标场景（如"用户不存在时登录失败"）

#### MEDIUM（中置信）—— 间接映射

满足以下**任意一条**：

- 用例通过执行完整流程，会间接触发该分支（如"成功下单"用例会走 `balance >= amount → true` 分支）
- 用例描述的场景逻辑上必然经过该路径（如"搜索结果为空"必然触发 `list.isEmpty()` 的 true 分支）
- 用例描述了父功能，该覆盖目标是子流程的一部分

#### LOW（低置信）—— 推断映射

- 用例描述笼统，无法确定是否触发了该分支（如"测试登录功能"无法确认是否覆盖了某异常路径）
- 用例步骤描述与代码逻辑有语义关联，但对应关系不明确

#### NONE —— 未映射

- 在所有用例中无法找到任何与该覆盖目标相关的描述

---

### 2.2 常见语义对照表

用于辅助判断用例描述是否覆盖了某类代码构造：

| 用例描述关键词 | 可能覆盖的代码构造 |
|-------------|----------------|
| "成功" / "正常" / "有效" | 条件为 true 的正向分支 |
| "失败" / "异常" / "错误" / "不合法" | 条件为 false 的异常分支、catch 块 |
| "为空" / "空列表" / "不存在" | null 检查、isEmpty()、Optional.empty() |
| "超过上限" / "超出范围" / "超过最大值" | 边界条件 > max |
| "恰好等于" / "边界值" | 边界条件 == threshold |
| "重复" / "已存在" / "唯一性" | DuplicateKey 异常路径 |
| "无权限" / "权限不足" / "未授权" | permission check 的 false 分支 |
| "并发" / "同时操作" / "竞态" | 乐观锁冲突、并发控制分支 |
| "超时" / "网络异常" | 超时异常捕获路径 |
| "回滚" | 事务异常分支 |

---

### 2.3 跨用例合并映射

一个覆盖目标可以被多个用例共同覆盖，记录时列出所有映射：

```
覆盖目标：[F] OrderService.createOrder(CreateOrderRequest)
映射用例：
  - TC-001 成功创建订单（HIGH，正向路径）
  - TC-002 余额不足创建失败（MEDIUM，同入口的异常路径）
  - TC-003 参数为空时创建失败（MEDIUM，同入口的参数校验路径）
综合覆盖评估：HIGH（函数级完整覆盖）
```

---

### 2.4 盲区严重性分级

`[GAP-MISSING]` 的盲区按以下规则分级：

| 严重性 | 判定条件 | 建议 |
|-------|---------|------|
| 🔴 严重 | P0/P1 目标 + 异常处理路径 未覆盖 | 必须补充用例 |
| 🟡 中等 | P1/P2 目标 + 条件分支 未覆盖 | 建议补充 |
| 🟢 低 | P2/P3 目标 + 循环边界/返回值分支 未覆盖 | 可选补充 |

## 三、多文件分析规则

### 3.1 分析范围优先级

用户提供多文件时，建议优先分析：
1. Service 层 / 业务逻辑层
2. Controller / API 层（接口入口）
3. Repository / DAO 层（数据访问）
4. 工具类（最低优先级）

### 3.2 跨文件引用处理

当一个方法调用了另一个文件中的方法时：
- **不深追调用链**（避免分析爆炸）
- 标注"跨文件调用"，提示用户如需深度分析可单独提供被调方法代码

## 四、语言无关的通用规则

### 4.1 条件复合处理

`if (A && B)` → 拆分为 2 个覆盖目标：
- `A = false`（A 短路，B 不执行）
- `A = true && B = false`
- `A = true && B = true`（正向路径）

`if (A || B)` → 拆分为：
- `A = true`（短路，B 不执行）
- `A = false && B = true`
- `A = false && B = false`（异常路径）

**注**：复杂复合条件（超过3个）可简化为"条件成立"和"条件不成立"两个目标，避免目标数量爆炸。

### 4.2 数量控制

单个文件的覆盖目标数量上限为 **50 个**。超过时：
1. 优先保留 P0/P1 目标
2. 合并同类循环边界目标
3. 提示用户文件过大，建议指定具体方法范围
