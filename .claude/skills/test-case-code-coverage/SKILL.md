# 文本用例 → 代码覆盖度分析

基于 LLM 语义推断，将纯文本测试用例描述映射到源码中的可覆盖单元，生成近似覆盖度报告。

> **重要说明**：本分析是**语义推断**，不是运行时测量。结果是概率性评估，置信度低的映射需人工复核。

| 输入 | 必须/可选 | 说明 |
|------|---------|------|
| 源代码文件 | 必须 | 支持 Java / Python / TypeScript / Go，可多文件 |
| 文本测试用例 | 必须 | Markdown 格式，含测试步骤和预期结果 |

## 核心工作流

### Phase 1：解析源码，构建覆盖目标清单

> **必须先加载** `references/analysis-rules.md` 中的「代码解析规则」章节。

逐文件扫描，提取以下可覆盖单元：

#### 1.1 函数/方法入口
每个公共方法 / 接口处理器 → 生成一个覆盖目标。

```
格式：[F] {类名}.{方法名}({参数类型列表})
示例：[F] OrderService.createOrder(CreateOrderRequest)
```

#### 1.2 条件分支
每个 if / else if / else / switch case / 三元运算符 → 每个分支独立为一个覆盖目标。

```
格式：[B] {文件}:{行号} {条件描述} → {分支方向}
示例：[B] OrderService.java:45 balance >= amount → true（余额充足）
       [B] OrderService.java:45 balance >= amount → false（余额不足）
```

#### 1.3 异常处理路径
每个 catch / error return / throw → 独立覆盖目标。

```
格式：[E] {文件}:{行号} {异常类型/错误条件}
示例：[E] OrderService.java:78 InsufficientBalanceException
```

#### 1.4 循环边界
每个 for / while / forEach → 提取三种边界状态（空集合、单元素、多元素）。

```
格式：[L] {文件}:{行号} {循环描述} → {边界状态}
示例：[L] OrderService.java:92 遍历订单列表 → 空列表
       [L] OrderService.java:92 遍历订单列表 → 多元素
```

#### 1.5 返回值分支
不同条件下的不同返回值 / 状态码 → 独立覆盖目标。

```
格式：[R] {文件}:{行号} 返回 {值/状态} ({触发条件})
示例：[R] OrderService.java:55 返回 400 (参数校验失败)
       [R] OrderService.java:60 返回 200 (创建成功)
```

**输出：覆盖目标清单**，每条记录包含：编号、类型、位置、自然语言描述。

---

### Phase 2：解析测试用例，提取覆盖意图

逐用例分析，从以下字段提取关键信息：

| 用例字段 | 提取内容 |
|---------|---------|
| 前置条件 | 初始状态（登录态、数据存在与否、余额状态等） |
| 测试步骤 | 触发动作、输入数据、操作序列 |
| 测试数据 | 具体数值、边界值、特殊值 |
| 预期结果 | 期望的系统响应、状态变化、返回值 |

**提取结构**（每条用例）：
```
用例编号：{ID}
用例名称：{名称}
触发条件摘要：{自然语言，描述"什么情况下 + 做了什么"}
期望行为摘要：{自然语言，描述"系统应该如何响应"}
数值边界标记：{如有明确数值/边界，列出}
错误场景标记：{是否为异常/错误路径}
```

---

### Phase 3：语义匹配，建立覆盖映射

> **必须参考** `references/analysis-rules.md` 中的「映射规则」章节。

对每个覆盖目标，在所有测试用例中找到匹配项：

#### 3.1 匹配算法

```
对每个覆盖目标 T：
  1. 将 T 的自然语言描述 + 上下文 作为"待匹配条件"
  2. 遍历所有用例的"触发条件摘要"和"期望行为摘要"
  3. 语义相似度判断：
     - HIGH（高置信）：用例明确描述了该分支的触发条件 + 期望结果
     - MEDIUM（中置信）：用例描述隐含覆盖了该分支（间接触发）
     - LOW（低置信）：推断可能覆盖，但用例描述不够明确
     - NONE：未找到任何覆盖该目标的用例
```

#### 3.2 映射记录格式

```
覆盖目标：[B] OrderService.java:45 balance >= amount → false（余额不足）
映射用例：TC-003 余额不足时下单失败
置信度：HIGH
映射依据：用例测试数据"余额=0，订单金额=100"+ 预期结果"返回余额不足错误"直接对应此分支
```

#### 3.3 未映射目标处理

对 NONE 的覆盖目标，分析原因：

| 原因类型 | 标识 | 说明 |
|---------|------|------|
| 用例缺失 | `[GAP-MISSING]` | 该路径确实没有对应用例，属于真实盲区 |
| 用例描述不清 | `[GAP-VAGUE]` | 可能有用例覆盖，但描述太模糊无法确认 |
| 防御性代码 | `[GAP-DEFENSIVE]` | 理论上不可达的代码路径（框架保证等） |
| 超出测试范围 | `[GAP-OUTSCOPE]` | 属于单元测试范畴，非功能测试覆盖对象 |

---

### Phase 4：生成用例-代码映射分析报告

> **报告定位说明**：本报告为语义推断结果（非运行时测量），用于发现用例盲区而非计算精确覆盖率。如需精确覆盖率，请使用 Istanbul（JS）/ Jacoco（Java）等运行时工具。

> **加载** `assets/coverage-report-template.md` 作为输出格式参考。

按模板输出报告，包含：

1. **执行摘要**：总覆盖目标数、已覆盖数、按模块分列已覆盖/未覆盖/不确定三列
   > **注意**：总覆盖率百分比为语义估算值，仅供参考。建议关注「未覆盖路径列表」而非追求百分比数字。
2. **覆盖矩阵**：每个覆盖目标 × 映射用例 的对应关系
3. **盲区清单**：`[GAP-MISSING]` 条目，按严重性排序
   > 每条 `[GAP-MISSING]` 应附「建议补充用例标题 + 优先级」，可直接作为 test-case-design Mode E 的输入。
4. **模糊区清单**：`[GAP-VAGUE]` 条目，建议补充描述
5. **精确覆盖率获取方式**：指引用户使用运行时覆盖率工具（Istanbul/Jacoco）获取精确数据，说明本报告与运行时报告的互补关系

## 使用说明

### 典型调用方式

```
请分析这批测试用例对以下代码的覆盖情况：
- 源代码：[粘贴代码 / 提供文件路径]
- 测试用例：[粘贴用例 / 提供文件路径]
```

### 分析粒度控制

| 用户指令 | 分析范围 |
|---------|---------|
| 默认 / 不指定 | 函数级 + 分支级（推荐） |
| "只看函数覆盖" | 仅分析 [F] 类型目标 |
| "只看分支覆盖" | 仅分析 [B][E] 类型目标 |
| "详细分析" | 全部类型，包含循环边界 [L] |

### 多文件处理

用户提供多个源码文件时：
1. 逐文件构建覆盖目标清单
2. 合并为统一清单，标注所属文件
3. 统一进行用例映射
4. 报告中按文件分组展示

---

## 结果解读说明

输出报告时，必须附上以下免责说明：

```
⚠️ 分析说明：
本报告基于语义推断，非运行时覆盖率测量。
- HIGH 置信映射：可靠性较高，但仍需人工抽样确认
- MEDIUM 置信映射：建议人工复核该用例是否真正触发了目标路径
- [GAP-MISSING] 盲区：优先补充对应测试用例
- [GAP-VAGUE] 模糊区：建议完善用例描述后重新分析
```
