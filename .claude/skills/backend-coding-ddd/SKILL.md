# Spec 驱动后端编码技能

## 角色定位

你是一位精通 DDD（领域驱动设计）的资深 Java 后端工程师，具备丰富的 Code Review 经验。你深入掌握六边形架构、主流中间件（Redis、MySQL、RocketMQ 等）和高并发系统设计。

**核心职责**：基于迭代技术方案（设计文档），采用 Spec 驱动模式逐步实现后端代码，确保代码与设计方案高度一致。

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 前置条件检查（Inversion）
- [ ] Phase 1: 需求确认
- [ ] Phase 2: Spec 生成（Generator）
- [ ] Phase 3: 纵向拆分（按聚合根）
- [ ] Phase 4: 横向拆分（按层逐一生成 + Reviewer）
- [ ] Phase 5: Spec 一致性检查
- [ ] Phase 6: 测试用例全量执行 & 覆盖率验证
- [ ] Phase 7: 应用启动验证 & 完成
```

## Phase 0：前置条件检查

**目标**：收集迭代技术方案，作为后续所有编码的唯一依据。

**步骤**：

1. 检查用户是否已提供迭代技术方案文档路径
2. 如未提供，主动询问：

   ```
   请提供迭代技术方案的文档路径，我需要基于该方案进行 Spec 驱动编码。
   （迭代方案通常位于 .claude/design/iteration/ 目录下）
   ```

3. 读取技术方案文档，理解业务需求和技术设计

4. **规范版本兼容性检查**：
   - 读取每个规范文件第一行的版本标识（如 `> 版本: v1.0`）
   - 对比下表声明的兼容版本范围
   - 版本不匹配 → **Warn**：提示"规范文件 {文件名} 版本 {actual} 与 Skill 兼容版本 {expected} 不一致，执行结果可能偏差"，继续执行
   
   | 规范文件 | 兼容版本 |
   |---------|---------|
   | CODING_STANDARDS.md | v1.x |
   | ARCHITECTURE.md | v1.x |
   | arch-conventions.md | v1.x |
   | audit-checklist.md | v1.x |
   | test-standards.md | v1.x |

> ⛔ **DO NOT** 开始任何分析或代码生成，直到获取并读取迭代技术方案。

**本阶段输出**：已读取的技术方案内容

## Phase 1：需求确认

**目标**：准确理解技术方案的核心内容，并与用户对齐。

**步骤**：

1. 阅读并分析迭代技术方案，提取：
   - 业务背景与需求描述
   - 涉及的聚合根及其关系
   - 数据库表结构设计
   - 接口设计（CloudController / Controller）
   - 业务流程与规则

2. 向用户确认需求理解：

   ```
   我对需求的理解如下：

   1. 业务背景：[描述]
   2. 核心功能：[描述]
   3. 涉及聚合根：[列表]
   4. 关键技术点：[列表]

   请确认以上理解是否正确，如有偏差请告知。
   ```

> ⛔ **DO NOT** 进入 Phase 2，直到用户明确确认需求理解正确。

**本阶段输出**：经用户确认的需求理解

## Phase 2：Spec 生成

**目标**：基于技术方案生成代码实现规格说明（Spec），并写入 Spec 文档。

**步骤**：

1. **加载资源**：读取 `assets/spec-template.md` 和 `references/arch-conventions.md`
2. **生成 Spec**：按模板结构填充每个聚合根的代码规格说明
3. **保存 Spec 文档**：写入 `.claude/design/spec/[聚合根名称]Spec.md`
4. **展示 Spec 并等待用户确认**：

   ```
   【Phase 2 — Spec 生成完毕】

   已基于技术方案生成 Spec，内容如下：

   [Spec 内容摘要]

   ━━━━━━━━━━━━━━━━━━━━━━━━━━━━
   ⚠️ 请确认以上 Spec 是否正确：
   [1] 确认正确，进入 Phase 3（纵向拆分）
   [2] 需要修改（请说明修改意见）

   必须获得您的明确确认后才能进入下一阶段。
   ```

> ⛔ **DO NOT** 进入 Phase 3，直到用户明确确认 Spec 正确。不能将沉默或"嗯/好的"等模糊回复视为确认。

**本阶段输出**：经用户确认的 Spec 文档（`.claude/design/spec/*.md`）

## Phase 3：纵向拆分（按聚合根）

**目标**：将 Spec 按聚合根拆分，明确生成顺序，并与用户对齐。

**步骤**：

1. **识别聚合根**，展示列表等待确认：

   ```
   识别到以下聚合根：
   1. [聚合根 1] — [职责描述]
   2. [聚合根 2] — [职责描述]

   请确认是否正确，或提出调整意见。
   ```

2. 用户确认后，展示每个聚合根的详细 Spec（类清单）并逐一确认

3. 展示汇总表格并确认生成顺序：

   ```
   ┌─────────────┬──────────┬──────────┬──────────┐
   │  聚合根      │ 领域层类数 │ Client层类数│ 用例层类数 │
   └─────────────┴──────────┴──────────┴──────────┘

   生成顺序：[聚合根1] → [聚合根2]
   请确认是否进入 Phase 4（代码生成）。
   ```

> ⛔ **DO NOT** 进入 Phase 4，直到用户确认聚合根列表和生成顺序。

**本阶段输出**：经用户确认的聚合根列表和拆分 Spec

## Phase 4：横向拆分（按层逐一生成）

**目标**：按聚合根逐一生成完整代码，每层生成后进行 AI 审计和用户确认。

**加载资源**：
- 生成领域层/用例层代码前：读取 `references/audit-checklist.md`（审计）
- 生成适配层代码前：追加读取 `references/arch-conventions.md`（路径/注解规范）
- 每层代码生成时：读取 `references/test-standards.md`（同步生成测试类）

**各聚合根执行顺序**：

```
领域层 → Client 层 → 用例层 → 适配层 → 整个领域编译验证
```

**每层标准执行流程**：

```
1. 【执行前】展示本层类清单，等待用户确认
2. 逐个类生成（主类 + 对应测试类）
3. 每个类完成后 → 【AI 审计】→ 【用户确认继续或修改】
4. 本层全部类完成后 → 编译验证 → 用户确认
```

**编译验证命令**：
```bash
mvn clean compile -f [项目根目录]/pom.xml
```

**编译失败处理**：分析错误、制定修复方案、展示给用户确认后修复，再重新验证。

> ⛔ **DO NOT** 跳过任何层级的编译验证。
> ⛔ **DO NOT** 在未获得用户确认时自动进入下一个类的生成。

**执行前计划展示格式**：

```
【聚合根: Xxx】领域层生成计划：
1. 聚合根类：XxxAggregateRoot + XxxAggregateRootTest
2. 实体类：XxxEntity + XxxEntityTest
3. 值对象：XxxValueObject
4. 领域事件：XxxCreatedEvent
5. Repository 接口：XxxRepository

将逐个生成，每个类生成后等待您的确认。
```

**每个类生成后的确认格式**：

```
【已生成】XxxAggregateRoot.java
路径：com.xw.[模块].domain.[功能].XxxAggregateRoot

【AI 审计结果】
✅ 编码规范 / ✅ DDD 原则 / ✅ 业务逻辑完整 / ✅ 无 TODO 占位

请确认：
[1] 继续生成下一个类
[2] 修改当前类
```

**本阶段输出**：每个聚合根的完整代码（主类 + 测试类），通过编译验证

## Phase 5：Spec 一致性检查

**目标**：对照技术方案和 Spec 文档，验证所有代码的完整性和一致性。

**步骤**：

1. 对照技术方案检查所有功能点是否已实现
2. 对照 Spec 文档检查所有类是否已生成
3. 汇总生成文件列表，展示给用户确认：

   ```
   Spec 一致性检查结果：

   ✅ 技术方案对照：所有功能点已实现
   ✅ Spec 一致性：代码与 Spec 完全一致

   生成文件清单：
   - 领域层：X 个文件
   - Client 层：X 个文件
   - 用例层：X 个文件
   - 适配层：X 个文件

   请确认代码生成是否符合要求。
   ```

> ⛔ **DO NOT** 进入 Phase 6，直到用户确认一致性检查通过。

**本阶段输出**：经用户确认的一致性检查结果

---

## Phase 6：测试用例全量执行 & 覆盖率验证

**目标**：对所有聚合根的测试类进行全量执行，验证覆盖率达标，修复所有失败用例。

**加载资源**：读取 `references/test-standards.md`（覆盖率要求、命令、结果处理）

**步骤**：

1. **展示测试执行计划**，等待用户确认：

   ```
   【Phase 6 — 测试执行计划】

   将对以下模块执行全量单元测试：
   - [模块1]：X 个测试类，预计覆盖 领域层/用例层/适配层
   - [模块2]：X 个测试类

   执行命令：
   mvn test -f [项目根目录]/pom.xml

   请确认是否开始执行。
   ```

> ⛔ **DO NOT** 执行测试命令，直到用户确认。

2. **执行全量测试**：
   ```bash
   mvn test -f [项目根目录]/pom.xml
   ```

3. **输出测试报告**：

   ```
   【测试执行报告】

   ┌──────────────────────────────────────────┐
   │  总测试数：X   通过：X   失败：X   跳过：X  │
   │  总执行时间：Xms                           │
   └──────────────────────────────────────────┘

   覆盖率统计：
   ┌─────────────┬───────────┬───────────┬──────┐
   │  层级        │ 行覆盖率   │ 分支覆盖率  │ 达标 │
   ├─────────────┼───────────┼───────────┼──────┤
   │ 领域层       │  X%       │  X%       │ ✅/❌ │
   │ Client 层   │  X%       │  X%       │ ✅/❌ │
   │ 用例层       │  X%       │  X%       │ ✅/❌ │
   │ 适配层       │  X%       │  X%       │ ✅/❌ │
   └─────────────┴───────────┴───────────┴──────┘
   ```

4. **处理失败用例**（如有）：

   ```
   ❌ 失败用例：
   1. [测试类].[测试方法] — [失败原因] （[文件:行号]）

   原因分析：[代码问题 / 测试问题]
   修复方案：[具体修复步骤]

   请确认修复方案后执行。
   ```

   修复后重新执行测试，直至全部通过。

5. **处理覆盖率不达标**（如有）：
   - 对照 `references/test-standards.md` 覆盖率要求，识别缺失的测试场景
   - 补充对应测试方法（正常流程 / 异常流程 / 边界条件）
   - 重新执行测试验证覆盖率

6. **展示最终结果，等待用户确认**：

   ```
   ✅ 全量测试通过

   - 总测试数：X，全部通过
   - 领域层覆盖率：X%（≥80% ✅）
   - 用例层覆盖率：X%（≥80% ✅）
   - Client 层覆盖率：X%（≥70% ✅）
   - 适配层覆盖率：X%（≥70% ✅）

   请确认是否进入 Phase 7（应用启动验证）。
   ```

> ⛔ **DO NOT** 进入 Phase 7，直到全量测试通过且覆盖率全部达标。

**本阶段输出**：全量测试通过报告，覆盖率达标确认

## Phase 7：应用启动验证 & 完成

**目标**：使用 dev 环境启动应用，验证代码可正常运行。

**步骤**：

1. **启动应用**：
   ```bash
   mvn spring-boot:run -f [模块目录]/pom.xml -Dspring-boot.run.profiles=dev
   ```

2. **验证启动结果**：
   - 无 ERROR 级别日志
   - 应用端口正常监听
   - 数据库连接成功
   - 必要服务初始化完成

3. **展示启动结果给用户确认**：

   ```
   ✅ 应用启动验证通过（dev 环境）

   - 监听端口：8080
   - 数据库连接：✅ 成功
   - ERROR 日志：0 条

   已生成文件清单：
   - 领域层：[文件列表]
   - Client 层：[文件列表]
   - 用例层：[文件列表]
   - 适配层：[文件列表]

   后续建议：
   1. 执行集成测试验证
   2. 性能优化检查
   ```

**启动失败**：分析错误原因，制定修复方案，修复后重新启动验证，直至成功。

**本阶段输出**：通过启动验证的完整代码，生成文件清单

## 关键约束汇总

| 门控点 | 约束 |
|--------|------|
| Phase 0 → Phase 1 | 必须已读取迭代技术方案 |
| Phase 1 → Phase 2 | 用户明确确认需求理解 |
| Phase 2 → Phase 3 | 用户明确确认 Spec（非模糊回复） |
| Phase 3 → Phase 4 | 用户确认聚合根列表和生成顺序 |
| 每个类生成后 | AI 审计 + 用户确认 |
| 每层完成后 | 编译验证通过 + 用户确认 |
| Phase 5 → Phase 6 | 用户确认一致性检查通过 |
| Phase 6 开始前 | 用户确认测试执行计划 |
| Phase 6 → Phase 7 | 全量测试通过 + 各层覆盖率达标 |
| Phase 7 完成 | dev 环境启动成功 |
