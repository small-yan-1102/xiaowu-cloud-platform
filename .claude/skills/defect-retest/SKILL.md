# 缺陷回归验证技能

> **版本**: v1.0
> **更新日期**: 2026-04-08
> **适用模式**: 开发修复缺陷 → AI 提取待回归列表 → 定向重执行关联用例 → 指导更新缺陷状态 → 输出回归报告

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 获取待回归缺陷清单
- [ ] Phase 2: 执行定向回归测试
- [ ] Phase 3: 指导更新缺陷状态
- [ ] Phase 4: 输出回归验证报告
```

---

### Phase 0: 定向收集

**目标**：确认缺陷来源和回归范围。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **缺陷来源**：
   - 云效实时查询（默认，需 `tools/yunxiao/config.yaml` 已配置）
   - 本地缺陷报告（默认：`bug_reports/bug-report-*.md`，取最新一份）
2. **回归范围**：
   - 所有「已解决」状态的缺陷（默认）
   - 指定缺陷编号列表
3. **迭代 ID**（可选）：用于云效查询时缩小范围

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户表示"按默认来" → 优先尝试云效实时查询，失败则回退到本地缺陷报告，进入 Phase 1

---

### Phase 1: 获取待回归缺陷清单

**目标**：确定哪些缺陷需要回归，并找到每条缺陷关联的测试用例。

**步骤**：

#### 1a. 从云效获取（优先）

检查 `tools/yunxiao/config.yaml` 配置完整性（需 `personal_access_token`、`organization_id`、`projex.project_id`、`projex.workitem_type_id`）：

```bash
# 查询所有「已解决」状态的缺陷明细
python tools/yunxiao/scripts/query_defects.py --list-resolved --json [--sprint-id <sprint_id>]
```

输出示例：
```json
[
  {"id": "WI-001", "subject": "[JQB-TEAM-042] 移除成员后列表未刷新", "case_id": "JQB-TEAM-042", "severity": "Major", "status": "已解决"},
  {"id": "WI-002", "subject": "[JQB-LOGIN-003] 密码错误提示缺失", "case_id": "JQB-LOGIN-003", "severity": "Critical", "status": "已解决"}
]
```

#### 1b. 从本地缺陷报告获取（降级）

读取 `bug_reports/bug-report-*.md`（取最新一份），提取：
- 所有缺陷记录（编号、标题、严重等级、关联用例编号 `BUG-MODULE-NNN → TC-XXX-NNN`）
- 请用户确认哪些缺陷已被开发标记为修复完成

#### 1c. 建立缺陷 → 用例映射表

对每条待回归缺陷，通过以下方式获取关联用例编号：

1. 优先从云效缺陷标题提取（格式：`[用例编号] 缺陷标题`）
2. 从本地缺陷报告的「关联测试用例」字段提取
3. 无法提取时，在缺陷标题中搜索关键词定位对应用例文件

对每条缺陷，确认：
- **用例编号**：如 `JQB-TEAM-042`
- **用例类型**：UI（在 `test_cases_*.md` 中）或 接口（在 `api_test_cases_*.md` 中）
- **所在文件路径**：通过 Glob 搜索 `**/*test_cases*.md` 定位

**输出待回归清单**：

| 缺陷 | 严重等级 | 关联用例 | 用例类型 | 用例文件 |
|------|---------|---------|---------|---------|
| [JQB-TEAM-042] 移除成员后列表未刷新 | Major | JQB-TEAM-042 | UI | test_cases_team_management.md |
| [JQB-LOGIN-003] 密码错误提示缺失 | Critical | JQB-LOGIN-003 | UI | test_cases_login.md |

> ⛔ **DO NOT** 进入 Phase 2，直到所有待回归缺陷均已建立用例映射。
> 若有无法映射的缺陷，列出并请用户确认是否跳过。

---

### Phase 2: 执行定向回归测试

**目标**：仅执行与待回归缺陷关联的测试用例，不重跑完整测试套件。

**步骤**：

#### 执行顺序原则

- **严重缺陷优先**：Fatal → Critical → Major → Minor
- **同文件合并**：同一用例文件中的多个待回归用例，合并为一次执行

#### 对每条待回归用例：

1. **读取用例文件**，定位到对应用例编号的用例内容
2. **执行前数据验证**：
   - 检查用例前置条件是否满足（参照原执行报告中的阻塞原因）
   - 测试数据已就绪 → 继续；数据缺失 → 标注「跳过：数据未就绪」
3. **执行用例**：
   - UI 用例 → 通过 browser-use MCP 操作浏览器执行
   - 接口用例 → 发送 HTTP 请求，断言响应
4. **记录结果**：
   - ✅ 通过：记录实际结果与期望一致
   - ❌ 失败：记录失败步骤、实际结果、截图（UI）或响应体（接口）
   - ⏭️ 跳过：记录跳过原因（数据未就绪 / 环境问题 / 无法映射用例）

#### 执行结果汇总（Phase 2 末尾输出）

```
回归执行完成:
  ✅ 通过: N 条
  ❌ 失败: M 条
  ⏭️ 跳过: K 条
```

> ⛔ **DO NOT** 进入 Phase 3，直到所有待回归用例均已执行或明确跳过。

---

### Phase 3: 指导更新缺陷状态

**目标**：根据回归结果，明确告知测试人员每条缺陷在云效中应做的操作。

**按结果分类输出操作指引**：

#### ✅ 回归通过 → 关闭缺陷

对每条通过的缺陷，输出：

> **[缺陷编号] {缺陷标题}**
> 回归结果：**通过**
> 建议操作：在云效将工作项状态更新为「**已关闭**」
> 云效工作项 ID：{WI-xxx}（如已知）

#### ❌ 回归失败 → 重新打开缺陷

对每条失败的缺陷，输出：

> **[缺陷编号] {缺陷标题}**
> 回归结果：**失败**
> 失败详情：步骤 {N} —— 期望「{expected}」，实际「{actual}」
> 建议操作：在云效将工作项状态更新为「**待处理**」，并在评论中注明「回归失败（{date}）: {actual}」
> 云效工作项 ID：{WI-xxx}（如已知）

#### ⏭️ 跳过 → 保持当前状态

> **[缺陷编号] {缺陷标题}**
> 跳过原因：{reason}
> 建议操作：暂不变更云效状态，待条件满足后重新回归

**给出所有操作指引后，询问用户**：
> 「以上缺陷状态是否已在云效更新完成？确认后生成回归报告。」

---

### Phase 4: 输出回归验证报告

**目标**：按 `assets/retest-report-template.md` 生成回归报告，并评估是否可重新发起 release-gate。

**步骤**：

1. **读取报告模板**：加载 `assets/retest-report-template.md`

2. **确定输出路径**：`test_reports/retest-report-YYYYMMDD.md`
   - 同日期已有报告 → 追加序号：`retest-report-YYYYMMDD-2.md`

3. **填写报告各章节**（参照模板结构）：
   - 回归概览（总数 / 通过 / 失败 / 跳过）
   - 逐条回归结果（通过清单 / 失败清单 / 跳过清单）
   - 残留风险（仍开放的 Fatal/Critical 缺陷）
   - 后续建议

4. **评估后续行动**，并告知用户：

   | 情况 | 建议 |
   |------|------|
   | 所有 Fatal/Critical 均已关闭 | → 建议重新触发 `release-gate` |
   | 仍有 Fatal 开放（含回归失败）| → 不建议发布，需开发继续修复 |
   | 仍有 Critical 开放 | → 视业务影响决定，建议触发 `release-gate` 评估 |
   | 有跳过项（含 Fatal/Critical）| → 标注待复测，谨慎发布 |

5. **向用户展示**：
   - 报告文件路径
   - 回归概览数字
   - 是否建议进入 `release-gate`

## 验收标准

### 覆盖完整性

- [ ] 所有「已解决」状态的缺陷均已回归（或明确跳过并注明原因）
- [ ] 每条缺陷均有明确的回归结论（通过 / 失败 / 跳过）
- [ ] 有失败用例时，失败详情已记录（步骤 / 期望 / 实际）

### 状态指引准确性

- [ ] 通过的缺陷已给出「关闭」操作指引，附云效工作项 ID（如已知）
- [ ] 失败的缺陷已给出「重新打开」指引，含回归失败说明文本
- [ ] 未对任何结果做假设（如「应该通过」）—— 实际执行后才给结论

### 报告完整性

- [ ] 报告已写入 `test_reports/retest-report-YYYYMMDD.md`
- [ ] 给出明确的后续行动建议（是否重新触发 release-gate）

## 与其他 Skill 的衔接

| 上游 Skill/工具 | 提供给本 Skill | 说明 |
|----------------|--------------|------|
| `bug-sync` | 缺陷清单（`bug_reports/bug-report-*.md`） | 缺陷-用例映射数据源（降级时使用）|
| `tools/yunxiao/scripts/query_defects.py` | 已解决缺陷列表（`--list-resolved`）| 云效实时状态，优先使用 |
| `test-execution` | 浏览器执行能力 | Phase 2 执行 UI 用例回归 |
| `api-test-execution` | HTTP 执行能力 | Phase 2 执行接口用例回归 |

| 下游 Skill | 触发条件 | 说明 |
|-----------|---------|------|
| `release-gate` | 所有 Fatal/Critical 回归通过后 | 重新评估发布条件 |
| `test-report` | 需要更新整体测试报告时 | 纳入回归结果后重新生成 |
