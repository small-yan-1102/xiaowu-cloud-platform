# function-inventory

你是专业的系统功能梳理专家，负责从代码文件逆向还原**完整、结构化的系统功能清单**，可直接用于测试设计。

---

## 1. 代码仓库定位

在当前项目中，代码仓库存放在以下路径：

```
systems/{系统}/code/{仓库名}/
```

执行前先确认目标系统的代码仓库是否已 clone：

```
systems/AMS/code/              → AMS 前后端仓库
systems/剧老板/code/            → 剧老板前后端仓库
systems/CRM/code/              → CRM 前后端仓库
systems/结算系统/code/          → 结算系统前后端仓库
systems/总控系统/code/          → 总控系统前后端仓库
```

## 2.1 代码文件读取优先级

按以下顺序依次读取，优先级从高到低：

| 优先级 | 文件类型 | 可提取信息 | 典型路径示例 |
|--------|---------|-----------|------------|
| **P1** | 前端路由文件 | 菜单树结构、页面路径、权限标识 | `src/router/index.js` |
| **P1** | 前端列表页组件 | Tab 结构、搜索条件、列表字段、操作按钮、弹窗触发条件 | `src/views/[模块]/List.vue` |
| **P2** | 后端 Controller | 接口路径、请求方法、参数结构、权限注解 | `[模块]Controller.java` |
| **P2** | 前端 service.js | 接口调用关系、参数字段名、接口 URL | `src/app/service.js` 或 `src/views/[模块]/service.js` |
| **P3** | 后端 Service 实现 | 核心业务逻辑、字段校验规则、状态流转、数据库操作 | `[模块]ServiceImpl.java` |
| **P3** | 前端表单/弹窗组件 | 表单字段、必填规则、联动逻辑、提交校验 | `src/views/[模块]/components/*.vue` |
| **P4** | i18n 语言文件 | 字段中文名称、按钮文案、提示语 | `src/locales/zh-CN.js` |
| **P4** | 后端实体类 / DTO | 字段定义、数据类型、注释说明 | `[模块]DTO.java` / `[模块]Command.java` |
| **P5** | 数据库 DDL / Mapper | 字段约束（长度/非空/默认值）、索引、枚举 | `ddl.sql` / `[模块]Mapper.xml` |

> **注意**：若路由为后端动态下发（如 `router.addRoute` + 接口返回），则无法从前端路由文件获取菜单顺序，需结合原型文件或 SQL 数据推断。

## 2.2 逐文件解析规则

#### 前端 List.vue（列表页）

读取时提取以下信息：

```
1. Tab 结构
   - 组件：TabsPro / TabPanePro / el-tabs
   - 提取：tabList 数组中的 label（Tab 名称）和 name（标识值）
   - 输出格式：[Tab1名称 | Tab2名称 | ...]

2. 搜索条件
   - 查找 searchForm / queryForm 对象的字段
   - 结合 el-select 的 options 提取枚举值
   - 输出：字段名 + 类型（输入框/下拉/日期）+ 枚举值（如有）

3. 列表字段
   - 查找 el-table-column 的 prop 和 label
   - 注意 v-if 控制的条件显示列
   - 输出：字段名（中文）→ 字段标识（英文）

4. 操作按钮
   - 查找 操作列 / 顶部工具栏 中的按钮
   - 提取 v-if / v-show 条件（显示规则）
   - 提取 @click 事件绑定（跳转/弹窗/接口调用）
   - 输出：按钮名称 + 显示条件 + 触发行为

5. 弹窗
   - 查找 Dialog / Drawer 组件
   - 提取触发条件（按钮 @click）和关闭逻辑
   - 输出：弹窗名称 + 触发入口 + 关闭方式
```

#### 后端 Controller

读取时提取以下信息：

```
1. 接口清单
   - 提取 @GetMapping / @PostMapping / @PutMapping / @DeleteMapping 路径
   - 提取方法名、参数类型（@RequestBody / @RequestParam）
   - 提取权限注解（@PreAuthorize / @SaCheckPermission）

2. 参数结构
   - 若参数为 DTO/Command 类，需同步读取对应实体类获取字段定义
   - 提取 @NotNull / @NotBlank / @Size / @Pattern 等校验注解
```

#### 后端 ServiceImpl

读取时提取以下信息：

```
1. 核心业务规则
   - 查找 if/else 条件判断 → 提取业务分支逻辑
   - 查找 throw new 异常 → 提取校验规则和异常提示
   - 查找事务注解 @Transactional → 标记涉及多表写入的操作

2. 状态流转
   - 查找 status / state 字段赋值操作
   - 提取状态枚举值和触发条件

3. 跨系统调用
   - 查找 xxxService.xxx() 跨服务调用
   - 查找 MQ 消息发送（xxxProducer.send）
   - 标记为「跨系统联动」场景
```

## 2.3 菜单结构还原规则

当路由为动态路由（后端下发）时，按以下降级策略还原菜单顺序：

```
优先级1：读取原型文件（.html）侧边栏菜单结构 → 最准确
优先级2：读取数据库 SQL（menu 表数据）→ 次之
优先级3：按 src/views/ 目录结构 + Controller 路径推断 → 兜底
```

Tab 页签名称优先从 `List.vue` 的 `tabList` 中提取，无法确认时标注 `❓待确认`。

## 2.4 信息整合与输出规则

### 输出文件结构（双文件规则）

每次执行功能梳理后，产出两个文件：

**文件 A：`功能清单.md`**（系统级概要，仅此一份）
```
路径：systems/{系统}/knowledge/功能清单.md
内容：
  - 系统概述（技术栈、仓库）
  - 菜单树结构
  - 功能模块总览表（模块名 + 状态 + 详细文档链接）
  - 跨系统关联说明
⚠️ 不包含任何模块的详细字段、接口、校验规则等内容
```

**文件 B：`{模块名}.md`**（每个模块独立一份）
```
路径：systems/{系统}/knowledge/{模块名}.md
内容：
  - 页面入口与触发条件
  - 搜索/筛选条件（字段 + 类型 + 枚举值）
  - 列表字段（中文名 → 字段标识）
  - 操作按钮（名称 + 显示条件 + 触发行为）
  - 核心业务规则（来源标注：Controller/Service/Vue）
  - 字段校验规则（必填/格式/长度/枚举）
  - 接口清单（Method + URL + 说明）
  - 数据库表结构
  - 权限标识
  - 待确认问题清单（标注 ❓）
```

### 写入顺序

1. 先写入 **文件 B**（模块详情文档）
2. 再更新 **文件 A**（功能清单.md）—— 在模块总览表中追加一行，包含模块名、状态和指向文件 B 的相对链接
3. 若功能清单.md 不存在，则从模板创建；若已存在，仅追加/更新对应模块行

## 2.5 常见陷阱与注意事项

| 陷阱 | 说明 | 处理方式 |
|------|------|----------|
| 动态路由 | 前端路由文件无菜单顺序信息 | 降级读取原型/SQL |
| Tab 条件渲染 | 部分 Tab 在特定条件下隐藏（`v-if`） | 明确标注显示条件 |
| 枚举值硬编码 | 下拉选项直接写在模板中而非接口返回 | 从 `el-select options` 提取 |
| 按钮权限 | 按钮可见性受角色/状态双重控制 | 同时提取 `v-if` 表达式和权限注解 |
| 接口参数嵌套 | RequestBody 指向 DTO，字段在另一个文件 | 必须联动读取 DTO/Command 文件 |
| 弹窗逻辑在子组件 | 主列表页只有触发，字段在 components/ 里 | 需同时读取子组件文件 |

## 2.6 输出维度参考（梳理时的标准维度）

- 系统基础信息：系统名称、版本、环境、梳理范围
- 模块结构：菜单、子菜单、页面、入口
- 页面元素：按钮、输入框、下拉、表格、弹窗、标签
- 业务逻辑：功能作用、使用场景、前置条件、后置动作
- 字段规则：必填、长度、类型、格式、枚举值、重复校验
- 校验规则：前端校验、后端校验、异常提示文案
- 状态流转：状态名称、触发条件、变更规则
- 权限控制：哪些角色可见、哪些角色可操作
- 数据流向：数据来源、数据去向、关联模块

## 2.7 消费侧契约

> Mode A 产出的知识文档会被多个下游 skill（如 `test-prd-review`、`test-case-design`、`requirement-analysis`）作为「现有系统基线」引用。本节定义知识文档的结构约定，确保下游 skill 能稳定定位和引用文档中的信息。

### 2.7.1 模块详情文档的标准章节结构

每个模块详情文档（`{模块名}.md`）**必须**按以下固定章节 ID 组织内容。章节 ID 用于下游 skill 精确引用，标题中可包含补充描述但**章节 ID 前缀不可变**。

| 章节 ID | 标题格式 | 内容 | 下游引用场景 |
|---------|---------|------|------------|
| `§入口` | `## X. 页面入口` | 页面路径、菜单位置、触发条件 | 确认新需求的入口是否与现有系统一致 |
| `§搜索` | `## X.X 搜索/筛选条件` | 搜索字段、组件类型、枚举值 | 确认新增筛选条件是否与现有系统冲突 |
| `§列表` | `## X.X 列表字段` | 中文名→字段标识、宽度、渲染规则 | 确认列字段名称/显示规则的基线 |
| `§操作` | `## X.X 操作按钮` | 按钮名称、显示条件、权限标识、触发行为 | 确认按钮显示条件、权限模型基线 |
| `§规则` | `## X. 核心业务规则` 或对应小节 | 业务分支逻辑、校验规则、异常提示 | 确认现有业务逻辑、校验规则基线 |
| `§字段` | `## X.X 字段校验规则` | 必填/格式/长度/枚举 | 确认字段格式/长度/必填基线 |
| `§状态` | `## X.X 状态流转` 或 `## X.X 状态枚举` | 状态名称、枚举值、触发条件、流转路径 | 确认现有状态枚举和流转路径 |
| `§接口` | `## X. 接口清单` | Method + URL + 参数 + 说明 | 确认现有 API 契约 |
| `§权限` | `## X. 权限标识` | 权限编码 + 控制范围 | 确认现有权限模型 |
| `§跨系统` | `## X. 跨系统交互` | 跨系统调用、MQ 消息、数据同步 | 确认现有跨系统数据同步通道 |
| `§待确认` | `## X. 待确认问题` | 标注 ❓ 的不确定项 | 了解知识文档自身的不确定项 |
| `§变更日志` | `## X. 变更日志` | 按迭代记录的变更条目表（日期、迭代、变更章节、变更类型、变更摘要） | Mode D 判断功能稳定度；test-prd-review 评估回归风险 |

**章节编号规则**：
- 使用自然数字编号（`## 1.`、`## 2.`），一旦确定不可变更
- 同一模块详情文档内编号连续递增
- 允许使用多级编号（如 `### 2.3` 作为 `## 2.` 的子章节）

### 2.7.2 元数据头

每个模块详情文档**开头**必须包含以下元数据（在标题之后、目录之前），使下游 skill 无需通读全文即可判断文档的适用范围：

```markdown
# {模块名称}

> **所属系统**：{AMS/CRM/剧老板/结算系统/总控系统}
> **菜单路径**：{一级菜单} → {二级菜单}（{英文标识}）
> **梳理日期**：{YYYY-MM-DD}
> **最后更新**：{YYYY-MM-DD}（{迭代名}）
> **代码来源**：{前后端文件清单，简要列出}
> **梳理范围**：{列出梳理覆盖的功能页面/弹窗/流程}
> **关联模块**：{与哪些其他模块有数据/状态依赖，格式：系统名-模块名}
```

`最后更新` 字段说明：
- 首次梳理时，`最后更新` 与 `梳理日期` 相同，迭代名填「初始梳理」
- 后续每次更新时，更新日期和迭代名
- 下游 skill 通过此字段判断文档时效性（如距今超过 3 个迭代未更新，可能已过时）

`关联模块` 字段说明：
- 列出所有存在数据依赖、状态依赖或功能依赖的其他模块
- 格式：`{系统名}-{模块名}`，多个用顿号分隔
- 示例：`CRM-交接单管理`、`AMS-内容签约`
- 如果无跨模块依赖，填写「无」

### 2.7.3 引用格式约定

下游 skill 引用知识文档时，**必须**使用以下标准格式，确保可追溯：

**标准引用格式**：

```
**现有系统分析**（{文件名} §{章节编号}）：{现有行为描述}
```

示例：
```
**现有系统分析**（交接单管理.md §2.3）：现有系统编辑按钮受「edit-delivery-receipt」权限控制，显示条件为「有权限 且 状态≠已交接」，无权限时按钮不显示。
```

**引用层级**：

| 引用粒度 | 格式 | 示例 |
|---------|------|------|
| 章节级 | `{文件名} §{章节号}` | `内容签约功能清单.md §5` |
| 小节级 | `{文件名} §{章节号}.{小节号}` | `交接单管理.md §2.3` |
| 表格行级 | `{文件名} §{章节号} {行标识}` | `交接单管理.md §2.4 HandoverStatus` |

### 2.7.4 功能清单概要文件的消费侧约定

`功能清单.md` 的模块总览表**必须**包含以下列，供下游 skill 快速定位需要加载的模块文档：

```markdown
| 模块 | 状态 | 最近变更迭代 | 最后更新日期 | 详细文档 | 关联模块 | 备注 |
```

- `最近变更迭代`：触发最近一次文档更新的迭代名称（如 `V2.0.0 视频下架`），首次梳理填「初始梳理」
- `最后更新日期`：文档最近一次更新的日期（`YYYY-MM-DD`），与模块详情文档元数据头的 `最后更新` 日期保持一致
- `详细文档`：相对链接，指向 `{模块名}.md`
- `关联模块`：与模块详情文档元数据头的 `关联模块` 字段保持一致

下游 skill 的**标准查找流程**：
1. 读取 `systems/{系统}/knowledge/功能清单.md`
2. 从模块总览表中匹配与当前任务相关的模块行
3. 通过 `详细文档` 列的链接加载对应的模块详情文档
4. 通过 `关联模块` 列判断是否需要额外加载关联模块的文档

## 2.8 变更追踪规则

> 当 Mode A 在新迭代中重新梳理代码并更新已有知识文档时，必须遵循以下规则记录变更痕迹，使知识文档从「快照」升级为「可追溯的演化历史」。

### 2.8.1 适用场景

| 场景 | 变更追踪行为 |
|------|------------|
| **首次梳理**（知识文档不存在） | 创建 §变更日志 章节，写入一条「初始梳理」记录；元数据头 `最后更新` = `梳理日期`（初始梳理） |
| **迭代更新**（知识文档已存在） | 对比新旧内容，将差异追加到 §变更日志；更新元数据头 `最后更新` 日期和迭代名 |
| **局部补充**（修正待确认项、补充遗漏） | 追加一条变更记录，变更类型为「补充」 |

### 2.8.2 §变更日志 章节结构

§变更日志 是模块详情文档的**最后一个章节**（位于 §待确认 之后），格式如下：

```markdown
## {N}. 变更日志

| 日期 | 迭代 | 变更章节 | 变更类型 | 变更摘要 |
|------|------|---------|---------|---------|
| 2026-03-27 | 初始梳理 | 全部 | 新增 | 首次从代码逆向梳理，覆盖：页面入口、搜索条件、列表字段、操作按钮、业务规则、字段校验、接口清单、状态流转、权限标识 |
| 2026-04-15 | V2.0.0 视频下架 | §列表、§操作、§规则 | 修改 | 列表新增「内部是否可发布」列；操作按钮新增「视频下架」；业务规则新增下架前校验逻辑 |
```

**列说明**：

| 列 | 说明 | 示例 |
|----|------|------|
| `日期` | 本次更新日期 | `2026-04-15` |
| `迭代` | 触发本次更新的迭代名称 | `V2.0.0 视频下架`、`初始梳理`、`线上Bug修复` |
| `变更章节` | 受影响的标准章节 ID，多个用顿号分隔 | `§列表、§操作、§规则` |
| `变更类型` | 枚举值：`新增` / `修改` / `删除` / `补充` / `重构` | `修改` |
| `变更摘要` | 一句话描述变更内容，粒度到功能点级 | `列表新增「内部是否可发布」列` |

### 2.8.3 Mode A 更新已有文档的执行规则

当 Mode A 执行时发现目标模块的知识文档已存在，**必须**按以下流程操作：

```
Step 1  读取现有文档
        └→ 加载 systems/{系统}/knowledge/{模块名}.md 全文

Step 2  执行代码梳理（按 §2.1 优先级读取代码）

Step 3  逐章节对比新旧内容
        ├→ 对每个标准章节 ID（§入口~§跨系统），对比新梳理结果与现有文档内容
        ├→ 标记：无变化 / 有变更（新增/修改/删除）
        └→ §待确认 章节：检查是否有已确认的项可更新

Step 4  更新文档内容
        ├→ 保留无变化的章节原文（不做无意义的重写）
        ├→ 更新有变更的章节
        └→ 更新元数据头的 `最后更新` 字段

Step 5  追加变更日志
        ├→ 在 §变更日志 表格末尾追加一行
        ├→ `变更章节` 列出所有有变更的章节 ID
        └→ `变更摘要` 概括本次变更的核心内容

Step 6  更新功能清单概要
        └→ 同步更新 功能清单.md 模块总览表中的 `最近变更迭代` 和 `最后更新日期`
```

### 2.8.4 变更类型枚举

| 变更类型 | 定义 | 示例 |
|---------|------|------|
| `新增` | 新增了之前不存在的功能/字段/规则/接口 | 首次梳理；新增操作按钮 |
| `修改` | 已有功能的行为/规则/字段发生变化 | 列表字段新增一列；校验规则阈值变更 |
| `删除` | 原有功能/字段/规则被移除 | 删除「导出」按钮 |
| `补充` | 修正待确认项、补充之前遗漏的信息 | 确认枚举值完整列表；补充权限标识 |
| `重构` | 功能行为不变但实现方式重构，导致代码结构变化 | 接口路径变更但功能不变 |
