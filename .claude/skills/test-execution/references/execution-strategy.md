> **替换此文件即可改变执行行为规范，无需修改 SKILL.md 中的指令逻辑。**

## 一、浏览器操作规范

### 1.1 元素定位优先级

AI 定位页面元素时，按以下优先级依次尝试：

| 优先级 | 定位方式 | 说明 |
|-------|---------|------|
| 1 | **可见文本** | 按钮、链接、标签等的显示文字 |
| 2 | **aria-label / role** | 无障碍属性标识 |
| 3 | **placeholder / title** | 输入框的占位文字或提示 |
| 4 | **相对位置描述** | 基于上下文的位置关系 |

**禁止**：使用 CSS 选择器、XPath、data-testid 等技术定位符——AI 直接执行模式不依赖 DOM 结构。

### 1.2 操作类型规范

| 操作类型 | 规范要求 |
|---------|---------|
| 点击 | 确保元素可见且可交互后再点击；被遮挡元素需先滚动到可视区域 |
| 文本输入 | 先清空输入框已有内容，再输入新值 |
| 下拉选择 | 先点击下拉框展开，再选择目标选项；等待下拉列表完全展开 |
| 文件上传 | 通过文件选择对话框上传；确认上传成功提示出现 |
| 拖拽 | 明确起点和终点元素；等待拖拽动效完成 |
| 滚动 | 目标元素不在可视区域时先滚动；长列表懒加载时需等待数据加载 |

### 1.3 键盘操作

| 场景 | 推荐操作 |
|------|---------|
| 表单字段间切换 | Tab 键 |
| 提交表单 | Enter 键（当焦点在提交按钮上时） |
| 关闭弹窗 | Escape 键（作为点击关闭按钮的备选） |
| 搜索触发 | Enter 键（在搜索框输入后） |

## 二、等待策略

### 2.1 核心原则

**永远不要使用固定时间等待（如"等待3秒"）**，应基于页面状态信号判断操作时机。

### 2.2 页面状态等待信号

| 等待信号 | 适用场景 |
|---------|---------|
| 加载指示器消失 | 页面加载、数据请求 |
| 目标元素出现 | 页面跳转、弹窗打开 |
| 文本内容变化 | 状态更新、数据刷新 |
| URL 变化 | 页面导航 |
| 网络请求完成 | AJAX 数据加载 |

### 2.3 超时设置

| 操作类型 | 默认超时 |
|---------|---------|
| 元素出现/可见 | **10 秒** |
| 页面导航完成 | **15 秒** |
| 文件上传 | **30 秒** |
| 数据加载（列表/表格） | **15 秒** |

超时后不立即标记失败，先执行一次**页面刷新重试**（见§三）。

### 2.4 SPA 路由切换

单页应用（SPA）路由切换完成判断信号（按顺序）：
1. URL hash 或路径变化
2. 页面主要内容区域更新
3. 导航菜单高亮状态切换
4. 面包屑导航内容更新

## 三、错误恢复与重试策略

### 3.1 重试决策矩阵

| 错误类型 | 处理策略 | 最大重试次数 |
|---------|---------|------------|
| 元素未找到 | 滚动页面 → 等待 → 重试点击 | 2 次 |
| 页面加载超时 | 刷新页面 → 重新执行当前步骤 | 1 次 |
| 意外弹窗遮挡 | 关闭弹窗 → 继续当前操作 | 1 次 |
| Session 过期 | 重新登录 → 导航回当前页面 → 重试 | 1 次 |
| 页面白屏/崩溃 | 刷新页面 → 检查是否恢复 | 1 次 |
| 网络请求失败 | 等待 5 秒 → 刷新页面 → 重试操作 | 1 次 |
| 验证码/人工介入 | 暂停并通知用户 | 不重试 |

### 3.2 重试原则

- **幂等性检查**：重试前确认之前的操作是否已经生效（如表单是否已提交成功），避免重复提交
- **重试上限**：单个步骤最多重试 2 次，超过则标记该步骤失败并继续执行后续步骤
- **整体熔断**：同一用例中累计 3 个步骤重试失败，终止该用例执行，标记为环境不稳定

### 3.3 意外弹窗处理优先级

1. Cookie 同意弹窗 → 点击"接受"或"同意"按钮
2. 系统通知/推送弹窗 → 点击关闭按钮或空白区域关闭
3. 首次登录引导弹窗 → 点击"跳过"或"我知道了"
4. 浏览器权限弹窗（通知、位置等）→ 拒绝或关闭
5. 无法识别的弹窗 → 截图记录 → 尝试 Escape 关闭 → 如仍存在，标记为阻塞问题

### 3.4 自愈边界规则

> **核心原则**：环境层问题可以自愈重试，业务层失败必须如实报告。禁止通过自愈机制掩盖真正的业务 Bug。

#### 问题分层定义

| 层级 | 典型现象 | AI 可否自愈 |
|------|---------|-----------|
| **环境层** | 页面加载超时、Session 过期、网络抖动、弹窗遮挡、元素懒加载未完成 | ✅ 可自愈 |
| **业务层** | 预期状态变更未发生、金额计算错误、权限验证失效、流程走不通 | ❌ 禁止自愈 |

#### 环境层 vs 业务层判断规则

| 判断信号 | 归类 |
|---------|------|
| 页面显示 Loading 超时 | 环境层 |
| 页面显示错误码（500/502/504） | 环境层 |
| 按钮点击无响应（JS 报错） | 环境层 |
| toast 提示"操作成功"但数据未变更 | 业务层 |
| 预期状态变更未发生 | 业务层 |
| 金额/积分数值与预期不符 | 业务层 |
| 元素不存在（页面正常加载的情况下） | 业务层 |

#### 自愈禁区（绝对不可自愈的场景）

| 场景 |
|------|
| L1 强验证预期结果与实际不符 |
| 确定性断言（`[断言: API]` / `[断言: DB-Query]`）失败 |
| 权限验证失败（能访问不应访问的资源） |
| 数据写入后查询不到 |
| 操作后状态未按预期变更 |

## 四、测试环境准备

### 4.1 浏览器初始状态

每次测试套件执行前，确保浏览器处于干净状态：

| 初始化项 | 操作 |
|---------|------|
| 窗口大小 | 设置为 1920×1080（桌面标准分辨率） |
| 语言设置 | 与被测系统语言一致（默认中文） |

### 4.2 登录态管理

| 策略 | 适用场景 |
|------|---------|
| **每次重新登录** | 用例间状态隔离要求高时 |
| **复用登录态** | 同模块连续用例执行时（需确认 Session 有效） |
| **多账号切换** | 权限验证类用例（需先登出再登入不同账号） |

**默认策略**：同一测试套件内复用登录态，每个套件开始时重新登录。执行过程中 Session 失效则自动重新登录。

### 4.3 部署预检（执行测试前必须通过）

目标环境部署完成后、执行测试或线上回归之前，AI 自动执行以下预检。预检不通过则不启动后续测试。

| # | 预检项 | 验证方式 | 失败处理 |
|---|-------|---------|---------|
| 1 | 目标环境可访问 | 访问目标环境 URL，页面返回非错误状态 | 终止执行，报告"环境不可用" |
| 2 | 登录功能正常 | 使用测试账号执行登录操作，验证登录成功 | 终止执行，报告"登录失败" |
| 3 | 核心页面可加载 | 登录后访问本次变更涉及的主要页面，页面无白屏、无 JS 报错 | 终止执行，报告"页面加载异常"并截图 |

**预检结果判定**：3 项全部通过 → 进入测试执行；任一项不通过 → 终止，输出预检报告。

## 五、AI 判断校准

### 5.1 文本匹配规则

| 匹配类型 | 规则 |
|---------|------|
| 精确匹配 | 预期结果明确指定完整文本时，必须看到完整这句话 |
| 包含匹配 | 预期结果使用"包含""出现"等描述时，任意位置出现关键词即可 |
| 忽略空格 | 对比文本时忽略前后空格和多余空白 |
| 大小写敏感 | 英文内容默认大小写敏感（除非用例特别说明） |

> **注意**：当前端实际渲染文案与 PRD 存在差异时（空格、标点、全角/半角等），以技术文档中标注的实际渲染值为准进行匹配。

### 5.2 数值比较容差

| 场景 | 容差规则 |
|------|---------|
| 精确金额（如积分余额） | 必须精确匹配，无容差 |
| 百分比显示 | 允许 ±1% 的显示误差 |
| 时间/日期显示 | 允许与当前时间 ±1 分钟误差 |
| 列表数量 | 必须精确匹配（如"共10条"） |

### 5.3 间歇性失败（Flaky）处理

当某个验证步骤失败时：

1. **首次失败**：截图记录失败现场（CP1）→ 等待 2 秒 → 重新检查一次
   - 重新检查通过 → 标记该步骤为 ⚡ Flaky，继续执行后续步骤
   - 重新检查失败 → 进入第 2 步

2. **二次失败**：截图记录二次失败现场（CP2）→ 刷新页面 → 重新检查一次
   - 重新检查通过 → 标记该步骤为 ⚡ Flaky，继续执行后续步骤
   - 重新检查失败 → 进入第 3 步

3. **三次失败**：截图记录最终失败现场（CP1）→ 确认标记为 ❌ 确认失败（Confirmed Failure），终止当前用例

每次重试截图文件名使用 `retry1` / `retry2` 后缀区分。

**用例结果分类规则**：

| 结果状态 | 判定条件 | 缺陷处理 |
|---------|---------|---------|
| ✅ 通过 | 所有步骤首次检查通过，无重试 | 无 |
| ⚡ 通过（含 Flaky）| 所有步骤最终通过，但 ≥1 个步骤经重试后通过 | 不创建缺陷；记录到执行报告「Flaky 警告」章节 |
| ❌ 确认失败 | 存在步骤三次均失败 | 创建缺陷（bug-sync 处理）|
| 🚫 阻塞 | 前置条件/数据就绪验证失败，用例无法执行 | 按环境问题处理，不创建功能缺陷 |

> **Flaky 处理原则**：⚡ Flaky 表示用例不稳定但未真正失败，不创建功能缺陷。执行报告的「Flaky 警告」章节按模块汇总，由测试工程师周期性评估修复方向（常见原因：等待时间不足、断言依赖动态内容、测试数据竞争）。

### Flaky 升级规则

报告生成时，统计本迭代内各用例的 Flaky 累计次数：

1. **扫描历史报告**：检索 `test_reports/test_report_*.md` 和 `test_reports/smoke_check_*.md`（当前迭代内的所有报告）
2. **计算累计次数**：统计同一用例编号在本迭代内出现 ⚡ Flaky 的报告总数
3. **升级判定**：

   | 累计 Flaky 次数 | 处理方式 |
   |--------------|---------|
   | 1 次 | 正常追踪，记入报告「Flaky 警告」章节 |
   | ≥ 2 次（本迭代内） | 🔺 **标记为"需关注 Flaky"**，在报告中高亮；通知 `bug-sync` 创建稳定性缺陷 |

4. **升级标注格式**（在执行报告「Flaky 警告」章节中）：
   ```
   🔺 [需关注] {用例编号} {用例名称}
      本迭代内累计 Flaky：N 次
      建议：由 bug-sync 创建稳定性缺陷，优先级 Major
   ```

### 5.4 确定性断言执行规则

#### 断言类型识别与执行

| 断言类型 | 标识 | 执行方式 | 失败处理 |
|---------|------|---------|---------|
| 视觉断言 | `[断言: 视觉]` 或无标识 | 通过 `browser_snapshot` 观察页面内容 | 按 Flaky 处理流程 |
| API 断言 | `[断言: API]` | 解析接口地址/参数/预期响应，执行 HTTP 调用 | **禁止重试**，失败即报告 |
| DB 断言 | `[断言: DB-Query]` | 解析 SQL 语句/预期结果，执行只读查询 | **禁止重试**，失败即报告 |

#### API 断言执行流程

```
1. 解析预期结果中的 API 断言块（接口地址 / 请求参数 / 预期响应 / 比对字段）
2. 构造 HTTP 请求（使用当前浏览器的登录态 Cookie/Token，GET 请求超时 10 秒）
3. 执行请求并解析响应
4. 按比对字段规则对比（精确匹配 / 包含匹配 / 数值比较）
5. 结果判定：匹配→通过；不匹配→失败，记录期望值和实际值；请求失败→标记"API调用失败"
```

#### DB 断言执行流程

```
1. 解析预期结果中的 DB 断言块（SQL语句 / 预期结果 / 比对规则）
2. 安全检查：确认为 SELECT 语句 + 有 WHERE 条件 + 有 LIMIT 限制
3. 执行查询（通过配置的数据库连接）
4. 按比对规则对比查询结果与预期结果
5. 结果判定：匹配→通过；不匹配→失败；查询失败→标记"DB查询失败"
```

#### 混合断言执行顺序

1. **视觉断言优先**：先验证 UI 层面的反馈
2. **API 断言次之**：验证接口返回的精确值
3. **DB 断言最后**：验证数据库层的最终状态

## 六、单条用例执行流程

```
1. 读取用例（解析编号、前置条件、步骤、预期结果）
      ↓
2. 检查前置（确认登录态、页面位置、数据准备策略）
      ↓
3. 数据就绪验证（按 [验证: ...] 标签确认数据存在且状态正确）
   - 无 [验证:] 标签 → 跳过，直接进入步骤 4
   - 验证通过 → 继续
   - 数据缺失 → 自愈（1次）→ 成功则继续 / 失败则阻塞
   - 状态异常 → 标记阻塞，进入下一用例
      ↓
4. 【CP1 必截】起始截图（执行第一个操作步骤之前）
      ↓
5. 逐步执行（操作 → 等待稳定 → browser_snapshot 验证）
   - 关键状态变更时【CP1 必截】
   - L1 失败时【CP1 必截】
      ↓
6. 【CP1 必截】最终截图（最后一步验证完成之后）
      ↓
7. 记录结果（通过/失败/警告 + 截图文件名 + 耗时）
      ↓
8. 状态恢复（为下一条用例准备干净的初始状态）
```

### 异常中断处理

| 中断类型 | 处理 |
|---------|------|
| 单步骤失败（非阻塞） | 记录失败，继续执行后续步骤 |
| L1 验证失败 | 记录失败，**终止当前用例**，进入下一用例 |
| 连续 3 步重试失败 | 标记环境不稳定，终止当前用例 |
| 人工介入点 | 暂停执行，通知用户 |

### 数据就绪验证规则

| 数据准备策略 | 验证方式 | 自愈策略 |
|------------|---------|---------|
| `[数据准备: UI自建]` | 导航到验证特征指定页面，搜索/查看数据 | 执行前置条件中的创建步骤（最多1次）|
| `[数据准备: API]` | 优先调用 GET 接口查询，备选 UI 验证 | 重新调用创建接口（最多1次）|
| `[数据准备: 依赖用例]` | 检查依赖用例执行状态 + 验证数据存在 | 不可自愈，标记阻塞 |
| `[数据准备: 环境预置]` | 导航到验证特征指定页面，确认数据存在 | 不可自愈，标记阻塞 |

## 七、截图检查点执行规范（双证据体系）

### 7.1 核心原则

AI 执行测试时，**必须（MUST）** 同时产生两种证据：

| 证据类型 | 工具 | 用途 |
|---------|------|------|
| **文本证据** | `browser_snapshot` | 精确文本断言（通过/失败判定依据） |
| **视觉证据** | `browser_take_screenshot` | 人可审计的页面截图 |

**判定依据仍然是 `browser_snapshot` 文本**——截图不参与通过/失败判定，仅作审计附件。

### 7.2 三级检查点模型

**CP1 必截（MUST）— 缺失则报告不完整**

| # | 场景 | 说明 |
|---|------|------|
| 1 | 用例起始状态 | 第一个操作步骤之前 |
| 2 | 用例最终状态 | 最后一步验证完成之后 |
| 3 | L1 断言失败时 | 立即截图保留现场 |
| 4 | 关键状态变更 | toast 出现、弹窗确认、列表增减等 |
| 5 | 异常/错误现场 | 意外弹窗、白屏、网络错误等 |
| 6 | L1 断言通过时 | 每个 L1 断言验证通过后立即截图，证明业务目标已达成；文件名状态字段用 `pass`。用例有多个 L1 断言时每个均截，步骤标识按实际步骤编号区分 |
| 7 | 页面导航完成后 | URL 发生变化且新页面内容已加载后截图，确认落点正确；步骤标识用 `nav`。同一步骤触发多次导航时每次均截 |

**CP2 条件截（SHOULD）— 满足条件时截图**

| # | 场景 | 触发条件 |
|---|------|---------|
| 1 | 重试操作 | 进入重试流程时（重试前后各一张） |
| 2 | L3 观察性验证 | 遇到 L3 验证点时 |
| 3 | 数据准备操作 | 执行非测试步骤的数据准备时 |
| 4 | 数据就绪自愈 | 数据就绪验证失败触发自愈时（前后各一张） |
| 5 | 破坏性操作完成后 | 删除确认完成、状态变更为终态（已关闭/已作废/已注销）、权限撤销生效后，在后续 L1 验证步骤执行之前截一张中间状态图；步骤标识用 `dest`（destructive） |

**CP3 可选截（MAY）** — AI 自主判断有审计价值时截图。

### 7.3 截图命名规范

```
{CP级别}_{用例编号}_{步骤标识}_{状态}_{时间戳}.png
```

| 字段 | 可选值 |
|------|-------|
| CP级别 | `CP1` / `CP2` / `CP3` |
| 步骤标识 | `init` / `step01` / `nav` / `dest` / `final` / `dataprep` |
| 状态 | `start` / `pass` / `fail` / `retry1` / `error` |

> 步骤标识说明：`nav` = 页面导航落点；`dest` = 破坏性操作完成后中间状态

**示例**：

- **路径**：`test_reports/screenshots/YYYY-MM-DD/`

- CP1 覆盖率低于 95% 时，报告中必须标注告警

| # | 熔断触发条件 | 处理动作 |
|---|------------|---------|

## 十、用例间变量传递（`[输出:]` → `{引用:}`）

### 10.1 机制概述

当上游用例产生系统自动生成的值（ID、编号、URL 参数等），而下游依赖用例需要精确引用该值时，使用本机制实现用例间数据传递，避免重复造数据。

```
上游用例执行 → 从页面提取 [输出:] 声明的值 → 写入运行时变量表
                                                      ↓
下游依赖用例执行前 → 将 {引用:变量名} 替换为运行时变量表中的实际值
```

### 10.2 变量表优先级

当同一变量名在多个来源中存在时，按以下优先级取值（高优先级覆盖低优先级）：

| 来源 | 优先级 |
|------|-------|
| `test-data-preparation` Mode A 生成的 data-prep-report | 最高（1）|
| 用例执行产生的 `[输出:]` 捕获值 | 次之（2）|
| `{引用:变量名}` 处理时的默认值（如有）| 最低（3）|

命名冲突时（同名变量在不同来源均有值），data-prep-report 的值始终胜出，并在就绪清单中输出 ⚠️ 命名冲突警告。

### 10.3 提取规则

**提取时机**：步骤 7b，即用例最终截图之后、状态恢复之前。

**提取方式**：优先使用 `browser_snapshot` 文本读取，按提取描述中给出的位置和格式特征定位值。

| 提取描述类型 | 提取方式 |
|-----------|---------|
| 页面可见文字（标题、标签、表格单元格）| `browser_snapshot` 文本匹配 |
| URL 参数（如 `order_id=xxx`）| 读取当前页面 URL，解析对应参数 |
| Toast / 提示文字中的值 | `browser_snapshot` + 正则匹配 |
| 表单回显值 | `browser_snapshot` 读取对应输入框的 value |

**提取结果处理**：

| 情况 | 处理 |
|------|------|
| 找到唯一匹配值 | 写入运行时变量表，标注 `[捕获成功]` |
| 找到多个候选值 | 取第一个，标注 `[捕获: 取第一个匹配]` |
| 未找到任何匹配 | 标注 `⚠️ [输出] 捕获失败`，依赖下游用例标记 🚫 阻塞 |

### 10.4 下游用例处理规则

执行到含 `{引用:变量名}` 的用例时：

1. 查找运行时变量表中该变量名的值
2. **已填充**：替换后正常执行
3. **待填充**（上游用例尚未执行，按拓扑排序不应出现）→ **Fatal**：报告执行顺序错误，终止
4. **捕获失败**：用例自动标记 🚫 阻塞（不尝试执行），阻塞原因：「`{引用:变量名}` 的上游输出变量捕获失败」

### 10.5 失败传播规则

| 上游用例状态 | 下游用例处理 |
|------------|------------|
| ✅ 通过 + 变量捕获成功 | 正常执行 |
| ✅ 通过 + 变量捕获失败 | 🚫 阻塞（不计入失败率）|
| ❌ 失败 | 🚫 阻塞（已由依赖图判定，原因：依赖用例失败）|
| 🚫 阻塞 | 下游用例同样 🚫 阻塞（级联）|

> **阻塞不等于失败**：变量捕获失败或上游用例失败导致的下游阻塞，不计入用例失败率，不触发缺陷创建流程。执行报告中独立列出阻塞原因，供测试工程师排查。
