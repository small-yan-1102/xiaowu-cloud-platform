# AI 测试执行技能

> **版本**: v2.0
> **更新日期**: 2026-04-02
> **适用模式**: AI 读取 Markdown 测试用例 → 通过 browser-use MCP 直接操作浏览器 → 自主判断结果 → 输出执行报告

---

## 附属资源文件

| 文件 | 路径 | 说明 |
|------|------|------|
| 执行策略 | `references/execution-strategy.md` | 浏览器操作规范、等待策略、错误恢复、自愈边界规则、截图规范、生产环境约束，Phase 2 执行时加载 |
| 执行报告模板 | `assets/execution-report-template.md` | 单条用例结果模板 + 汇总报告模板 + 线上回归报告模板 + 证据索引模板，Phase 3 输出时加载 |

---

## 执行模式

| 模式 | 触发特征 | 说明 |
|------|---------|------|
| **冒烟验证（Mode S）** | 包含"冒烟验证"/"冒烟检查"/"独立冒烟"/"提测冒烟" | 只跑 Mode D 冒烟套件（P0），输出通过/打回结论，不进入全量执行 |
| **标准执行** | 指定测试用例文件，无"线上/生产"关键词 | 在测试环境执行全量用例，含内部 P0 门禁 |
| **线上回归（D5）** | 包含"线上回归"/"D5"/"生产环境"/"发版验证" | 在生产环境执行，额外执行 §八 生产约束和熔断机制 |

> **Mode S 与标准执行的区别**：标准执行内部的 P0 门禁是流程中的一步；Mode S 是独立调用，只跑冒烟套件，失败直接输出"打回开发"结论，不继续后续步骤。

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 依赖检查与预执行准备
- [ ] Phase 2: 执行测试用例
- [ ] Phase 3: 输出执行报告
```

### Phase 0: 定向收集

**目标**：在任何执行行为开始前，确认测试范围和环境配置。

向用户提出以下问题，**等待用户全部回答后再进行任何操作**：

1. **测试用例文件路径**：要执行的测试用例文件是什么（默认：`test_suites/*.md`）？
2. **执行范围**：执行全部用例，还是指定优先级（P0/P1）或特定模块？
3. **环境配置**：
   - 测试环境 URL 是什么？
   - 测试账号（用户名/密码）是什么？
   - 如果是线上回归：生产环境 URL？生产测试账号？关联发版版本？
4. **构建版本号**：本次提测的构建版本标识是什么？（如 commit hash / tag / build 号 / 版本号，用于在冒烟报告中记录并计算同版本连续失败次数；无法提供时填"未知"）
   > **版本号获取**：优先从页面 footer/about 读取，次选从 API 响应头（X-Version/X-Build）获取，均失败则提示用户手动输入。
5. **数据准备报告**（如有）：是否已有数据准备报告（`test_suites/data-prep-report-*.md`）？

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 启动浏览器，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 冒烟验证模式（Mode S）→ 进入 Phase 1，完成后执行 Phase 2-S 和 Phase 3-S，**不执行** Phase 2 和 Phase 3
- 标准执行模式 → 进入 Phase 1 → Phase 2 → Phase 3
- 线上回归模式 → 进入 Phase 1 → Phase 2（含 §八 生产约束）→ Phase 3

### Phase 2-S: 冒烟验证执行（Mode S 专属）

**目标**：只执行 Mode D 冒烟套件（P0 用例），输出明确的通过/打回结论。

**步骤**：

1. **确认冒烟套件文件**：
   - 查找 `test_suites/smoke_*.md` 或 Phase 0 中用户指定的路径
   - 未找到 → 提示用户指定 Mode D 冒烟套件文件路径

2. **加载 P0 用例**：
   - 只读取优先级为 P0 的用例
   - 统计 P0 用例总数

3. **执行部署预检**（同 Phase 1 §5）：
   - 目标环境可访问、登录正常、核心页面可加载
   - 预检不通过 → 直接输出「环境异常，无法冒烟」，终止

4. **逐条执行 P0 用例**（含单次重试规则）：
   - 记录**执行开始时间**（用于判断是否触发"快速失败"阈值：30 分钟内出现 P0 确认失败，视为代码提交前未经充分自测）
   - 按 `references/execution-strategy.md` §六 规范执行
   - 每条用例执行后：
     - **通过** → 记录 ✅，继续下一条
     - **失败/阻塞** → **立即重试 1 次**（过滤环境抖动）：
       - 重试后通过 → 记录 ⚡ 环境抖动（计入通过，但在报告中单独标注）
       - 重试后仍失败 → 记录 ❌ 确认冒烟失败，加入打回清单
   - 出现确认失败后**不中断**，继续执行其余 P0 用例，收集全量失败清单

   > **重试上限**：每条用例最多重试 **1 次**，不做 Flaky 判定，冒烟结论只有通过/失败两态。

5. **读取连续失败次数**：
   - 检索 `test_reports/smoke_check_*.md`，按 Phase 0 提供的**构建版本号**精确匹配（而非按日期）
   - 匹配规则：报告头部 `构建版本` 字段 = 当前版本号 → 计为同版本失败记录
   - 构建版本号为"未知"时 → 降级策略：按本次提测日期当天的报告数量估算，在报告中注明「版本号未知，失败次数为当日估算值」
   - 统计已有几次确认冒烟失败（不含通过和升级处理后停止计入的记录）

6. **判定冒烟结论**：

   | 情况 | 连续失败次数 | 快速失败（≤30分钟） | 结论 |
   |------|------------|-------------------|------|
   | 全部 P0 通过（含环境抖动） | — | — | ✅ 冒烟通过，可进入全量执行 |
   | 有 P0 确认失败 | 第 1 次 | 否 | ❌ 冒烟失败，打回开发修复 |
   | 有 P0 确认失败 | 第 1 次 | 是（≤30分钟） | ❌ 冒烟失败，打回并在报告中标注「⚠️ 快速失败：主流程 P0 在 30 分钟内暴露，代码提交前自测不足」 |
   | 有 P0 确认失败 | 连续 2 次（均为快速失败） | 是 | 🚨 升级处理：下次提测须由**技术负责人陪同走查**后，方可重新触发提测门禁 |
   | 有 P0 确认失败 | 第 2 次及以上（非快速失败） | 否 | 🚨 升级处理，禁止继续单纯打回重提 |

   **升级处理触发后的动作**：
   - 输出升级通知，明确标注「同版本第 N 次冒烟失败」及是否触发"快速失败"路径
   - **连续 2 次快速失败路径**：通知技术负责人，要求陪同开发走查代码后确认签字，**测试门禁不接受下次提测，除非提供技术负责人走查记录**
   - **其他升级路径**：暂停提测流程，组织团队会议排查根因（环境问题？设计缺陷？回归策略问题？），不再由测试单方面打回，需团队共同决策

> ⛔ **DO NOT** 执行任何 P1/P2/P3 用例。Mode S 只验证核心链路可用性。

### Phase 3-S: 冒烟验证结论输出（Mode S 专属）

**目标**：输出轻量冒烟验证报告，结论明确。

**步骤**：

1. **输出冒烟报告**，路径：`test_reports/smoke_check_YYYYMMDD_HHmmss.md`

   报告包含：
   - 冒烟结论：✅ 通过 / ❌ 失败 / 🚨 升级处理
   - 本次为同版本第 N 次冒烟（N ≥ 2 时高亮标注）
   - P0 用例总数 / 通过数 / ⚡ 环境抖动数 / ❌ 确认失败数
   - **失败用例清单**（用例编号 + 失败步骤 + 现象描述 + 是否重试后仍失败）
   - 环境信息（URL + 执行时间）
   - **构建版本**（来自 Phase 0 用户提供；填"未知"时标注）

2. **向用户展示**：

   ```
   冒烟验证结论：[✅ 通过 / ❌ 失败（第 N 次）/ 🚨 升级处理（第 N 次，已达阈值）]
   P0 用例：{总数} 条，通过 {n}，⚡ 环境抖动 {n}，❌ 确认失败 {n}

   [如失败，第 1～2 次]
   失败用例：
   - {用例编号} {用例名称}：{失败步骤} — {现象描述}
   建议：通知开发修复上述问题后重新提测。

   [如失败，第 1 次但为快速失败（≤30分钟）]
   ⚠️ 快速失败：主流程 P0 在执行开始 {elapsed} 分钟内暴露，代码提交前自测不足。
   建议：通知开发认真自测（主流程 + 异常场景）后重新提测。

   [如失败，连续 2 次快速失败]
   🚨 同版本已连续 {N} 次快速失败，已触发 AI 代码质量红线。
   下次提测须由技术负责人陪同走查代码并签字确认后，方可重新触发提测门禁。
   测试部门不接受该开发人员的下次提测，除非附技术负责人走查记录。

   [如失败，第 2 次及以上（非快速失败）]
   ⚠️ 同版本已连续冒烟失败 {N} 次，请勿继续单纯打回重提。
   建议：暂停提测，组织团队会议排查根因后再决策。

   [如通过]
   建议：执行全量测试（api-test-execution → test-data-preparation → test-execution）。
   ```

### Phase 1: 依赖检查与预执行准备

**目标**：确认资源文件存在，执行部署预检，加载数据准备报告。

**步骤**：

1. **检查执行策略文件**：
   - 检查 `references/execution-strategy.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「执行策略文件缺失」

2. **检查报告模板文件**：
   - 检查 `assets/execution-report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「报告模板文件缺失」

3. **读取测试用例文件**：
   - 加载用户指定的用例文件
   - 识别并分类标注：
     - `[执行方式: 人工]` 用例 → 单独列为**人工执行清单**，不进入 AI 执行队列；从 `test_suites/data-prep-report-*.md` §9b 中提取其变量实际值
     - `[数据准备: 依赖用例]` 用例 → 提取前置条件中的依赖用例编号，建立**依赖图**
     - `[输出: 变量名 = 提取描述]` 声明 → 扫描所有用例的预期结果，收集输出变量声明，建立**输出变量清单**（变量名 + 提取描述 + 所属用例编号），标记为「待填充」状态
   - 排列执行顺序：先按优先级分组（P0 → P1 → P2 → P3），同优先级内按依赖关系拓扑排序，确保前序用例早于依赖它的用例执行；若存在循环依赖 → **Fatal**：终止执行，报告循环依赖链
   - 统计用例总数、各优先级分布、AI 执行用例数、人工执行用例数

4. **构建运行时变量表**：
   - 读取 `test_suites/data-prep-report-*.md`（如有），提取变量映射表作为初始值
   - 将步骤 3 中扫描到的 `[输出:]` 声明合并入运行时变量表，初始值标记为「待填充」
   - 命名冲突检查：若 `[输出:]` 变量名与 data-prep-report 中的变量名重复，在就绪清单中输出 ⚠️ 命名冲突警告（data-prep-report 中的值优先，`[输出:]` 捕获结果不覆盖）
   - 运行时变量表在整个套件执行期间持续更新，每条用例执行后写入新的捕获值

5. **执行部署预检**（参照 `references/execution-strategy.md` §四「4.3 部署预检」）：
   - ✅ 目标环境可访问
   - ✅ 登录功能正常
   - ✅ 核心页面可加载
   - 预检不通过 → **Fatal**：终止执行，输出预检报告，通知用户排查环境

6. **输出就绪清单**：已加载 N 条用例，环境预检通过，进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到部署预检通过且所有依赖均已就绪。

### Phase 2: 执行测试用例

**目标**：按照 `references/execution-strategy.md` 中的规范，逐条执行测试用例，记录结果。

**步骤**：

1. **读取执行策略**：加载 `references/execution-strategy.md`，确认当前模式的执行约束

2. **按套件执行顺序执行**：

   ```
   冒烟测试 (P0) → 全部通过才继续
        ↓ 有失败 → 终止执行，报告冒烟失败
   E2E 测试 (P1)
        ↓
   异常测试 (P2)
        ↓
   边界测试 (P3)
   ```

3. **对每条用例，按以下流程执行**（详见 `references/execution-strategy.md` §六）：

   - **步骤 0**：检查执行方式标记
     - 用例标注 `[执行方式: 人工]` → **跳过 AI 执行**，记录结论「⏭️ 人工执行（待人工完成）」，加入人工执行清单，直接进入步骤 8
     - 用例无标记或标注 `[执行方式: AI]` / `[执行方式: AI+人工]` → 继续执行后续步骤
   - **步骤 1**：读取用例，解析编号、前置条件、步骤、预期结果；替换前置条件中的 `{引用:变量名}` 为数据准备报告中的实际值
   - **步骤 2**：检查前置条件（登录态、页面位置、数据准备策略）
   - **步骤 3**：数据就绪验证（按 `[验证: ...]` 标签执行；无标签则跳过）

     **验证失败的结论分类**：

     | 验证失败场景 | 结论 | 原因 |
     |------------|------|------|
     | `[数据准备: API]` 的 `[验证:]` 失败 | 🚫 **阻塞** | 数据准备未就绪，用例本身无问题 |
     | `[数据准备: 环境预置]` 的 `[验证:]` 失败 | 🚫 **阻塞** | 环境配置缺失，用例本身无问题 |
     | `[数据准备: UI自建]` 的 `[验证:]` 失败 | 🚫 **阻塞** | 人工创建数据步骤未完成 |
     | 步骤中的 L1 断言失败 | ❌ **失败** | 用例逻辑缺陷或被测功能 Bug |

     > 阻塞（🚫）≠ 失败（❌）：阻塞不计入用例失败率，不触发缺陷创建；失败才触发缺陷流程。

   - **步骤 4**：【CP1 必截】起始截图
   - **步骤 5**：逐步执行操作（操作 → 等待稳定 → `browser_snapshot` 验证）；L1 断言失败 → 终止该用例，记录 ❌；L2 断言失败 → 继续执行，记录 L2 警告；L3 → 仅记录观察值
   - **步骤 6**：【CP1 必截】最终截图
   - **步骤 7**：记录单条用例结果（参照 `assets/execution-report-template.md` §模板一）
   - **步骤 7b**：提取输出变量（如当前用例有 `[输出:]` 声明）

     | 当前用例状态 | 处理方式 |
     |------------|---------|
     | ✅ 通过 / ⚡ Flaky 通过 | 使用 `browser_snapshot` 按提取描述定位并读取变量值，写入运行时变量表 |
     | ❌ 确认失败 | 跳过提取，下游依赖用例已因依赖失败标记 🚫 阻塞，无需额外处理 |
     | 🚫 阻塞 | 跳过提取，同上 |

     **提取成功**：写入运行时变量表（`变量名 = 实际值`），后续用例通过 `{引用:变量名}` 使用。

     **提取失败**（用例通过但无法从页面找到对应值）：
     - 在执行记录中追加 `⚠️ [输出] 捕获失败: {变量名}（提取描述: {描述}）`
     - 将依赖该变量（`{引用:变量名}`）的所有下游用例标记为 🚫 阻塞，阻塞原因：「上游输出变量 {变量名} 捕获失败」
     - **不中断**当前套件执行，继续执行不依赖该变量的用例

   - **步骤 8**：状态恢复（为下一条用例准备干净初始状态）

4. **执行规范遵守**：
   - 元素定位按优先级（可见文本 > aria-label > placeholder > 相对位置）
   - 等待信号优先于固定时间等待
   - 错误恢复按重试决策矩阵执行
   - 自愈边界规则：环境层问题可自愈，业务层失败必须如实报告

5. **线上回归模式额外约束**（参照 `references/execution-strategy.md` §八「生产环境执行约束」）：
   - 严格遵守只读优先、幂等安全、数据隔离原则
   - 监控熔断触发条件，出现任一情况立即停止
   - 每步操作记录审计日志

> ⛔ **DO NOT** 进入 Phase 3，直到所有待执行用例均已执行完毕（通过/失败/阻塞/跳过均记录完整）。

### Phase 3: 输出执行报告

**目标**：按 `assets/execution-report-template.md` 中的模板，生成完整的执行报告。

**步骤**：

1. **读取报告模板**：加载 `assets/execution-report-template.md`

2. **确定报告类型和路径**：

   | 模式 | 报告模板 | 输出路径 |
   |------|---------|---------|
   | 标准执行 | 模板二：测试执行汇总报告 | `test_reports/test_report_YYYYMMDD_HHmmss.md` |
   | 线上回归（D5） | 模板三：线上回归报告 | `test_reports/prod_regression_report_YYYYMMDD.md` |

   > **与 test-result-sync 的衔接**：test-result-sync 的默认输入路径为 `test_reports/test_report_*.md`（标准执行）或 `test_reports/prod_regression_report_*.md`（线上回归）。如存在多个文件，取最新生成的一份。

3. **填写报告内容**：
   - 执行概览（总数 / ✅ 通过 / ⚡ Flaky（重试后通过）/ ❌ 失败 / 🚫 阻塞 / ⏭️ 人工执行 / 跳过）
   - **运行时变量表**（如本次执行有 `[输出:]` 声明）：列出所有输出变量的实际捕获值（格式：`{变量名}` = `{实际值}` [来源: 用例编号]；捕获失败的标注 `[捕获失败]`）；供下次执行或调试时参考
   - 证据摘要（CP1/CP2/CP3 覆盖率，人工执行用例不计入截图统计）
   - 冒烟测试结果 + 结论
   - E2E/回归测试结果
   - 失败用例详情（❌ 确认失败，提供给 bug-sync）
   - 🚫 阻塞用例清单（独立于失败，说明阻塞原因：数据未就绪 / 环境预置缺失 / 人工创建步骤未完成）
   - ⏭️ 人工执行用例清单（用例编号 + 名称 + 变量实际值速查，供测试工程师人工执行参考）
   - ⚡ Flaky 警告汇总（按模块列出，注明步骤编号和重试次数，供测试工程师评估）
   - L2 警告汇总（**醒目提示**：报告摘要中必须包含 `⚠️ 本次执行有 N 条 L2 警告未处理`，N>0 时）
   - 环境问题记录
   - 审计清单（逐项自查后勾选）
   - 线上回归模式额外填写：操作审计日志、回归结论

4. **生成证据索引文件**（模板四）：
   - 路径：`test_reports/screenshots/YYYY-MM-DD/evidence_index.md`
   - 记录所有截图文件清单和统计

5. **向用户展示**：报告路径 + 执行概览摘要 + 失败用例列表 + CP1 覆盖率

6. **数据清理引导**（仅标准执行模式；线上回归 D5 跳过，其回滚已在用例内处理）：

   **触发条件**：Phase 0 中用户提供了数据准备报告，或 Phase 1 成功加载了 `test_suites/data-prep-report-*.md`。

   满足条件时，在向用户展示报告后输出：

   ```
   ---
   📋 数据清理提醒

   本次执行使用了数据准备报告：{data-prep-report 路径}
   测试数据尚未清理，残留数据可能影响下次执行的唯一性校验。

   如需清理，请触发 test-data-preparation Mode B：
     → 说"清理测试数据"或"数据清理"
   ```

   未使用数据准备报告（用例均为 `[数据准备: UI自建]` / `[数据准备: 依赖用例]` / `[数据准备: 环境预置]`）→ 跳过本步骤，不输出提示。

## 验收标准

### 冒烟验证（Mode S）

- [ ] 只执行了 P0 用例，未执行 P1/P2/P3
- [ ] 每条失败用例已重试 1 次，重试结果已记录（⚡ 环境抖动 或 ❌ 确认失败）
- [ ] 冒烟报告已写入 `test_reports/smoke_check_*.md`
- [ ] 报告中已标注构建版本号（或注明"未知"）
- [ ] 报告中已标注本次为同版本第 N 次冒烟（按构建版本号精确计算）
- [ ] 结论三态明确：✅ 通过 / ❌ 失败（打回）/ 🚨 升级处理
- [ ] 失败时已记录执行开始时间，并判断是否为快速失败（≤30分钟）
- [ ] 快速失败已在报告中单独标注「⚠️ 快速失败」及耗时
- [ ] 连续 2 次快速失败已触发技术负责人陪同走查要求，并在报告中注明下次提测前置条件
- [ ] 非快速失败的升级处理（第 2 次+）：已输出升级通知，建议暂停提测并组织团队排查

### 执行完整性

- [ ] 所有待执行用例均已记录执行结果（通过 / 失败 / 阻塞 / ⏭️ 人工执行 / 跳过）
- [ ] `[执行方式: 人工]` 用例已记录为「⏭️ 人工执行（待人工完成）」，未尝试 AI 浏览器执行
- [ ] `[数据准备: 依赖用例]` 用例已按依赖关系拓扑排序执行，不存在前序用例晚于依赖用例执行的情况
- [ ] 数据就绪验证失败一律标记为「🚫 阻塞」而非「❌ 失败」，阻塞原因已说明
- [ ] 有 `[输出:]` 声明的用例执行通过后，已尝试从页面提取变量值并写入运行时变量表
- [ ] 输出变量捕获失败时，已标注 ⚠️ 警告并将依赖该变量的下游用例标记为 🚫 阻塞（原因说明完整）
- [ ] 报告中已包含「运行时变量表」小节（如本次执行有 `[输出:]` 声明）
- [ ] 每条 AI 执行用例有单条执行结果记录（含步骤明细）
- [ ] CP1 截图覆盖率 ≥ 95%（人工执行用例不计入分母）
- [ ] 每条失败用例附有 ≥ 2 张截图（失败前 + 失败时）
- [ ] 每个 L1 断言通过时均有对应 CP1 截图（步骤标识含实际步骤编号，状态为 `pass`）
- [ ] 每次页面导航完成后均有对应 CP1 截图（步骤标识含 `nav`）
- [ ] 破坏性操作（删除/终态变更/权限撤销）完成后、L1 验证前有对应 CP2 截图（步骤标识含 `dest`）

### 报告完整性

- [ ] 汇总报告已写入正确路径
- [ ] 证据索引文件（`evidence_index.md`）已生成
- [ ] 报告中的截图路径可访问（相对路径正确）
- [ ] 截图命名符合 `{CP级别}_{用例编号}_{步骤}_{状态}_{时间戳}.png` 规范
- [ ] 使用了数据准备报告时，Phase 3 步骤 6 已输出数据清理引导提示

### 线上回归专项

- [ ] 操作审计清单已完整填写并逐项自查
- [ ] 所有 PROD 写操作已记录并确认回滚
- [ ] 测试标记数据（`_AITEST_`）已清理
- [ ] 未执行任何 `[ENV: TEST]` 标记用例
- [ ] 回归结论（通过/部分通过/建议回滚）已给出
