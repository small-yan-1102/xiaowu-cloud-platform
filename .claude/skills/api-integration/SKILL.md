# 接口联调执行技能

> **设计模式**：Inversion（收集范围）→ Pipeline（收集上下文 → 规则预读 → 逐字段对照 → 执行联调 → 验证），差异必须逐一确认

**⚠️ 核心约束：本阶段专注于接口联调执行，直接修改代码替换 Mock 数据。**

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集（Inversion）
- [ ] Phase 1: 收集需求上下文
- [ ] Phase 2: 规则预读
- [ ] Phase 3: Mock 数据与接口文档对照分析
- [ ] Phase 4: 执行联调修改
- [ ] Phase 5: 联调验证（Generator）
```

### Phase 0: 范围收集（Inversion 模式）

**⚠️ 硬门控：DO NOT proceed until scope is confirmed.**

使用 AskUserQuestion 收集：
- API 文档路径
- 对应的 mock 数据路径
- 相关页面代码路径
- 版本号

---

### Phase 1: 收集需求上下文

- 读取 API 设计文档
- 读取现有 mock 数据定义
- 读取相关页面代码（service 文件和组件）

---

### Phase 2: 规则预读

加载项目编码规范和组件使用规范。

---

### Phase 3: Mock 数据与接口文档对照分析

**⚠️ 关键步骤：逐字段对照，发现差异立即暂停询问**

1. **枚举值对照**：对比 mock 枚举与接口文档的枚举定义
2. **接口参数对照**：对比请求参数的字段名、类型、是否必填
3. **接口返回对照**：对比 mock 数据结构与接口文档返回结构
4. **差异处理**：
   - **发现任何差异** → 立即暂停，向用户确认
   - **接口文档缺失** → 立即暂停，询问用户
   - **接口定义有疑问** → 立即暂停，向用户澄清

**⚠️ 硬门控：DO NOT proceed to Phase 4 until all differences are confirmed.**

**差异对照输出格式**：

```markdown
## 接口对照：{接口名称}

### 请求参数对照
| 字段 | Mock 定义 | 接口文档 | 状态 |
|------|----------|---------|------|
| id | `taskId` | `task_id` | ⚠️ 不一致 |

### 返回结构对照
| 字段 | Mock 定义 | 接口文档 | 状态 |
|------|----------|---------|------|

### 枚举对照
| 枚举名 | Mock 定义 | 接口文档 | 状态 |
|--------|----------|---------|------|
```

---

### Phase 4: 执行联调修改

**⚠️ 硬门控：DO NOT proceed to Phase 5 until all modifications are complete.**

根据 Phase 3 的对照分析结果,直接修改代码:

1. **修改 service.js**:
   - 取消注释真实接口代码
   - 修改 `const USE_MOCK = true` → `const USE_MOCK = false`
   - 修改请求参数(如字段名不一致)
   - 修改响应数据处理(如结构不一致)

2. **删除 Mock 数据**:
   - 删除 `mock/` 目录(如有)
   - 删除 service.js 中的 `const MOCK_XXX = [...]`
   - 删除 `delay` 函数
   - 删除所有 `if (USE_MOCK)` 分支

3. **清理代码**:
   - 删除调试代码(console.log)
   - 清理注释掉的代码
   - 确保代码格式规范

---

### Phase 5: 联调验证（Generator 模式）

**检查清单**:

1. ✅ **接口是否全部对接完成**
   - [ ] 所有接口都切换成真实接口
   - [ ] 无遗漏的接口
   - [ ] 接口调用参数正确

2. ✅ **Mock 数据是否完全删除干净**
   - [ ] mock/ 目录已删除
   - [ ] service.js 中的 Mock 数据已删除
   - [ ] USE_MOCK 已改为 false
   - [ ] 无遗留的 Mock 相关代码

3. ✅ **对项目的影响是否完全消除**
   - [ ] 无调试代码(console.log)
   - [ ] 无注释代码未清理
   - [ ] 无影响其他模块
   - [ ] 代码格式规范

4. ✅ **是否可以执行测试**
   - [ ] 联调通过
   - [ ] 数据展示正常
   - [ ] 错误处理完善
   - [ ] 可以进入测试阶段

**输出**: 联调验证报告

```markdown
# 接口联调验证报告 - {模块名}

**联调时间**: {YYYY-MM-DD HH:mm}
**接口文档**: {路径}

## 联调概览

| 检查项 | 状态 | 说明 |
|--------|------|------|
| 接口对接 | ✅ | 所有接口已对接完成 |
| Mock清理 | ✅ | Mock数据已完全删除 |
| 代码影响 | ✅ | 无遗留影响 |
| 测试就绪 | ✅ | 可以执行测试 |

## 接口对接清单

| 接口 | 状态 | 修改内容 |
|------|------|----------|
| GET /api/users | ✅ | 字段名调整 |
| POST /api/users | ✅ | 无修改 |

## 结论

**联调状态**: ✅ 完成
**测试就绪**: ✅ 可以进入测试阶段
```

---

## 核心约束

- **严禁跳过对照**：必须逐字段对比 mock 数据与接口文档
- **严禁自行定义**：不确定的字段、枚举值必须暂停询问
- **严禁字段映射**：mock 数据必须与接口文档完全一致，禁止前端做字段转换映射
- **接口文档优先**：以接口文档为准，mock 数据必须同步修改
