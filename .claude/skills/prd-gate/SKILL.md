## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载 PRD 审阅报告
```

---

### Phase 0: 定向收集

**目标**：确认 PRD 审阅报告位置。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **PRD 审阅报告路径**：test-prd-review 生成的审阅报告位于哪里？（默认：`review/test-prd-review-*.md`，取最新一份）

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

**决策点**：
- 用户回答或表示"按默认来" → 使用默认路径，进入 Phase 1

---

### Phase 1: 加载 PRD 审阅报告

**目标**：读取审阅报告，确认其可用且包含必要结构。

**步骤**：

1. **检查门禁报告模板**：
   - 检查 `assets/gate-report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「门禁报告模板文件缺失」

2. **读取 PRD 审阅报告**：
   - 加载用户指定路径（默认取 `review/test-prd-review-*.md` 中最新一份）
   - 不存在 → **Fatal**：终止执行，提示「PRD 审阅报告不存在，请先执行 test-prd-review」

3. **提取检查数据**：从审阅报告中提取：
   - 所有被标记为**阻塞级（P0）**的问题
   - 所有被标记为**严重缺陷**的问题（逻辑矛盾、核心流程缺失等）
   - 所有标注为「待定」「TBD」「待确认」且直接影响测试设计的条目

> ⛔ **DO NOT** 进入 Phase 2，直到报告加载成功且结构完整。

---

### Phase 2: 执行门禁判定

**目标**：按阻塞条件逐项核验，确定门禁结论。

#### 阻塞条件（任一成立即为不通过）

| 编号 | 阻塞条件 | 说明 |
|------|---------|------|
| B-1 | 核心功能流程描述严重缺失 | 无法推断主要用户操作路径，无法设计主流程用例 |
| B-2 | 存在未解决的明显逻辑矛盾 | PRD 中同一功能在不同章节的描述相互矛盾，且未有定论 |
| B-3 | 关键业务规则标注为「待定/TBD/待确认」 | 核心权限规则、触发条件、计算规则等无法合理推断 |
| B-4 | 无法为主流程定义通过/失败判断准则 | 缺少验收标准，测试人员无法判断功能是否正确 |
| B-5 | 核心功能缺少可判定的输入/输出定义 | PRD 对核心操作的触发条件、输入参数或预期结果缺失，测试人员无法写出"期望结果"一栏（例如：仅写"支持导出"，但未说明触发条件、文件格式、包含哪些数据字段） |
| B-6 | 核心业务规则存在隐性缺失 | 字段或操作存在且**未标注「待定」**，但其校验规则、计算规则或错误处理规则完全未描述——测试人员无法判断输入是否合法、计算是否正确、异常如何处理（与 B-3 的区别：B-3 是"明确标注待定"，B-6 是"根本没提，只能靠猜"。例如：表单有"金额"字段，但 PRD 未描述最大值、小数位、负数处理） |
| B-7 | 上下游数据关系未定义 | 模块的数据输入来源、输出去向或上游失败降级行为完全未描述，导致测试人员无法设计集成场景和边界用例（例如：订单模块依赖库存模块，但 PRD 未说明库存不足时的降级行为；或导出功能未说明数据从哪张表/哪个接口取，无法验证数据一致性） |

#### 放行条件

以下情况**不构成阻塞**，放行并在报告中注明风险：
- 边界值未明确，但核心流程清晰可推断（注意：若字段根本无任何校验规则描述，则构成 B-6 阻塞）
- 部分交互描述模糊，但不影响主流程用例设计
- 存在建议性补充项（优化建议、体验改进等）
- 非核心功能模块标注「待定」，核心模块已清晰
- 上下游模块仅有部分描述缺失，核心数据流可从已有描述推断（注意：若完全未提及数据来源/去向，则构成 B-7 阻塞）

#### 门禁结论判定

| 条件 | 结论 |
|------|------|
| 无任何阻塞条件成立 | **通过** |
| 任一阻塞条件成立 | **不通过** |

> ⛔ **DO NOT** 进入 Phase 3，直到所有阻塞条件均已逐一核验。

---

### Phase 3: 输出门禁结论报告

**目标**：按 `assets/gate-report-template.md` 生成门禁结论报告。

**步骤**：

1. **读取门禁报告模板**：加载 `assets/gate-report-template.md`

2. **确定输出路径**：`review/prd-gate-result.md`
   - 文件已存在 → **覆盖写入**（本次为最新结果）

3. **填写报告内容**：
   - 门禁结论（**通过** / **不通过**）—— 置于报告首行，醒目展示
   - 阻塞项列表（仅不通过时）：每项标注阻塞条件编号（B-1～B-7）、具体问题描述、建议修复方向
   - 放行注意事项（仅通过时，如有风险点）：列出需在用例设计时特别关注的模糊项
   - 后续步骤建议

4. **向用户展示**：
   - 直接输出门禁结论（通过 / 不通过）
   - 通过 → 提示可进入 `test-point-extraction`（提炼测试点），并列出需关注的风险点
   - 不通过 → 列出每条阻塞项和建议修复方向，提示通知产品修复后重新执行 `test-prd-review` + `prd-gate`

## 验收标准

### 结论准确性

- [ ] 任一阻塞条件（B-1～B-7）成立时，结论必须为**不通过**
- [ ] B-6 判定时已区分「标注待定（B-3）」与「根本未提及（B-6）」，不混淆
- [ ] B-7 判定时已区分「部分描述缺失（放行+风险注记）」与「完全未描述数据来源/去向（B-7）」
- [ ] 仅有放行条件问题时，结论为**通过**（附风险注意事项）
- [ ] 门禁结论不出现除「通过」「不通过」以外的第三种状态

### 报告完整性

- [ ] 门禁结论已写入 `review/prd-gate-result.md`
- [ ] 报告首行显示门禁结论
- [ ] 不通过时，每个阻塞项均已标注条件编号、具体描述和建议修复方向
- [ ] 通过时，风险注意事项已列出（如有）

### 下游衔接

- [ ] 通过时，报告中给出了进入 `test-point-extraction` 的建议
- [ ] 不通过时，报告中给出了重新进入门禁所需的前置步骤（修复 → test-prd-review → prd-gate）
