## 二、数据模型问题（DM 系列）

<!-- 为什么需要这部分：数据模型是后端架构的基石。PRD 中数据定义的缺陷如果不在需求阶段暴露，到技术设计甚至编码阶段才发现，返工成本是指数级增长的。后端关注的是"字段能不能存储、查询、关联"，而非仅仅"字段能不能展示"。 -->

> 数据模型问题 = PRD 中数据实体、字段、关系的定义存在缺失、歧义或不合理，导致后端无法确定数据库设计方案。

| DM-2 | | | | | | |

<!-- 常见的数据模型问题类型：
1. 实体关系不明：如"用户"和"组织"是 1:1 还是 1:N？用户能属于多个组织吗？
2. 字段定义缺失：如"金额"字段未说明精度（分/元/小数位数），"名称"未说明长度上限
3. 枚举值不完整：如"订单状态"只列了部分状态，无法确定完整的状态机
4. 唯一性约束不明：如"编号"是否全局唯一？同一组织内唯一？
5. 数据生命周期未定义：数据保留多久？是否需要归档？软删除还是硬删除？
6. 冗余与一致性矛盾：多处定义了同一数据但口径不一致
7. 索引/查询模式不明：PRD 要求按某维度筛选，但数据字典中无对应字段
-->

## 四、性能风险（PF 系列）

<!-- 为什么需要这部分：性能是 PRD 中最常被忽略的非功能需求。产品经理说"展示列表"，但不说数据量级是 100 条还是 100 万条，后端方案天差地别。如果不在需求阶段确认，后端可能按小数据量设计，上线后才发现性能崩溃。 -->

> 性能风险 = PRD 中的功能描述存在潜在的性能隐患，需要产品确认数据量级、并发要求或实时性要求，以便后端选择合适的技术方案。

| PF-1 | {模块名} | {风险描述} | {PRD 章节编号} | {需要产品确认什么} | {量级不同方案如何变化} | |
| PF-2 | | | | | | |

<!-- 常见的性能风险类型：
1. 大数据量列表：未说明数据量级，可能需要分页、虚拟滚动、ES 等不同方案
2. 复杂聚合查询：统计/报表功能涉及多表关联和聚合计算，需确认实时性要求
3. N+1 查询风险：列表中每条记录需关联查询其他实体
4. 导出功能量级：导出数据量级影响是同步导出还是异步导出+下载

## 五、安全隐患（SC 系列）

<!-- 为什么需要这部分：安全漏洞的修复成本远高于功能 Bug。后端是安全的最后防线，如果 PRD 对权限描述模糊（如"相关人员可见"），后端无法实现精确的权限控制。安全问题一旦上线后才发现，可能导致数据泄露等严重后果。 -->

> 安全隐患 = PRD 中的功能涉及权限控制、敏感数据、外部输入等安全相关场景，但规则定义不够明确，后端无法确定安全策略。

| SC-2 | | | | | | |

<!-- 常见的安全隐患类型：
1. 权限粒度不明：只说"管理员可操作"，未区分超级管理员/普通管理员/数据管理员
2. 数据可见性未定义：谁能看到什么数据？是按角色、按组织、还是按数据归属？
3. 敏感数据处理规则缺失：身份证、手机号、银行卡等是否需要加密存储、脱敏展示？
4. 操作审计未说明：哪些关键操作需要记录审计日志？

7. 外部接口认证：调用第三方系统的认证方式和密钥管理策略未说明

## 六、业务逻辑问题（BL 系列）

<!-- 为什么需要这部分：后端是业务逻辑的实际承载者。PRD 中"看起来简单"的一句话，到后端可能需要复杂的状态机、分布式事务、消息队列配合。如果规则本身就有矛盾，编码阶段才发现会导致大量返工。 -->

> 业务逻辑问题 = PRD 中的业务规则存在矛盾、遗漏或过度复杂，后端无法确定正确的实现逻辑。

| # | 所属模块 | 问题描述 | PRD 位置 | 对实现的影响 | 建议处理方式 | 产品回复 |
|---|---------|---------|---------|------------|------------|---------|
| BL-1 | {模块名} | {具体问题描述} | {PRD 章节编号} | {导致什么实现困难} | {建议如何补充/修改} | |
| BL-2 | | | | | | |

<!-- 常见的业务逻辑问题类型：
1. 状态机不完整：只定义了部分状态转换路径，缺少"从A能否回到B"等转换规则
2. 状态转换缺少四要素：状态转换未包含触发条件、输入（参数/数据）、输出（副作用/返回值）、目标状态中的一项或多项，导致后端无法确定状态机实现
3. 业务规则无结构化描述：PRD 只写"按规则校验"或"按条件判断"，未给出具体的触发条件、输入数据源、处理逻辑和输出结果
4. 规则矛盾：PRD 不同章节对同一场景的处理逻辑不一致
5. 异常流程缺失：只描述了正常流程，未说明超时、失败、部分成功等异常的处理方式
6. 并发冲突规则不明：多人同时操作同一数据时的预期行为未定义
7. 定时/异步任务触发条件模糊："定期执行"但未定义执行频率和时间窗口
8. 第三方依赖容错未定义：依赖外部系统时，外部系统不可用时如何降级
9. 计算公式不明确：涉及金额、比例、分成等计算但公式或精度规则未给出
10. 跨模块副作用未说明：A 模块的操作对 B 模块数据的影响未定义
-->

## 七、需求歧义（AM 系列）

<!-- 为什么需要这部分：与测试审阅的歧义章节类似，但后端关注的是"不同理解导致不同技术方案"。比如 PRD 说"支持撤销操作"，理解为"软删除+恢复"和"操作日志+逆向操作"是两种完全不同的后端架构。提前对齐可避免架构级返工。 -->

> 需求歧义 = PRD 文字有多种理解方式，不同理解会导致后端选择不同的技术方案。

| # | 所属模块 | PRD 原文 / 位置 | 理解 A（及技术方案） | 理解 B（及技术方案） | 后端倾向 | 产品回复 |
|---|---------|---------------|-------------------|-------------------|---------|---------|
| AM-1 | {模块名} | {原文引用或章节} | {理解 A + 对应的后端方案} | {理解 B + 对应的后端方案} | {倾向哪个及理由} | |
| AM-2 | | | | | | |

<!-- 填写指引：
- 关键在于说明"不同理解对后端方案的影响差异"，让产品经理理解为什么需要澄清
- 后端倾向应附带技术理由（如性能、复杂度、可维护性）
- 如果两种理解的实现成本差异巨大，务必标明
- 常见的歧义类型包括：
  1. 方案差异型：模糊描述导致后端可选择完全不同的技术方案（如同步vs异步、关系型vs文档型）
  2. 术语不一致型：PRD 中的业务术语与开发侧理解不一致（如产品说"子集"，开发理解为"发布通道"），同一概念使用不同名称导致实现偏差
  3. 隐含条件型：PRD 未显式表述但需要开发凭经验推断的规则（如"汇总收益"的分母范围）
-->

| CP-2 | | | | | | |

<!-- 常见的完整性缺失类型：
1. 异常流程未定义：接口超时、第三方回调失败、数据不一致等场景的处理方式
2. 边界条件未说明：空值、零值、最大值、最小值的处理规则
3. 参数校验规则缺失：涉及精度（如金额小数位数）、长度上限、取值范围（是否允许负值）、格式约束的字段未定义校验规则

6. PRD-原型不一致：原型中存在的交互元素（如弹窗格式、特殊展示方式、操作确认流程）在 PRD 中没有对应描述，或 PRD 描述与原型展示矛盾
7. 并发场景未考虑：多人同时编辑、重复提交、竞态条件下的预期行为
8. 数据迁移方案缺失：新功能上线时，历史数据如何处理？需要做数据迁移吗？
9. 向前/向后兼容：新版本 API 对旧客户端的兼容策略
10. 操作确认/撤销机制：危险操作是否需要二次确认？操作后是否支持撤销？
11. 通知/触达规则：哪些事件需要通知用户？通知方式（站内信/邮件/短信）？
-->
