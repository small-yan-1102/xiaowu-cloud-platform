# 后端 PRD 审阅规范

> **目标**：为后端开发工程师提供 PRD 审阅的标准化参照基准，确保审阅维度完整、问题分类准确、建议可操作
> **核心原则**：从"能否确定技术方案"的角度审视每个需求——如果一个功能的 PRD 描述不足以让后端确定数据模型、API 设计或关键技术选型，那就是需要提出的问题

---

## 一、审阅维度总览

后端审阅 PRD 应覆盖以下 8 个核心维度，每个维度对应审阅报告中的一个章节。此外还有 1 个补充章节（SG 技术实现建议）用于记录优化建议：

```
┌── 核心审阅维度（8 个）──────────────────────────┐
│  DM  数据模型可行性   ← 能不能建表、建什么样的表  │
│  API 接口设计合理性   ← 能不能定接口、定什么样的接口│
│  PF  性能风险         ← 数据量和并发下能不能撑住   │
│  SC  安全隐患         ← 权限和数据安全有没有漏洞   │
│  BL  业务逻辑完整性   ← 规则有没有矛盾和遗漏      │
│  AM  需求歧义         ← 不同理解是否导致不同方案   │
│  CP  完整性缺失       ← 异常、边界、兼容是否缺失   │
│  NF  非功能需求       ← 性能/可用性/安全标准是否明确│
├── 补充章节（1 个）──────────────────────────────┤
│  SG  技术实现建议     ← 降低复杂度或规避风险的建议  │
└─────────────────────────────────────────────┘
```

---

## 二、各维度审阅检查清单

### DM - 数据模型可行性检查

对 PRD 中涉及的每个数据实体，逐项检查：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 实体识别 | PRD 中提到的业务对象是否都能映射为明确的数据实体？ | DM 系列 |
| 字段完整性 | 每个实体的字段是否列全？字段类型、长度、精度是否明确？ | DM 系列 |
| 唯一性约束 | 哪些字段或字段组合需要唯一？唯一范围是什么（全局/租户/组织）？ | DM 系列 |
| 实体关系 | 实体间关系类型是否明确（1:1 / 1:N / M:N）？ | DM 系列 |
| 枚举值完整性 | 状态、类型等枚举字段的可选值是否列全？ | DM 系列 |
| 数据生命周期 | 数据保留策略、归档策略、删除策略（软删/硬删）是否定义？ | DM 系列 |
| 索引需求推导 | PRD 中的筛选、排序、搜索场景是否能推导出合理的索引设计？ | PF 系列 |

### API - 接口设计合理性检查

对 PRD 中描述的每个用户操作，评估其 API 设计可行性：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 操作粒度 | 单条操作和批量操作是否分别说明？批量上限是否定义？ | API 系列 |
| 事务边界 | 涉及多步操作时，是全部成功还是允许部分成功？ | API 系列 |
| 幂等性 | 重复提交同一请求时的预期行为是否明确？ | API 系列 |
| 分页排序 | 列表类功能的分页大小、默认排序、最大查询范围是否说明？ | API 系列 |
| 筛选逻辑 | 多条件筛选时的组合关系（AND/OR）是否明确？ | API 系列 |
| 文件操作 | 上传/下载的文件大小限制、格式限制、并发限制是否说明？ | API 系列 |
| 实时性 | 操作结果是否需要立即可见？是否允许最终一致？ | API/PF 系列 |

### PF - 性能风险检查

对每个功能评估其性能风险等级：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 数据量级 | PRD 是否说明了数据的预期量级（当前/1年后/3年后）？ | PF 系列 |
| 查询复杂度 | 是否存在多表 JOIN、子查询、聚合计算等复杂查询？ | PF 系列 |
| N+1 风险 | 列表中每条记录是否需要额外关联查询？ | PF 系列 |
| 导出量级 | 数据导出功能的数据量上限是否定义？ | PF 系列 |
| 写入频率 | 高频写入场景（如日志、消息）的 QPS 预期是否明确？ | PF 系列 |
| 缓存需求 | 读多写少的热点数据是否识别？缓存失效策略是否可推导？ | PF 系列 |
| 搜索方案 | 全文搜索/模糊搜索在预期数据量下是否需要专门的搜索引擎？ | PF 系列 |

### SC - 安全隐患检查

对每个功能评估其安全风险：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 权限模型 | 访问控制是角色级还是数据级？权限粒度是否足够？ | SC 系列 |
| 数据可见性 | "谁能看到什么数据"的规则是否明确？ | SC 系列 |
| 敏感数据 | 身份证、手机号等敏感字段的加密/脱敏要求是否定义？ | SC 系列 |
| 审计日志 | 关键操作（创建/修改/删除/导出）是否需要审计记录？ | SC 系列 |
| 二次验证 | 高风险操作是否需要二次身份验证？ | SC 系列 |
| 输入校验 | 用户可输入字段的字符集、长度上限是否约束？ | SC/CP 系列 |
| 外部认证 | 第三方系统调用的认证方式是否说明？ | SC 系列 |

### BL - 业务逻辑完整性检查

对 PRD 中的每条业务规则进行完整性审查：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 规则明确性 | 业务规则是否使用了"如果...则...否则..."的明确表述？ | BL/AM 系列 |
| 业务规则结构化 | 每条关键业务规则是否包含：**触发条件**（什么时候执行）、**输入**（数据来源/参数）、**处理逻辑**（具体计算或判断）、**输出**（结果/副作用）？ | BL 系列 |
| 状态机完整性 | 实体的所有状态、所有合法转换路径是否定义？**每个状态转换是否包含四要素：触发条件、输入（参数/数据）、输出（副作用/返回值）、目标状态？** | BL 系列 |
| 规则一致性 | 不同章节对同一功能的规则是否矛盾？ | BL 系列 |
| 异常处理 | 操作失败、超时、部分成功等异常场景的处理方式是否定义？ | BL/CP 系列 |
| 并发处理 | 多用户同时操作同一数据时的冲突解决策略是否明确？ | BL 系列 |
| 外部依赖 | 依赖的外部系统不可用时，是阻塞还是降级？降级策略是什么？ | BL 系列 |
| 跨模块影响 | A 模块的操作对 B 模块数据的副作用是否说明？ | BL/CP 系列 |

### AM - 需求歧义检查

对 PRD 中的描述逐项检查是否存在理解歧义：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 方案差异放大 | 模糊描述是否可能导致不同的后端技术方案？方案差异大则为阻塞级歧义 | AM 系列 |
| 术语一致性 | PRD 中的业务术语是否可能与开发侧理解不一致？是否存在同一概念产品和开发使用不同名称的情况？ | AM 系列 |
| 隐含条件 | PRD 中是否存在未显式表述但需要开发凭经验推断的业务规则？ | AM/BL 系列 |
| 量词模糊 | "大量"、"实时"、"快速"等量词是否缺乏具体标准？ | AM/PF 系列 |

### CP - 完整性缺失检查

对 PRD 中的每个功能检查是否存在关键信息遗漏：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 异常流程 | 接口超时、第三方回调失败、数据不一致等异常场景的处理方式是否定义？ | CP 系列 |
| 边界条件 | 空值、零值、最大值、最小值的处理规则是否说明？ | CP 系列 |
| 参数校验规则 | 涉及精度（如金额小数位）、长度上限、取值范围（如是否允许负值）、格式约束的字段是否定义了校验规则？ | CP/DM 系列 |
| 跨系统数据链路 | PRD 中提到的外部数据来源是否说明了完整的数据流转链路（而非只写"来源于XX系统"）？是否清楚数据经过了哪些中间系统、是否有转换或过滤？ | CP 系列 |
| 列表页面要素 | 列表类功能是否完整定义了：展示字段、筛选条件、排序规则、操作按钮？四项是否齐全？ | CP/API 系列 |
| 并发场景 | 多人同时编辑、重复提交、竞态条件下的预期行为是否说明？ | CP/BL 系列 |
| 数据迁移 | 新功能上线时，历史数据如何处理？是否需要数据迁移？ | CP 系列 |
| PRD-原型一致性 | PRD 文字描述是否与原型页面展示一致？原型中有的内容 PRD 是否都有对应描述？ | CP 系列 |

## 三、问题严重度分级标准

审阅发现的问题应按严重度分级，便于产品经理优先处理：

| 严重度 | 判定标准 | 对后端的影响 |
|--------|---------|------------|
| **阻塞级** | 不解决则无法确定技术方案，技术设计无法启动 | 数据模型无法设计、API 无法定义、核心逻辑无法编码 |
| **重要级** | 不解决则技术方案存在重大风险，可能导致架构级返工 | 方案选型有误、性能不达标、安全漏洞 |
| **一般级** | 不解决则实现细节不确定，但不影响整体架构 | 个别接口行为不确定、边界case处理不明 |
| **建议级** | 可以按后端理解实现，但建议产品确认以避免后期争议 | 后端有合理假设，但不确定是否符合产品意图 |

**分级原则**：
- 问题类型不等于严重度——DM 系列问题可以是阻塞级也可以是一般级
- 严重度取决于"不解决对技术方案的影响程度"
- 在审阅报告中，每个问题应在描述中体现其严重度

## 四、审阅思维方法

### 1. 技术可行性追问

对 PRD 中的每个功能，追问以下问题链：

```
PRD 说"做什么" → 追问"数据从哪来、存到哪去"
                → 追问"通过什么接口触发"
                → 追问"多大数据量、多高并发"
                → 追问"谁有权限、数据怎么隔离"
                → 追问"失败了怎么办"
```

如果任一环节答不上来，就是需要提给产品的问题。

### 2. 方案差异放大

遇到 PRD 中的模糊描述时，刻意构造两种不同的后端方案，评估方案差异：

- **方案差异小**（如：字段命名不同）→ 一般级问题或建议级问题
- **方案差异大**（如：同步 vs 异步、关系型 vs 文档型）→ 阻塞级问题

核心：**如果不同理解导致不同架构，就必须澄清。**

### 3. 生产环境推演

对每个功能，想象它在生产环境运行 1 年后的状态：

- 数据量从 100 条增长到 100 万条，还能正常工作吗？
- 并发用户从 10 人增长到 1000 人，性能有瓶颈吗？
- 某个外部服务挂了 10 分钟，业务能降级运行吗？
- 某个运营人员误操作了，能恢复吗？

PRD 通常描述的是"Day 1"状态，后端需要考虑"Day 365"的状态。

### 4. 奥卡姆剃刀

与测试审阅相同，审阅报告本身也要遵循简约原则：

- 不提"凑数"问题——每个问题都应有实际技术影响
- 同一根因不拆成多个问题——合并为一个问题
- 问题描述直击要点——产品经理应能在 10 秒内理解

## 五、常见反模式（PRD 中的后端雷区）

| PRD 中的写法 | 后端风险 | 应追问的问题 |
|-------------|---------|------------|
| "支持大量数据" | 无法确定存储方案和查询优化策略 | 具体量级是多少？万级/百万级/亿级？ |
| "实时更新" | 轮询/WebSocket/SSE 方案差异巨大 | 实时到什么程度？秒级/分钟级？ |
| "根据权限展示" | 无法确定权限模型和数据过滤策略 | 权限维度是什么？角色/组织/数据归属？ |
| "支持搜索" | 模糊搜索在大数据量下可能需要 ES | 搜索范围和数据量？需要全文检索吗？ |
| "批量操作" | 无法确定事务范围和超时策略 | 批量上限？部分失败怎么办？ |
| "历史记录" | 无法确定数据保留和清理策略 | 保留多久？要归档吗？可查询范围？ |
| "导出数据" | 大数据量导出需要异步方案 | 最大导出量？需要异步下载吗？ |
| "通知用户" | 通知方案选型差异大 | 什么事件触发？什么渠道？实时还是批量？ |
| "高可用" | 无法确定部署架构和容灾方案 | SLA 等级？RTO/RPO 要求？ |
| "安全存储" | 无法确定加密方案和密钥管理 | 哪些字段加密？传输加密还是存储加密？ |
| "来源为XX系统透传" | 不清楚完整数据链路，中间可能有转换/过滤/聚合 | 完整的数据流转链路是什么？经过了哪些中间系统？有无数据转换？ |
| "按规则校验" / "按条件判断" | 无法确定具体校验逻辑和判断条件 | 具体的校验规则是什么？输入/判断条件/通过/不通过各是什么结果？ |
| "状态变更" / "流转到下一步" | 缺少结构化的状态转换定义，无法确定状态机 | 触发条件是什么？输入参数？目标状态？有什么副作用？ |
| "展示列表" （仅描述筛选条件） | 展示字段、排序、操作按钮缺失，无法确定完整 API | 列表展示哪些字段？默认排序？有什么操作按钮？ |

## 六、审阅报告编号规则

| 前缀 | 含义 | 示例 |
|------|------|------|
| DM-{序号} | 数据模型问题 | DM-1：用户实体缺少组织关联关系定义 |
| API-{序号} | 接口设计问题 | API-1：批量删除的事务边界未定义 |
| PF-{序号} | 性能风险 | PF-1：订单列表未说明数据量级 |
| SC-{序号} | 安全隐患 | SC-1：数据导出接口未定义权限控制 |
| BL-{序号} | 业务逻辑问题 | BL-1：订单状态"已取消"能否重新激活未说明 |
| AM-{序号} | 需求歧义 | AM-1："相关人员"可理解为按角色或按数据归属 |
| CP-{序号} | 完整性缺失 | CP-1：支付超时场景的订单处理方式未定义 |
| NF-{序号} | 非功能需求 | NF-1：核心接口响应时间标准未定义 |
| SG-{序号} | 技术实现建议 | SG-1：建议将实时统计改为定时预计算 |
