# 前端测试用例审阅技能

> **设计模式**：Reviewer（检查清单驱动）+ Generator（模板化输出报告）

## 输入/输出契约

### 输入
| 输入项 | 来源 | 缺失时处理 |
|-------|------|----------|
| 测试用例文档 | 用户指定路径 | Fatal：向用户索取 |
| 用户故事文档 | 用户指定路径 | Fatal：向用户索取 |
| 前端代码 | 当前项目代码 | Warn：跳过代码实现检查 |
| 审阅检查清单 | `references/review-checklist.md` | 自动加载 |

### 输出
| 输出产物 | 格式 |
|---------|------|
| 前端测试覆盖度报告 | 基于 `assets/coverage-report-template.md` 生成 |

---

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集（Inversion）
- [ ] Phase 1: 读取用户故事文档
- [ ] Phase 2: 读取测试用例文档
- [ ] Phase 3: 覆盖度分析（Reviewer）
- [ ] Phase 4: 生成覆盖度报告（Generator）
- [ ] Phase 5: 输出结果
```

### Phase 0: 范围收集（Inversion 模式）

**⚠️ 硬门控：DO NOT proceed to Phase 1 until:**
- [ ] 测试用例文档路径已确认
- [ ] 用户故事文档路径已确认
- [ ] 目标模块名称已确认
- [ ] 用户确认所有输入信息无误

使用 AskUserQuestion 收集：
- 测试用例文档路径
- 用户故事文档路径
- 目标模块名称

---

### Phase 1: 读取用户故事文档

**⚠️ 硬门控：DO NOT proceed to Phase 2 until user stories are extracted.**

1. **提取功能清单**：
   - 读取用户故事文档
   - 提取所有功能点（用户故事/场景）
   - 建立功能点清单：
     - 功能编号
     - 功能描述
     - 优先级
     - 验收标准

2. **分类功能点**：
   - 核心功能（必须测试）
   - 交互场景（建议测试）
   - UI状态（建议测试）
   - 边界条件（建议测试）

3. **建立功能追踪表**：
   ```
   功能编号 | 功能描述 | 优先级 | 测试用例编号 | 覆盖状态
   ```

---

### Phase 2: 读取测试用例文档

**⚠️ 硬门控：DO NOT proceed to Phase 3 until test cases are extracted.**

1. **提取测试用例清单**：
   - 读取测试用例文档
   - 提取所有测试场景/用例
   - 建立测试用例清单：
     - 用例编号
     - 用例标题
     - 前置条件
     - 测试步骤
     - 预期结果
     - 优先级

2. **分类测试用例**：
   - 冒烟测试用例
   - 功能测试用例
   - 边界值测试用例
   - 异常场景测试用例
   - UI测试用例

3. **建立测试追踪表**：
   ```
   用例编号 | 用例标题 | 关联功能 | 优先级 | 测试类型
   ```

---

### Phase 3: 覆盖度分析（Reviewer 模式）

**⚠️ 硬门控：DO NOT proceed to Phase 4 until coverage analysis is complete.**

按 `references/review-checklist.md` 中的四个维度逐一分析：

#### 1. 功能覆盖度
- 测试用例是否覆盖了用户故事的所有功能点?
- 核心功能是否都有对应测试用例?
- 功能优先级与测试优先级是否匹配?
- 是否有高优先级功能未覆盖?

#### 2. 交互场景覆盖度
- 正常路径（Happy Path）是否覆盖?
- 异常路径（Exception Path）是否覆盖?
- 表单校验场景是否完整（必填、格式、长度、唯一性）?
- 按钮状态是否测试（禁用、加载、成功、失败）?
- 页面跳转逻辑是否测试?

#### 3. UI状态覆盖度
- 空状态是否测试（无数据、无权限）?
- 加载中状态是否测试（Loading、Skeleton）?
- 错误状态是否测试（网络错误、接口异常）?
- 成功提示是否测试?
- 弹窗/抽屉交互是否测试?
- 表单验证提示是否测试?

#### 4. 前端专属测试点
- 响应式布局是否测试（不同屏幕尺寸）?
- 浏览器兼容性是否测试?
- 数据边界值是否测试（空列表、超大数据量）?
- 特殊字符是否测试（表情、HTML标签、SQL注入）?
- 性能体验是否测试（首屏加载、大数据渲染）?
- 无障碍访问是否测试（键盘导航）?

**分析流程**：
1. 对比功能清单和测试用例清单
2. 建立映射关系（功能 ↔ 测试用例）
3. 计算各维度覆盖度
4. 识别未覆盖场景
5. 按 Critical/Warning/Info 分级
6. 给出补充建议

---

### Phase 4: 生成覆盖度报告（Generator 模式）

> **补充用例建议**：审核发现的每条覆盖缺失项，应附「建议补充用例标题」和优先级，可直接作为 test-case-design Mode E 的输入。

基于 `assets/coverage-report-template.md` 模板生成覆盖度报告，包含：

1. **覆盖度概览**：各维度覆盖度统计
2. **未覆盖场景**：按 Critical/Warning/Info 分级
3. **覆盖度详情**：功能点与测试用例映射表
4. **补充建议**：按优先级分层的建议清单
5. **审阅结论**：是否满足测试准入条件

---

### Phase 5: 输出结果

**输出内容**：
1. 前端测试覆盖度报告（Markdown格式）
2. 未覆盖场景清单
3. 优先级补充建议

---

## 覆盖度计算标准

### 覆盖度公式

```
覆盖度 = (已覆盖功能点数 / 总功能点数) × 100%
```

### 覆盖度等级

| 覆盖度 | 等级 | 说明 |
|--------|------|------|
| ≥ 90% | ✅ 优秀 | 可以进入测试执行 |
| 80-89% | ⚠️ 良好 | 建议补充后进入测试 |
| 70-79% | ⚠️ 一般 | 必须补充核心功能测试 |
| < 70% | ❌ 不足 | 不建议进入测试执行 |

---

## 问题严重度分级

| 级别 | 定义 | 处理要求 |
|------|------|---------|
| **Critical** | 核心功能未覆盖、高优先级场景缺失 | 必须补充测试用例 |
| **Warning** | 交互场景不完整、UI状态缺失 | 建议补充测试用例 |
| **Info** | 前端专属测试点缺失、优化建议 | 可选补充测试用例 |

---

## 约束

- 审阅聚焦测试用例覆盖度，不评判测试用例编写质量
- 所有未覆盖场景必须给出具体补充建议
- 覆盖度计算必须准确，基于实际对比结果
- 必须与用户故事文档对比，确保功能点全覆盖
- 问题分级必须准确，不允许夸大或弱化
