# 测试数据准备技能

> **版本**: v2.1
> **更新日期**: 2026-04-08
> **适用模式**: AI 直接执行模式
> **Mode A（Setup）**：测试执行前准备数据，输出数据准备报告 + 变量映射表
> **Mode B（Teardown）**：测试执行完毕后读取数据准备报告中的清理指引，执行 API 删除，防止脏数据累积
> **质量标准**: 数据准备结果必须可通过被测系统 UI 或 API 验证其存在和正确性

## 适用场景 vs 不适用场景

| 适用 | 不适用 |
|------|--------|
| 用例前置条件标注了 `[数据准备: API]` | 数据可通过 UI 简单创建（≤5 步）→ 使用 `[数据准备: UI自建]` |
| UI 创建步骤超过 5 步，API 创建更高效 | 数据由其他用例产生 → 使用 `[数据准备: 依赖用例]` |
| 需要批量创建测试数据（如 10+ 条记录） | 需要运维预配置的系统级数据 → 使用 `[数据准备: 环境预置]` |
| 数据结构复杂，UI 表单分多步骤多页面 | |
| 测试数据有上下游依赖关系，需按顺序创建 | |

### 非 API 策略的执行归属

本 Skill 仅执行 `[数据准备: API]` 策略。其余三种策略的执行主体如下：

| 策略 | 执行主体 | 本 Skill 的职责 |
|------|---------|---------------|
| `[数据准备: UI自建]` | 测试工程师 | Phase 2 中识别并标注「需测试工程师执行前人工创建」，不执行任何 API 调用 |
| `[数据准备: 依赖用例]` | test-execution skill | Phase 2 中识别并标注「由 test-execution 保证执行顺序」，提示执行侧注意顺序依赖 |
| `[数据准备: 环境预置]` | 运维 / 测试环境管理员 | Phase 2 中读取 `[验证:]` 标签，生成**环境预置检查清单**；Phase 5 输出该清单，由测试工程师在执行前逐条确认 |

## Mode A 核心工作流（数据准备）

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 解析技术文档
- [ ] Phase 2: 匹配用例数据需求
- [ ] Phase 3: 执行 API 调用（含执行前残留数据检测）
- [ ] Phase 4: 验证数据创建成功
- [ ] Phase 5: 输出数据准备报告
```

### Phase 0: 定向收集

**目标**：在任何操作开始前，确认技术文档路径和目标用例范围。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **技术文档路径**：API 接口文档位于哪里（默认：`input/tech/*.md`）？
2. **用例文件路径**：包含 `[数据准备: API]` 标注的用例文件路径是什么（默认：`test_suites/*.md`）？
3. **目标范围**：准备全部用例的数据，还是只针对特定用例/模块？
4. **环境配置**：测试环境 Base URL 和测试账号是否已在用例文件中写明？如未写明，请提供。

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 开始任何操作，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答 → 进入 Phase 1
- 用户表示"按默认来" → 使用默认路径，进入 Phase 1

---

### Phase 1: 解析技术文档

**目标**：从技术文档中提取可用的 API 接口清单。

**步骤**：

1. 读取技术文档
2. 识别所有 API 接口定义（endpoint、method、参数、响应结构）
3. 提取认证方式（Token 获取方式、登录接口路径）
4. 识别接口间的依赖关系（如创建 B 需要先创建 A 的 ID）
5. 提取业务约束（字段唯一性规则、枚举值、格式要求）
6. 输出**接口清单表**

**输出格式**：

| # | 接口名称 | Method | Endpoint | 必填参数 | 依赖接口 |
|---|---------|--------|----------|---------|---------|
| 1 | 用户登录 | POST | /auth/login | username, password | 无 |
| 2 | 创建企业 | POST | /company | name, creditCode | #1(获取token) |

> ⛔ **DO NOT** 进入 Phase 2，直到技术文档已完整解析并输出接口清单。

---

### Phase 2: 匹配用例数据需求

**目标**：分析用例前置条件，确定需要调用哪些 API，生成数据准备计划和清理计划。

**步骤**：

1. 扫描所有测试用例，按策略分类处理：
   - **`[数据准备: API]`**：提取前置条件描述（字段、值、数量、唯一性要求），进入后续匹配流程
   - **`[数据准备: 环境预置]`**：提取前置条件中的 `[验证:]` 标签内容，生成**环境预置检查清单**（供 Phase 5 §9a 输出）；无 `[验证:]` 标签时标注「⚠️ 缺少验证方式，请测试工程师在执行前确认」
   - **`[数据准备: UI自建]`**：记录用例编号和数据描述，在报告中标注「需测试工程师执行前人工创建」，不做 API 调用
   - **`[数据准备: 依赖用例]`**：记录用例编号和依赖的前序用例，在报告中标注「由 test-execution 保证执行顺序，请勿调整执行顺序」

2. 从 `[数据准备: API]` 前置条件中提取需要创建的数据描述
3. 将数据需求与 Phase 1 的接口清单进行匹配
4. 根据依赖关系确定 API 调用顺序
5. 生成**数据准备计划**（调用顺序 + 依赖关系图）
6. 生成**数据清理计划**（按依赖关系逆序）

**调用顺序格式**：

| 序号 | 目标数据 | 使用接口 | 关键参数 | 供给用例 |
|-----|---------|---------|---------|---------|
| 1 | 获取认证Token | POST /auth/login | admin/Test1234 | 全部 |
| 2 | 创建企业 | POST /company | name="测试公司_{时间戳}" | PRJ-CERT-003 |

> ⛔ **DO NOT** 进入 Phase 3，直到数据准备计划和清理计划均已生成。

---

### Phase 3: 执行 API 调用

**目标**：按计划顺序调用 API，创建测试数据。包含执行前残留数据检测。

**步骤**：

#### Step 3.0 执行前残留数据检测

在创建数据前，逐条检测数据是否已存在（防止唯一性冲突和脏数据）：

1. 调用对应的查询接口（GET/LIST），检查数据是否已存在
2. 数据已存在且有删除接口 → 调用 DELETE 清除残留，然后重新创建
3. 数据已存在且无删除接口 → 更换时间戳后缀，使用新的唯一值创建；**同步更新变量映射表中对应变量的值为新名称**，并在残留数据检测记录中标注「变更原因：无删除接口，已更换唯一值」
4. 数据不存在 → 正常创建

#### Step 3.1 获取认证凭证

- 调用登录接口获取 Token/Cookie
- 验证认证成功（检查响应 code 和 token 有效性）

#### Step 3.2 按依赖顺序逐个调用

**请求构造规则**：

| 规则 | 说明 |
|------|------|
| Content-Type | 默认 `application/json`，文件上传用 `multipart/form-data` |
| 唯一性字段 | 自动追加时间戳后缀 `_{yyyyMMddHHmmss}`，避免冲突 |
| 枚举字段 | 使用技术文档中定义的合法枚举值 |
| 关联字段 | 从前序接口响应中提取（如 companyId 从创建企业接口获取） |
| 选填字段 | 除非用例明确要求，否则不填（测试最小必填场景） |

**错误处理规则**：

| 响应状态 | 处理方式 |
|---------|---------|
| 2xx | 解析响应体，提取 ID 和关键字段，写入变量映射表 |
| 401 | 认证过期 → 自动重新登录并重试 |
| 400 | 参数错误 → 记录完整错误信息，**不重试** |
| 409 | 唯一性冲突 → 更换时间戳后缀重试一次 |
| 500 | 服务端错误 → 等待 3 秒后重试一次 |
| 超时(>10s) | 记录超时 → 重试一次 |

**安全规则**：
- API 调用的 Base URL 必须是**测试环境**地址，禁止调用生产环境
- Token、密码等仅在内存中持有，报告中使用脱敏格式（如 `eyJ...`）

> ⛔ **DO NOT** 进入 Phase 4，直到所有 API 调用均已执行（成功或失败均记录）。

---

### Phase 4: 验证数据创建成功

**目标**：确认通过 API 创建的数据确实存在于系统中。

**验证方式**（按优先级）：

| 优先级 | 验证方式 | 说明 |
|-------|---------|------|
| 1 | **API 查询验证** | 调用对应的 GET/LIST 接口，确认数据存在 |
| 2 | **UI 页面验证** | 通过浏览器导航到对应页面，确认数据可见 |

**验证规则**：
- 每条创建的数据必须至少通过一种方式验证
- API 查询验证：响应中包含创建时使用的关键字段值
- UI 页面验证：在对应列表页面能看到创建的数据记录
- 验证失败的数据标记为「数据准备失败」，对应的测试用例标记为「阻塞」

**`[验证:]` 标签优先原则**：

若用例前置条件中存在 `[验证:]` 标签，以标签描述的验证目标为准，而非仅做通用的记录存在性检查：
- 标签指定了具体字段值（如 `[验证: 查询企业列表，确认 name = {公司名称}]`）→ 验证时必须确认该字段值匹配
- 标签指定了状态（如 `[验证: 确认订单状态 = 待审核]`）→ 验证时必须确认状态字段
- 标签未指定具体内容（如 `[验证: 企业记录已存在]`）→ 使用通用验证（记录存在即可）
- 用例无 `[验证:]` 标签 → 使用通用验证

> ⛔ **DO NOT** 进入 Phase 5，直到全部数据验证结果均已记录。

---

### Phase 5: 输出数据准备报告

**目标**：按 `assets/data-prep-report-template.md` 中的模板，输出完整的数据准备报告。

**步骤**：

1. **读取报告模板**：加载 `assets/data-prep-report-template.md`

2. **填写报告各章节**：
   - §1 准备概况（成功/失败统计）
   - §2 调用计划（接口清单 + 依赖关系图）
   - §3 执行记录（每次调用的入参、响应、耗时）
   - §4 创建数据清单（含验证特征列）
   - §5 变量映射表（供用例中 `{引用:变量名}` 使用）
   - §6 残留数据检测记录
   - §7 失败项（如有）
   - §8 数据清理指引（按逆序）
   - §9 数据就绪验证清单
   - §9a 环境预置检查清单（如有 `[数据准备: 环境预置]` 用例）：逐条列出用例编号 + `[验证:]` 标签内容，供测试工程师在执行前人工核对
   - §9b 人工执行用例变量速查表（如有 `[执行方式: 人工]` 用例）：将前置条件中的 `{引用:变量名}` 展开为实际值，格式如下，供测试工程师在人工执行时直接查阅

     | 用例编号 | 变量名 | 实际值 |
     |---------|-------|-------|
     | TC-001 | company_id | comp_20260408153021 |
     | TC-001 | company_name | 测试公司_20260408153021 |

3. **确定输出路径**：`test_suites/data-prep-report-{YYYY-MM-DD}-{HHmmss}.md`

4. **向用户展示**：输出文件路径 + 准备概况摘要 + 失败项列表（如有）

## Mode B 核心工作流（数据清理）

```
Task Progress:
- [ ] Phase B-0: 定向收集
- [ ] Phase B-1: 读取清理指引
- [ ] Phase B-2: 执行清理 API 调用
- [ ] Phase B-3: 更新清理执行记录
```

---

### Phase B-0: 定向收集

**目标**：确认数据准备报告路径。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **数据准备报告路径**：本次测试对应的数据准备报告在哪里？（默认：`test_suites/data-prep-report-*.md`，取最新一份）
2. **清理范围**：清理全部创建数据，还是指定某个模块/用例范围？（默认：全部）

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

**决策点**：
- 用户已回答或表示"按默认来" → 使用默认路径，进入 Phase B-1

---

### Phase B-1: 读取清理指引

**目标**：从数据准备报告中提取 §8 数据清理指引，确认清理范围和顺序。

**步骤**：

1. **读取数据准备报告**：
   - 加载用户指定路径（默认取最新一份）
   - 不存在 → **Fatal**：终止执行，提示「数据准备报告不存在，无法执行清理」

2. **提取 §8 数据清理指引**：
   - 读取清理条目（数据描述、清理接口、清理顺序、清理时机）
   - 确认依赖关系逆序排列（子数据先于父数据清理）

3. **核对 §5 变量映射表**：
   - 提取每条数据的实际 ID（创建时分配的唯一标识）
   - 将 ID 填入清理接口路径（如 `DELETE /xxx/{id}` 中的 `{id}`）

4. **输出清理计划**，向用户确认后进入 Phase B-2

> ⛔ **DO NOT** 进入 Phase B-2，直到清理计划已确认、所有 ID 已映射。

---

### Phase B-2: 执行清理 API 调用

**目标**：按清理顺序逐条调用 API 删除测试数据。

**步骤**：

1. **获取认证凭证**：调用登录接口重新获取 Token（测试执行后 Token 可能已过期）

2. **按逆序执行 DELETE 调用**：

   | 规则 | 说明 |
   |------|------|
   | 执行顺序 | 严格按 §8 清理指引的「清理顺序」列，子数据先于父数据 |
   | 404 响应 | 数据已不存在 → 标记「无需清理（已缺失）」，继续下一条 |
   | 403 响应 | 权限不足 → 标记「需人工清理」，记录清理路径，继续下一条 |
   | 500 响应 | 服务端错误 → 等待 3 秒后重试一次；仍失败 → 标记「清理失败」|
   | 无删除接口 | 已在 §8 中标注「无删除接口，需人工处理」→ 跳过，汇总到人工清理列表 |

3. **逐条记录结果**（供 Phase B-3 填写 §10）：
   - 成功（200/204）
   - 无需清理（404，数据已缺失）
   - 需人工清理（403，权限限制）
   - 清理失败（500，记录原因）

> ⛔ **DO NOT** 进入 Phase B-3，直到全部清理条目均已执行（成功/失败均记录）。

---

### Phase B-3: 更新清理执行记录

**目标**：将清理结果写回数据准备报告的 §10，输出清理摘要。

**步骤**：

1. **更新 §10 数据清理执行记录**：将 Phase B-2 的执行结果逐条填入数据准备报告

2. **输出清理摘要**：
   - 共清理 N 条，成功 X 条，无需清理 Y 条，需人工处理 Z 条，失败 W 条
   - 如有「需人工清理」或「清理失败」项，逐条列出并提示处理方式

3. **环境状态结论**：
   - 全部成功 → ✅ 测试环境已还原，可进行下一轮执行
   - 有待处理项 → ⚠️ 部分数据需人工清理，输出 `manual-cleanup-checklist.md` 包含：数据标识、所在表/接口、建议清理 SQL/API、优先级

4. **API 不可用降级**：当 `[数据准备: API]` 策略执行失败（服务不可用/鉴权失败）时：
   - 输出 UI 创建步骤脚本（按操作顺序列出页面路径、填写字段、点击按钮），供 test-execution 浏览器执行
   - 在报告中标注 `[降级: UI自建]`，提示执行效率降低

## 验收标准

### 数据准备报告完整性

- [ ] 报告已写入正确路径
- [ ] 变量映射表已生成，变量名与用例前置条件中的 `{引用:变量名}` 一致
- [ ] 数据就绪验证清单已生成（供执行阶段验证）
- [ ] 数据清理指引已生成（按依赖关系逆序）

### 数据质量

- [ ] 所有 `[数据准备: API]` 标注的数据需求均已处理
- [ ] 每条创建的数据均已通过 API 或 UI 验证存在；有 `[验证:]` 标签的按标签内容验证，而非仅验证记录存在
- [ ] 唯一性字段均使用了时间戳后缀（无硬编码）
- [ ] API 调用仅针对测试环境（非生产环境）

### 非 API 策略覆盖

- [ ] `[数据准备: 环境预置]` 用例：§9a 环境预置检查清单已生成，逐条包含 `[验证:]` 内容

## 变量传递机制

数据准备阶段创建的数据通过**变量映射表**传递给测试用例：

- 数据准备报告变量映射表中的变量名 → 对应测试用例前置条件中的 `{引用:变量名}`
- AI 执行测试用例时，从数据准备报告中查找变量值并替换

**示例**：
```
数据准备报告：company_id = comp_001

测试用例前置条件：
3. [数据存在][数据准备: API] 企业记录已创建，ID 为 {引用:company_id}

AI 执行时替换为：
3. [数据存在][数据准备: API] 企业记录已创建，ID 为 comp_001
```
