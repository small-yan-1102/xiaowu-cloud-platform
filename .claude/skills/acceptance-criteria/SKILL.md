# 验收标准检查技能

> **设计模式**：Inversion（先收集验收范围）+ Reviewer（基于验收标准逐条核验）

## 输入/输出契约

### 输入
| 输入项 | 来源 | 缺失时处理 |
|-------|------|----------|
| 需求文档（含验收标准） | 用户指定路径 | Fatal：向用户索取 |
| 对应源码文件 | 项目目录 | Fatal：无法验收 |

### 输出
| 输出产物 | 格式 |
|---------|------|
| 验收报告 | Markdown（控制台或文件输出） |

---

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集（Inversion）
- [ ] Phase 1: 定位需求文档
- [ ] Phase 2: 提取验收标准
- [ ] Phase 3: 逐条对照源码
- [ ] Phase 4: 输出验收报告
```

### Phase 0: 范围收集（Inversion 模式）

**⚠️ 硬门控：DO NOT proceed until scope is confirmed.**

**两种输入方式**（任一即可）：
- **A. 用户预先提供**：用户在调用命令时直接给出模块名 + 文档路径 → 跳过交互，直接进入 Phase 1
- **B. 交互收集**：用户未提供足够信息 → 使用 AskUserQuestion 收集：
  - 验收哪个模块 / 功能 / Quest？
  - 需求文档路径（或版本号）

---

### Phase 1: 定位需求文档

**定位规则**（按优先级）：
1. Phase 0 已获得完整路径 → 直接使用
2. 用户给出版本号 → 在 `iterations/` 下匹配 `*_{版本号}_*` 或 `*_V{版本号}_*` 目录，读取 `input/prd/` 下的需求文档
3. 均未提供 → 扫描 `iterations/` 下所有目录，按目录名中的日期/季度排序，取最新迭代的 `input/prd/`

找不到时 Fatal 终止，向用户索取正确路径。

---

### Phase 2: 提取验收标准

从文档中找到目标功能的**验收标准**章节，提取所有验收条目。

---

### Phase 3: 逐条对照源码

针对每条验收标准，阅读对应的源码文件，判断是否实现：

- **✅ 通过**：代码中有对应实现
- **❌ 未实现**：代码中缺少该功能
- **⚠️ 部分实现**：功能存在但不完整或有偏差

**审查重点**：
- **数据与配置核查**：枚举值、计费规则、校验规则、禁用状态是否与文档一致（不允许自行推断）
- **条件显示逻辑核查**：条件渲染、状态驱动 UI、表单联动
- **交互行为核查**：路由路径、弹窗触发、按钮禁用条件

---

### Phase 4: 输出验收报告

```markdown
## 验收报告：{模块名} - {功能/Quest}

### 验收结果概览
- ✅ 通过：X 项
- ⚠️ 部分实现：X 项
- ❌ 未实现：X 项

### 逐条验收

| # | 验收标准 | 状态 | 说明 |
|---|---------|------|------|
| 1 | xxx | ✅ | 已在 ComponentA.vue 实现 |
| 2 | xxx | ❌ | 未找到对应实现，需补充 |
| 3 | xxx | ⚠️ | 实现了但缺少 xxx 情况的处理 |

### 需修复问题

#### 问题 1（❌ 未实现）
- **验收标准**：xxx
- **修复建议**：xxx
- **涉及文件**：`src/views/xxx/xxx.vue`
```

---

## 不确定时的处理

遇到以下情况时，**暂停验收并询问用户**：

1. 文档描述模糊，无法判断代码是否符合
2. 代码实现与文档有出入但可能是合理演进
3. 功能尚未开发（标注为"待开发"而非"未实现"）

## 约束

- 验收基于需求文档中的验收标准，不额外发明验收项
- 数据和配置必须与文档完全一致，不允许自行推断
- 验收报告客观呈现结果，不在报告中做需求变更建议
