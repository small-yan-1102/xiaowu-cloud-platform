# 测试 PRD 审阅报告（模板）

> **产出人**：测试工程师
> **接收人**：产品经理
> **用途**：测试从可测试性视角审阅 PRD 和产品原型后，向产品经理反馈需要澄清、补充或修正的问题，推动 PRD 达到可测试标准。
> **使用说明**：`{花括号}` 内为需要填写的内容，`<!-- 说明 -->` 为填写指引（最终文档中删除）。

## 二、阻塞性问题

<!-- 为什么需要这部分：这些问题不解决，测试无法设计用例或无法判断预期结果。产品经理必须优先处理这些问题。归类为"阻塞性"是为了让产品经理理解优先级——这些是测试工作的前置依赖。 -->

> 阻塞性问题 = PRD 中缺失的关键规则或存在的逻辑矛盾，导致测试无法确定正确的预期行为。

| # | 所属模块 | 问题描述 | PRD 位置 | 影响范围 | 建议处理方式 | 产品回复 |
|---|---------|---------|---------|---------|------------|---------|
| B-1 | {模块名} | {具体问题描述} | {PRD 章节编号} | {影响哪些测试场景} | {建议如何补充/修改} | |
| B-2 | | | | | | |

<!-- 填写指引：
- "问题描述"要具体，写清楚"PRD 说了什么"和"缺了什么 / 矛盾在哪"
- "影响范围"说明这个问题会阻塞哪些测试场景，让产品理解严重性
- "建议处理方式"给出测试的建议方案，降低产品的决策成本
  - AI 产品设计场景：改为写「需要明确：{缺失的信息项清单}」，不给出具体补充方案
- "产品回复"列留空，由产品经理填写澄清结论
-->

<!-- 常见的阻塞性问题类型，供参考：
1. 业务规则缺失：如边界条件未定义（最大值、最小值、空值怎么处理）
2. 逻辑矛盾：PRD 不同章节对同一功能的描述不一致
3. 异常流程未定义：只描述了正常流程，未说明失败/异常时的行为
4. 关键状态流转不明：实体的状态机（如订单状态）缺少完整定义
5. 跨模块交互未明确：A 模块操作对 B 模块的影响未说明
-->

## 三、验收标准不足 / 需补充项

<!-- 为什么需要这部分：PRD 中的功能描述虽然可以理解，但缺少明确的验收标准，测试只能"猜"预期结果。这些问题不阻塞测试启动，但会导致测试用例的预期结果不准确，后期可能产生大量"这算不算 Bug"的争议。提前补充可以减少后期的沟通成本。 -->

> 需补充项 = 功能描述能理解，但验收标准不够精确，测试无法写出确定的预期结果。

|---|---------|-------|-------------|------------|---------|---------|
| S-1 | {模块名} | {功能名} | {PRD 当前怎么写的} | {需要补充什么} | {为什么需要补充} | |
| S-2 | | | | | | |

<!-- 填写指引：
- "当前 PRD 描述"引用原文或概括，让产品经理知道你在说哪段内容
- "需补充的内容"要具体到测试需要知道的信息粒度
- 常见需补充类型：
  a. 输入约束缺失（字段长度、字符集、格式要求）
  b. 默认值未说明
  c. 列表排序规则未说明
  d. 分页/搜索/筛选行为未说明
  e. 操作成功/失败的提示文案未说明
  f. 并发操作的预期行为未说明
  g. 性能指标未说明（响应时间、并发量）
-->

## 四、描述不精确 / 存在歧义

<!-- 为什么需要这部分：PRD 中某些描述可能有多种理解方式，测试和产品的理解可能不同。如果不提前对齐，测试可能按错误理解编写用例，导致后期返工。这部分的核心是"我理解的是 X，你的意思是不是 X？" -->

> 描述不精确 = PRD 文字有多种理解方式，测试需要与产品确认唯一正确的理解。

| # | 所属模块 | PRD 原文 / 位置 | 可能的理解 A | 可能的理解 B | 测试倾向的理解 | 产品回复 |
|---|---------|---------------|------------|------------|-------------|---------|
| A-1 | {模块名} | {原文引用或章节} | {理解方式 A} | {理解方式 B} | {测试认为哪个更合理} | |
| A-2 | | | | | | |

<!-- 填写指引：
- 明确列出两种（或多种）可能的理解，而不是只说"描述不清楚"
- "测试倾向的理解"表达测试的判断，降低产品的回复成本——产品只需要说"同意"或"应该是 B"
-->
