# PRD 功能点编写规范模板

> **目标**：让产品写得出、开发能对齐、测试可直接转用例
> **核心原则**：产品负责"做什么+业务规则"，技术负责"怎么实现+接口字段"，双方职责在文档中物理隔离

---

## 一、模板总览

每个功能点由 **5 个区块** 组成：

```
┌─────────────────────────────────────┐
│  ① 功能描述（产品填写）              │  ← 用户视角：谁在哪做什么
│  ② 业务规则（产品填写）              │  ← 逻辑判断、枚举值、边界
│  ③ 验收标准（产品填写）              │  ← 可验证的 checklist
│  ④ 技术确认项（产品标记→技术填写）    │  ← 接口、字段、关联系统
│  ⑤ 异常与降级（产品+技术协作）        │  ← 失败场景的兜底行为
└─────────────────────────────────────┘
```

**颜色标记约定**（适用于 Confluence / 飞书文档）：
- 🟢 绿色 = 产品已确认
- 🟡 黄色 = 待技术确认
- 🔴 红色 = 有分歧待讨论

---

## 二、功能点模板（可直接复制使用）

---

### 【功能编号】功能名称

**所属模块**：xxx模块
**需求来源**：（需求单号 / 用户反馈 / 业务目标）
**关联功能**：→ 参见 [功能编号] （如果本功能依赖其他功能的规则）

---

#### ① 功能描述（产品填写）

**用户场景**：
> 用户在 [什么页面]，处于 [什么状态] 时，执行 [什么操作]，系统应 [产生什么结果]。

**触发方式**：
- [ ] 用户主动操作（按钮点击 / 表单提交 / 筛选）
- [ ] 系统自动触发（定时任务 / 状态变更 / MQ消息）
- [ ] 第三方回调

---

#### ② 业务规则（产品填写）

> **要求**：规则必须使用"如果…则…否则…"的明确表述，禁止使用"根据情况"、"适当处理"等模糊用语。

**规则清单**：

| 规则编号 | 条件 | 结果 | 备注 |
|---------|------|------|------|
| R1 | 如果 xxx = A | 则显示"xxx" | |
| R2 | 如果 xxx = B | 则显示"yyy" | |
| R3 | 如果数量 > 上限值 | 则拒绝操作并提示"xxx" | 上限值 = 500 |

**枚举值定义**（如涉及状态、类型等，必须列全）：

| 枚举字段 | 可选值 | 中文显示 | 说明 |
|---------|--------|---------|------|
| 视频状态 | normal | 正常 | 可发布 |
| | expired | 已过期 | 不可发布，可重新获取 |
| | redoing | 重做中 | 不可操作 |
| | redo_failed | 重做失败 | 可重新发起 |
| | damaged | 已损坏 | 不可操作 |

**边界值定义**：

| 边界场景 | 阈值 | 超过后行为 |
|---------|------|----------|
| 备片区上限 | 500条 | 全部拒绝，提示"备片区已满，请先清理" |
| 批量操作上限 | 15个 | 超出部分忽略，提示"最多选择15个" |
| 有效期预警 | ≤48小时 | 列表标记⚠️图标 |

---

#### ③ 验收标准（产品填写）

> **要求**：每条标准必须是**可直接验证**的陈述句，以"应"或"应该"开头，一条标准只验一件事。

- [ ] AC1：选择 voice_source=cloning 的视频发布时，通道套餐**应**显示"译制代运营"
- [ ] AC2：选择 voice_source=original 的视频发布时，通道套餐**应**显示"代运营"
- [ ] AC3：选择 voice_source=tts 的视频发布时，通道套餐**应**显示"译制代运营"
- [ ] AC4：备片区已有500条时，加入操作**应**被拒绝并显示提示文案"xxx"
- [ ] AC5：筛选"正常"状态时，结果列表中**不应**出现"重做中/已损坏/重做失败"的记录

> ⚠️ **检查点**：如果产品无法写出 AC，说明业务规则还没想清楚，需要先完善规则再写 AC。

---

#### ④ 技术确认项（产品标记 → 技术填写）

> **说明**：此区块由产品列出需要技术回答的问题，技术在评审前或评审中填写。
> 未填写的项目在评审会上作为**必须闭环的待办项**跟踪。

**4.1 接口信息** 🟡

| 确认项 | 产品的问题 | 技术回答 | 状态 |
|-------|-----------|---------|------|
| 接口路径 | 本功能是否需要新接口？还是复用现有接口？ | 🟡 待确认 | |
| 请求方式 | — | 🟡 待确认（GET/POST/PUT） | |
| 请求参数 | 需要传哪些参数？参数类型是什么？ | 🟡 待确认 | |
| 响应结构 | 返回哪些字段？字段命名是什么？ | 🟡 待确认 | |

**4.2 字段映射** 🟡

| 产品说的 | 技术确认的字段名 | 字段类型 | 状态 |
|---------|----------------|---------|------|
| "有效期至" | 🟡 待确认（expiresAt? expires_at?） | 🟡 | |
| "视频时长" | 🟡 待确认（duration? durationStr?） | 🟡 | |
| "文件大小" | 🟡 待确认（fileSize? fileSizeStr?） | 🟡 | |
| "人声来源" | 🟡 待确认（voiceSource? voice_source?） | 🟡 | |

**4.3 关联系统** 🟡

| 确认项 | 产品的问题 | 技术回答 | 状态 |
|-------|-----------|---------|------|
| 是否调用外部系统 | 本功能是否依赖剧译宝/AMS/其他系统？ | 🟡 待确认 | |
| 数据来源 | 数据从哪个库/表读取？ | 🟡 待确认 | |
| 状态同步 | 状态变更是否需要通知其他系统？ | 🟡 待确认 | |
| MQ依赖 | 是否涉及消息队列？延迟时间？ | 🟡 待确认 | |

**4.4 已有接口变更** 🟡

| 原接口路径 | 变更内容 | 技术回答 | 状态 |
|-----------|---------|---------|------|
| （如有变更的接口在此登记） | 新增字段/修改字段/废弃字段 | 🟡 待确认 | |

---

#### ⑤ 异常与降级（产品 + 技术协作）

> **说明**：产品定义"用户看到什么"，技术定义"系统怎么兜底"。

| 异常场景 | 用户看到什么（产品填写） | 系统如何处理（技术填写） |
|---------|----------------------|----------------------|
| 网络超时 | 🟢 提示"网络异常，请稍后重试" | 🟡 待确认（重试次数/超时时间） |
| 数据为空 | 🟢 展示空状态页，文案"暂无数据" | 🟡 待确认（返回空数组还是null） |
| 无权限 | 🟢 提示"无操作权限" | 🟡 待确认（返回401还是403） |
| 接口报错 | 🟢 提示"系统异常，请联系管理员" | 🟡 待确认（错误码约定） |
| 并发冲突 | 🟢 提示"数据已被修改，请刷新" | 🟡 待确认（乐观锁/悲观锁） |

## 三、使用示例（以本项目实际功能为例）

---

### 【PUB-001】发布抽屉页-通道套餐展示

**所属模块**：发布模块
**需求来源**：译制成品模块迭代 V4.0
**关联功能**：→ 参见 [LIST-005] 人声来源标记字段

---

#### ① 功能描述

**用户场景**：
> 用户在译制成品列表页，选择一个或多个正常状态的视频，点击"发布"按钮，系统打开发布抽屉页，自动展示该视频对应的通道名称和套餐类型。

**触发方式**：
- [x] 用户主动操作（点击"发布"按钮）

---

#### ② 业务规则

| 规则编号 | 条件 | 结果 |
|---------|------|------|
| R1 | voice_source = original（原始人声） | 套餐显示"代运营" |
| R2 | voice_source = cloning（音色克隆） | 套餐显示"译制代运营" |
| R3 | voice_source = tts（TTS合成） | 套餐显示"译制代运营" |
| R4 | voice_source 为空或未知值 | 🔴 待讨论：默认走哪个套餐？ |

**枚举值定义**：

| 枚举字段 | 可选值 | 说明 |
|---------|--------|------|
| voice_source | original | 原始人声-无后期处理 |
| | cloning | 音色克隆-AI克隆原声 |
| | tts | TTS合成-文本转语音 |

---

#### ③ 验收标准

- [x] AC1：voice_source=cloning 的视频，通道套餐**应**显示"译制代运营"
- [x] AC2：voice_source=original 的视频，通道套餐**应**显示"代运营"
- [x] AC3：voice_source=tts 的视频，通道套餐**应**显示"译制代运营"
- [ ] AC4：同时选择不同 voice_source 的视频批量发布时，**应**各自显示对应套餐

---

#### ④ 技术确认项

**4.1 接口信息**

| 确认项 | 产品的问题 | 技术回答 | 状态 |
|-------|-----------|---------|------|
| 接口路径 | 打开发布抽屉时调什么接口？ | `GET /dispatcher/pipeline/list` | 🟢 已确认 |
| 请求参数 | 传视频ID还是作品ID？ | `ids: number[]`（传 dubbed_product_video.id） | 🟢 已确认 |
| 套餐字段 | 套餐信息在哪个字段返回？ | `pipelineType: string` | 🟢 已确认 |

**4.2 字段映射**

| 产品说的 | 技术确认的字段名 | 字段类型 | 状态 |
|---------|----------------|---------|------|
| "人声来源" | voiceSource | string | 🟢 已确认 |
| "套餐类型" | pipelineType | string | 🟢 已确认 |
| "通道名称" | pipelineName | string | 🟢 已确认 |

---

| 异常场景 | 用户看到什么 | 系统如何处理 |

## 四、评审检查清单

产品提交 PRD 后，测试在评审会上按此清单逐项检查：

### 产品侧（评审前必须完成）

- [ ] 每个功能点是否有明确的用户场景描述？
- [ ] 业务规则是否使用了"如果…则…否则…"的明确表述？
- [ ] 涉及状态/类型的枚举值是否列全？
- [ ] 边界值是否有明确阈值和超限行为？
- [ ] 是否写了可直接验证的验收标准（AC）？
- [ ] 跨功能依赖是否有显式引用（→ 参见 xxx）？

### 技术侧（评审中必须闭环）

- [ ] 所有 🟡 标记的技术确认项是否已回答？
- [ ] 接口路径和请求方式是否已确认？
- [ ] 返回字段命名是否与前端对齐？
- [ ] 是否有接口变更需要登记？
- [ ] 异常场景的系统处理方式是否已明确？

### 遗留项管理

- [ ] 所有 🔴 标记的争议项是否已记录为待办？
- [ ] 待办项是否指定了负责人和截止日期？
- [ ] 下次评审前是否需要补充评审？

## 五、常见反模式（避坑指南）

| ❌ 不要这样写 | ✅ 应该这样写 | 原因 |
|-------------|-------------|------|
| "根据情况展示不同套餐" | "voice_source=cloning时展示译制代运营" | 测试无法验证"情况" |
| "支持多种状态筛选" | "状态枚举：normal/expired/redoing/redo_failed/damaged" | 不列全=测试漏覆盖 |
| "展示视频时长" | "展示视频时长（🟡待技术确认字段名）" | 字段名不确认=断言写错 |
| "调用相关接口" | "🟡待技术确认：是否新增接口？复用哪个？" | 不标记=没人跟进 |
| "异常时给出提示" | "网络超时时提示'网络异常，请稍后重试'" | 不写具体文案=前端自由发挥 |
| 把规则写在A功能，B功能不引用 | B功能写"→ 参见 A功能 R2规则" | 不引用=测试找不到 |
