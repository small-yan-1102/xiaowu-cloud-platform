# test-prd-review 规则体系说明

## 整体定位

`test-prd-review` 是一套**测试工程师视角的 PRD 审阅技能**，包含 1 个主规则文件 + 3 个附属参考文件，共计 4 个文件（约 77KB）。其职责是：接收 PRD + 原型作为输入 → 产出需求理解总结 + 审阅报告两份产物。

---

## 文件结构与各自作用

```
test-prd-review/
├── SKILL.md                           (33.7KB) ← 主规则：工作流 + 审阅维度 + 验收标准
├── test-prd-review-template.md        (9.4KB)  ← 审阅报告的输出模板
├── prd-feature-writing-standard.md    (11.2KB) ← PRD 功能点编写规范（审阅参照基准）
└── prd-dependency-and-path-standard.md(18.5KB) ← 依赖与路径分析方法论
```

## 1. SKILL.md — 主规则（工作流引擎 + 审阅清单）

**作用**：定义技能的完整执行流程、所有审阅维度和验收标准。是整个规则体系的核心骨架。

### 核心内容

| 模块 | 内容 |
|------|------|
| **运行模式** | 独立模式 / 工作流模式，决定输入来源和输出位置 |
| **角色定位** | 资深 QA 的职责边界（做什么 / 不做什么） |
| **输入/输出契约** | 5 项必需输入 + 2 项可选输入 → 2 份输出产物 |
| **4-Phase 工作流** | Phase 1 依赖检查 → Phase 2 需求理解 → Phase 3 审阅报告 → Phase 4 自我补漏 |
| **19 个审阅维度（a~s）** | 覆盖从 PRD 结构完整性到数据流转、权限矩阵、通知场景的全方位检查 |
| **6 类问题编号** | B-阻塞 / S-补充 / A-歧义 / D-不一致 / Q-质量 / N-非功能 |
| **验收标准** | 4 大类 38 项 checklist（结构完整性 / 内容准确性 / 审阅思维质量 / 格式规范） |
| **错误处理** | Fatal / Error / Warn 三级分级 + 幂等性保证 |
| **核心原则** | 6 条审阅立场 + 3 个思维工具（苏格拉底提问 / 第一性原理 / 奥卡姆剃刀） |

### 4-Phase 工作流

```
Phase 1: 依赖检查与输入验证
  ↓ 全部就绪
Phase 2: 需求理解与总结 → 产出 test-prd-summary.md
  ↓ 用户确认理解正确
Phase 3: 基于模板的审阅报告编写 → 产出 test-prd-review-report.md 草稿
  ↓
Phase 4: 自我审阅与补漏循环（最多 3 次）→ 产出最终版两份文件
```

### 19 个审阅维度速览

| 编号 | 维度名称 | 关注重点 |
|------|---------|---------|
| a | PRD 功能点结构完整性 | 逐功能对照测试相关的 4 个区块（功能描述/业务规则/AC/异常），④技术确认项不在测试审阅范围 |
| b | 阻塞性问题 | 关键规则缺失、逻辑矛盾、跨模块依赖、变更传播 |
| c | 验收标准不足 | 字段约束、默认值、提示文案、并发行为、搜索/筛选/排序/分页详细行为、时间相关行为 |
| d | 描述歧义 | 苏格拉底提问法拆解模糊描述 |
| e | 文档不一致 | PRD vs 原型、PRD 内部各章节一致性 |
| f | 文档质量 | 编号、术语、格式 |
| g | 依赖关系审阅 | 基于 DEP-1~DEP-7 检查清单 |
| h | 操作路径审阅 | 基于 PATH-1~PATH-8 检查清单 |
| i | 版本一致性 | 对比历史 PRD，标记回归风险 |
| j | 外部平台限制 | 第三方平台限制的校验规则和用户提示 |
| k | 用户体验合理性 | 重复操作、批量支持、操作成本 |
| l | 工单类实体 | 工单创建→流转→时效→结果处理 |
| m | 数据流转完整性 | 数据实体全生命周期（产生→存储→变更→销毁→中断恢复→一致性→追溯） |
| n | 多端实现对齐 | 各端功能点/数据项/业务规则是否对齐 |
| o | 消息/通知场景 | 通知触发条件、对象、渠道、内容模板、失败处理 |
| p | 权限矩阵交叉验证 | 各角色功能可见性、数据可见范围、越权预期表现 |
| q | 迭代边界与范围声明 | 本期范围、排除范围的界面处理、"后续优化"类措辞 |
| r | 非功能需求 | 性能/兼容性/安全/数据安全/容错分项检查 |
| s | 测试工作量预估 | 用例数/优先级/前置依赖 |

### 6 类问题编号体系

| 前缀 | 类型 | 定义 |
|------|------|------|
| B- | 阻塞性问题 | PRD 中缺失的关键规则或逻辑矛盾，导致测试无法确定正确预期行为 |
| S- | 需补充项 | 功能描述能理解，但验收标准不够精确，测试无法写出确定的预期结果 |
| A- | 歧义项 | PRD 文字有多种理解方式，测试需要与产品确认唯一正确的理解 |
| D- | 不一致项 | PRD 与原型、或 PRD 内部各章节在同一功能上的描述存在差异 |
| Q- | 文档质量 | 不影响功能理解，但影响文档的准确性和可维护性 |
| N- | 非功能需求 | PRD 中未提及或未明确标准的非功能维度 |

### 3 个思维工具

| 工具 | 核心思想 | 应用场景 |
|------|---------|---------|
| **苏格拉底提问** | 用结构化反问拆解模糊描述 | 遇到"根据情况"、"相关用户"等模糊用语时 |
| **第一性原理** | 追问功能的根本目的 | PRD 描述了方案但没说清要解决的核心问题时 |
| **奥卡姆剃刀** | 质疑不必要的复杂性 | PRD 过于复杂或审阅报告堆砌琐碎发现时 |

## 2. test-prd-review-template.md — 审阅报告输出模板

**作用**：定义审阅报告的标准输出结构。审阅发现按此模板的 8 个章节填入，确保报告格式统一、便于产品经理逐项回复。

### 8 个章节

| 章节 | 内容 | 问题编号前缀 |
|------|------|------------|
| 一、审阅结论摘要 | 问题统计 + 整体评估（可进入测试/修改后可进入/需重大补充） | — |
| 二、阻塞性问题 | 缺失的关键规则或逻辑矛盾，必须修改才能进入测试 | B- |
| 三、验收标准不足 | 功能可理解但 AC 不精确，测试只能「猜」预期结果 | S- |
| 四、描述不精确/歧义 | 有多种理解方式，需与产品确认唯一正确的理解 | A- |
| 五、PRD 与原型不一致 | PRD 与原型或 PRD 内部各章节的描述存在差异 | D- |
| 六、文档质量问题 | 编号错误、术语不一致、格式问题 | Q- |
| 七、非功能需求确认 | 安全/性能/兼容/数据安全/容错 | N- |
| 八、测试范围与工作量预估 | 用例数/测试类型/优先级/前置依赖 | — |
| 附录 | 问题状态跟踪表（待回复/已回复/已关闭） | — |

**关键设计**：每个问题都含「产品回复」列（留空），使报告成为产品经理可直接填写回复的协作文档。

## 3. prd-feature-writing-standard.md — PRD 功能点编写规范

**作用**：定义 PRD 功能点应该「长什么样」的标准。不是直接用于审阅，而是作为审阅的**参照基准** — 告诉审阅者「好的 PRD 功能点应具备哪些内容」，从而发现差距。

### 核心内容

| 部分 | 内容 |
|------|------|
| **5 区块结构** | ① 功能描述（用户场景+触发方式） → ② 业务规则（条件-结果表+枚举+边界） → ③ 验收标准（可验证的 AC 陈述句） → ④ 技术确认项（接口/字段/关联系统，🟡标记）**※不在测试审阅范围** → ⑤ 异常与降级（用户看到什么+系统如何处理） |
| **颜色标记约定** | 🟢 已确认 / 🟡 待技术确认 / 🔴 有分歧 |
| **完整示例** | 以「发布抽屉页-通道套餐展示」为例，展示 5 个区块的标准写法 |
| **评审检查清单** | 产品侧 6 项 + 技术侧 5 项 + 遗留项管理 3 项 |
| **反模式列表** | 6 种常见的 PRD 错误写法及对应的正确写法 |

### 在 SKILL.md 中的引用点

- Phase 1 步骤 2：加载规范，理解 5 区块结构
- Phase 3 维度 a：逐功能对照 5 区块标准进行结构完整性检查
- Phase 3 步骤 3：使用评审检查清单做最终扫描

## 4. prd-dependency-and-path-standard.md — 依赖与路径分析方法论

**作用**：提供系统性的**分析方法论**，指导审阅者如何构建依赖关系图和操作路径清单，以及如何判断问题严重程度。是规则体系中「技术含量最高」的参考文件。

### 核心内容

| 部分 | 内容 |
|------|------|
| **1.1 依赖类型定义** | 4 类依赖（数据/状态/功能/规则）+ 识别隐性依赖的 5 种信号 |
| **1.2 依赖图构建规范** | 依赖关系清单表格 + Mermaid 拓扑图 + 变更影响分析表 |
| **1.3 依赖完整性检查清单** | DEP-1~DEP-7（上游完整性/下游覆盖/循环依赖/外部系统登记/数据契约/状态触发/引用对称） |
| **1.4 变更传播分析方法** | 4 步法：识别变更点 → 沿依赖图追踪 → 检查 PRD 覆盖 → 输出影响表 |
| **1.5 依赖问题归类规则** | 7 种问题场景 → B/A/S/D/Q 系列的映射表 |
| **2.1 操作路径梳理方法** | 5 段结构（入口→导航→操作→结果→跳转）+ 3 类路径（主/分支/异常）+ 7 步梳理方法 |
| **2.2 路径表达规范** | 路径清单表格 + 编号规则 + PRD 覆盖状态标记（✅/⚠️/❌）+ 跨模块端到端路径表 |
| **2.3 路径完整性检查清单** | PATH-1~PATH-8（入口可达/路径连通/退出路径/主路径覆盖/异常路径覆盖/分支完备/端到端/深层直达） |
| **2.4 路径可达性验证方法** | 4 步法：准备路径清单 → 原型中逐条验证 → 更新覆盖状态 → 输出差异 |
| **2.5 路径问题归类规则** | 7 种问题场景 → B/S/A/D/Q 系列的映射表 |
| **三、反模式速查** | 10 种 PRD 中关于依赖/路径的常见问题写法及归类 |

### 在 SKILL.md 中的引用点

- Phase 1 步骤 3：加载规范，理解 4 种依赖类型和 5 段路径结构
- Phase 2 步骤 4：构建依赖关系清单 + Mermaid 图 + 操作路径清单
- Phase 3 维度 g：使用 DEP-1~DEP-7 检查清单审阅依赖关系
- Phase 3 维度 h：使用 PATH-1~PATH-8 检查清单审阅操作路径
- Phase 4 覆盖度检查：核对依赖关系清单和路径清单的完整性

## 4 个文件的协作关系

```
                    ┌─ prd-feature-writing-standard.md
                    │   (告诉你"好的 PRD 长什么样")
                    │
SKILL.md ──引用──→ ├─ prd-dependency-and-path-standard.md
(工作流引擎)        │   (告诉你"怎么分析依赖和路径")
                    │
                    └─ test-prd-review-template.md
                        (告诉你"审阅报告怎么写")
```

- **SKILL.md** 是引擎，定义「做什么、按什么顺序做、做到什么标准」
- **3 个附属文件** 是被引用的参照标准，各有独立职责，SKILL.md 在工作流的不同阶段加载并引用它们

## 输入与输出一览

### 输入

| 输入项 | 必需 | 说明 |
|-------|------|------|
| PRD 文档 | 是 | Markdown 格式的产品需求文档 |
| 产品原型 | 是 | HTML 文件或可访问的 URL |
| 审阅报告模板 | 是 | 技能附属文件（自带） |
| PRD 功能点编写规范 | 是 | 技能附属文件（自带） |
| 需求依赖与路径分析规范 | 是 | 技能附属文件（自带） |
| 相关历史测试用例 | 否 | 用于了解已覆盖场景 |
| 历史版本 PRD | 否 | 用于版本一致性检查 |

### 输出

| 输出产物 | 文件名 | 说明 |
|---------|--------|------|
| PRD 需求理解总结 | `test-prd-summary.md` | 覆盖所有功能模块的测试视角理解，含依赖图、路径清单、数据流转表 |
| 测试 PRD 审阅报告 | `test-prd-review-report.md` | 严格按模板 8 章节输出的审阅发现，产品回复列留空 |
