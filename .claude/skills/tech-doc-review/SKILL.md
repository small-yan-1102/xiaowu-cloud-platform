# 技术文档审阅技能

> **版本**: v2.0
> **更新日期**: 2026-04-02
> **适用模式**: AI 读取开发技术文档 → 对照 PRD 和审阅维度标准 → 输出结构化审阅报告

---

## 附属资源文件

| 文件 | 路径 | 说明 |
|------|------|------|
| 审阅维度标准 | `references/review-dimensions.md` | 文档类型识别指南 + 全部审阅维度（C/U/F/O/D/A/T/R/K）+ 问题分级标准，Phase 3 执行审阅时加载 |
| 审阅报告模板 | `assets/review-templates.md` | 单文档审阅报告模板 + 审阅汇总报告模板，Phase 5 输出时加载 |

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 输入收集与确认
- [ ] Phase 2: 文档分类与维度选择
- [ ] Phase 3: 执行审阅（核心）
- [ ] Phase 4: 问题汇总与去重分类
- [ ] Phase 5: 输出审阅报告
```

---

### Phase 0: 定向收集

**目标**：在任何审阅行为开始前，收集必要信息，避免审阅后大范围返工。

向用户提出以下问题，**等待用户全部回答后再进行任何操作**：

1. **审阅范围**：全量审阅所有文档，还是聚焦某个特定模块或文档类型？
2. **审阅模式**：全量审阅 / 增量审阅（仅变更部分）？
3. **模块标识**：当前迭代是否涉及多个功能模块？如果是，请提供各模块的英文标识（如 `order`, `payment`）。命名约束：仅允许 `[a-z0-9-]`，长度 ≤ 30 字符。
4. **文档路径**：技术文档位于哪个路径（默认：`input/tech/*.md`）？PRD 位于哪个路径（默认：`input/prd/*.md`）？
5. **历史汇总报告**：是否已有审阅汇总报告（`review/tech-review-summary.md`）？如果有，本次是追加审阅（增量更新汇总报告）还是从头开始？

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 开始任何审阅，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答 → 进入 Phase 1
- 用户表示"按默认来" → 使用默认路径，单模块默认标识 `main`，进入 Phase 1

---

### Phase 1: 输入收集与确认

**目标**：确认资源文件存在，验证输入材料完整。

**步骤**：

1. **检查审阅维度标准**：
   - 检查 `references/review-dimensions.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「审阅维度标准文件缺失」

2. **检查审阅报告模板**：
   - 检查 `assets/review-templates.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「审阅报告模板文件缺失」

3. **验证输入材料**：
   - 扫描技术文档路径（`input/tech/*.md`）
   - 未找到 → 🔵 使用 AskUserQuestion 请用户提供文件路径
   - 扫描 PRD 文档路径（`input/prd/*.md`）
   - 未找到 → **Fatal**：终止执行，提示「PRD 文档缺失，无法执行需求一致性检查」

4. **识别模块标识**：
   - 根据 Phase 0 用户回答和文档数量，确认模块标识列表
   - 多模块 → 为每个模块分配英文标识（仅允许 `[a-z0-9-]`，长度 ≤ 30）
   - 单模块 → 默认 `main`

5. **汇总检查结果**：
   - 有 Fatal 缺失 → 一次性列出所有缺失项，不分多次告知
   - 全部就绪 → 输出就绪清单，进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到所有必要依赖均已就绪。

---

### Phase 2: 文档分类与维度选择

**目标**：识别每份技术文档的类型，确定本次审阅的维度范围。

**步骤**：

1. **读取审阅维度标准**：加载 `references/review-dimensions.md`，使用「§一 文档类型识别指南」

2. **对每份技术文档识别类型**：

   | 文档特征 | 判定为 |
   |---------|--------|
   | 数据库表结构、ER图、SQL DDL | 数据库设计文档 |
   | API 路径、请求/响应 JSON | 接口定义文档 |
   | 页面布局、组件交互、前端路由 | 前端用户故事/开发规格 |
   | data-testid 映射表 | data-testid 映射文档 |
   | 系统架构图、状态机、时序图 | 后端技术设计文档 |
   | 包含以上多种内容 | 综合技术方案 |

3. **确定审阅维度**：按 `references/review-dimensions.md` 中「文档类型→维度」映射表确定：
   - 必审维度（全部执行）
   - 重点维度（重点执行）
   - 可选维度（按情况执行）

4. **输出分类结果**，向用户确认后进入 Phase 3

---

### Phase 3: 执行审阅

**目标**：按审阅维度标准逐项执行审阅，记录所有发现的问题。

> 审阅过程全程参照 `references/review-dimensions.md` 中各维度的检查项

**步骤**：

#### Step 3.1 需求一致性审阅（C-1～C-6）

- 按 `references/review-dimensions.md` §二 C-1～C-6 检查项逐项执行
- 对照 PRD 逐项检查，记录问题并标记级别（B/S/Q）

检查项：C-1 功能点覆盖 / C-2 业务规则一致 / C-3 状态机一致 / C-4 枚举值一致 / C-5 文案一致 / C-6 超范围设计

#### Step 3.2 风险场景识别（C-7～C-12）

- 按 `references/review-dimensions.md` §二 C-7～C-12 检查项逐项执行
- **全部检查，按去重判定规则分类产出**（详见 `references/review-dimensions.md` §二「去重判定规则」）

#### Step 3.3 AI 可测试性维度审阅

按 Phase 2 确定的维度，逐一执行 `references/review-dimensions.md` 中对应维度的检查项：

- **U（UI元素文案）**：U-1～U-8
- **F（表单字段约束）**：F-1～F-9
- **O（状态可观测性）**：O-1～O-5
- **D（测试数据准备）**：D-1～D-6
- **A（异步等待策略）**：A-1～A-5
- **T（data-testid覆盖）**：T-1～T-4（可选维度）
- **R（排序筛选规则）**：R-1～R-5
- **K（跨系统可执行性）**：K-1～K-4

> ⛔ **DO NOT** 进入 Phase 4，直到当前文档所有必审维度均已执行完毕。

---

### Phase 4: 问题汇总与去重分类

**目标**：按级别分类汇总所有问题，执行风险项去重处理，给出整体评估结论。

**步骤**：

1. **按严重级别分类**（参照 `references/review-dimensions.md` §十一 问题分级标准）：

   | 级别 | 代码 | 处理要求 |
   |------|------|---------|
   | 阻塞 | B | 必须在测试设计前解决 |
   | 补充 | S | 建议补充，可并行处理 |
   | 疑问 | Q | 需要开发回复确认 |
   | 风险 | R | 按去重判定规则处理 |
   | 建议 | A | 可选改进 |

2. **风险项去重处理**（按 `references/review-dimensions.md` §二「去重判定规则」）：
   - C-9, C-12（已覆盖）→ 仅标记确认，不产出补充场景
   - C-8, C-11（视PRD而定）→ 检查PRD定义后归类
   - C-7, C-10（未覆盖）→ 必须产出补充AI测试场景

3. **给出整体评估**（按 `references/review-dimensions.md` §十一「整体评估判定条件」）：
   - B = 0 → 可进入测试设计
   - B ≤ 3 且不涉及核心设计缺陷 → 需补充后进入
   - B > 3 或存在核心设计缺陷 → 需重新设计

> ⛔ **DO NOT** 进入 Phase 5，直到所有问题分类完毕且整体评估结论已确定。

---

### Phase 5: 输出审阅报告

**目标**：按 `assets/review-templates.md` 中的模板，生成单文档审阅报告并更新汇总报告。

**步骤**：

1. **读取审阅报告模板**：加载 `assets/review-templates.md`

2. **确定输出路径**：

   | 输出物 | 路径 | 说明 |
   |-------|------|------|
   | 单文档审阅报告 | `review/tech-review-report-{module}-{type}.md` | 每审阅一份技术文档生成一份 |
   | 单文档审阅报告（复审） | `review/tech-review-report-{module}-{type}-r{N}.md` | 同模块同类型第N轮复审 |
   | 审阅汇总报告 | `review/tech-review-summary.md` | 聚合所有模块完成度 + 补充场景 |

   > **`{module}`**：功能模块英文标识，单模块默认 `main`
   > **`{type}`**：`backend` / `frontend` / `api` / `db` / `testid` / `full`

3. **按模板一生成单文档审阅报告**，填写全部章节

4. **按模板二更新审阅汇总报告**：
   - 更新「§〇 审阅完成度」表格（含模块级判定）
   - 追加本次审阅的补充场景到「§二 风险场景→补充AI测试场景（聚合）」
   - 追加本次审阅的用例建议到「§三 风险场景补充的AI测试用例建议（聚合）」

5. **输出技术文档审阅门禁结论**：

   在审阅汇总报告末尾追加「技术文档审阅门禁」章节，给出明确的放行/阻塞判定：

   | 结论 | 判定条件 |
   |------|---------|
   | ✅ 可进入用例补全 | 无 🔴 模块，且整体评估为「可进入测试设计」或「需补充后进入」 |
   | ❌ 阻塞，禁止进入用例补全 | 存在 🔴 模块，或整体评估为「需重新设计」 |

   阻塞时，列出：
   - 每个 🔴 模块名称及阻塞原因
   - 建议开发修复的具体问题（B 级问题列表）
   - 修复后需重新执行 `tech-doc-review` 的范围

   部分阻塞时（仅部分模块为 🔴），说明：
   - 🟢/🟡 模块可正常进入 `api-test-case-design` / `test-case-design` Mode B
   - 🔴 模块暂不触发任何用例设计，等待修复后复审

6. **向用户展示**：输出文件路径 + 审阅结论摘要 + 整体评估 + 技术文档审阅门禁结论

## 验收标准

### 输出完整性

- [ ] 单文档审阅报告已写入正确路径
- [ ] 审阅汇总报告已同步更新
- [ ] 报告包含「审阅结论摘要」「需求一致性与风险场景」「AI可测试性问题清单」「对AI测试设计的影响评估」四个主要章节

### 审阅质量

- [ ] 已加载 `references/review-dimensions.md` 执行审阅
- [ ] C-1～C-6 需求一致性检查项均已执行（或标注不适用原因）
- [ ] C-7～C-12 风险场景检查项均已执行，并按去重规则分类
- [ ] 所有必审维度的检查项均已执行
- [ ] 每个问题已标注 B/S/Q/R/A 级别
- [ ] 整体评估结论已给出（可进入测试设计 / 需补充后进入 / 需重新设计）

### 下游衔接

- [ ] 审阅汇总报告的「§〇 审阅完成度」已更新模块级判定（🟢/🟡/🔴）
- [ ] 补充AI测试场景（C-7、C-10 类）已写入汇总报告「§二」和「§三」
- [ ] 汇总报告可供 `test-case-design` Mode B 直接读取

## 与 test-case-design 的衔接说明

审阅汇总报告 `review/tech-review-summary.md` 中的以下内容将作为 `test-case-design` Mode B 的输入：

- **§二**「风险场景→补充AI测试场景（聚合）」
- **§三**「风险场景补充的AI测试用例建议（聚合）」

模块级判定结果决定用例生成策略：

| 模块级判定 | 用例设计策略 |
|-----------|------------|
| 🟢 可生成E2E用例 | 可立即调用 `test-case-design` Mode B，scope = 该模块 |
| 🟡 技术文档不完整 | 跳过 `test-case-design` Mode B；调用 `api-test-case-design` 针对该模块设计接口用例 |
| 🔴 建议等待 | **阻塞**：该模块禁止触发任何用例设计，等待开发修复后重新执行 `tech-doc-review` |

> **门禁规则**：任何模块出现 🔴 时，汇总报告末尾输出「技术文档审阅门禁：❌ 阻塞」结论，并列出阻塞原因和修复建议。🟢/🟡 模块不受影响，可正常推进。
