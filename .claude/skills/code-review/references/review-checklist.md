# 代码审查检查清单

本文件定义了前端代码审查的多维度检查清单,按文件类型分类。

## 一、通用检查维度

### 1. 正确性 (Correctness)
- [ ] 逻辑正确,无潜在 bug
- [ ] 边界条件处理完善
- [ ] 类型使用准确
- [ ] 异步操作正确处理(async/await、Promise 链)
- [ ] 事件监听器和定时器正确清理
- [ ] 错误处理完善(try-catch、错误边界)

### 2. 安全性 (Security)
- [ ] 无 XSS 风险(动态 HTML 插入安全)
- [ ] 敏感信息不硬编码(API密钥、密码)
- [ ] 输入验证完善
- [ ] 无不安全的外部资源引用
- [ ] CSRF 防护(如适用)

### 3. 性能 (Performance)
- [ ] 无不必要的重渲染
- [ ] 大数据量使用虚拟滚动或分页
- [ ] 图片/资源懒加载
- [ ] 避免内存泄漏(清理定时器、事件监听)
- [ ] 无无效的重复请求
- [ ] 计算属性/监听器使用合理

### 4. 可读性 (Readability)
- [ ] 命名清晰有意义
- [ ] 函数单一职责,长度适中(≤ 50 行)
- [ ] 复杂逻辑有注释
- [ ] 魔法数字有常量定义
- [ ] 代码结构清晰,缩进一致

---

## 二、按文件类型专项检查

### Vue 组件 (.vue)

**组件结构**
- [ ] 组件接口定义完整(Props/Emit/Slots)
- [ ] Props 有类型定义和默认值
- [ ] 使用 scoped 样式(如项目要求)
- [ ] 组件命名符合 PascalCase 规范

**响应式数据**
- [ ] ref/reactive 使用正确
- [ ] 计算属性优于手动监听
- [ ] 避免深层嵌套响应式对象

**生命周期**
- [ ] onMounted/onUnmounted 配对使用
- [ ] 副作用清理逻辑完善
- [ ] 避免在生命周期中执行耗时操作

**模板**
- [ ] v-for 使用 :key
- [ ] v-if/v-show 使用合理
- [ ] 无内联样式(除非必要)
- [ ] 表单元素有 label 关联

---

### 接口层 (service.js)

**Mock 数据规范**
- [ ] Mock 数据使用 `MOCK_` 前缀 + 大写下划线
- [ ] Mock 数据字段名与后端接口一致
- [ ] 提供 USE_MOCK 开关
- [ ] 模拟网络延迟(setTimeout)
- [ ] Mock 数据集中管理,未分散在组件中

**接口定义**
- [ ] 函数命名: 动词开头 + 驼峰(getUserList)
- [ ] 参数类型定义完整
- [ ] 返回值结构统一({ code, data, message })
- [ ] 错误处理完善

**联调友好**
- [ ] 真实接口代码已注释但保留
- [ ] 切换开关位置明显
- [ ] Mock 数据便于删除

---

### 样式文件 (.css/.scss/.less)

**命名规范**
- [ ] 类名使用 kebab-case
- [ ] 避免使用 ID 选择器
- [ ] 避免深层嵌套(≤ 3 层)

**可维护性**
- [ ] 使用 CSS 变量(如项目要求)
- [ ] 避免 !important
- [ ] 样式复用合理(公共样式提取)

**性能**
- [ ] 避免通配符选择器
- [ ] 避免过度使用动画/过渡
- [ ] 大图使用适当格式(webp)

---

### 工具函数 (utils.js/hooks)

**函数设计**
- [ ] 函数单一职责
- [ ] 参数数量适中(≤ 3 个)
- [ ] 纯函数优先(无副作用)
- [ ] 有完善的类型定义

**错误处理**
- [ ] 边界条件处理
- [ ] 异常情况返回合理默认值
- [ ] 有必要的错误提示

**复用性**
- [ ] 避免重复代码
- [ ] 通用逻辑提取为公共方法
- [ ] 有使用示例或注释

---

## 三、项目规范检查

### 组件使用规范
- [ ] 优先使用项目已封装公共组件
- [ ] 禁止直接使用底层 UI 库原生组件(除非必要)
- [ ] 搜索表单使用项目标准搜索组件
- [ ] 表格使用项目标准表格组件

### 代码风格
- [ ] 遵循项目 ESLint 规则
- [ ] 遵循项目 Prettier 配置
- [ ] 文件命名: kebab-case(user-list.vue)
- [ ] 组件命名: PascalCase(UserList)
- [ ] 变量命名: 驼峰(userList)
- [ ] 常量命名: 大写下划线(MOCK_USER_LIST)

### 提交规范
- [ ] Commit message 符合规范
- [ ] 变更范围清晰
- [ ] 无调试代码(console.log 等)

---

## 四、问题严重度分级

| 级别 | 定义 | 处理要求 |
|------|------|---------|
| **Critical** | 必须修复 — bug、安全问题、数据损坏风险 | 阻断合并 |
| **Warning** | 建议修复 — 性能、可读性、潜在风险 | 合并前建议修复 |
| **Info** | 仅供参考 — 风格、优化建议 | 可选修复 |

---

## 五、审查检查流程

1. **收集变更**: 列出所有变更文件
2. **分类文件**: 按文件类型分组
3. **逐项检查**: 对照本清单逐文件审查
4. **分级问题**: 按 Critical/Warning/Info 分类
5. **输出报告**: 生成结构化审查报告

---

## 使用建议

- **常规审查**: 使用"通用检查维度" + 对应文件类型专项检查
- **安全专项**: 重点关注"安全性"维度
- **性能专项**: 重点关注"性能"维度
- **新成员代码**: 全量检查所有维度
- **紧急修复**: 仅检查"正确性"和"安全性"
