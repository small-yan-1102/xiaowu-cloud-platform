> **版本**: v1.0
> **更新日期**: 2026-04-08
> **适用模式**: AI 读取测试执行报告和缺陷清单 → 核验发布就绪条件 → 输出发布门禁结论（通过 / 不通过）

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载报告
- [ ] Phase 2: 执行门禁检查
- [ ] Phase 3: 输出门禁结论报告
```

---

### Phase 0: 定向收集

**目标**：确认检查依据的报告文件和通过率阈值。缺陷状态通过自动查询获取，无需用户手动填报。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **执行报告路径**：
   - UI 执行报告（默认：`test_reports/test_report_*.md`，取最新一份）
   - 接口执行报告（默认：`test_reports/api_report_*.md`，取最新一份）
   - 可指定具体文件，或只用其中一类
2. **测试报告路径**：test-report 生成的测试报告（默认：`test_reports/test-report-*.md`，取最新一份）
3. **通过率阈值**（可选，默认 90%）：P0+P1 用例最低通过率要求是多少？
4. **阻塞/跳过比例上限**（可选，默认 20%）：P0+P1 用例中阻塞+跳过占总用例数的最大允许比例是多少？超过则视为执行覆盖不足。
5. **Major 缺陷阈值**（可选，默认：仅展示不阻塞）：开放 Major 缺陷超过多少条时阻塞发布？不设则只在报告中展示数量供参考。

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。
> ℹ️ 当前开放缺陷数由 Phase 1 自动获取，**无需用户手动填报**。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答 → 进入 Phase 1
- 用户表示"按默认来" → 使用全部默认路径，通过率阈值 90%，阻塞/跳过上限 20%，Major 缺陷仅展示不阻塞，进入 Phase 1

---

### Phase 1: 加载报告

**目标**：读取执行报告、测试报告和缺陷清单，自动获取当前开放缺陷数，无需用户手动填报。

**步骤**：

1. **检查门禁报告模板**：
   - 检查 `assets/gate-report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「门禁报告模板文件缺失」

2. **读取执行报告**（按 Phase 0 指定路径）：
   - 文件不存在 → **Fatal**：终止执行，提示「执行报告不存在，请先执行测试」
   - 从执行报告提取：
     - 用例总数 / 通过 / 失败 / 阻塞 / 跳过
     - P0 用例执行结果列表（逐条）
     - P1 用例执行结果列表（逐条）
     - P0+P1 通过率（= P0+P1 通过数 / P0+P1 执行数）

3. **获取当前开放缺陷数**（按以下优先级依次尝试）：

   **优先级 1：读取 test-report 生成的测试报告**
   - 加载 Phase 0 指定路径的测试报告（默认 `test_reports/test-report-*.md`）
   - 文件存在 → 提取「缺陷汇总」章节中的开放 Fatal / Critical / Major 数量
   - 提取来源标注为「来自测试报告（test-report 生成）」

   **优先级 2：云效实时查询**
   - 仅当测试报告不存在或缺陷数据缺失时触发
   - 检查 `tools/yunxiao/config.yaml` 配置完整性
   - 配置完整 → 运行：
     ```bash
     python tools/yunxiao/scripts/query_defects.py --json
     ```
   - 解析 JSON，提取各严重等级的 `open` 数量
   - 来源标注为「来自云效实时查询」

   **优先级 3：手动输入（最终降级）**
   - 仅当以上两种方式均失败时，向用户询问：
     > 「无法自动获取缺陷状态（测试报告不含缺陷数据，云效查询失败）。请手动提供当前开放 Fatal 数和 Critical 数。」

4. **汇总检查数据**：
   - P0 用例列表（编号 / 名称 / 执行结果）
   - P0+P1 总用例数 / 通过数 / 失败数 / 阻塞数 / 跳过数
   - P0+P1 通过率（执行数为分母，不含阻塞和跳过）
   - P0+P1 阻塞+跳过占总用例比例
   - 当前开放 Fatal 数 / Critical 数 / Major 数（来源标注）

> ⛔ **DO NOT** 进入 Phase 2，直到报告加载完毕且所有必要数据已提取。

---

### Phase 2: 执行门禁检查

**目标**：逐项核验发布就绪条件，确定整体门禁结论。

**发布门禁检查项**：

#### 检查 1：P0 用例全部通过

- 逐一核查每条 P0 用例的执行结果
- 发现任一 P0 用例结果为**失败**或**阻塞** → 记录该用例，贡献 FAIL
- 全部 P0 用例为**通过** → 检查 1 通过

> **冒烟失败即阻塞发布**：P0 代表核心业务链路，任一失败均不可接受。

#### 检查 2：无开放 Fatal 缺陷

- 用户提供的当前开放 Fatal 数 > 0 → 贡献 FAIL（列出数量）
- 开放 Fatal 数 = 0 → 检查 2 通过

#### 检查 3：无开放 Critical 缺陷

- 当前开放 Critical 数 > 0 → 贡献 FAIL（列出数量）
- 开放 Critical 数 = 0 → 检查 3 通过

> **与 test-report 🟡 的区别**：`test-report` 输出 🟡（基本达标）时，允许存在开放 Critical 缺陷——那是**质量评估视角**，反映"有已知风险但主流程可用"。`release-gate` 对 Critical 执行零容忍——这是**发布准入视角**，保护用户不受严重缺陷影响。两者维度不同：🟡 报告不等于门禁通过，团队应在 release-gate 之前完成 Critical 缺陷的修复或风险评估（如确认 Critical 有 workaround、不影响核心链路，可由负责人临时豁免并在报告中记录决策依据）。

#### 检查 4：P0+P1 通过率 ≥ 阈值

- 计算 P0+P1 通过率 = P0+P1 通过数 ÷ P0+P1 执行数（不含阻塞和跳过）
- 通过率 < 用户设定阈值（默认 90%）→ 贡献 FAIL（列出实际通过率和阈值）
- 通过率 ≥ 阈值 → 检查 4 通过

#### 检查 5：P0+P1 执行覆盖完整性

- 计算阻塞+跳过比例 = （P0+P1 阻塞数 + P0+P1 跳过数）÷ P0+P1 总用例数
- 比例 > 用户设定上限（默认 20%）→ 贡献 FAIL，列出原因：「执行覆盖不足，当前通过率基于缩小后的分母，不能真实反映质量」；同时列出所有阻塞/跳过用例清单
- 比例 ≤ 上限 → 检查 5 通过

> **为什么要单独设这一项**：检查 4 的通过率以"已执行用例"为分母，大量阻塞/跳过会让分母缩水，90% 看起来达标但实际有大量用例从未跑过。检查 5 从总量角度守住执行覆盖底线。

#### 检查 6：开放 Major 缺陷数

- 获取当前开放 Major 缺陷数（来源同检查 2/3：测试报告 → 云效查询 → 手动输入）
- 用户未设阈值（默认）→ **不贡献 FAIL**，在报告中展示数量供决策者参考
- 用户设定阈值且 Major 数 > 阈值 → 贡献 FAIL（列出阈值和实际数量）
- Major 数 ≤ 阈值（或未设阈值）→ 检查 6 通过 / 通过（仅展示）

**门禁结论判定规则**：

| 条件 | 结论 |
|------|------|
| 检查 1～6 全部通过（含"仅展示"项） | **通过** |
| 任一检查不通过 | **不通过**（列出全部不通过项）|

> 门禁无第三态。发布时机和范围（灰度/全量）由发布决策者决定，门禁不介入。

> ⛔ **DO NOT** 进入 Phase 3，直到所有检查项均已完成且结论已确定。

---

### Phase 3: 输出门禁结论报告

**目标**：按 `assets/gate-report-template.md` 中的模板，生成发布门禁结论报告。

**步骤**：

1. **读取门禁报告模板**：加载 `assets/gate-report-template.md`

2. **确定输出路径**：`review/release-gate-result.md`
   - 文件已存在 → **覆盖写入**（本次门禁检查为最新结果）

3. **填写报告内容**：
   - 门禁结论（**通过** / **不通过**）—— 置于报告首行，醒目展示
   - 各检查项结果明细表（含检查 5 执行覆盖率、检查 6 Major 缺陷数）
   - P0+P1 执行覆盖摘要：总用例数 / 已执行 / 阻塞 / 跳过 / 阻塞+跳过占比
   - 开放缺陷汇总：Fatal / Critical / Major 各自数量（来源标注）
   - 不通过时：阻塞项列表（检查项编号 / 具体原因 / 建议解决方向）
   - P0 用例执行结果逐条列出（通过/失败/阻塞）
   - 通过时：后续步骤建议

4. **向用户展示**：
   - 门禁结论（通过/不通过）
   - 通过 → 提示可进入线上验证（调用 `test-execution` Mode C / `api-test-execution`）
   - 不通过 → 列出阻塞项和解决方向

## 验收标准

### 结论准确性

- [ ] 任一 P0 用例失败/阻塞时，结论必须为**不通过**
- [ ] 开放 Fatal 或 Critical 缺陷 > 0 时，结论必须为**不通过**
- [ ] P0+P1 通过率低于阈值时，结论必须为**不通过**
- [ ] P0+P1 阻塞+跳过比例超过上限时，结论必须为**不通过**
- [ ] 用户设定 Major 阈值且开放 Major 数超过阈值时，结论必须为**不通过**
- [ ] 用户未设定 Major 阈值时，Major 缺陷数仅展示不阻塞，结论不因此变为**不通过**
- [ ] 六项检查全部通过时，结论为**通过**
- [ ] 门禁结论不出现除「通过」「不通过」以外的第三种状态
- [ ] 开放缺陷数来源已在报告中标注（测试报告 / 云效查询 / 手动输入）
- [ ] 报告中已展示 P0+P1 执行覆盖摘要（总数 / 已执行 / 阻塞 / 跳过 / 占比）

### 报告完整性

- [ ] 门禁结论已写入 `review/release-gate-result.md`
- [ ] 报告首行显示门禁结论
- [ ] 报告包含六项检查的逐项结果（含检查 5 执行覆盖率、检查 6 Major 缺陷数）
- [ ] P0 用例执行结果已逐条列出
- [ ] 不通过时，每个阻塞项均有具体原因和解决方向

## 与其他 Skill 的衔接

| 上游 Skill | 提供给本 Skill | 说明 |
|-----------|--------------|------|
| `test-execution` | UI 执行报告（`test_reports/test_report_*.md`） | P0/P1 通过率 |
| `api-test-execution` | 接口执行报告（`test_reports/api_report_*.md`） | P0/P1 通过率 |
| `bug-sync` | 缺陷清单（`bug_reports/bug-report-*.md`） | 参考缺陷分布 |
| `test-report` | 测试报告（`test_reports/test-report-*.md`） | 优先从中读取开放 Fatal/Critical 数；缺失时降级云效查询或手动输入 |

| 下游动作 | 触发条件 | 说明 |
|---------|---------|------|
| 线上验证 | 门禁通过 | 调用 `test-execution` Mode C 执行线上回归 |
| 修复缺陷 | 门禁不通过 | 修复 Fatal/Critical 缺陷后，重新执行测试 → 重走 bug-sync → 重新触发发布门禁 |
