# 前端用户故事修改技能

> **设计模式**：Pipeline — 范围收集 → 审阅分析 → 文档定位 → 增量修改 → 关联检查 → 结果输出,严格顺序执行

**⚠️ 核心约束：本技能仅同步更新用户故事文档,不涉及代码开发。**

## 输入/输出契约

### 输入
| 输入项 | 来源 | 缺失时处理 |
|-------|------|----------|
| 审阅反馈文档 | 用户指定路径(测试/产品/后端审阅输出) | Fatal: 向用户索取 |
| 目标用户故事版本 | 用户指定,或通过版本号自动定位 | Warn: 列出可用版本供选择 |
| PRD文档(可选) | 用户指定路径 | 如有PRD变更则必须提供 |
| 后端接口文档(可选) | 用户指定路径 | 如有接口变更则必须提供 |

### 输出
| 输出产物 | 格式 |
|---------|------|
| 更新后的用户故事文档 | 原地更新 `requirements/user-stories/{version}/{version}.md` |
| 变更日志 | Markdown 格式输出,记录所有修改内容 |

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集
- [ ] Phase 1: 审阅意见分析
- [ ] Phase 2: 用户故事定位
- [ ] Phase 3: 增量修改
- [ ] Phase 4: 关联性检查
- [ ] Phase 5: 输出结果
```

### Phase 0: 范围收集

**⚠️ 硬门控：DO NOT proceed to Phase 1 until:**
- [ ] 修改类型已确认（测试/产品/后端/综合）
- [ ] 审阅反馈文档路径已确认
- [ ] 目标用户故事版本号已确认
- [ ] PRD 文档变更情况已确认（如有）
- [ ] 后端接口文档变更情况已确认（如有）
- [ ] 用户确认所有输入信息无误

使用 AskUserQuestion 收集：
- 修改类型：`测试反馈` / `产品反馈` / `后端反馈` / `综合反馈`
- 审阅反馈文档路径
- 目标用户故事版本号
- 是否有 PRD 文档变更
- 是否有后端接口文档变更

---

### Phase 1: 审阅意见分析

> **三路意见冲突处理**：当测试/产品/后端意见矛盾时，按以下优先级裁决：产品意见优先（需求所有者）> 后端意见（技术约束）> 测试意见（质量建议）。冲突项标记 `[CONFLICT]` 由用户最终确认。

**读取审阅反馈文档,提取并分类修改意见:**

1. **解析审阅反馈**
   - 读取测试/产品/后端审阅反馈文档
   - 提取所有审阅意见和问题清单

2. **意见分类**
   - **需求变更类**: PRD理解偏差、功能遗漏、需求补充
   - **接口变更类**: 接口路径调整、参数变更、新增/删除接口
   - **体验优化类**: 交互改进、UI调整、用户流程优化
   - **技术问题类**: 组件选型不当、实现方案调整、性能问题
   - **验收标准类**: 补充边界场景、完善验收条件

3. **映射到 Quest**
   - 将每条审阅意见映射到对应的 Quest
   - 识别新增 Quest(审阅中发现的遗漏功能)
   - 识别需删除 Quest(审阅中确认不需要的功能)
   - 审阅意见分类标准详见 `references/review-feedback-types.md`

**⚠️ 硬门控：DO NOT proceed to Phase 2 until:**
- [ ] 所有审阅意见已提取并分类
- [ ] 每条意见已映射到对应 Quest
- [ ] 用户确认审阅意见分析结果

---

### Phase 2: 用户故事定位

**定位并读取目标用户故事文档:**

1. **版本定位**
   - 根据版本号定位 `requirements/user-stories/{version}/{version}.md`
   - 如未指定版本,列出可用版本供用户选择

2. **文档读取**
   - 读取现有用户故事文档
   - 解析所有 Quest 结构
   - 建立 Quest ID 与内容的映射关系

3. **基准确认**
   - 向用户展示当前用户故事概览
   - 确认修改基准版本

---

### Phase 3: 增量修改

**根据审阅意见修改用户故事,遵循增量修改原则:**

#### 修改规则

1. **需求变更类意见**
   - 更新"产品原文"(如PRD有变更)
   - 更新"需求解读"
   - 更新"前端设计"中的布局和交互描述

2. **接口变更类意见**
   - 更新"接口对接"表
   - 修改接口路径、请求方法、前端参数、响应处理
   - 更新接口状态(✅ 已有 / ⚠️ 需新增)
   - 更新"前端设计"中的数据流和错误处理

3. **体验优化类意见**
   - 更新"前端设计"中的交互描述
   - 补充用户体验相关的验收标准

4. **技术问题类意见**
   - 更新"组件选型表"
   - 调整"前端设计"中的修改位置
   - 更新实现方案说明

5. **验收标准类意见**
   - 补充或修改"验收标准"
   - 增加边界场景、异常处理验证项

#### 增量修改原则

- ✅ **只修改受影响的 Quest**: 未涉及的 Quest 保持原样
- ✅ **保留修改痕迹**: 在变更日志中记录修改原因和来源
- ✅ **保持结构一致**: 修改后仍符合 user-story-template.md 模板结构
- ✅ **引用审阅来源**: 在变更日志中注明每条修改的审阅意见来源

**⚠️ 硬门控：DO NOT proceed to Phase 4 until:**
- [ ] 所有审阅意见已处理或标记为待确认
- [ ] 用户审阅修改差异(diff)
- [ ] 用户确认修改结果

---

### Phase 4: 关联性检查

**交叉验证修改后的用户故事:**

1. **Quest 依赖关系检查**
   - 验证 Quest 间的依赖关系是否保持一致
   - 检查新增/删除 Quest 是否影响其他 Quest

2. **接口一致性检查**
   - 如有接口文档,验证接口对接表与接口文档一致
   - 检查接口参数、响应处理是否完整

3. **组件完整性检查**
   - 验证组件选型表中的组件路径有效
   - 检查是否有组件遗漏或冲突

4. **验收标准覆盖检查**
   - 验证每个 Quest 都有验收标准
   - 检查验收标准是否覆盖审阅意见中提到的场景

**⚠️ 硬门控：DO NOT proceed to Phase 5 until:**
- [ ] Quest 依赖关系验证通过
- [ ] 接口一致性验证通过（如有接口文档）
- [ ] 组件完整性验证通过
- [ ] 验收标准覆盖验证通过

---

### Phase 5: 输出结果

**生成变更日志并输出修改结果:**

基于 `assets/change-log-template.md` 模板生成变更日志,包含:

1. **修改概览**: 统计修改/新增/删除的 Quest 数量
2. **详细变更**: 每条修改的 Quest、类型、内容、审阅来源、状态
3. **审阅意见处理情况**: 区分已处理和待确认的意见
4. **修改影响范围**: 涉及文件和下游影响
5. **人工复核建议**: 需要人工确认的事项

**输出内容:**
1. 变更日志(控制台展示)
2. 更新后的文件路径
3. 待确认问题清单(如有)

---

## 不确定时的处理

遇到以下情况时,**暂停修改并询问用户**:

1. 审阅意见模糊,无法确定如何修改用户故事
2. 审阅意见与现有用户故事冲突,需要用户决策
3. 审阅建议新增 Quest,需确认是否纳入当前版本
4. 接口变更但未提供新接口文档,需确认接口细节
5. 组件选型建议与项目现有组件冲突

---

## 约束

- 严格按照审阅意见修改,不添加主观推测
- 只修改受影响的 Quest,保持未变更内容完整
- 接口信息必须基于后端接口文档,不自行推断
- 产品原文必须严格引用 PRD,不修改或简化
- 如有任何问题,立即向用户发起提问
- 建议修改前自动备份原文件
- 所有修改必须在变更日志中记录来源和原因
