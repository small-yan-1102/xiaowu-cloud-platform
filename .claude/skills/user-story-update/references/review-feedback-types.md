# 审阅反馈分类标准

本文件定义了测试/产品/后端三路审阅反馈的分类标准和处理策略。

## 审阅反馈分类体系

### 1. 需求变更类

**来源**: 产品审阅为主

**特征**:
- PRD 理解有偏差
- 功能遗漏或多余
- 业务规则调整
- 用户流程变更

**处理方式**:
- 更新"产品原文"(如 PRD 有变更)
- 更新"需求解读"
- 更新"前端设计"中的布局和交互描述
- 必要时新增或删除 Quest

**示例**:
```
审阅意见: "用户列表应该支持按部门筛选,PRD 中已明确"
处理方式: 更新 Quest 1 需求解读,补充筛选功能的前端设计
```

---

### 2. 接口变更类

**来源**: 后端审阅为主

**特征**:
- 接口路径调整
- 请求方法变更(GET → POST)
- 参数增减或类型变更
- 响应结构变化
- 新增或删除接口

**处理方式**:
- 更新"接口对接"表
- 更新接口路径、请求方法、前端参数、响应处理
- 更新接口状态(✅ 已有 / ⚠️ 需新增)
- 更新"前端设计"中的数据流和错误处理

**示例**:
```
审阅意见: "用户列表接口已从 /api/users 调整为 /api/users/v2,新增了 departmentId 筛选参数"
处理方式: 更新 Quest 1 接口对接表,补充 departmentId 参数说明
```

---

### 3. 体验优化类

**来源**: 产品审阅为主

**特征**:
- 交互流程不顺畅
- UI 布局不合理
- 用户操作路径过长
- 缺少必要的反馈提示

**处理方式**:
- 更新"前端设计"中的交互描述
- 补充用户体验相关的验收标准
- 调整页面布局描述

**示例**:
```
审阅意见: "删除操作应该增加二次确认,避免误操作"
处理方式: 更新 Quest 1 交互描述,补充删除确认的验收标准
```

---

### 4. 技术问题类

**来源**: 后端审阅为主

**特征**:
- 组件选型不当
- 实现方案复杂
- 性能隐患
- 可维护性问题

**处理方式**:
- 更新"组件选型表"
- 调整"前端设计"中的实现方案
- 补充性能优化建议

**示例**:
```
审阅意见: "用户列表数据量大时,建议使用虚拟滚动组件而非分页"
处理方式: 更新 Quest 1 组件选型表,将分页组件改为虚拟滚动组件
```

---

### 5. 可测性类

**来源**: 测试审阅为主

**特征**:
- 缺少边界条件处理
- 异常场景未覆盖
- 状态管理不清晰
- 难以自动化测试

**处理方式**:
- 补充"验收标准"中的边界场景
- 增加异常处理验证项
- 明确状态流转逻辑

**示例**:
```
审阅意见: "缺少空状态、加载中状态、接口失败状态的测试场景"
处理方式: 补充 Quest 1 验收标准,增加空状态/加载中/失败状态的验证项
```

---

### 6. 场景覆盖类

**来源**: 测试审阅为主

**特征**:
- 特殊角色权限未考虑
- 极端数据量未处理
- 并发场景未覆盖
- 兼容性测试缺失

**处理方式**:
- 补充"验收标准"中的场景验证
- 增加多角色权限测试项
- 补充性能测试标准

**示例**:
```
审阅意见: "未考虑超级管理员和普通用户的权限差异"
处理方式: 补充 Quest 1 验收标准,增加不同角色的权限验证项
```

---

## 审阅意见优先级

### P0 - 阻塞性 (必须修改)

**标准**:
- 接口不匹配,无法联调
- 业务规则错误
- 安全漏洞
- 核心功能遗漏

**处理**: 立即修改,阻塞后续流程

---

### P1 - 重要 (应该修改)

**标准**:
- 用户体验问题
- 边界场景缺失
- 性能隐患
- 组件选型不当

**处理**: 优先修改,可在人工查看环节确认

---

### P2 - 建议 (可以修改)

**标准**:
- 代码风格建议
- 注释完善
- 优化建议
- 技术债

**处理**: 记录在变更日志,由用户决定是否纳入当前版本

---

## 审阅意见冲突处理

### 场景 1: 产品 vs 后端

**示例**:
- 产品: "需要实时搜索"
- 后端: "接口不支持实时搜索,建议改为手动触发"

**处理方式**:
1. 在变更日志中标记为"待确认"
2. 说明冲突点和建议方案
3. 由用户决策或组织三方会议

---

### 场景 2: 测试 vs 产品

**示例**:
- 测试: "需要增加接口失败重试机制"
- 产品: "需求中没有这个要求"

**处理方式**:
1. 评估是否为合理的体验优化
2. 如合理,标记为 P2 建议
3. 由用户决定是否纳入

---

### 场景 3: 多个审阅方提出相同问题

**示例**:
- 测试: "缺少空状态处理"
- 产品: "空状态应该引导用户创建数据"

**处理方式**:
1. 合并为一条修改
2. 在变更日志中注明多个来源
3. 综合各方意见给出完整方案

---

## 审阅意见映射规则

### 映射到 Quest 的策略

1. **直接映射**: 意见明确指出是哪个功能
   ```
   意见: "用户列表的搜索功能有问题"
   映射: Quest 1: 用户列表管理
   ```

2. **间接映射**: 意见描述模糊,需要分析
   ```
   意见: "表单提交后应该有成功提示"
   分析: 涉及所有表单提交功能
   映射: 所有包含表单的 Quest
   ```

3. **新增 Quest**: 意见提出全新功能
   ```
   意见: "应该支持批量导出功能"
   映射: 新增 Quest N: 批量导出
   状态: ⚠️ 待确认(需确认是否纳入当前版本)
   ```

4. **删除 Quest**: 意见确认功能不需要
   ```
   意见: "PRD 已确认暂不需要导入功能"
   映射: 删除 Quest N: 数据导入
   状态: ✅ 已确认删除
   ```

---

## 使用建议

1. **分类是手段,不是目的**: 分类帮助理解意见本质,选择正确的修改策略
2. **优先级指导决策**: P0 必须改,P1 优先改,P2 可商议
3. **冲突必须上报**: 审阅意见冲突时,不要自行判断,交由用户决策
4. **记录来源**: 变更日志中必须标注每条修改的审阅来源,便于追溯
