# 前端审阅后端方案技能

> **设计模式**：Reviewer（检查清单驱动）+ Generator（模板化输出报告）

## 输入/输出契约

### 输入
| 输入项 | 来源 | 缺失时处理 |
|-------|------|----------|
| 产品PRD/原型 | 用户指定路径 | Fatal：向用户索取 |
| 后端接口文档 | 用户指定路径 | Fatal：向用户索取 |
| 审阅检查清单 | `references/review-checklist.md` | 自动加载 |

### 输出
| 输出产物 | 格式 |
|---------|------|
| 后端方案审阅报告 | 基于 `assets/review-report-template.md` 生成 |

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集（Inversion）
- [ ] Phase 1: 现有代码分析
- [ ] Phase 2: 接口文档分析
- [ ] Phase 3: 审阅报告生成（Generator）
- [ ] Phase 4: 输出结果
```

### Phase 0: 范围收集（Inversion 模式）

**⚠️ 硬门控：DO NOT proceed to Phase 1 until:**
- [ ] 产品PRD/原型路径已确认
- [ ] 后端接口文档路径已确认
- [ ] 目标模块名称已确认
- [ ] 用户确认所有输入信息无误

使用 AskUserQuestion 收集：
- 产品PRD/原型路径
- 后端接口文档路径
- 目标模块名称

---

### Phase 1: 现有代码分析

**⚠️ 硬门控：DO NOT proceed to Phase 2 until existing code analysis is complete.**

1. **搜索相似功能实现**：
   - 搜索 `src/api/` 或 `src/services/` 目录
   - 查找与目标模块功能相似的接口文件
   - 提取现有接口的：
     - 响应结构（如 `{ code, data, message }`）
     - 字段命名风格（驼峰 vs 下划线）
     - 分页参数格式（`page/size` vs `pageNum/pageSize`）
     - 日期格式（`YYYY-MM-DD` vs 时间戳）

2. **提取项目规范**：
   - 请求封装方式（axios/fetch）
   - 错误处理模式
   - 数据转换逻辑

3. **建立对比基准**：
   - 记录现有接口的共性特征
   - 作为审阅新接口的参考标准

---

### Phase 2: 接口文档分析

**⚠️ 硬门控：DO NOT proceed to Phase 3 until all APIs are reviewed.**

按 `references/review-checklist.md` 中的四个维度逐一审阅：

1. **接口设计合理性**
   - 接口路径是否符合RESTful规范
   - 请求方法是否合理（GET查询/POST创建/PUT更新/DELETE删除）
   - 参数定义是否完整（必填/选填/类型/默认值）
   - 响应结构是否与项目统一
   - 状态码使用是否规范

2. **前端消费友好性**
   - 字段命名是否前端友好（驼峰命名）
   - 是否需要额外数据转换
   - 枚举值是否完整（前端下拉选项）
   - 分页参数是否与项目一致
   - 日期格式是否统一
   - 数据类型是否准确（数字/布尔/字符串）

3. **性能与体验影响**
   - 是否支持分页/懒加载
   - 大数据量是否有优化
   - 响应时间是否可接受
   - 是否有缓存策略（如适用）
   - 是否需要额外的loading状态

4. **与现有功能一致性**
   - 接口风格是否与项目一致
   - 数据结构是否统一
   - 是否有重复接口
   - 是否可复用现有模式

**审阅流程**：
1. 逐个接口分析
2. 对照检查清单逐项检查
3. 与现有接口对比差异
4. 按 Critical/Warning/Info 分级问题
5. 给出具体改进建议

---

### Phase 3: 审阅报告生成（Generator 模式）

基于 `assets/review-report-template.md` 模板生成审阅报告，包含：

1. **审阅概览**：各维度问题统计
2. **问题详情**：按 Critical/Warning/Info 分级
3. **现有功能对比**：新旧接口差异对比表
4. **改进建议**：分优先级的修改建议
5. **字段合理性分析**：字段命名、类型、完整性检查

---

### Phase 4: 输出结果

**输出内容**：
1. 审阅报告（Markdown格式）
2. 问题汇总清单
3. 优先级修改建议

---

## 字段合理性判断标准

### 1. 命名风格一致性
- ✅ 与项目现有接口命名风格一致（驼峰/下划线）
- ❌ 混用不同命名风格

### 2. 字段粒度合理性
- ✅ 字段拆分合理，前端可直接使用
- ❌ 过度聚合（前端需再次拆分）
- ❌ 过度拆分（增加请求次数）

### 3. 数据类型合理性
- ✅ 数字字段是Number类型
- ✅ 布尔字段是Boolean类型
- ✅ 枚举字段提供完整选项列表
- ❌ 数字用字符串表示
- ❌ 布尔用"true"/"false"字符串

### 4. 字段完整性
- ✅ 包含前端必需的字段
- ✅ 包含审计字段（创建人/时间、更新人/时间）
- ❌ 缺少前端展示必需的字段

### 5. 命名语义清晰
- ✅ 字段名能自解释（`userName` vs `name`）
- ✅ 与业务术语一致
- ❌ 使用模糊命名（`flag`, `type`, `data`）

### 6. 嵌套层级合理性
- ✅ 嵌套层级 ≤ 3层
- ❌ 嵌套过深（访问困难）

---

## 问题严重度分级

| 级别 | 定义 | 处理要求 |
|------|------|---------|
| **Critical** | 必须修改 — 与项目规范冲突、无法直接使用 | 阻断后端方案通过 |
| **Warning** | 建议修改 — 增加前端适配成本、体验不佳 | 建议后端修改 |
| **Info** | 仅供参考 — 优化建议、锦上添花 | 可选修改 |

---

## 约束

- 审阅聚焦后端接口文档，不对后端实现细节提出建议
- 所有问题必须给出具体修改建议，而非仅指出问题
- 问题分级必须准确，不允许夸大或弱化
- 必须与项目现有接口对比，给出一致性检查
- 字段合理性判断基于前端消费视角，不评判后端设计优劣
