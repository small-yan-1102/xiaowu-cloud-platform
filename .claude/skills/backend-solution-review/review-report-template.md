  "result": {...}
}

**建议**:
```json

  "code": 200,
  "data": {...},

## ⚠️ Warning 问题（建议修改）

### 问题 1: {问题标题}

**接口**: `GET /api/users`

**问题描述**: 
字段命名使用下划线，与项目驼峰命名不一致

**现状**:
```json
{
  "user_name": "张三",
  "dept_id": 1,
  "created_at": "2024-01-01"
}
```

**建议**:
```json
{
  "userName": "张三",
  "deptId": 1,
  "createdAt": "2024-01-01"
}
```

**原因**: 
- 项目其他接口均使用驼峰命名
- 保持一致可减少前端字段映射成本
- 提升代码可维护性

## 现有功能对比

### 对比表

| 检查项 | 现有接口 | 新接口 | 是否一致 | 差异说明 |
|--------|---------|--------|---------|---------|
| 响应结构 | `{ code, data }` | `{ success, result }` | ❌ | 结构不统一 |
| 字段命名 | 驼峰（userName） | 下划线（user_name） | ❌ | 命名风格不同 |
| 分页参数 | `page, size` | `pageNum, pageSize` | ❌ | 参数名不同 |
| 日期格式 | 时间戳 | `YYYY-MM-DD` | ❌ | 格式不同 |
| 枚举值 | 提供options | 仅提供value | ❌ | 缺少选项列表 |

### 相似接口参考

**用户列表接口**（`src/api/user.js`）:
```javascript
// 现有接口示例
export function getUserList(params) {
  return request({
    url: '/api/users',
    method: 'get',
    params: { page: params.page, size: params.size }
  })
}

// 响应格式
{
  code: 200,
  data: {
    list: [...],
    total: 100,
    page: 1,
    size: 20
  }
}
```

**建议**: 新接口应遵循相同的响应结构和参数命名

### 问题字段清单

| 接口 | 字段 | 问题类型 | 现状 | 建议 | 原因 |
|------|------|---------|------|------|------|
| GET /api/users | user_name | 命名风格 | 下划线 | userName | 与项目不一致 |

## 改进建议

### 🔴 必须修改（Critical）

1. **统一响应结构**
   - 修改所有接口为 `{ code, data, message }` 格式
   - 影响接口: X 个
   - 优先级: P0

2. **修正字段类型**
   - `age` 字段改为 Number 类型
   - `isEnabled` 字段改为 Boolean 类型
   - 影响接口: X 个
   - 优先级: P0

### 🟡 建议修改（Warning）

1. **统一字段命名**
   - 所有字段改为驼峰命名
   - 影响接口: X 个
   - 优先级: P1

2. **统一分页参数**
   - 使用 `page, size` 替代 `pageNum, pageSize`
   - 影响接口: X 个
   - 优先级: P1

3. **补充枚举选项**
   - 所有枚举字段添加 `xxxOptions` 字段
   - 影响接口: X 个
   - 优先级: P1

### 🟢 可选优化（Info）

   - 优先级: P2

2. **提供Mock数据**

   - 优先级: P2

## 使用指南

### 1. 报告结构说明

- **审阅概览**: 快速了解整体质量
- **Critical问题**: 必须修改的问题，详细说明原因和影响
- **Warning问题**: 建议修改的问题，给出优化方案
- **Info建议**: 可选优化，锦上添花
- **现有功能对比**: 与项目现有接口的差异对比
- **字段合理性分析**: 字段命名、类型、完整性专项检查
- **改进建议**: 按优先级分层的修改建议
- **审阅结论**: 是否通过的明确判断

### 2. 状态标记

| 标记 | 含义 |
|------|------|
| ✅ 通过 | 无Critical问题，Warning ≤ 5个 |
| ⚠️ 有建议 | 无Critical问题，Warning > 5个 |
| ❌ 需修改 | 有Critical问题 |

### 3. 填写建议

- **问题描述**: 客观描述，基于事实
- **现状/建议**: 使用JSON示例，清晰对比
- **原因**: 说明为什么需要修改，帮助理解
- **影响范围**: 评估修改成本

### 4. 字段合理性判断

参考 SKILL.md 中的"字段合理性判断标准"：
1. 命名风格一致性
2. 字段粒度合理性
3. 数据类型合理性
4. 字段完整性
5. 命名语义清晰
6. 嵌套层级合理性
