# 前端审阅后端方案检查清单

本文件定义了前端审阅后端方案的多维度检查清单。

## 一、接口设计合理性

### 1.1 RESTful规范
- [ ] 接口路径使用名词复数（`/api/users` 而非 `/api/getUser`）
- [ ] 资源层级清晰（`/api/users/{id}/orders`）
- [ ] 避免动词在路径中（使用HTTP方法表达动作）

### 1.2 HTTP方法
- [ ] GET：查询操作，无副作用
- [ ] POST：创建操作
- [ ] PUT：完整更新操作
- [ ] PATCH：部分更新操作
- [ ] DELETE：删除操作

### 1.3 参数定义
- [ ] 必填参数明确标注
- [ ] 选填参数有默认值说明
- [ ] 参数类型定义完整（String/Number/Boolean/Object）
- [ ] 参数长度/范围限制说明
- [ ] Query参数 vs Body参数使用合理

### 1.4 响应结构
- [ ] 响应结构与项目统一（如 `{ code, data, message }`）
- [ ] 成功响应包含完整数据
- [ ] 错误响应包含错误码和错误信息
- [ ] 分页响应包含 `total`, `page`, `size` 字段

### 1.5 状态码
- [ ] 200：请求成功
- [ ] 400：参数错误
- [ ] 401：未授权
- [ ] 403：禁止访问
- [ ] 404：资源不存在
- [ ] 500：服务器错误

---

## 二、前端消费友好性

### 2.1 字段命名
- [ ] 使用驼峰命名（`userName` 而非 `user_name`）
- [ ] 与项目现有接口命名风格一致
- [ ] 字段名语义清晰，能自解释
- [ ] 避免模糊命名（`flag`, `type`, `data`）
- [ ] 与业务术语一致

### 2.2 数据类型
- [ ] 数字字段使用Number类型（非字符串）
- [ ] 布尔字段使用Boolean类型（非"true"/"false"字符串）
- [ ] 金额字段使用Number或String（明确说明）
- [ ] 日期格式统一（`YYYY-MM-DD HH:mm:ss` 或时间戳）
- [ ] 数组字段即使为空也返回 `[]`（非null）

### 2.3 枚举值
- [ ] 枚举字段提供完整选项列表
- [ ] 选项包含 `value` 和 `label`
- [ ] 枚举值与项目其他接口一致
- [ ] 枚举值有明确说明文档

示例：
```javascript
// ✅ 合理
{
  status: 1,
  statusOptions: [
    { value: 1, label: "启用" },
    { value: 0, label: "禁用" }
  ]
}

// ❌ 不合理
{
  status: 1  // 前端不知道1代表什么
}
```

### 2.4 分页参数
- [ ] 与项目其他接口分页参数一致（`page/size` 或 `pageNum/pageSize`）
- [ ] 响应包含 `total` 字段
- [ ] 支持按需选择字段（fields参数，如适用）

### 2.5 字段粒度
- [ ] 字段拆分合理，前端可直接使用
- [ ] 避免过度聚合（如 `"张三-技术部-13800138000"`）
- [ ] 避免过度拆分（增加请求次数）
- [ ] 常用字段在上层，避免深层嵌套

### 2.6 嵌套层级
- [ ] 对象嵌套层级 ≤ 3层
- [ ] 深层嵌套数据考虑扁平化
- [ ] 列表数据独立返回，不嵌套在其他对象中

## 三、性能与体验影响

### 3.1 分页与懒加载
- [ ] 列表接口支持分页
- [ ] 大数据量支持懒加载或虚拟滚动
- [ ] 分页参数有最大值限制（防止一次性拉取过多数据）

### 3.2 响应性能
- [ ] 接口响应时间 < 1s（常规查询）
- [ ] 复杂查询响应时间 < 3s
- [ ] 支持字段筛选（fields参数，减少数据传输）
- [ ] 支持按需加载（避免返回前端不需要的字段）

### 3.3 缓存策略
- [ ] 静态数据支持缓存（字典、枚举等）
- [ ] 缓存时间合理（Cache-Control头）
- [ ] 实时性要求高的数据不缓存

### 3.4 Loading状态
- [ ] 接口响应时间可预期，前端可合理展示loading
- [ ] 长时间操作支持进度查询（如适用）
- [ ] 支持取消请求（AbortController）

### 3.5 批量操作
- [ ] 支持批量创建/更新/删除（如适用）
- [ ] 批量操作有数量限制
- [ ] 批量操作返回详细结果（成功/失败列表）

---

## 四、与现有功能一致性

### 4.1 接口风格
- [ ] 接口路径风格与项目一致
- [ ] 请求/响应格式与项目一致
- [ ] 错误处理模式与项目一致

### 4.2 数据结构
- [ ] 相同业务实体字段一致（如用户信息在所有接口中字段相同）
- [ ] 通用字段统一（id, createdAt, updatedAt等）
- [ ] 类型定义统一（同一字段在不同接口中类型一致）

### 4.3 避免重复
- [ ] 无功能重复的接口
- [ ] 优先复用现有接口
- [ ] 相似接口考虑合并

### 4.4 模式复用
- [ ] 复用项目现有的查询模式
- [ ] 复用现有的分页模式
- [ ] 复用现有的排序/筛选模式

---

## 五、现有接口对比检查

### 5.1 对比项目相似接口

| 检查项 | 现有接口 | 新接口 | 是否一致 |
|--------|---------|--------|---------|
| 响应结构 | `{ code, data }` | `{ code, data }` | ✅ |
| 字段命名 | 驼峰 | 驼峰 | ✅ |
| 分页参数 | `page, size` | `pageNum, pageSize` | ❌ |
| 日期格式 | 时间戳 | `YYYY-MM-DD` | ❌ |

### 5.2 不一致项处理

对于不一致的项，需要：
1. 标注差异点
2. 评估影响范围
3. 给出统一建议
4. 说明统一的好处

## 六、问题分级标准

### Critical（必须修改）
- 与项目规范冲突，无法直接使用
- 字段类型错误，前端无法处理
- 缺少必需字段，功能无法实现
- 响应结构不统一，需要额外适配层

### Warning（建议修改）
- 增加前端适配成本
- 用户体验不佳
- 性能可优化
- 命名不规范但不影响使用

### Info（仅供参考）
- 优化建议
- 锦上添花的功能
- 代码风格建议

---

## 使用建议

- **常规审阅**：全量检查所有维度
- **快速审阅**：重点关注"接口设计合理性"和"前端消费友好性"
- **性能专项**：重点关注"性能与体验影响"
- **规范化审阅**：重点关注"与现有功能一致性"
