# 用例审核技能

> **版本**: v1.1
> **更新日期**: 2026-04-13
> **适用模式**: AI 扫描用例集 → 对照测试点与技术文档 → 标注问题 → 输出审核报告
> **v1.1 变更**: 新增维度 5「断言可执行性」（覆盖 CH-7/CH-8/CH-9）；明确适用边界

---

> **适用场景**：用例刚生成后的结构质量门禁——覆盖完整性、边界/异常遗漏、预期结果准确性、步骤可执行性、断言可执行性。
>
> **不适用**：PRD 版本变更后需要识别哪些用例受影响、或排查用例与 PRD 的偏差 → 使用 `test-case-prd-consistency`。

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载用例集与基准文件
- [ ] Phase 2: AI 扫描审核
- [ ] Phase 3: 输出审核报告
```

---

### Phase 0: 定向收集

**目标**：确认输入文件路径，避免读错文件。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **UI 用例路径**：test-case-design 生成的 UI 用例文件位于哪里？（默认：`review/test-cases-ui.md`）
2. **接口用例路径**：api-test-case-design 生成的接口用例文件位于哪里？（默认：`review/test-cases-api.md`）
3. **测试点清单路径**：test-point-extraction 生成的测试点清单位于哪里？（默认：`review/test-points.md`）
4. **技术文档路径**：后端技术文档和前端技术文档位于哪里？（默认：`input/tech/*.md`）
5. **审核范围**：全量审核所有用例，还是聚焦某个特定模块？（默认：全部模块）

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

**决策点**：
- 用户回答或表示"按默认来" → 使用默认路径，进入 Phase 1
- 用户指定仅审核 UI 用例或仅审核接口用例 → 跳过未指定部分，在报告中注明

---

### Phase 1: 加载用例集与基准文件

**目标**：读取所有输入文件，确认可审核。

**步骤**：

1. **检查审核报告模板**：
   - 检查 `assets/case-review-template.md` 是否存在
   - 不存在 → **Warn**：跳过模板加载，按内置格式输出报告

2. **读取 UI 用例文件**：
   - 加载用户指定路径（默认 `review/test-cases-ui.md`）
   - 不存在 → **Warn**：跳过 UI 用例审核，在报告中注明「UI 用例文件未找到」

3. **读取接口用例文件**：
   - 加载用户指定路径（默认 `review/test-cases-api.md`）
   - 不存在 → **Warn**：跳过接口用例审核，在报告中注明「接口用例文件未找到」

4. **读取测试点清单**：
   - 加载用户指定路径（默认 `review/test-points.md`）
   - 不存在 → **Fatal**：终止执行，提示「测试点清单不存在，请先执行 test-point-extraction」

5. **读取技术文档**：
   - 加载 `input/tech/*.md` 下所有文件
   - 不存在 → **Warn**：跳过预期结果一致性校验，在报告中注明「技术文档未找到，预期结果一致性检查不可用」

6. **确认用例总数**：统计 UI 用例条数、接口用例条数，向用户展示概览后进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到文件加载完成（或已记录所有 Warn/Fatal）。

---

### Phase 2: AI 扫描审核

**目标**：从 4 个维度系统性扫描用例集，标注问题。

> **原则**：测试工程师主导最终判断，AI 负责扫描盲点和一致性校验。

---

#### 维度 1：覆盖完整性（对照 test-points.md）

逐条遍历 `review/test-points.md` 中的测试点：

- **P0 测试点**：是否至少有一条用例与之对应？
  - 无对应用例 → 标注 `[覆盖缺失-P0]`：`{测试点编号} {测试点描述}` 未被任何用例覆盖
- **P1 测试点**：是否至少有一条用例与之对应？
  - 无对应用例 → 标注 `[覆盖缺失-P1]`
- **P2 测试点**：统计覆盖率，低于 60% 时整体标注 `[P2 覆盖率不足]`
- **冗余用例**：发现与任何测试点均无法对应的用例 → 标注 `[疑似冗余]`，供工程师判断是否保留

---

#### 维度 2：边界值与异常路径识别

扫描所有用例，检查以下场景是否有对应用例：

| 场景类型 | 检查要点 |
|---------|---------|
| 数值边界 | 最大值/最小值/临界值+1 是否均有用例 |
| 文本边界 | 最大长度、空值、特殊字符是否有用例 |
| 必填校验 | 每个必填字段是否有「空值提交」用例 |
| 格式校验 | 格式不合法场景是否有用例 |
| 重复操作 | 重复提交/重复创建是否有用例 |
| 状态拦截 | 非法状态转换是否有拒绝行为用例 |
| 并发/竞态 | 有并发场景描述时，是否有对应用例 |

- 发现缺失 → 标注 `[边界/异常缺失]`：`{缺失场景描述}`

---

#### 维度 3：预期结果与技术文档一致性

> 仅当 Phase 1 成功读取技术文档时执行。

逐条检查用例的预期结果，与技术文档（接口定义、业务规则、状态机、交互规范）对照：

- **接口用例**：HTTP 状态码、响应字段、错误码是否与接口定义一致
- **UI 用例**：
  - 成功路径的页面跳转/提示文案是否与交互规范一致
  - 失败路径的错误提示是否与接口错误码映射一致
  - 数据展示格式（日期格式、金额格式、枚举值显示文案）是否与规范一致
- 发现不一致 → 标注 `[预期结果偏差]`：`{用例编号}` 预期结果为「{用例描述}」，技术文档定义为「{技术文档内容}」

---

#### 维度 4：步骤可执行性

逐条检查用例步骤，识别以下问题：

| 问题类型 | 判定规则 |
|---------|---------|
| 前置条件缺失 | 用例步骤中依赖特定数据状态（如"已创建订单"），但前置条件未说明如何准备 |
| 步骤描述模糊 | 步骤使用「打开某页面」「点击相关按钮」等无法定位的描述 |
| 预期结果缺失 | 用例无预期结果，或预期结果为「正常」「成功」等无法验证的描述 |
| 测试数据未指定 | 步骤需要具体输入值（如手机号格式），但未给出示例值或范围 |

- 发现问题 → 标注 `[可执行性问题]`：`{用例编号}` + 问题类型 + 具体描述

---

#### 维度 5：断言可执行性

> 本维度覆盖 CH-7～CH-9 三项断言质量检查，此处执行后，`test-case-prd-consistency` 不再重复检查这三项。

逐条检查用例的预期结果，识别以下问题：

| 检查项 | 编号 | 判定规则 |
|-------|------|---------|
| 预期结果可观测性 | CH-7 | 预期结果是否精确可观测？含禁用词（"正常""成功""无报错""加载完成"等）→ 标注 `[断言模糊]`，并给出改写建议 |
| 验证层级标注 | CH-8 | 每条预期结果是否标注了 L1/L2/L3？缺失 → 标注 `[层级缺失]`，并根据验证意图建议补充层级 |
| 数值断言类型 | CH-9 | 涉及精确数值（金额、积分、计数）的预期结果是否使用了 `[断言: API]` 或 `[断言: DB-Query]`？仍用视觉断言 → 标注 `[数值断言应升级]`，给出对应接口/SQL 建议 |

**禁用词速查表**（CH-7 命中则必须改写）：

| 禁用词 | 替换方向 |
|-------|---------|
| 页面正常显示 / 加载成功 | 说明"页面包含{具体元素}"或"标题显示{具体文案}" |
| 数据正确 / 数据正常 | 说明"列表第一行显示{字段}={值}" |
| 提交成功 | 说明"出现绿色 toast 提示{包含:成功}" |
| 无报错 / 无异常 | 说明"页面无错误弹窗、无红色提示文字" |
| 功能正常 / 显示正常 | 说明可观测的具体现象 |

- 发现 CH-7 问题 → 标注 `[断言模糊]`：`{用例编号}` 第 N 条预期结果，给出改写建议
- 发现 CH-8 问题 → 标注 `[层级缺失]`：`{用例编号}` 第 N 条预期结果缺少 L1/L2/L3 标注
- 发现 CH-9 问题 → 标注 `[数值断言应升级]`：`{用例编号}` 第 N 条预期结果中 `{数值字段}` 需改用 API 断言

---

### Phase 3: 输出审核报告

**目标**：汇总扫描结果，输出结构化审核报告。

**步骤**：

1. **确定输出路径**：`review/case-review-report.md`
   - 文件已存在 → **覆盖写入**（本次为最新版本）

2. **填写报告内容**：

   ```markdown
   # 用例审核报告

   > 执行时间：{日期}
   > 审核范围：UI 用例 {N} 条 / 接口用例 {N} 条
   > 测试点基准：{test-points.md 总测试点数}

   ## 一、覆盖完整性

   ### P0 覆盖缺失（{N} 项）
   | 测试点编号 | 测试点描述 | 建议补充用例 |
   |-----------|-----------|------------|
   | ... | ... | ... |

   ### P1 覆盖缺失（{N} 项）
   ...

   ### 疑似冗余用例（{N} 条）
   ...

   ## 二、边界值与异常路径缺失（{N} 项）
   | 场景类型 | 缺失描述 | 所属模块 |
   |---------|---------|---------|
   | ... | ... | ... |

   ## 三、预期结果偏差（{N} 项）
   | 用例编号 | 当前预期结果 | 技术文档定义 | 建议修正 |
   |---------|------------|------------|---------|
   | ... | ... | ... | ... |

   ## 四、步骤可执行性问题（{N} 项）
   | 用例编号 | 问题类型 | 具体描述 | 建议修正 |
   |---------|---------|---------|---------|
   | ... | ... | ... | ... |

   ## 五、断言可执行性问题（{N} 项）

   | 用例编号 | 检查项 | 当前预期结果 | 问题类型 | 改写建议 |
   |---------|-------|------------|---------|---------|
   | ... | CH-7/CH-8/CH-9 | ... | 断言模糊/层级缺失/数值断言应升级 | ... |

   ## 六、审核结论

   - 🟢 通过：发现问题 0 项，用例质量良好，可直接进入补充阶段
   - 🟡 需修正：发现问题 {N} 项，请按报告修正后继续
   - 🔴 重新设计：P0 覆盖缺失 {N} 项，建议重新触发 test-case-design 补全后再审核

   ## 七、待测试工程师确认事项

   - 疑似冗余用例是否删除
   - 预期结果偏差项的最终判断（AI 建议仅供参考）
   - [AI+人工] / [人工] 标签用例的人工介入计划
   ```

3. **向用户展示**：
   - 各维度问题数量汇总
   - P0 覆盖缺失清单（优先级最高，须立即处理）
   - 审核结论（🟢/🟡/🔴）
   - 提示：问题修正后可触发 `test-case-design` 或 `api-test-case-design` 补充用例，再合并至主用例集

## 验收标准

### 覆盖检查

- [ ] 所有 P0 测试点已逐一检查，缺失项已标注
- [ ] 所有 P1 测试点已逐一检查，缺失项已标注
- [ ] 边界值与异常场景已按 7 类全面扫描

### 一致性检查

- [ ] 已读取技术文档时，接口用例预期结果已与接口定义对照
- [ ] UI 用例的成功/失败路径预期结果已与交互规范对照
- [ ] 未读取技术文档时，报告中已注明「预期结果一致性检查不可用」

### 可执行性检查

- [ ] 每条用例已检查前置条件、步骤描述、预期结果、测试数据
- [ ] 发现问题的用例已逐条标注问题类型和修正建议

### 断言可执行性检查

- [ ] 所有预期结果已检查 CH-7（禁用词命中则给出改写建议）
- [ ] 所有预期结果已检查 CH-8（缺少 L1/L2/L3 标注则建议补充）
- [ ] 所有涉及精确数值的预期结果已检查 CH-9（视觉断言则建议升级为 API/DB-Query 断言）

### 文件输出

- [ ] `review/case-review-report.md` 已写入
- [ ] 报告包含六个章节，结论清晰（🟢/🟡/🔴）

### 下游衔接

- [ ] 🟢 结论：可直接进入 `CASE_SUPPLEMENT`（用例补充）阶段
- [ ] 🟡/🔴 结论：需修正后重新触发本 skill 或重新触发 `test-case-design`
