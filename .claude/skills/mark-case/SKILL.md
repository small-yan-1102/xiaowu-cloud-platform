# mark-case 用例状态勾选技能

> **受众**：工程师人工勾选用例执行状态（不是 AI 自动执行，AI 走 `test-execution` / `api-test-execution` 自动回写）。
> **核心价值**：替代手工修改套件 md，通过 AskUserQuestion 按钮式选择，避免拼写/格式错误。

---

## 用途

替代手工修改套件 md。工程师调用 `/mark-case {用例编号}` 即可通过 AskUserQuestion 按钮式勾选完成状态回写，符合 `.claude/skills/test-case-design/OVERRIDES.md` §新增 10 定义的「执行清单（状态记录入口）」格式。

---

## 参数解析

调用形式：

- `/mark-case` — 无参数，先问工程师要勾选哪些用例
- `/mark-case S01-001` — 单条
- `/mark-case S01-001 S01-002 S02-003` — 批量（空格分隔）
- `/mark-case S02-004 失败` — 编号 + 状态短词，跳过按钮勾选（见下方状态短词映射）

**状态短词映射**（可选，用户懒得点按钮时直接打）：

| 短词 | 映射到 |
|------|-------|
| `通过` / `pass` / `p` | ✅ 通过（勾选 `- [x]`） |
| `失败` / `fail` / `f` | ❌ 失败（不勾，行尾补 `· ❌ {备注}`） |
| `阻塞` / `block` / `b` | 🚫 阻塞（不勾，行尾补 `· 🚫 {原因}`） |
| `跳过` / `skip` / `s` | ⏭ 跳过（不勾，行尾补 `· ⏭ {原因}`） |
| `待执行` / `reset` / `r` | 清空备注（回滚为 `- [ ] **编号** 名称（优先级）`，不勾选） |

---

## 执行步骤

### Phase 0：解析输入与定位套件

1. 解析斜杠命令参数，提取用例编号列表和可选的状态短词
2. 如果没有用例编号 → 用 AskUserQuestion 问工程师："要勾选哪个用例？可输入编号，或列出待选"。若工程师说"列出待选"则列出所有当前迭代套件中**未勾选**的用例编号（即清单段内 `- [ ] **{编号}**` 行）
3. 对每个用例编号，使用 Grep 在 `iterations/**/testcase/suite_*.md` 和 `iterations/**/testcase/api_suite_*.md` 中按 `**{编号}**`（加粗格式）查找，确定所属套件文件
4. 匹配不到 → 报错："用例编号 {编号} 未在任何套件的执行清单段中找到，请确认"；匹配到多个 → 让工程师选择哪个迭代的

### Phase 1：交互式选择状态

**如果命令行已包含状态短词**（如 `/mark-case S01-001 失败`）：直接跳到 Phase 2。

**否则**：对每个待勾选用例（如有多个，逐个处理），用 AskUserQuestion 弹出：

```
问题："用例 {编号} 的执行结果？"
选项（5 个按钮）：
  - ✅ 通过（勾选）
  - ❌ 失败
  - 🚫 阻塞
  - ⏭ 跳过
  - 重置为未执行（回滚）
```

工程师点击一个按钮。

### Phase 2：根据状态补充备注（按需）

| 状态 | 额外询问 |
|------|---------|
| ✅ 通过 | 无需询问，直接 Edit |
| ❌ 失败 | AskUserQuestion："失败备注（BUG-id 或一句话原因）？" 提供常见选项：选填 BUG-id / 无 BUG-id 填原因 / 常见 BUG 前缀如 BUG- / TAPD- 让工程师只输数字 |
| 🚫 阻塞 | AskUserQuestion："阻塞原因？" 提供常见选项：登录失败 / 环境不可达 / 依赖用例未通过 / 数据异常 / 自定义 |
| ⏭ 跳过 | AskUserQuestion："跳过理由？" 提供选项：本轮不测 / 已被其他用例覆盖 / 暂缓 / 自定义 |
| 重置 | 无需询问（清空行尾备注） |

### Phase 3：生成清单段行内容

按 `.claude/skills/test-case-design/OVERRIDES.md` §新增 10 的格式约定：

| 状态 | 清单段行示例（保留原 `**{编号}** {名称}（{优先级}）`） |
|------|-----------------------------------------------|
| ✅ 通过 | `- [x] **S01-001** 跨期正常分发 0→3（P0）` |
| ❌ 失败 | `- [ ] **S01-001** 跨期正常分发 0→3（P0） · ❌ BUG-12` 或 `- [ ] **S01-001** 跨期正常分发 0→3（P0） · ❌ 返回 500` |
| 🚫 阻塞 | `- [ ] **S01-001** 跨期正常分发 0→3（P0） · 🚫 登录失败` |
| ⏭ 跳过 | `- [ ] **S01-001** 跨期正常分发 0→3（P0） · ⏭ 本轮不测` |
| 重置 | `- [ ] **S01-001** 跨期正常分发 0→3（P0）` |

**保留原行字段**：`**{编号}**`、`{名称}`、`{优先级}` 均来自原清单段行，**不得改动**；人工 `/mark-case` **不写 AI 标识、不写日期、不写报告链接**（这是 AI 执行的标志，人工勾选保持简洁）。

### Phase 4：Edit 回写清单段

1. 用 Read 读取定位到的套件文件
2. Grep 找到清单段内该用例的行（如 `^- \[[ x]\] \*\*OVERDUE-S01-001\*\* `）
3. 用 Edit 工具替换整行
   - old_string：完整的原行（含前缀 checkbox + 编号 + 名称 + 优先级 + 可能已有的行尾备注）
   - new_string：按 Phase 3 的新格式生成
4. 若用户在一次 `/mark-case` 中勾选了多条，逐条 Edit

### Phase 5：输出确认

单次简短输出，不长篇大论：

```
✅ 已回写 3 条：
- S01-001 → ✅ 通过（已勾选）
- S01-002 → ❌ 失败 · BUG-12
- S02-003 → 🚫 阻塞 · 登录失败

文件：iterations/2026-Q2_结算系统_逾期结算处理优化/testcase/suite_set01_status_dispatch.md
      iterations/2026-Q2_结算系统_逾期结算处理优化/testcase/suite_set02_settlement.md
```

---

## 失败处理

| 情况 | 处理 |
|------|------|
| 用例编号在所有套件的清单段都找不到 | 停止并告知工程师，列出用例编号，建议检查拼写 |
| 同一编号在多个套件中出现（跨迭代） | AskUserQuestion 让工程师选哪个迭代 |
| Edit 因清单段行格式异常失败 | 停止并告知原因，不中断后续用例的回写 |
| 工程师取消 AskUserQuestion | 该条用例不回写，继续下一条 |

## 与其他 Skill 的衔接

- **test-execution / api-test-execution**：AI 自动执行时会直接回写清单段（附 AI 标记 + 日期 + 报告链接），无需走 `/mark-case`。本命令**专供人工使用**。
- **真源约定**：本命令只改套件内「执行清单」段。失败详情、截图、步骤明细仍需由 AI 执行时落 `report/execution_report_*.md`。`/mark-case` **不创建**执行报告。
- **格式约束**：完全遵循 `.claude/skills/test-case-design/OVERRIDES.md` §新增 10、`.claude/skills/api-test-case-design/OVERRIDES.md` §新增 3 和 `.claude/skills/test-execution/OVERRIDES.md` §覆盖 2。
