# 测试用例与PRD一致性排查技能

> **版本**: v2.2
> **更新日期**: 2026-04-13
> **适用模式**: Reviewer Pattern — 读取用例和 PRD → 执行一致性审查 → 输出根因分析和修复方案

---

> **适用场景**：PRD 版本变更后的受影响用例识别（强制）、怀疑用例与 PRD 偏差、全量质检。
>
> **不适用**：评估用例结构质量（断言模糊/层级缺失/数值断言未升级等）→ 使用 `test-case-review`（CH-7～CH-9 已在该 skill 中统一检查）。

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 依赖检查与输入定位
- [ ] Phase 2: 一致性审查
- [ ] Phase 3: 根因分析与修复方案
- [ ] Phase 4: 输出审查报告
```

---

### Phase 0: 定向收集

**目标**：在任何审查行为开始前，明确触发原因和审查范围。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **触发原因**：本次触发是以下哪种情况？
   - **A：PRD 版本变更**（强制触发）→ 提供变更前后版本号，输出受影响用例清单
   - **B：怀疑偏差**（按需触发）→ 提供具体可疑用例编号或症状描述
   - **C：全量质检**（按需触发）→ 全量审查所有用例
2. **PRD 路径**：PRD 文档路径是什么（默认：`input/prd/*.md`）？当前版本号是多少？
3. **用例文件路径**：测试用例文件路径是什么（默认：`test_suites/*.md`）？

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 开始任何审查，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 触发原因 A（PRD 变更）→ 进入 Phase 1，执行**变更影响分析模式**
- 触发原因 B/C → 进入 Phase 1，执行**常规审查模式**

---

### Phase 1: 依赖检查与输入定位

**目标**：确认检查清单存在，定位问题用例和 PRD。

**步骤**：

1. **检查一致性检查清单**：
   - 检查 `references/consistency-checklist.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「一致性检查清单文件缺失」

2. **定位 PRD 文档**：
   - 读取用户指定路径的 PRD 文档
   - 未找到 → 🔵 使用 AskUserQuestion 请用户提供文件路径
   - 确认 PRD 版本号

3. **定位问题用例**：
   - 用户指定了具体编号 → 在用例文件中定位对应用例
   - 用户要求全量审查 → 读取全部用例文件
   - 未找到指定用例 → 🔵 使用 AskUserQuestion 提示用户确认路径

4. **汇总检查结果**：
   - 全部就绪 → 输出就绪信息（已定位 N 条问题用例，PRD 版本 VX.X），进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到 PRD 和用例文件均已定位。

---

### Phase 2: 一致性审查

**目标**：加载检查清单，对问题用例执行一致性审查。根据触发原因选择执行路径。

**步骤**：

1. **读取一致性检查清单**：加载 `references/consistency-checklist.md`

#### 触发原因 A：PRD 变更影响分析

1. **识别 PRD 变更内容**：
   - 对比 PRD 变更前后版本（若只有新版本，根据用户描述的变更范围推断）
   - 列出变更章节和变更类型（新增功能 / 修改规则 / 删除功能 / 文案变更 / 枚举值变更）

2. **映射受影响用例**：
   - 对每处变更，在用例文件中查找引用了对应 PRD 章节的用例
   - 同时扫描步骤文本中含变更关键词（功能名、枚举值、文案）的用例

3. **输出受影响用例清单**：

   | 用例编号 | 用例名称 | 关联 PRD 章节 | 变更类型 | 影响判断 | 建议处理 |
   |---------|---------|-------------|---------|---------|---------|
   | PRJ-XXX-001 | {用例名} | §3.2 | 规则修改 | 预期结果需更新 | 方案 A |

4. **对受影响用例执行 CH-1～CH-9 检查**（同下方常规审查模式）

> 触发原因 A 的重点输出：受影响用例清单 + 每条的修复建议，而非全量扫描。

---

#### 触发原因 B/C：常规审查模式

2. **对每条问题用例，执行以下检查**（参照检查清单「§二 逐项审查清单」）：

   **PRD 一致性检查**：
   - **CH-1**：PRD 是否明确描述了该步骤或预期结果？
   - **CH-2**：用例引用的 PRD 版本是否是当前最新版？
   - **CH-3**：用例中的术语是否与 PRD 精确一致？
   - **CH-4**：是否参考了历史用例（是否有旧版本污染）？
   - **CH-5**：步骤顺序是否与 PRD 流程描述一致？
   - **CH-6**：测试设计扩展是否有 PRD 依据？

   > **断言可执行性（CH-7～CH-9）已由 `test-case-review` 统一检查，本 skill 不重复执行。**

3. **识别根因类型**（参照检查清单「§一 不一致根因分类」）：
   - 类型 A：AI 理解偏差
   - 类型 B：测试设计方法过度扩展
   - 类型 C：历史用例干扰
   - 类型 D：PRD 与系统不同步

4. **执行快速检查清单**（参照检查清单「§四 快速检查清单」）：
   - 逐项核对 8 个检查项
   - 标记未通过的检查项

> ⛔ **DO NOT** 进入 Phase 3，直到所有问题用例的审查记录均已完成。

---

### Phase 3: 根因分析与修复方案

**目标**：对每条问题用例给出根因结论和具体修复建议。

**步骤**：

1. **确定修复方案**（参照检查清单「§三 修复方案决策表」）：

   | 场景 | 方案 |
   |------|------|
   | PRD 正确，用例错误 | **方案 A**：修正用例，按 PRD 原文改写 |
   | 用例正确，PRD 过时 | **方案 B**：标记 PRD 需更新，用例暂按系统行为执行 |
   | AI 合理补充（有测试价值） | **方案 C**：保留并在「备注」字段中标注扩展理由 |
   | 重复或无效 | **方案 D**：删除或标记跳过 |

2. **对选择方案 A 的用例**，在报告的「修复内容」章节中输出修正后的完整用例内容（**不直接覆盖原用例文件**，由用户确认后手动应用或通过 test-case-design 重新生成）：
   - 修正有问题的步骤文本（与 PRD 精确匹配）
   - 修正预期结果（与 PRD 一致）
   - 更新「关联需求」字段（标注 PRD 版本和章节）

3. **对选择方案 C 的用例**，输出需要追加的备注内容：
   - 格式：`备注：基于 PRD 的合理扩展：{扩展理由}`

---

### Phase 4: 输出审查报告

**目标**：汇总所有审查结果，生成结构化报告，便于追踪修复进度。

**输出内容**：

```markdown
# 用例与PRD一致性审查报告

**审查日期**：{YYYY-MM-DD}
**触发原因**：A：PRD版本变更 / B：怀疑偏差 / C：全量质检
**PRD 版本**：V{X.X}（变更触发时：V{旧} → V{新}）
**审查用例数**：{N} 条

<!-- 触发原因 A 时额外输出 -->
## PRD 变更影响摘要

| 变更章节 | 变更类型 | 受影响用例数 | 需立即修复 |
|---------|---------|-----------|---------|
| §{X.X} | 规则修改 / 文案变更 / 新增 / 删除 | N | Y/N |

**受影响用例清单**：（详见「问题用例详情」章节，标注「PRD变更」标签）

---

## 审查摘要

| 根因类型 | 数量 | 推荐方案 |
|---------|------|---------|
| A：AI理解偏差 | N | 方案A/C |
| B：过度设计扩展 | N | 方案A/C/D |

## 问题用例详情

| 用例编号 | 不一致位置 | 根因类型 | 修复方案 | 修复状态 |
|---------|---------|---------|---------|---------|

> 以下内容为建议修正版本，**未直接修改原用例文件**。确认无误后，请将对应用例替换到 `test_suites/*.md` 中对应位置，或触发 `test-case-design` 针对该模块重新生成。

{逐条输出修正后的用例内容}

```
