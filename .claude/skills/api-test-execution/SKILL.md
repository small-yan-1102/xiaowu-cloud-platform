# 接口测试执行技能

> **版本**: v1.0
> **更新日期**: 2026-04-08
> **适用模式**: AI 读取 Markdown 接口用例 → 直接执行 HTTP 请求 → 自主断言响应 → 输出执行报告

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 依赖检查与预执行准备
- [ ] Phase 2: 执行接口用例
- [ ] Phase 3: 输出执行报告
```

---

### Phase 0: 定向收集

**目标**：在任何执行行为开始前，确认测试范围和环境配置。

向用户提出以下问题，**等待用户全部回答后再进行任何操作**：

1. **用例文件路径**：要执行的接口用例文件是什么（默认：`test_suites/api_suite_*.md`）？
2. **执行范围**：执行全部用例，还是指定优先级（P0/P1）或特定接口模块？
3. **环境配置**：
   - 测试环境 Base URL 是什么？
   - 认证账号（用于获取 Token 的用户名/密码）是什么？
4. **数据准备报告**（如有）：是否已有数据准备报告（`test_suites/data-prep-report-*.md`）？

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 发起任何 HTTP 请求，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

---

### Phase 1: 依赖检查与预执行准备

**目标**：确认资源文件存在，验证测试环境可达，加载变量映射表。

**步骤**：

1. **检查报告模板文件**：
   - 检查 `assets/execution-report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「报告模板文件缺失」

2. **读取接口用例文件**：
   - 加载用户指定的用例文件
   - 按执行顺序表排列用例（P0 → P1 → P2 → P3，有依赖的按依赖链排序）
   - 统计用例总数、各优先级分布、依赖关系

3. **加载数据准备报告**（如有）：
   - 读取 `test_suites/data-prep-report-*.md`，取最新一份
   - 提取变量映射表（供执行时替换 `{引用:变量名}`）

4. **环境预检**：
   - 访问 `{Base URL}/` 或文档头部指定的健康检查路径，确认环境可达
   - 使用认证账号执行登录请求，获取 Token，记录到**会话变量表**
   - 任一步骤失败 → **Fatal**：终止执行，提示「环境不可用或认证失败」

5. **输出就绪清单**：已加载 N 条用例，环境预检通过，进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到环境预检通过且所有依赖均已就绪。

---

### Phase 2: 执行接口用例

**目标**：按顺序逐条执行用例，记录每条结果。

**执行顺序策略**：

```
P0 用例 → 全部通过才继续
     ↓ 有失败 → 终止执行，报告 P0 失败
P1 用例
     ↓
P2 用例
     ↓
P3 用例
```

**对每条用例，按以下流程执行**：

**步骤 1：解析用例**
- 读取：用例编号、维度、前置条件、请求（Method / URL / Headers / Body）、预期结果、清理步骤、依赖用例

**步骤 2：检查前置条件**
- `[认证]`：确认会话变量表中 Token 有效（若已过期则重新登录刷新）
- `[数据存在]`：按 `[验证: ...]` 标签执行查询请求，确认数据就绪
  - 验证失败 → 标记为**阻塞**，进入下一用例

**步骤 3：变量替换**
- 将用例中所有 `{引用:变量名}` 替换为会话变量表中对应的值
- 将 `{时间戳}` 替换为当前时间戳（yyyyMMddHHmmss）
- 有引用但变量不存在 → 标记为**阻塞**，提示「变量 {变量名} 未定义」

**步骤 4：构造并执行 HTTP 请求**
- 按用例中的 Method / URL / Headers / Body 构造请求
- 超时设置：默认 10 秒；文件上传类 30 秒
- 记录完整的请求详情（URL / Headers 脱敏 / Body / 耗时）

**步骤 5：断言响应**

| 断言项 | 规则 |
|-------|------|
| HTTP 状态码 | 精确匹配预期状态码（L1，失败则终止该用例） |
| 响应体字段（L1） | 精确匹配或包含匹配，失败则用例判定为失败 |
| 响应体字段（L2） | 不通过时记录警告，不影响用例判定 |
| 响应体字段（L3） | 仅记录观察值，不判定 |

**断言失败处理**：
- L1 断言失败 → 记录期望值和实际值，终止当前用例，标记**失败**
- 不重试（接口响应是确定性的，重试掩盖真实问题）

**步骤 6：提取并记录变量**
- 将预期结果中 `{记录为:变量名}` 标注的响应字段值写入会话变量表
- 供后续依赖该变量的用例使用

**步骤 7：执行清理步骤**
- 按用例中的清理步骤逐条执行
- 清理失败时：记录警告，**不影响当前用例判定**，但在报告中标注「需人工清理」

**步骤 8：记录单条结果**
- 参照 `assets/execution-report-template.md` §模板一 记录

---

#### 错误恢复规则

| 错误类型 | 处理策略 | 重试次数 |
|---------|---------|---------|
| 网络超时 | 等待 2 秒后重试 | 1 次 |
| HTTP 5xx | 等待 3 秒后重试 | 1 次 |
| HTTP 4xx | 直接记录失败，**不重试** | 0 次 |
| Token 过期（401） | 自动重新登录刷新 Token，重试 | 1 次 |
| 依赖用例失败 | 标记当前用例为**跳过** | — |

> ⛔ **DO NOT** 进入 Phase 3，直到所有待执行用例均已记录结果（通过/失败/阻塞/跳过）。

---

### Phase 3: 输出执行报告

**目标**：按 `assets/execution-report-template.md` 中的模板，生成完整的执行报告。

**步骤**：

1. **读取报告模板**：加载 `assets/execution-report-template.md`

2. **确定输出路径**：`test_reports/api_report_YYYYMMDD_HHmmss.md`

3. **填写报告内容**：
   - 执行概览（总数 / 通过 / 失败 / 阻塞 / 跳过）
   - 各接口通过率汇总
   - 失败用例详情（完整请求 + 响应 + 断言差异）
   - L2 警告汇总
   - 会话变量表（调试用）
   - 需人工清理的数据清单（如有）
   - 审计清单（逐项自查后勾选）

4. **向用户展示**：报告路径 + 执行概览摘要 + 失败用例列表

## 验收标准

### 执行完整性

- [ ] 所有待执行用例均已记录执行结果
- [ ] 每条用例有完整的请求详情记录（URL / 状态码 / 关键响应字段）
- [ ] 每条失败用例记录了期望值和实际值的对比

### 报告完整性

- [ ] 汇总报告已写入 `test_reports/api_report_*.md`
- [ ] 报告包含各接口维度的通过率统计
- [ ] 需人工清理的数据已在报告中列出

### 变量管理

- [ ] 所有 `{记录为:变量名}` 的值已写入会话变量表
- [ ] 变量表已在报告中输出（便于调试依赖链问题）
