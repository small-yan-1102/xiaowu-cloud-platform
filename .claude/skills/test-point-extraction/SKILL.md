## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载 PRD 审阅摘要
```

### Phase 0: 定向收集

**目标**：确认输入文件路径，避免读错文件。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **PRD 审阅摘要路径**：test-prd-review 生成的审阅摘要位于哪里？（默认：`review/test-prd-summary.md`）
2. **PRD 审阅报告路径**：test-prd-review 生成的审阅报告位于哪里？（默认：`review/test-prd-review-report.md`，用于提取 B/S/A 系列高风险问题加密覆盖）
3. **提炼范围**：全量提炼所有模块，还是聚焦某个特定模块？（默认：全部模块）
4. **历史知识补充**（可选）：系统将在 Phase 1 自动检查知识库（`knowledge/risk-profiles/` 和 `knowledge/defect-patterns/`）是否有当前模块的历史档案，有则自动加载。如有知识文件未覆盖的额外信息（如"本次迭代特别关注 XX 场景"），可在此补充。（默认：无补充，完全依赖知识库或均衡覆盖）

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

**决策点**：
- 用户回答或表示"按默认来" → 使用默认路径，提炼全部模块，进入 Phase 1
- 用户提供了额外补充 → 记录为**手动风险补充**，Phase 1 知识加载后与知识库内容合并

---

### Phase 1: 加载 PRD 审阅摘要

**目标**：读取审阅摘要，确认包含必要章节。

**步骤**：

1. **检查测试点清单模板**：
   - 检查 `assets/test-points-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「测试点清单模板文件缺失」

2. **读取 PRD 审阅摘要**：
   - 加载用户指定路径（默认 `review/test-prd-summary.md`）
   - 不存在 → **Fatal**：终止执行，提示「PRD 审阅摘要不存在，请先执行 test-prd-review」

3. **读取 PRD 审阅报告**（用于风险映射）：
   - 加载用户指定路径（默认 `review/test-prd-review-report.md`）
   - 文件存在 → 提取以下内容，记录为**风险映射表**：
     - **B 系列**（阻塞性问题）：问题编号 / 所属模块 / 问题描述 → 对应测试点需升级至 P0 并标注来源
     - **S 系列**（需补充项）：问题编号 / 所属模块 → 对应测试点标注「⚠️ PRD 待补充，用例设计时确认」
     - **A 系列**（歧义项）：问题编号 / 所属模块 / 两种可能理解 → 对应测试点标注「⚠️ 歧义待确认：{理解A} / {理解B}」
   - 文件不存在 → **Warn**：跳过风险映射，继续执行；在输出摘要中注明「未读取审阅报告，风险加密覆盖不可用」

4. **确认摘要结构**，提取以下内容：
   - **§一 功能模块总览**：模块列表、各模块核心目标
   - **§二 各模块详细理解**：业务规则、数据约束、状态定义、权限要求
   - **§三 模块间依赖关系**：依赖链和集成边界
   - **§四 操作路径清单**：用户操作路径（入口、导航步骤、核心操作、预期结果）

   > 如摘要缺少某章节 → 记录缺失，从其他章节尽量补充，不终止执行

5. **加载历史知识**（知识库自动检查，静默处理）：

   从步骤 4 识别的模块列表出发，对每个模块逐一检查以下路径（模块名即文件名，允许模糊匹配前缀）：

   | 知识文件 | 路径 | 加载后用途 |
   |---------|------|---------|
   | 风险档案 | `knowledge/risk-profiles/{模块名}.md` | 标注历史高风险维度，Phase 2 对应维度加深覆盖 |
   | 缺陷模式 | `knowledge/defect-patterns/{模块名}.md` | 识别历史高频缺陷场景，对应测试点升至 P1 以上并标注来源 |

   **加载规则**：
   - 文件存在 → 加载，与 Phase 0 手动补充内容合并，汇总为**风险权重上下文**
   - 文件不存在 → 静默跳过，不提示用户，按 PRD 内容均衡覆盖
   - 知识内容与 PRD 有冲突时 → 以 PRD 为准，知识内容仅作权重参考

   **Phase 2 标注约定**：
   - 来自知识库的风险维度 → 在测试点备注中标注 `📚 知识库`
   - 来自手动补充的风险维度 → 标注 `📝 手动补充`

> ⛔ **DO NOT** 进入 Phase 2，直到摘要加载成功。

### Phase 2: 按维度提炼测试点

**目标**：从摘要内容中系统性地提炼测试点，覆盖以下 7 个维度。

> **原则**：每个测试点只记录"要验证什么"，不展开具体步骤。步骤由 test-case-design 负责。

---

#### 维度 1：正向流程（PF）

来源：§四 操作路径清单

- 每条操作路径对应至少一个 P0 正向测试点
- 含多个分支的路径，每个分支单独记录
- 格式：`{主体}在{场景}下执行{操作}，{预期结果}`

---

#### 维度 2：边界值（BV）

来源：§二 各模块详细理解中的数据约束

检查以下内容，存在则提炼：
- 数值字段：最大值、最小值、临界值（如：金额上限 9999.99）
- 文本字段：最大长度、最小长度、空值
- 数量限制：列表最多N条、批量操作上限
- 日期/时间：过去日期、未来日期、跨天跨月边界

---

#### 维度 3：异常与错误场景（EX）

来源：§二 业务规则 + §四 操作路径（失败路径）

检查以下内容，存在则提炼：
- 必填字段为空提交
- 格式不合法（非数字/非日期/特殊字符）
- 重复提交（创建相同名称/编号的记录）
- 依赖资源不存在（关联对象已删除/被占用）
- 操作冲突（并发编辑、状态不允许的操作）

---

#### 维度 4：权限与角色（AU）

来源：§二 权限要求

检查以下内容，存在则提炼：
- 无权限角色访问/操作受限功能的拒绝行为
- 仅特定角色可见的入口、按钮、菜单
- 数据隔离（A 用户不可见 B 用户的数据）
- 越权操作的拒绝行为

> 无权限控制描述 → 此维度标注"PRD 未涉及，跳过"

---

#### 维度 5：状态机转换（SM）

来源：§二 状态定义

检查以下内容，存在则提炼：
- 合法状态转换路径（如：草稿 → 审核中 → 已通过）
- 非法状态转换的拒绝行为（如：已发布不可直接回到草稿）
- 终态的不可逆性验证

> 无状态机描述 → 此维度标注"PRD 未涉及，跳过"

---

#### 维度 6：集成与依赖（IN）

来源：§三 模块间依赖关系

检查以下内容，存在则提炼：
- 上游模块数据变化对当前模块的影响
- 关联模块同步更新的一致性（如：订单取消后库存是否回滚）
- 跨模块状态联动
- 当前模块数据被其他模块消费的场景（如：本模块的审核结果被下游报表/结算引用）
- 跨系统数据一致性（如：AMS 创建任务单后剧老板侧是否同步展示）
- 上游依赖链路中断后的降级行为（如：SSO 登录服务不可用时各子系统的表现）

> 无跨模块依赖 → 此维度标注"PRD 未涉及，跳过"

---

#### 维度 7：UI 渲染状态（UI）

来源：§四 操作路径清单 + 原型 + PRD 中的界面说明

对每个有列表/表单/详情页的功能模块，检查以下场景并提炼测试点：

| 场景 | 检查要点 |
|------|---------|
| 空状态 | 列表/搜索结果无数据时，是否有明确的空状态提示（文案/图示）而非空白页 |
| 加载态 | 数据加载中的 loading 指示器或骨架屏，是否在数据就绪后消失 |
| 错误态 | 接口失败/网络异常时，页面是否显示错误提示而非白屏或静默失败 |
| 临界量渲染 | 列表 1 条 / N 条 / 最大条数时，分页、滚动、布局是否正常 |
| 表单回填 | 编辑场景下，已有数据是否正确回填到各字段（含下拉、日期、文件等） |

> PRD 通常不描述这些渲染态，但测试工程师凭经验知道必须覆盖。若原型中有对应设计则作为验证依据；若原型和 PRD 均未提及，仍须提炼测试点，标注「PRD/原型未明确，执行时以实际表现为准」。

优先级：核心页面的空态/错误态 → P1；非核心页面 / 临界量渲染 / 表单回填 → P2

---

#### 审阅报告风险映射

> 仅当 Phase 1 成功读取审阅报告时执行。

对风险映射表中的每条问题，执行以下操作：

| 问题类型 | 操作 |
|---------|------|
| **B 系列**（阻塞性问题） | 检查已提炼测试点中是否覆盖该问题所涉及的场景；若有 → 升级为 P0 并追加标注 `[来源: 审阅报告 {编号}]`；若无 → 新增一条 P0 测试点并标注来源 |
| **S 系列**（需补充项） | 检查相关测试点，追加标注「⚠️ PRD 待补充，用例设计时需与产品确认该项的最终规则」 |
| **A 系列**（歧义项） | 检查相关测试点，追加标注「⚠️ 歧义待确认：{理解A} / {理解B}，执行前需产品明确」 |

---

#### 历史高风险域权重规则

若 Phase 0 用户提供了历史高风险域，在对应维度执行时：
- 标注「⚠️ 历史高风险」
- 覆盖更细的子场景（如 AU 维度：不仅检查入口权限，还检查数据隔离、越权操作、角色切换边界）
- P1 测试点在同等条件下可升级为 P0

> 若未提供历史高风险域 → 按 PRD 内容均衡覆盖，不做权重调整。

---

#### 提炼规则

- **优先级判定**：

  | 优先级 | 判定规则 |
  |-------|---------|
  | P0 | 主流程正向路径、核心业务规则、权限拦截 |
  | P1 | 重要边界值、重要异常场景、状态机合法路径 |
  | P2 | 次要边界值、非核心状态转换、集成联动 |

- **编号规则**：`TP-{模块缩写}-{序号}`，如 `TP-ORDER-001`；单模块时直接 `TP-001`
- **不重复**：同一验证意图只记录一次，合并方向不同但目的相同的点
- **不展开**：只写"验证什么"，不写"怎么操作"
- **AI 可执行性标签**：每个测试点必须标注以下之一，供 test-case-design 决策用例设计方式：

  | 标签 | 含义 | 典型场景 |
  |------|------|---------|
  | `[AI]` | AI 可完全自动执行 | 表单提交、列表查询、状态变更、权限拒绝验证 |
  | `[AI+人工]` | AI 执行主流程，特定步骤需人工介入 | 验证码/短信验证、扫码操作、第三方授权弹窗 |
  | `[人工]` | 纯人工执行，AI 无法完成 | 纯视觉走查（色彩/字体/图片内容）、物理设备操作、无法自动化的第三方系统 |

  标注依据：
  - 涉及验证码、短信验证码、图形验证 → `[AI+人工]`
  - 涉及微信支付、人脸识别、扫码等第三方操作 → `[AI+人工]` 或 `[人工]`
  - 纯视觉审查（"颜色是否正确"、"图片是否清晰"）→ `[人工]`
  - 其余默认 → `[AI]`

> ⛔ **DO NOT** 进入 Phase 3，直到所有 7 个维度均已处理（或标注"跳过"）。

---

### Phase 3: 输出测试点清单

**目标**：按 `assets/test-points-template.md` 生成测试点清单文件。

**步骤**：

1. **读取测试点清单模板**：加载 `assets/test-points-template.md`

2. **确定输出路径**：`review/test-points.md`
   - 文件已存在 → **覆盖写入**（本次为最新版本）

3. **填写报告内容**：
   - 概览统计（各优先级数量 / 各维度数量 / AI 可执行性分布）
   - 按功能模块分组的完整测试点清单（含 AI 可执行性标签、风险来源标注、歧义/待确认标注）
   - 各维度覆盖统计
   - 审阅报告风险映射摘要（仅当读取了审阅报告时）：B/S/A 系列问题各处理了几条，新增了几条测试点

4. **向用户展示**：
   - 总测试点数量（按优先级 / 维度 / AI 可执行性分布）
   - `[AI+人工]` 和 `[人工]` 测试点清单（提醒测试工程师提前安排人工介入计划）
   - 歧义/待确认测试点清单（提醒在用例设计前与产品确认）
   - 提示：`test-case-design` Mode A 应以 `review/test-points.md` 为覆盖基准，每个 P0 测试点至少对应一条用例

## 验收标准

### 覆盖完整性

- [ ] 7 个维度均已处理（未涉及的已标注"跳过"，并说明原因）
- [ ] 所有操作路径均已映射到正向测试点
- [ ] 存在数据约束的字段已产出边界值测试点
- [ ] 存在权限描述的功能已产出权限测试点
- [ ] 有列表/表单/详情页的模块均已产出 UI 渲染状态测试点（维度 7）
- [ ] 审阅报告中所有 B 系列问题均已映射到至少一条 P0 测试点（读取了审阅报告时）
- [ ] A 系列歧义问题已在对应测试点中标注「⚠️ 歧义待确认」

### 输出质量

- [ ] 每个测试点描述明确"验证什么"，无歧义
- [ ] 优先级标注符合规则（P0/P1/P2）
- [ ] 编号唯一，格式统一
- [ ] 无重复意图的测试点
- [ ] 每个测试点均已标注 AI 可执行性标签（`[AI]` / `[AI+人工]` / `[人工]`）
- [ ] `[AI+人工]` 和 `[人工]` 测试点已在输出摘要中单独汇总

### 文件输出

- [ ] `review/test-points.md` 已写入
- [ ] 包含概览统计和按模块分组的完整清单

### 下游衔接

- [ ] 文件可被 `test-case-design` Mode A 直接读取作为覆盖基准
- [ ] P0 测试点清晰标注，便于用例设计优先覆盖
