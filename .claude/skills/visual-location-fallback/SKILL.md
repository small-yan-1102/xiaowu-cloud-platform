# 视觉定位兜底技能

> **版本**：v1.0
> **更新日期**：2026-04-13
> **定位**：test-execution 元素定位失败时的最后兜底手段，不替代语义定位，仅在其完全失效时介入

---

## 使用前提

本 skill 仅在以下条件**全部满足**时启用：

1. test-execution 已按优先级依次尝试：可见文本 → aria-label/role → placeholder/title → 相对位置描述
2. 上述四种方式均未能定位到目标元素
3. 页面已正常加载（非白屏、非加载中状态）

> ⚠️ **不得作为第一手段**：视觉坐标定位脆弱（受分辨率、布局变化影响），优先修复用例的语义定位描述。

---

## 用例中的标注方式

**方式一（推荐）：隐式触发**
- 不在用例中标注，由 test-execution 语义定位失败后自动触发本 skill
- 触发后在执行记录中标注 `[视觉定位兜底]`

**方式二：显式标注（已知语义定位无效时预声明）**
```markdown
步骤：
3. [定位: 视觉兜底] 点击画布左上角的"新建节点"图标按钮（蓝色圆形，内含加号）
```
显式标注时，跳过语义定位尝试，直接进入本 skill。

---

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 确认触发条件
- [ ] Phase 1: 截图采集与目标描述
- [ ] Phase 2: AI 视觉定位分析
- [ ] Phase 3: 坐标点击与结果验证
- [ ] Phase 4: 输出定位记录
```

---

### Phase 0: 确认触发条件

**由 test-execution 自动触发时**：
- 接收上下文：当前页面状态、目标元素的语义描述、已尝试的定位方式及失败原因
- 跳过用户问答，直接进入 Phase 1

**独立调用时**，向用户确认：

1. **目标元素描述**：想要定位的元素是什么？（包括视觉特征：颜色、形状、位置、图标样式）
2. **预期操作**：定位后执行什么操作？（点击 / 悬停 / 输入）
3. **当前页面**：是否已在目标页面？

---

### Phase 1: 截图采集与目标描述

1. 调用 `browser_take_screenshot` 采集当前完整页面截图
2. 整理目标元素的视觉特征描述（从用例步骤描述或用户输入中提取）：

   **高质量目标描述要素**：
   - 外观特征（颜色、形状、图标样式）
   - 大致位置（页面左上角、工具栏区域、弹窗内）
   - 相邻参照物（"位于搜索框右侧"、"列表第一行操作列"）

   **描述质量判断**：
   - 描述充分 → 直接进入 Phase 2
   - 描述模糊（如"找一个按钮"）→ 在执行记录中标注「视觉定位描述不足，结果可信度低」，仍继续执行

---

### Phase 2: AI 视觉定位分析

将截图与目标元素描述提交 AI 视觉模型，输出定位结果：

**定位输出结构**：

```
目标元素：{描述}
定位结果：{找到 / 未找到 / 不确定}
视觉置信度：{高(≥85%) / 中(60-84%) / 低(<60%)}
坐标区域：{x: N, y: N, width: N, height: N}（相对于页面左上角，单位 px）
点击建议坐标：{x: N, y: N}（目标区域中心点）
参照特征：{AI 观察到的视觉特征，说明为何认为这是目标元素}
```

**置信度处理规则**：

| 置信度 | 处理方式 |
|-------|---------|
| 高（≥ 85%） | 直接进入 Phase 3 执行点击 |
| 中（60-84%） | 进入 Phase 3，执行前在记录中标注「⚠️ 视觉定位置信度中等，结果需验证」|
| 低（< 60%） | **不执行点击**，标记「🔵 视觉定位失败：无法从截图中确认目标元素位置」，中止当前步骤 |

**无法定位时的处理**：
- 重新截图一次（防止截图时元素未完全渲染）→ 再分析一次
- 仍无法定位 → 标记步骤为 🚫 阻塞，附上截图，提示「需人工确认元素是否存在」

---

### Phase 3: 坐标点击与结果验证

1. **执行坐标点击**：
   - 使用 Phase 2 输出的点击建议坐标执行点击操作
   - 点击前记录当前页面状态（用于与点击后对比）

2. **点击后验证**（执行以下其中一种）：
   - 等待可观测信号出现（参照用例步骤中的等待描述）
   - `browser_snapshot` 检查页面状态是否发生变化
   - 若用例有后续语义定位步骤可成功执行，视为点击生效

3. **验证结果判定**：

   | 结果 | 判定依据 | 处理方式 |
   |------|---------|---------|
   | ✅ 点击生效 | 页面状态发生预期变化 | 继续执行用例后续步骤 |
   | ❌ 点击无效 | 页面无响应，状态未变化 | 重试一次（更换坐标为区域边缘点）；仍无效则标记失败 |
   | ⚠️ 点击错误目标 | 页面变化但非预期（弹出错误内容、跳到错误页面）| 撤销操作（浏览器后退或关闭弹窗）→ 标记定位失败，中止步骤 |

---

### Phase 4: 输出定位记录

在 test-execution 执行记录中追加以下信息：

```
[视觉定位兜底] {目标元素描述}
触发原因：语义定位全部失败（{已尝试的方式}）/ 用例显式标注
定位结果：{✅ 成功 / ❌ 失败 / 🔵 无法判定}
视觉置信度：{高/中/低}
点击坐标：({x}, {y})
验证结果：{点击生效 / 点击无效 / 点击错误目标}
截图：{截图路径（标注目标区域）}
⚠️ 建议：{视觉定位本质上脆弱，建议开发补充 aria-label 或 data-testid，后续迭代改用语义定位}
```

**截图命名**：`VLF_{用例编号}_{步骤标识}_{结果}_{时间戳}.png`（VLF = Visual Location Fallback）

---

## 验收标准

- [ ] 仅在语义定位全部失败后触发（或用例显式标注）
- [ ] 置信度低时未强行执行点击
- [ ] 点击错误目标时已撤销操作，未对测试状态造成污染
- [ ] 每次使用均在执行记录中附上「建议开发补充语义标注」的提示
- [ ] 定位截图已存档（含目标区域标注）

---

## 与 test-execution 的衔接

**引入方式**：在 test-execution `references/execution-strategy.md` §一.1 元素定位优先级末尾，追加第 5 级：

```
| 5 | **视觉定位兜底**（需加载 multimodal-visual-assertion / visual-location-fallback skill）| 前四级均失败且页面已正常加载时启用 |
```

**未引入时的处理**：未加载本 skill 时，语义定位失败则按现有规则标记步骤为阻塞，不做任何坐标点击尝试。

---

## 局限性

| 局限 | 说明 |
|------|------|
| 坐标依赖分辨率 | 浏览器窗口尺寸变化会导致坐标失效；固定 1920×1080 可缓解 |
| 布局变化敏感 | 页面结构调整后坐标失效，需重新定位 |
| 不适合高频使用 | 每次都需截图+AI分析，速度明显慢于语义定位 |
| 不适合精确操作 | 坐标点击误差约 ±5px，不适合小于 10px 的目标 |
| 根本解 | 本 skill 是临时兜底，根本解是被测系统补充无障碍语义标注 |
