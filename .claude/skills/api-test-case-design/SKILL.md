# 接口测试用例设计技能

> **版本**: v1.1
> **更新日期**: 2026-04-13
> **适用模式**: AI 读取接口定义文档 → 设计接口测试用例 → 输出可供 api-test-execution 执行的用例套件
> **v1.1 变更**: Phase 1 新增 test-points.md 可选加载（存在时作为覆盖基准，对照补全用例）；新增 Phase 6 输出前自检

## 附属资源文件

| 文件 | 路径 | 说明 |
|------|------|------|
| 用例模板 | `assets/case-template.md` | 接口用例标准格式模板，Phase 3 生成时加载 |
| 质量检查清单 | `references/quality-checklist.md` | 接口用例审查规则 + 禁用词表 + 覆盖度检查，Phase 4 审查时加载 |

## 模式路由

| 用户请求特征 | 执行模式 |
|------------|---------|
| "设计/生成接口用例" + 无变更说明 | **Mode A**：基于接口文档全量生成 |
| "变更接口" / "接口回归" / 引用变更列表 | **Mode B**：针对变更接口设计回归用例 |

---

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 依赖检查与输入验证
- [ ] Phase 2: 接口解析
- [ ] Phase 3: 用例设计
- [ ] Phase 4: 质量审查
- [ ] Phase 5: 输出组装
- [ ] Phase 6: 输出前自检
```

---

### Phase 0: 定向收集

**目标**：在任何生成行为开始前，收集必要信息，避免生成后大范围返工。

向用户提出以下问题，**等待用户全部回答后再进行任何操作**：

1. **模式确认**（根据模式路由自动判断，有歧义时询问）：
   - Mode A：接口文档路径是什么（默认：`input/tech/*.md`）？
   - Mode B：变更的接口列表是什么（接口路径 + HTTP Method）？接口文档路径是什么？
2. **测试范围**：全量覆盖所有接口，还是聚焦特定模块或接口？
3. **优先级范围**：需要覆盖 P0-P3 全部优先级，还是只需要 P0-P1？
4. **环境信息**：测试环境 Base URL 是什么？认证方式是什么（Bearer Token / Cookie / API Key）？

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 开始任何分析或生成，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答 → 进入 Phase 1
- 用户表示"按默认来" → 使用默认路径，进入 Phase 1

---

### Phase 1: 依赖检查与输入验证

**目标**：确认模板和规范文件存在，验证输入材料完整。

**步骤**：

1. **检查用例模板**：
   - 检查 `assets/case-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「用例模板文件缺失」

2. **检查质量检查清单**：
   - 检查 `references/quality-checklist.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「质量检查清单缺失」

3. **验证接口文档**：
   - 扫描用户指定路径或默认路径（`input/tech/*.md`）
   - 未找到 → 🔵 使用 AskUserQuestion 请用户提供文件路径
   - 仍不存在 → **Fatal**：终止执行，提示「接口文档缺失」

4. **加载测试点清单（可选覆盖基准）**：
   - 检查 `review/test-points.md` 是否存在
   - 存在 → 加载，记录 P0/P1 测试点总数，在 Phase 3 用例设计时对照补全覆盖
   - 不存在 → **Warn**：跳过覆盖基准对照，在输出摘要中注明「未加载测试点清单，接口用例覆盖基准不可用」

5. **Mode B 额外检查**：
   - 确认变更接口列表不为空
   - 在接口文档中定位变更接口，未找到任何一条 → **Fatal**：提示「变更接口在文档中未找到」

6. **汇总检查结果**：
   - 有 Fatal 缺失 → 一次性列出所有缺失项
   - 全部就绪 → 输出就绪清单（含是否加载了测试点清单），进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到所有必要依赖均已就绪。

---

### Phase 2: 接口解析

**目标**：从接口文档中提取所有接口的完整定义，构建接口清单。

**步骤**：

1. **逐文件读取接口文档**，提取每个接口的：
   - HTTP Method + 路径（如 `POST /api/order`）
   - 请求参数（Path / Query / Header / Body 字段，含类型、是否必填、约束条件）
   - 响应结构（字段名、类型、枚举值、说明）
   - 错误码列表（code、HTTP 状态码、含义）
   - 认证要求（是否需要 Token、权限角色）
   - 业务约束（唯一性、关联依赖、幂等性说明）

2. **构建接口清单表**：

   | # | 接口名称 | Method | 路径 | 认证 | 关键约束 |
   |---|---------|--------|------|------|---------|
   | 1 | 创建订单 | POST | /api/order | Bearer Token | amount > 0；orderId 唯一 |

3. **Mode B**：在接口清单中标记变更接口（新增 / 修改 / 删除），仅对标记接口生成用例。

4. **识别接口间依赖**（如创建后查询、创建后删除），记录依赖关系，用于后续用例排序。

> ⛔ **DO NOT** 进入 Phase 3，直到接口清单已完整输出。

---

### Phase 3: 用例设计

**目标**：按照 `assets/case-template.md` 中的模板，对每个接口从七个维度生成用例。

**步骤**：

1. **读取用例模板**：加载 `assets/case-template.md`

2. **对每个接口，按维度生成用例**（根据优先级范围过滤）：

   | 维度 | 优先级 | 生成条件 |
   |------|-------|---------|
   | 正向（Happy Path） | P0 | 必须；合法参数，验证响应结构和核心字段值 |
   | 鉴权 | P0 | 接口有认证要求时；无 Token / 过期 Token / 无权限角色 |
   | 错误码覆盖 | P1 | 文档定义了错误码时；逐一覆盖 4xx 场景 |
   | 参数边界 | P1 | 存在数值/长度约束时；上下限及 ±1 |
   | 等价类 | P1 | 存在枚举/格式约束时；有效类 + 无效类各一条 |
   | 响应结构验证 | P1 | 必须；验证所有必填响应字段的存在性和类型 |
   | 幂等性 | P2 | 文档说明接口幂等或涉及重复提交时 |

3. **用例编号规则**：
   - Mode A：`API-[MODULE]-[SEQ]`（如 `API-ORDER-001`）
   - Mode B：`API-[MODULE]-R[SEQ]`（R = Regression）

4. **用例排序**：
   - 依赖链：被依赖的接口用例优先（如先 Create 再 Delete）
   - 同接口内：P0 → P1 → P2 → P3

5. **测试点覆盖对照**（仅当 Phase 1 成功加载 test-points.md 时执行）：
   - 遍历 P0/P1 测试点，检查每个测试点是否有对应的接口用例
   - 无对应用例的 P0 测试点 → 补充一条 P0 正向或鉴权用例并标注 `[来源: 测试点 TP-XXX]`
   - 无对应用例的 P1 测试点 → 标注 `[测试点 TP-XXX 未覆盖]`，在输出摘要中列出

5. **数据标记**：
   - 唯一性字段使用 `{时间戳}` 动态标记
   - 跨用例传递的值使用 `{记录为:变量名}` / `{引用:变量名}` 标记

> ⛔ **DO NOT** 进入 Phase 4，直到所有接口的全部维度用例均已生成完毕。

---

### Phase 4: 质量审查

**目标**：读取检查清单，逐条审查，发现问题直接修复。

**步骤**：

1. **读取质量检查清单**：加载 `references/quality-checklist.md`

2. **逐条执行检查**（发现问题立即修复，不跳过）：
   - **§一 覆盖度检查**：每个接口是否有 P0 正向用例；有认证要求的接口是否有鉴权用例
   - **§二 请求规范检查**：必填参数是否全部传入；动态值是否使用标记语法；认证头是否标注
   - **§三 预期结果检查**：是否标注 HTTP 状态码；响应字段验证是否具体（禁用模糊断言）；L1/L2/L3 是否标注
   - **§四 禁用词检查**：预期结果中是否含禁用词，命中则立即改写
   - **§五 依赖与清理检查**：有写操作的用例是否有清理步骤；依赖其他用例的是否声明依赖

3. **记录审查结果**：
   ```
   质量审查完成：共检查 N 条用例，发现并修复 M 处问题
   - 改写模糊断言：X 处
   - 补充清理步骤：X 处
   - 修正参数标记：X 处
   - 其他：X 处
   ```

> ⛔ **DO NOT** 进入 Phase 5，直到所有检查项均已通过或处理完毕。

---

### Phase 5: 输出组装

**目标**：按模板结构组装最终产物，写入文件。

**步骤**：

1. **确定输出路径**：

   | 模式 | 输出文件 |
   |------|---------|
   | Mode A | `test_suites/api_suite_{模块名}.md` |
   | Mode B | `test_suites/api_suite_{模块名}_regression.md` |

2. **组装文档结构**：
   - 文档头部：关联接口文档、生成日期、Base URL、认证方式、模式标记
   - 接口依赖关系图（如有跨用例依赖）
   - 用例主体
   - 统计摘要

3. **写入文件**，向用户展示产物路径和统计摘要

---

### Phase 6: 输出前自检

**目标**：在向用户展示最终产物前，AI 自主逐项验收，确保输出质量达标。

**自检清单**：

```
覆盖度
  ☐ 每个接口至少有 1 条 P0 正向用例
  ☐ 有认证要求的接口均有无 Token / 越权用例
  ☐ 文档定义的错误码均有对应用例
  ☐ 存在数值/长度约束的字段均有边界值用例
  ☐ test-points.md 已加载时：P0 测试点均已对照覆盖（或已在摘要中标注未覆盖项）

用例质量
  ☐ 每条用例标注了 HTTP 状态码预期
  ☐ 响应字段验证具体（字段名 + 预期值/类型），无模糊断言
  ☐ 每条预期结果标注了 L1/L2/L3 级别
  ☐ 动态值使用 {时间戳} 标记，无硬编码
  ☐ 写操作用例包含清理步骤或唯一性标识策略

输出完整性
  ☐ 文件已写入正确路径
  ☐ 文档头部包含 Base URL 和认证方式说明
  ☐ 有跨用例依赖时包含依赖关系图
  ☐ 统计摘要已生成（按接口 + 维度汇总）
```

**自检结论**：
- 全部通过 → 输出产物路径和统计摘要
- 发现未通过项 → 先修复，再输出

> ⛔ **DO NOT** 跳过自检直接输出，即使 Phase 4 质量审查已通过。

---

## 验收标准

### 覆盖度

- [ ] 每个接口至少有 1 条 P0 正向用例
- [ ] 有认证要求的接口均有无 Token / 越权用例
- [ ] 文档中定义的错误码均有对应用例
- [ ] 存在数值/长度约束的字段均有边界值用例

### 用例质量

- [ ] 每条用例标注了 HTTP 状态码预期
- [ ] 响应字段验证具体（字段名 + 预期值/类型），无模糊断言
- [ ] 每条预期结果标注了 L1/L2/L3 级别
- [ ] 动态值使用 `{时间戳}` 标记，无硬编码
- [ ] 写操作用例包含清理步骤或唯一性标识策略

### 输出完整性

- [ ] 文件已写入正确路径
- [ ] 文档头部包含 Base URL 和认证方式说明
- [ ] 有跨用例依赖时包含依赖关系图
- [ ] 统计摘要已生成（按接口 + 维度汇总）
