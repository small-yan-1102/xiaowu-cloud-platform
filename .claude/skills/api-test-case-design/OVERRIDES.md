# api-test-case-design 项目定制

## 新增 1：数据准备策略标注

**HE 原文位置**：Phase 3 → 用例设计 → 数据标记之前
**HE 原文摘要**：无数据准备策略标注规则
**定制为**：新增第 6 项「数据准备策略标注」：
- 需要特定状态数据的用例 → 标注 `[数据准备: API]`
- 有依赖链的用例 → 标注 `[数据准备: 依赖用例]` 并声明依赖编号
- 无需特殊数据 → 无需标注

## 新增 2：执行方式判定（Phase 0 询问 + Phase 1 标签兜底）

**HE 原文位置**：Phase 0 → 问题列表之后（新增第 5 个问题）+ Phase 1 → 验证接口文档之后（新增步骤）
**HE 原文摘要**：无执行方式判定，默认全部 AI 自动执行
**定制为**：

### Phase 0 新增问题（第 5 个，必问）

在 Phase 0 的 4 个问题之后，**必须**追加第 5 个问题：

> **5. 用例适用对象**：生成的用例由谁执行？
>    - 全部 `[AI]`：AI 自动执行（api-test-execution 直接跑）
>    - 全部 `[人工]`：人工执行（测试工程师按菜谱操作）
>    - 混合：我来指定哪些 AI、哪些人工
>    - 按默认：读取 test-points.md 标签自动判定

⛔ **DO NOT** 跳过此问题，即使 test-points.md 已有标签。用户回答是最终标准。

### Phase 1 兜底判定

仅当用户在 Phase 0 回答"按默认"时，读取 test-points.md 标签：

1. **检查 test-points.md**（路径：`review/test-points.md`）：
   - 存在且含可执行性标签 → 按标签分组记录
   - 存在但无标签 → 默认全部视为 `[AI]`
   - 不存在 → 默认全部视为 `[AI]`

### 按标签决定用例写法

   | 标签 | 用例写法 | 排序 |
   |------|---------|------|
   | `[AI]` | 全自动格式：完整 HTTP 请求 + 精确断言 + 清理步骤，步骤写到 api-test-execution 可直接执行 | 优先执行 |
   | `[AI+人工]` | 在需要人工介入的步骤处标注 `⚠️ [人工介入] {介入内容}`（如：等待人工触发 MQ / 手动构造 DB 数据 / 切换登录账号）。其余步骤仍为全自动格式 | 正常排序 |
   | `[人工]` | 人工可读菜谱格式：步骤逐条写清楚（操作对象 + 操作动作 + 输入数据），预期结果逐条明确（观察什么 + 期望值是什么）。与 `[AI]` 的区别仅在于不含 HTTP Method/URL/Headers/Body 等机器执行细节，但步骤和预期结果的颗粒度与 `[AI]` 一致 | 排至末尾 |

3. **用例头部标注执行方式**：每条用例新增 `**执行方式**` 字段：
   ```
   **执行方式**：[执行方式: AI] / [执行方式: AI+人工] / [执行方式: 人工]
   ```

4. **Phase 5 统计摘要**中增加执行方式分布：
   ```
   **执行方式分布**：[AI] N 条 / [AI+人工] N 条 / [人工] N 条
   ```

## 新增 3：API 套件「执行清单」段作为唯一状态记录入口

**HE 原文位置**：Phase 3 生成套件首部 → 执行顺序表结构 / 套件末尾章节
**HE 原文摘要**：执行顺序表只含用例元数据，无执行进度字段；套件末尾为统计摘要和人工校验提示
**定制为**：所有 `api_suite_*.md` **必须**：
1. 执行顺序表**不得**包含「执行记录」列（状态不进表格）
2. 在**执行顺序表之后、首个用例分节之前**插入一段 `## 执行清单（状态记录入口）`，作为**人工 + AI 共享的唯一状态记录位置**

### 设计动机

同 test-case-design 新增 10——VSCode 不支持表格内 checkbox 切换，独立段 Markdown 任务列表可鼠标点击。单源原则避免双轨不同步；完整证据在 `execution/execution_report_*.md`。

### 段落骨架（必填）

```markdown
---

## 执行清单（状态记录入口）

> **操作说明**：
> - **人工**：鼠标点击 `- [ ]` 切换为 `- [x]` 表示**执行通过**；失败/阻塞/跳过**不勾选**，行尾追加 ` · ❌ BUG-{id}` / ` · 🚫 {原因}` / ` · ⏭ {原因}`
> - **AI（test-execution / api-test-execution）**：执行完成后自动勾选并追加 ` · ✅ AI {日期} · [报告](...)` 或 ` · ❌ AI {日期} · [失败详情](...)`
> - **真源定位**：本清单为**进度真源**；完整执行证据（步骤/断言/截图/堆栈）在 `execution/execution_report_*.md`

{按阶段/优先级分组的勾选列表}
```

### 分组规则

- API 套件通常按 `① ② ③ ...` 阶段分组，勾选段按同样阶段分组
- 组内按执行顺序排列

### 行格式

**默认**：
```
- [ ] **{用例编号}** {用例名称}（{优先级}）
```

**AI 执行后**（由 api-test-execution 回写）：
```
- [x] **API-S03-001** 任务互斥：重复导入被拒（P0） · ✅ AI 2026-04-17 · [报告](execution/execution_report_20260417.md#api-s03-001)
- [ ] **API-S03-002** 任务互斥：超时失效（P1） · ❌ AI 2026-04-17 · [失败详情](execution/execution_report_20260417.md#api-s03-002)
```

### 硬性约束

1. **执行顺序表不得含「执行记录」列**——状态唯一入口是本清单段
2. Phase 3 新生成的 API 套件，所有用例行默认 **未勾选** `- [ ]`
3. 不使用 Unicode checkbox（`☐/☑`），仅使用 Markdown 任务列表语法
4. 不嵌入表格内
5. Phase 5 自检硬性检查项：
   ```
   ☐ 执行顺序表最右列不是「执行记录」（已废除）
   ☐ 套件存在 `## 执行清单（状态记录入口）` 段，位于执行顺序表之后
   ☐ 段首含标准操作说明块
   ☐ 每条可执行用例有 `- [ ] **编号** 名称（P?）` 形式的行
   ☐ 勾选行数 = 执行顺序表中非暂缓用例数
   ```

### 与执行回写的衔接

- 本覆盖项约束**生成阶段**的格式
- 执行阶段的回写逻辑由 `api-test-execution` OVERRIDES 覆盖 2 定义
- 回写目标：**本清单段对应行**（勾选 + 行尾追加 AI 标记 + 报告链接）
- 审计证据：`execution/execution_report_*.md` 为**真源**
