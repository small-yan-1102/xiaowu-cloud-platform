# 接口测试用例质量检查清单

> 本文件是 api-test-case-design skill 的 Reviewer 知识库，由 SKILL.md 在 Phase 4 质量审查时加载。
> **替换此文件即可改变审查标准，无需修改 SKILL.md 中的指令逻辑。**

---

## 一、覆盖度检查

- [ ] 每个接口至少有 1 条 P0 正向用例
- [ ] 有认证要求的接口均有无 Token 和越权用例（P0）
- [ ] 接口文档中定义的每个错误码均有对应用例
- [ ] 存在数值/长度约束的字段均有边界值用例（上限、下限、上限+1、下限-1）
- [ ] 存在枚举值的字段均有无效枚举用例
- [ ] 文档说明幂等的接口有重复请求用例

---

## 二、请求规范检查

- [ ] 必填字段是否全部传入（正向用例）
- [ ] 认证 Header 是否标注（需 Token 的接口必须有 Authorization 字段）
- [ ] 唯一性字段是否使用 `{时间戳}` 动态标记（禁止硬编码）
- [ ] 跨用例传递的值是否使用 `{记录为:变量名}` / `{引用:变量名}` 标记
- [ ] Body 是否为合法 JSON 格式（字符串加引号，数值不加）
- [ ] Path 参数是否使用变量引用（如 `/api/order/{引用:order_id}`）

---

## 三、预期结果检查

- [ ] 每条用例是否标注了预期 HTTP 状态码
- [ ] 每条预期结果是否标注了 L1/L2/L3 级别
- [ ] 响应体验证是否具体（字段名 + 预期值），无模糊断言（见§四禁用词表）
- [ ] 写操作（POST/PUT/DELETE）是否验证了数据副作用（数据确实被创建/修改/删除）
- [ ] 错误码用例是否验证了数据未被修改（副作用为空）

---

## 四、预期结果禁用词表

以下描述出现在预期结果中时必须改写：

| 禁用描述 | 问题 | 改写示例 |
|---------|------|---------|
| `请求成功` / `接口正常` | 无法定义"正常" | HTTP 状态码 200，`code` = 0（L1） |
| `返回数据正确` | 不知道"正确"长什么样 | `data.name` = "{测试数据值}"，`data.status` = "ACTIVE"（L1） |
| `响应不为空` | 不知道期望什么字段 | `data.id` 不为空，`data.createdAt` 格式符合 `{格式:yyyy-MM-dd}`（L1） |
| `无错误` / `无异常` | 无法穷举 | HTTP 状态码 200，`code` = 0，`data` 字段存在（L1） |
| `返回错误提示` | 不知道什么提示 | `message` 包含 "参数不能为空"，HTTP 状态码 400（L1） |
| `字段存在` | 缺少类型和值约束 | `data.amount` 为数值类型且 > 0（L1） |

**自检公式**：能否回答"调用方收到【什么状态码 + 什么响应字段 = 什么值】就判定通过"？
- 能明确回答 → 合格
- 回答不出 → 必须改写

---

## 五、依赖与清理检查

- [ ] 依赖其他用例数据的用例是否声明了 `**依赖用例**` 字段
- [ ] 有写操作（创建/修改）的用例是否包含清理步骤
- [ ] 清理步骤是否验证了清理结果（如 GET 返回 404）
- [ ] 依赖链中的清理步骤是否按逆序编写（先删子数据，后删父数据）

---

## 六、接口优先级判定规则

| 优先级 | 判定条件 |
|-------|---------|
| **P0** | 正向 Happy Path；无 Token 鉴权 |
| **P1** | 错误码覆盖；边界值；等价类；响应结构完整性；越权鉴权 |
| **P2** | 幂等性；组合参数；非必填字段边界 |
| **P3** | 极端值（超长字符串、特殊字符）；低频错误码 |
