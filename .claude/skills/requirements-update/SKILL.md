# 需求变更同步技能

> **设计模式**：Pipeline — 环境准备 → 需求对比 → 文档修改 → 关联校验 → 结果输出，严格顺序执行

**⚠️ 核心约束：本流程仅同步更新文档，不涉及代码开发。**

## 输入参数

| 参数 | 必填 | 说明 |
|------|------|------|
| `update_type` | 是 | 修改类型：`user-story` / `quests` / `both` |
| `version` | 是 | 当前需求版本号 |
| `old_document` | 是 | 旧需求文档 |
| `new_document` | 是 | 新需求文档 |

---

## 核心工作流

```
Task Progress:
- [ ] Step 1: 环境准备
- [ ] Step 2: 需求对比分析
- [ ] Step 3: 用户故事修改
- [ ] Step 4: 实施计划修改
- [ ] Step 5: 关联性检查
- [ ] Step 6: 结果输出
```

### Step 1: 环境准备

- 根据版本号自动定位用户故事文件和实施计划文件
- 验证文件是否存在，不存在则提示用户

---

### Step 2: 需求对比分析

- 读取旧需求文档和新需求文档
- 逐条对比分析需求变更点
- 识别新增、修改、删除的需求项
- 标记涉及的关联 Quest 和功能模块

**⚠️ 硬门控：DO NOT proceed to Step 3 until user confirms the change analysis.**

---

### Step 3: 用户故事修改（当 update_type 包含 user-story 时）

**增量修改原则**：
- 只修改新增或变更的 Quest 条目
- 检查关联 Quest 的依赖关系并同步更新
- 保持未变更 Quest 的完整性和一致性

---

### Step 4: 实施计划修改（当 update_type 包含 quests 时）

**精准修改原则**：
- 只修改受影响的实施计划部分
- 检查关联 Quest 是否需要同步调整
- 保持实施计划的可执行性和时序合理性

---

### Step 5: 关联性检查

**交叉验证**：
- 用户故事修改后 → 验证实施计划是否仍能满足需求
- 实施计划修改后 → 验证是否完全覆盖用户故事要求
- 检查各 Quest 间的依赖关系是否保持一致
- 确认变更不会引入新的冲突或遗漏

---

### Step 6: 结果输出

- 生成变更日志，记录具体的修改内容
- 输出更新后的文件路径列表
- 提供修改影响范围说明
- 给出人工复核建议清单

---

## 核心约束

- 严格按照最小范围修改原则，避免不必要的变更
- 只记录客观需求变更信息，不添加主观推测
- 如有任何问题，立即向用户发起提问
- 建议修改前保留备份文件

---

## 变更影响检查清单

执行 Step 2 对比分析时，必须逐项检查：

1. **影响范围识别**：新增/修改/删除的需求项分别影响哪些用户故事和 Quest？
2. **依赖冲突检测**：新需求是否与已有用户故事的验收标准矛盾？
3. **关联文档同步**：以下文档是否需要联动更新？
   - [ ] 实施计划（时序/依赖/排期）
   - [ ] 接口文档（字段/枚举/状态变更）
   - [ ] 测试用例（已有用例是否需要修改或废弃）
4. **数据兼容性**：变更是否影响已有数据结构或存量数据？

## 变更验证标准

变更后文档需通过以下 3 项自检：

1. **变更标记**：每处修改必须有 `[变更]` 标签或 diff 标记，说明变更原因和日期
2. **无孤立引用**：已删除的功能/Quest 是否仍被其他文档引用？引用处是否已同步清理？
3. **版本号递增**：文档版本号是否已更新（如 V4.2 → V4.3）？

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| 文档路径不存在 | 提示用户提供正确路径，或询问是否需要创建新文档 |
| 变更内容与现有文档矛盾 | 标记 `[CONFLICT]`，暂停修改，由用户确认后继续 |
| 无法判断影响范围 | 通过 AskUserQuestion 询问用户确认受影响的模块和文档 |
| 变更涉及已废弃功能 | Warn：标注"该功能已在 V{X} 中废弃"，建议用户确认是否恢复 |
