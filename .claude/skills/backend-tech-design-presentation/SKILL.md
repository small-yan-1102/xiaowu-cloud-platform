# 迭代设计文档 → 方案评审演讲稿

## 模式说明

本 Skill 采用 **Inversion → Pipeline → Generator** 组合模式：
- **Inversion**：先收集文档来源，信息完整前禁止开始生成
- **Pipeline**：五阶段严格顺序执行，每阶段有明确门控
- **Generator**：基于 `assets/presentation-template.md` 模板填充，保证结构一致

## 资源路径

| 资源 | 路径 |
|------|------|
| 演讲稿输出模板 | `./assets/presentation-template.md` |
| 图表规范参考 | `./references/chart-guide.md` |
| 项目演讲稿模板（优先）| `.claude/design/template/迭代设计文档演讲稿.md` |
| 输出目录 | `.claude/design/id/{迭代名称}-演讲稿.md` |

## Phase 0：信息收集（Inversion 门控）

**DO NOT load the template or start any analysis until this phase is complete.**

使用 AskUserQuestion 收集以下信息：

**问题 1**：请选择迭代设计文档来源
- 当前上下文中的文档（已打开或正在讨论）
- 指定文件路径

**如选择指定路径**，追问：请提供完整的文件路径。

**门控条件**：确认已获取到可读取的迭代设计文档后，方可进入 Phase 1。

## Phase 1：文档读取与分析

**DO NOT proceed to Phase 2 until this phase outputs the analysis summary.**

### 1.1 加载资源

按优先级读取演讲稿模板：
1. `.claude/design/template/迭代设计文档演讲稿.md`
2. `./assets/presentation-template.md`（内置模板）

同时加载图表规范：`./references/chart-guide.md`

### 1.2 提取核心信息

从迭代设计文档提取：

| 提取项 | 说明 |
|--------|------|
| 业务背景 | 本次迭代的业务驱动因素 |
| 项目目标 | 本次迭代要达成的目标 |
| 系统现状 | 当前系统状态和能力边界 |
| 变更类型 | 新增功能 / 功能修改 / 功能优化 |
| 涉及角色 | 业务流程中的所有角色 |
| 实体变更 | 新增/修改的实体和聚合根 |
| 待确认点 | 需要在评审会确认的问题 |

### 1.3 判断驱动类型与变更粒度

**驱动类型**：

| 类型 | 判断依据 | 分析角度 |
|------|----------|----------|
| 业务驱动 | 用户体验差、流程繁琐、功能缺失 | 按角色分组展示痛点 |
| 技术驱动 | 性能瓶颈、架构耦合、扩展性差 | 思维导图分类展示 |
| 混合驱动 | 业务需求导致的技术改造 | 分层展示 |

**影响粒度**：

| 粒度 | 判断依据 | 推荐图表 |
|------|----------|----------|
| 服务级 | 涉及多个微服务/系统 | 服务关系图（多 subgraph） |
| 模块级 | 单服务内多个模块 | 模块关系图 |
| 功能级 | 单模块内功能变更 | 功能列表图 |
| 类级 | 实体/类的变更 | classDiagram |

### 1.4 功能类型判断

| 类型 | 判断依据 | 处理方式 |
|------|----------|----------|
| 全新功能 | 无「变更前」描述 | 重点展示功能设计 |
| 功能修改 | 有「变更前/后」对比 | 生成前后对比分析 |

**输出**：打印分析摘要（驱动类型、影响粒度、功能类型、涉及角色列表、待确认点数量），供下一阶段使用。

## Phase 2：演讲稿生成（Generator 填充）

**DO NOT proceed to Phase 3 until all sections are filled.**

按 `assets/presentation-template.md` 的结构**逐节填充**，遵循 `references/chart-guide.md` 的图表规范。

### 核心填充规则

**图优先原则**：能用图不用表格，能用表格不用文字。

#### 一、业务背景与痛点

- **1.1 当前现状**：生成多角色泳道图，展示现有业务/功能流程
- **1.2 问题与痛点**：在现状图中标注痛点环节（`:::problem` 样式），并生成痛点思维导图

> 业务驱动时**必须**按角色分组（subgraph）展示各角色痛点

#### 二、项目目标

- 生成「问题 → 解决方案 → 预期效果」三列对照表
- 确保每个问题都有对应解决方案，每个目标可追溯到具体问题

#### 三、影响范围

- 根据粒度识别结果选择对应图表
- **功能修改时必须**生成前后对比：
  - 业务流程对比（flowchart LR，左：变更前，右：变更后）
  - 技术架构对比表（数据模型 / API接口 / 业务规则 三列）

#### 四、整体设计

- 架构图：使用 subgraph 分层，**标注**本次变更涉及的组件
- 多角色泳道图：sequenceDiagram，体现角色间交互

#### 五、功能详解

每个功能独立一节，包含：
1. 功能说明（一句话）
2. 流程图（异步处理用 `-.->` 虚线箭头 + `:::async` 样式）
3. 变更对比（功能修改时必须）

#### 六、核心设计

- 实体关系图（突出新增实体）
- 新增聚合根专节：说明设计为独立聚合根的 3 个理由
- 核心 API 列表

#### 七、待确认点

| 序号 | 问题描述 | 涉及角色 | 建议方案 | 影响范围 |
|------|----------|----------|----------|----------|

#### 八、风险与依赖

列出技术风险、外部依赖。

## Phase 3：预览确认（检查点）

**DO NOT save the file until user confirms.**

使用 AskUserQuestion 展示已生成演讲稿摘要，询问：

```
演讲稿已生成，请确认：
1. 确认，继续评估演讲时间并保存
2. 补充待确认点
3. 调整图表内容
4. 重新生成
```

若用户选择 2/3/4，执行对应操作后**重新进入本检查点**。

## Phase 4：时间评估与保存

**仅在 Phase 3 用户确认后执行。**

### 4.1 演讲时间评估

| 内容类型 | 预估时间 |
|----------|----------|
| 业务背景介绍 | 3-5 分钟 |
| 整体设计讲解 | 5-8 分钟 |
| 单个功能详解 | 3-5 分钟/功能 |
| 领域模型讲解 | 5-10 分钟 |
| 待确认点讨论 | 2-3 分钟/问题 |
| 风险与依赖 | 3-5 分钟 |
| Q&A 环节 | 10-15 分钟 |

生成时间评估表，附在演讲稿末尾：

```markdown
## 演讲时间评估

| 章节 | 内容量 | 预估时间 |
|------|--------|----------|
| 业务背景 | {X} 个图表 | {X} 分钟 |
| 整体设计 | {X} 个图表 | {X} 分钟 |
| 功能详解 | {X} 个功能 | {X} 分钟 |
| 领域模型 | {X} 个实体 | {X} 分钟 |
| 待确认点 | {X} 个问题 | {X} 分钟 |
| 风险依赖 | - | {X} 分钟 |
| Q&A | - | 15 分钟 |
| **合计** | - | **{总时长} 分钟** |

**建议会议时长**：{向上取整到30分钟的倍数} 分钟
```

**建议标准**：≤30 分钟预约 30 分钟；30-60 分钟预约 60 分钟；60-90 分钟预约 90 分钟或拆分评审。

### 4.2 保存文件

将完整演讲稿（含时间评估）保存至：
`.claude/design/id/{迭代名称}-演讲稿.md`

## 错误处理

| 场景 | 处理方式 |
|------|----------|
| 模板文件不存在 | 使用内置默认结构，继续执行 |
| 迭代设计文档不完整 | 提示用户补充缺失信息，保持在 Phase 0 |
| 无法识别变更类型 | 使用 AskUserQuestion 询问用户确认 |
