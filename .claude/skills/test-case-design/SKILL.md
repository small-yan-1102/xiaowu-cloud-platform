# 测试用例设计技能

> **版本**: v4.0
> **更新日期**: 2026-04-13
> **适用模式**: AI 读取 Markdown 用例 → 直接操作浏览器执行 → 自主判断结果
> **v4.0 变更**: 重构生成流程为「大纲确认 → 逐模块生成」；Phase 2b 新增大纲确认等待；Phase 3 改为逐模块循环并内联质量审查；Phase 4 缩减为跨模块一致性检查


## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 依赖检查与输入验证
- [ ] Phase 2: 输入分析与设计方法应用
- [ ] Phase 2b: 测试场景大纲确认（⏸ 等待用户确认）
- [ ] Phase 3: 逐模块用例生成（含内联质量审查）
- [ ] Phase 4: 跨模块一致性审查
- [ ] Phase 5: 输出组装
- [ ] Phase 6: 输出前自检
```

### Phase 0: 定向收集

**目标**：在任何生成行为开始前，收集必要信息，避免生成后大范围返工。

向用户提出以下问题，**等待用户全部回答后再进行任何操作**：

1. **测试范围**：全量测试所有模块，还是聚焦某个特定模块？
2. **优先级范围**：需要覆盖 P0-P3 全部优先级，还是只需要 P0-P1？（Mode D 固定为 P0，无需询问）
3. **模式确认**（根据模式路由自动判断，有歧义时询问）：
   - Mode A：PRD / 流程图文件路径是什么？
   - Mode B：审阅汇总报告路径是什么？目标模块是哪个？
   - Mode C：变更影响范围是什么？已有用例库路径是什么？生产环境 URL 和测试账号？
   - Mode D：是为新模块从零设计冒烟套件，还是更新/扩充已有冒烟套件？PRD / 流程图文件路径是什么？如已有冒烟套件，路径是什么？
   - Mode E：用例审核报告路径是什么？（默认：`review/case-review-report.md`）需要补充 UI 用例、接口用例，还是两者都补？

> ⛔ **DO NOT** 读取任何文件，**DO NOT** 开始任何分析或生成，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答 → 进入 Phase 1
- 用户表示"按默认来"或全量模式 → 记录为全量模式，进入 Phase 1
- Mode E → 进入 Phase 1 Mode E 分支，读取审核报告，按缺失项生成补充用例

4. **业务约束文档检查**：

   | 输入材料 | 必须/建议 | 缺失时的处理 |
   |---------|---------|------------|
   | PRD 文档 / 流程图 | 必须 | 提示用户提供，无法继续 |
   | 业务约束文档（`assets/business-constraint-template.md`） | **强烈建议** | 缺失时主动提示："建议先填写业务约束文档，这能显著提升用例质量。是否继续仅基于 PRD 生成？" |
   | 参考样例（历史高质量用例） | 建议 | Phase 1 自动检查 `knowledge/case-samples/{模块名}/`，有则加载；无则按团队规范风格生成 |
   | 知识库业务约束（`knowledge/constraints/{模块名}.md`） | 自动 | Phase 1 自动检查并加载，无需用户提供；当前迭代业务约束文档内容优先级高于知识库 |

   > 用户确认继续后，进入 Phase 1。

### Phase 1: 依赖检查与输入验证

**目标**：确认模板和规范文件存在，验证输入材料完整。

**步骤**：

1. **检查用例模板**：
   - 检查 `assets/case-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「用例模板文件缺失」

2. **检查质量检查清单**：
   - 检查 `references/quality-checklist.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「质量检查清单缺失」

3. **验证输入材料**（根据模式）：

   **Mode A**：
   - 扫描用户指定路径或默认路径（`input/prd/*.md`、`docs/PRD-*.md`）的 PRD 文档
   - 未找到 → 🔵 使用 AskUserQuestion 请用户提供文件路径
   - 仍不存在 → **Fatal**：终止执行，提示「PRD 文档缺失」
   - **检查测试点清单**（`review/test-points.md`）：
     - 存在 → 加载，记录为**测试点覆盖基准**；Phase 3 生成时每个 P0 测试点至少对应一条用例
     - 存在且含可执行性标签 → 按 `[AI]` / `[AI+人工]` / `[人工]` 分组记录：
       - `[人工]` 测试点 → 对应用例头部标注 `[执行方式: 人工]`，排至执行顺序表末尾，不进入 AI 自动执行批次
       - `[AI+人工]` 测试点 → 对应用例的人工介入步骤处标注具体介入内容（如"等待人工输入短信验证码"）
       - `[AI]` 测试点 → 正常生成，无需特殊处理
     - 存在但无可执行性标签（旧版本）→ 默认全部视为 `[AI]`，在 Phase 5 输出摘要中注明「test-points.md 无可执行性标签，建议重新执行 test-point-extraction」
     - 不存在 → 跳过，按 PRD 直接生成；在 Phase 5 输出摘要中注明「未使用 test-point-extraction 基准，建议先执行 test-point-extraction」

   - **加载知识库**（自动检查，静默处理）：

     对每个目标模块，依次检查以下路径（模块名即文件名，允许模糊匹配前缀）：

     | 知识文件 | 路径 | 加载后用途 |
     |---------|------|---------|
     | 业务约束 | `knowledge/constraints/{模块名}.md` | 补充历史业务规则；当前迭代业务约束文档内容优先级更高，有冲突以当前文档为准 |
     | 用例样本 | `knowledge/case-samples/{模块名}/` | 风格和覆盖深度参考；不直接复制，仅参照格式和场景颗粒度 |

     - 文件 / 目录存在 → 加载，记录已加载的知识文件列表（供 Phase 5 输出摘要列出）
     - 文件 / 目录不存在 → 静默跳过，不提示用户

   **Mode B**：
   - 检查 `review/tech-review-summary.md`（审阅汇总报告）
   - 未找到 → **Fatal**：终止执行，提示「审阅汇总报告缺失」
   - 读取「§〇 审阅完成度」→ 识别**模块判定**列（`🟢/🟡/🔴`符号，与「审阅状态」`✅/⏳/➖`列不同）：
     - 🟢 可生成E2E用例 → 正常执行
     - 🟡 技术文档不完整 → 跳过该模块，提示用户可使用 `api-test-case-design` 针对该模块设计接口用例
     - 🔴 建议等待 → **Fatal**：提示用户先完成审阅
   - 检查 `input/tech/*.md`（正式技术文档）
   - 未找到 → **Fatal**：终止执行，提示「正式技术文档缺失」
   - **去重扫描**（防止与 Mode A 已有用例重复）：
     - 扫描 `test_suites/suite_*.md` 下已有用例，提取各用例的「所属模块 + 风险类型关键词 + 操作对象」
     - 建立「已覆盖场景集合」，Phase 3 生成每条补充用例前先比对
     - 重复判定：同一模块 + 同一风险类型（如 C-7 并发竞态）+ 同一操作对象 → 视为重复，跳过生成
     - Phase 5 输出摘要中注明「跳过 N 条已覆盖场景」

   **Mode C**：
   - 检查已有用例库（`test_suites/*.md`）和冒烟用例（`suite_smoke.md`）
   - 未找到 → **Fatal**：终止执行，提示「用例库或冒烟用例缺失」

   **Mode D**：
   - 检查 `references/smoke-test-standard.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「冒烟测试标准文件缺失」
   - 从零设计：扫描 PRD 文档路径（同 Mode A）
   - 更新已有套件：读取用户指定的现有冒烟套件文件

   **Mode E**：
   - 读取用例审核报告（默认 `review/case-review-report.md`）
   - 不存在 → **Fatal**：终止执行，提示「用例审核报告缺失，请先执行 test-case-review」
   - 解析报告中的缺失项，按章节分类：
     - §一 覆盖完整性 → P0/P1 覆盖缺失项，优先生成
     - §二 边界值与异常路径缺失 → 补充对应场景用例
     - §四 步骤可执行性问题 → 修正对应用例的步骤/前置条件/预期结果
   - 读取现有用例集（`review/test-cases-ui.md` 或 `review/test-cases-api.md`）
   - 不存在 → **Warn**：无法去重，在输出摘要中注明；仍继续生成补充用例
   - 执行去重扫描（同 Mode B），避免生成已有用例

4. **汇总检查结果**：
   - 有 Fatal 缺失 → 一次性列出所有缺失项，不分多次告知
   - 全部就绪 → 输出就绪清单，进入 Phase 2

> ⛔ **DO NOT** 进入 Phase 2，直到所有必要依赖均已就绪。

### Phase 2: 输入分析与设计方法应用

**目标**：解析输入材料，构建功能矩阵，确定每个功能点的测试设计方法。

**步骤**：

#### Mode A：需求分析

1. **通读 PRD**，提取：
   - 功能模块清单
   - 每个功能点的输入项及约束条件
   - 业务规则和状态流转
   - 跨模块依赖关系

2. **识别跨边界数据流**（参照 `references/quality-checklist.md` §〇「跨边界用例触发规则」）：

   | 数据流类型 | 触发条件 | 使用模板 |
   |-----------|---------|---------|
   | 跨模块 E2E | 模块 A 的操作产生数据，模块 B 消费/展示该数据，且两者均在同一产品内 | 模板六 |
   | 跨系统 | 数据流跨越独立部署的系统边界（不同域名/不同登录态） | 模板二 |
   | 单模块内依赖 | 同一模块内 CRUD 链依赖（Create → Update/Delete） | 模板一 + `**依赖用例**` 字段 |

   输出跨边界数据流清单：
   | 数据流编号 | 源模块/系统 | 目标模块/系统 | 数据流描述 | 类型 |
   |-----------|-----------|-------------|----------|------|
   | FLOW-001 | {模块A} | {模块B} | {A产生XX数据 → B读取展示} | 跨模块E2E / 跨系统 |

   > 无跨边界数据流 → 跳过此步骤，仅生成模板一用例。

3. **对每个功能点标注业务重要性**（参照 `references/quality-checklist.md` §〇「功能重要性判定规则」）：

   | 功能点 | 业务重要性 | 判定依据 | 建议优先级范围 |
   |-------|-----------|---------|--------------|
   | {功能点} | Core / Major / Minor | {判定理由} | P0 / P1-P2 / P2-P3 |

   > 此表作为**功能矩阵的前置输入**，后续用例生成深度和优先级均以此为准。

4. **构建功能矩阵**：对每个功能点确定设计方法：

   | 功能点 | 业务重要性 | 功能类型 | 推荐设计方法 |
   |-------|-----------|---------|------------|
   | {功能点} | Core / Major / Minor | 有边界输入 / 分支流程 / CRUD / 多条件组合 | 等价类 / 场景法 / CRUD全覆盖 / 判定表 |

5. **等价类划分**：将输入划分为有效等价类和无效等价类
6. **边界值分析**：测试点 = 最小值-1 / 最小值 / 最小值+1 / 最大值-1 / 最大值 / 最大值+1
7. **场景法**：识别主路径 → 分支路径 → 循环路径 → 每条路径对应一个或多个用例

#### Mode B：技术文档补充分析

1. **读取审阅汇总报告**：
   - 提取 `§二 风险场景→补充AI测试场景（聚合）`，按目标模块过滤
   - 提取 `§三 风险场景补充的AI测试用例建议（聚合）`，按目标模块过滤

2. **读取技术文档**，提取：
   - 字段约束（精确边界值）
   - API 接口（数据准备策略）
   - 异步等待时间
   - 并发竞态场景

3. **补充场景→用例类型映射**：

   | 审阅报告风险类型 | 用例类型 | 优先级 |
   |---------------|---------|-------|
   | C-7 并发竞态 | 功能测试 | P1 |
   | C-8 异步失败可观测性 | 异常测试 | P2 |
   | C-10 数据依赖完整性 | 功能测试 | P1 |
   | C-11 跨系统数据不一致 | E2E测试 | P1 |

#### Mode C：变更影响分析

1. **读取变更范围**（审阅报告 / 用户描述）
2. **构建影响矩阵**：

   | 影响类型 | 说明 | 必选 |
   |---------|------|------|
   | 直接变更 | 变更功能的所有用例 | 是 |
   | 上游依赖 | 依赖该功能数据的用例 | 是 |
   | 同模块 P0 | 同模块冒烟用例 | 是 |
   | 跨系统关联 | 涉及同一数据流的跨系统用例 | 有则必选 |

#### Mode D：冒烟范围分析

1. **读取冒烟测试标准**：加载 `references/smoke-test-standard.md`

2. **识别核心价值链**：从 PRD 或现有套件中提取所有功能模块，对每个功能点执行选取决策树（参照标准 §二）

3. **构建冒烟候选清单**：

   冒烟套件由两部分组成：

   **① 核心正向流程（Happy Path）**：主体部分，每个模块 3-5 条

   | 功能模块 | 核心流程 | 是否纳入 | 判定理由 |
   |---------|---------|---------|---------|
   | {模块} | {核心链路描述} | 是/否 | {决策树判定理由} |

   **② 关键异常核心（Critical Exception Core）**：每个模块最多 2 条，供开发自测覆盖关键异常场景

   选取规则：
   - 无需复杂数据准备，30 秒内可执行（不超过 3 步操作）
   - 仅限以下两类：
     - **必填字段为空提交** → 系统拒绝并显示错误提示（几乎所有表单类功能必选）
     - **权限拦截** → 无权限角色访问/操作受限功能被正确拒绝（仅当 PRD 有权限控制时）
   - 不包含：边界值、复杂业务规则、并发冲突、需要特定数据状态的场景

   | 功能模块 | 关键异常 | 是否纳入 | 判定理由 |
   |---------|---------|---------|---------|
   | {模块} | {异常类型} | 是/否 | {无权限控制→跳过 / 无表单→跳过} |

   > 无表单且无权限控制的模块（如纯展示页）→ 关键异常核心为空，在清单中标注「无适用异常」

4. **容量检查**：
   - Happy Path 每个模块候选数量 > 5 条 → 按业务重要度裁剪至 3-5 条
   - 关键异常核心每个模块 > 2 条 → 裁剪至 2 条（权限拦截优先于必填为空）
   - 预估单条执行时间 > 2 分钟 → 拆分或降级为 E2E P1
   - 预估套件总时间 > 8 分钟 → 裁剪 Happy Path，关键异常核心不裁剪

   **执行时间估算参考**（每步骤耗时，含等待）：

   | 操作类型 | 参考耗时 |
   |---------|---------|
   | 页面导航 / 点击按钮 | ~15 秒 |
   | 表单填写（单字段） | ~10 秒 |
   | 下拉选择 / 弹窗确认 | ~15 秒 |
   | 等待数据加载（列表/刷新） | ~10 秒 |
   | 断言验证 | ~5 秒 |

   **临界线参考**：6 步以内 ≈ 90 秒（安全）；8 步 ≈ 2 分钟（临界）；超过 8 步应考虑拆分。
   Happy Path 套件总步骤数 ≤ 20 步；加上关键异常核心，全套件 ≤ 28 步（约 8 分钟）。

5. **更新已有套件时**（增量更新模式）：
   - 读取现有冒烟套件文件
   - 对比候选清单，识别：新增项 / 需更新项 / 可删除项
   - 仅生成差异部分，保留未变更用例

对**每个确认的模块**，依次执行 Step A → Step B → Step C：

#### Step A：生成当前模块用例

按 Phase 2 确定的功能矩阵和设计方法，仅针对当前模块生成所有用例。

**读取用例模板**：加载 `assets/case-template.md`，按对应模板格式生成。

**L1/L2/L3 断言级别判定规则**（生成每条预期结果时必须参照）：

| 级别 | 含义 | AI 执行时行为 | 判定依据 |
|------|------|------------|---------|
| **L1 强验证** | 核心业务结果，失败立即终止用例 | 失败 → 终止当前用例，标记 ❌ | 业务意图是否达成（状态变更 / 数据写入 / 权限拦截） |
| **L2 中等验证** | 辅助确认，失败记录但继续执行 | 失败 → 记录告警，继续后续步骤 | 关键字段值 / 界面反馈 / 次要提示 |
| **L3 观察性验证** | 信息收集，不影响用例结论 | 仅记录，不计入通过/失败 | 非核心文案 / 样式 / 顺序 / 响应时间 |

**快速判定规则**：
- 核心业务操作的成功/失败标识（登录成功、创建成功、权限被拒绝）→ **L1**
- 精确数值字段（金额、积分、数量）→ **L1** + `[断言: API]` 或 `[断言: DB-Query]`
- 唯一的成功反馈（Toast 消息是唯一成功标识时）→ **L1**；有其他 L1 验证时 → **L2**
- 列表中的记录出现 / 状态列显示正确 → **L1**
- 页面元素存在性 / 按钮可用状态 → **L2**
- 非核心提示文案 / 页面副标题 / 加载时间 → **L3**
- 冒烟套件（Mode D）L1 占比应在 60-70%，低于此比例时将关键验证点升为 L1

**预期结果准确性规范**（生成每条预期结果时必须参照）：

##### 3a. 动态值写法规则

系统自动生成的值（订单号、ID、时间戳、计算结果）不可写死具体值，按以下方式处理：

| 值类型 | 错误写法 | 正确写法 |
|-------|---------|---------|
| 自动编号/ID | `验证订单号显示"20240115001"` | `验证订单号为 8 位纯数字` |
| 时间戳 | `验证创建时间为"2024-01-15 10:00"` | `验证创建时间在当前时间 ±1 分钟内` |
| 计算结果（金额、积分） | `验证合计金额为"199.00"` | `[断言: API] response.data.total === 输入金额之和`（精确值从接口取，不在用例中写死）|
| 系统生成文案 | `验证提示为"订单ORD20240115001创建成功"` | `验证提示包含"创建成功"` |

> 含动态计算值的字段，必须用 `[断言: API]` 或 `[断言: DB-Query]` 取实际值比对，不在预期结果中写入任何具体数值。

##### 3b. 负向用例预期结果完整性要求

异常/失败场景的预期结果必须包含以下三类验证，缺一不可：

| 验证类型 | 说明 | 级别 |
|---------|------|------|
| **操作被拒绝的反馈** | 错误提示出现、按钮置灰、弹窗拦截等 | L1 |
| **原始状态保持** | 表单数据保留（用户无需重填）、页面未跳转 | L2 |
| **数据层无副作用** | 相关记录未被创建/修改/删除（用 `[断言: API]` 或 `[断言: DB-Query]` 验证）| L1 |

> 若 PRD 未明确原状态保持行为，仍须提炼为 L2 验证并标注「PRD 未明确，执行时以实际表现为准」。

##### 3c. 断言精度与类型匹配规则

| 断言类型 | 精度要求 | 正确示例 | 错误示例 |
|---------|---------|---------|---------|
| `[断言: 视觉]` | 包含匹配，允许模糊 | `页面包含文字"操作成功"` | `页面标题精确为"用户管理 - 系统 v2.1"` |
| `[断言: API]` | 精确字段名 + 精确值或类型 | `response.data.status === "ACTIVE"` | `返回结果中有状态字段` |
| `[断言: DB-Query]` | 完整 SQL + 精确比对规则 + WHERE 条件 | `SELECT status FROM orders WHERE id='{订单ID}' 结果为 'PAID'` | `数据库中订单状态已更新` |

追加规则：
- `[断言: API]` 和 `[断言: DB-Query]` 的预期结果中，比对字段必须写出具体字段名，不允许只描述含义
- `[断言: 视觉]` 不得用于验证精确数值（金额、积分、数量），这类数据必须升级为 `[断言: API]`
- 同一预期结果同时包含视觉和接口验证时，分两行分别标注断言类型

##### 3d. 数据准备策略选择规则

生成每条用例的前置条件时，按以下决策树选择策略标签：

```
需要的测试数据是否可通过 API 调用创建？
├── 是 → [数据准备: API]（首选，速度快、稳定、可验证）
└── 否 → 是否依赖前一条用例的执行结果（CRUD 链中的 Update/Delete）？
         ├── 是 → [数据准备: 依赖用例]（标注依赖的用例编号）
         └── 否 → 是否必须通过 UI 操作才能创建（如需要截图、手动审批触发）？
                  ├── 是 → [数据准备: UI自建]（在前置条件中写出完整 UI 创建步骤）
                  └── 否 → [数据准备: 环境预置]（仅用于无法通过任何自动化手段创建的静态基础数据，如系统配置、角色权限）
```

| 策略 | 适用场景 | 禁止场景 |
|------|---------|---------|
| `[数据准备: API]` | 有接口可创建的业务数据（订单、用户、商品等）| 不可用于需要审批流才能到达目标状态的数据 |
| `[数据准备: 依赖用例]` | CRUD 链中 Update/Delete 依赖 Create 结果 | 不可跨套件依赖（只能依赖同文件内用例） |
| `[数据准备: UI自建]` | 无 API 支持、必须走 UI 流程创建的数据 | 有 API 时不得选此项（速度慢且不稳定） |
| `[数据准备: 环境预置]` | 系统角色、全局配置、基础字典数据 | 不可用于业务数据（执行时可能不存在或被其他用例修改）|

##### 3e. 前置条件可验证特征要求

每条前置条件必须包含 `[验证: ...]` 标签，描述 AI 执行时如何确认数据已就绪。无 `[验证:]` 的前置条件视为不完整：

| 数据准备策略 | 验证标签写法 |
|------------|------------|
| `[数据准备: API]` | `[验证: 调用 GET /orders?id={id} 返回 status=PENDING]` |
| `[数据准备: UI自建]` | `[验证: 在审批列表中可搜索到该申请且状态列显示"待审批"]` |
| `[数据准备: 依赖用例]` | `[验证: 依赖用例 {编号} 执行结果为通过，且 {记录名称} 在列表中可查询到]` |
| `[数据准备: 环境预置]` | `[验证: 导航至{配置页面}，确认{配置项}已存在]` |

> 验证标签描述的是 AI 执行前的检查动作，不是预期结果。验证失败 → 用例标记为"阻塞"，不计入失败统计。

##### 3f. 唯一性字段动态生成规则

| 字段类型 | 禁止写法 | 正确写法 |
|---------|---------|---------|
| 用户名 / 编号 | `"testuser"` | `"testuser_{时间戳后6位}"` 或 `"testuser_AITEST_{随机数}"` |
| 手机号 | `"13800138001"` | `"138{8位随机数}"` |
| 邮箱 | `"test@example.com"` | `"test_{时间戳}@example.com"` |
| 业务名称 | `"测试订单"` | `"测试订单_AITEST_{时间戳}"` |

追加规则：
- 生产环境（Mode C）测试数据必须加 `_AITEST_` 标记，便于事后清理和识别
- 用例依赖该动态值时，在依赖用例的前置条件中通过 `[验证:]` 读取上一步创建的实际值，不重复写死

##### 3g. 测试环境数据清理规则

| 场景 | 清理方式 |
|------|---------|
| API 可删除 | 套件统计摘要后追加「数据清理步骤」：调用 DELETE 接口删除本次创建的测试记录 |
| 仅 UI 可删除 | 在 CRUD 链的最后一条用例末尾追加清理步骤（Delete 用例本身即完成清理）|
| 无法删除（软删除/归档）| 前置条件使用唯一标识区分（`_AITEST_` 标记），并在套件说明中注明「本套件产生的测试数据需人工定期清理」|

> Mode C（生产环境）清理规则更严格，每条写操作用例执行后立即执行回滚步骤，不等到套件末尾。

##### 3h. browser-use 可执行性分类规则

生成每条用例时，根据核心操作类型判定 `[执行方式:]` 标注。若 `test-points.md` 已加载且含明确可执行性标签，以测试点标签为准；否则按本规则判定。

**判定决策树**：

```
核心操作是否完全在浏览器 DOM 内可完成？
├── 是 → 是否包含以下需人工介入的操作？
│         ├── 短信/邮件验证码（读取真实收件箱）
│         ├── 文件上传（OS 文件选择器弹窗）
│         ├── 第三方 OAuth 扫码（微信/钉钉/企微等外部 App）
│         ├── 图形 CAPTCHA 验证码
│         ├── 复杂拖拽排序（多项/精确位置）
│         └── 富文本特殊格式操作（嵌入表格/公式/媒体等）
│         ├── 是 → [执行方式: AI+人工]
│         └── 否 → [执行方式: AI]
└── 否（OS 级别 / 硬件级别交互）
    └── [执行方式: 人工]
```

**分类速查表**：

| 操作类型 | 分类 | 人工介入步骤标注示例 |
|---------|------|-------------------|
| 表单填写（文本/数字/下拉/日期选择器） | `[AI]` | — |

| 弹窗确认/关闭（Modal/Toast/Alert） | `[AI]` | — |
| 分页/排序/筛选/搜索 | `[AI]` | — |

| 简单拖拽（单次、目标元素明确） | `[AI]` | — |
| 富文本基本输入/加粗/斜体 | `[AI]` | — |

**`[AI+人工]` 标注要求**：
- 用例头部标注 `[执行方式: AI+人工]`
- 在人工介入步骤处精确写出：`「等待人工：{具体操作描述，说明操作对象和完成标准}」`
- 人工完成后，AI 继续执行后续步骤和验证

##### 3i. 等待步骤信号规则

每条含"等待"的步骤必须写明**可观测的等待信号**，禁止使用固定时间等待或模糊描述。browser-use 依赖可观测信号判断继续时机，无信号则无法自动推进。

**禁止写法 vs 正确写法**：

| 禁止写法（模糊） | 正确写法（信号明确） |
|---------------|------------------|
| 等待 3 秒 | 等待加载指示器消失 |
| 等待页面加载完成 | 等待页面出现"{模块名}"标题 |
| 等待响应 | 等待成功提示"{提示文字}"出现 |
| 等待数据刷新 | 等待列表中出现"{记录标识}" |
| 等待片刻后继续 | 等待"{按钮名}"按钮变为可点击（非置灰） |
| 等待操作完成 | 等待弹窗关闭 / 等待页面跳转至{目标页特征} |
| 等待提交成功 | 等待跳转至详情页（URL 含 `/detail`）或等待"保存成功"提示出现 |

**可接受的等待信号类型**：

| 信号类型 | 写法示例 |
|---------|---------|
| 加载指示器消失 | `等待加载动画/转圈消失` |
| 元素出现 | `等待"{文字或组件名}"出现在页面` |
| 元素消失 | `等待弹窗关闭` / `等待遮罩层消失` |
| 页面跳转 | `等待页面跳转至{目标页特征，如 URL 含 "/detail"}` |
| 状态变更 | `等待状态标签变更为"{新状态}"` |
| 列表更新 | `等待列表中出现"{记录标识}"` |
| 按钮状态变化 | `等待"{按钮名}"按钮变为可点击（非置灰）` |
| Toast/通知 | `等待"{提示文字}"Toast 消失后继续` |

> 若确实无可观测信号（如纯防抖延迟），在步骤备注中注明原因并标注 `[L3: 环境等待，不影响用例结论]`，此类情况应极少出现。

##### 3j. 用例间变量传递规则（`[输出:]` 注解）

当一条用例执行后产生的值（系统自动生成的 ID、编号、URL 参数等）需要被后续依赖用例引用时，在**预期结果**块末尾追加 `[输出:]` 注解，声明需要捕获的值及提取描述。

**格式**：
```
[输出: 变量名 = 提取描述]
```

**提取描述要求**：
- 明确说明从页面哪个位置提取（元素、URL 参数、Toast 文字等）
- 说明格式特征，帮助 AI 精确定位（如"格式为 #YYYYMMDD-NNN"）
- 一条用例可声明多个 `[输出:]`，每行一个

**示例（上游用例）**：
```markdown
**预期结果**：
- 页面显示"创建成功"，并跳转至详情页 [L1]
- [输出: 订单号 = 详情页标题行中的订单编号（格式 #YYYYMMDD-NNN）]
- [输出: 订单ID = 当前页面 URL 的 order_id 参数值]
```

**示例（下游依赖用例）**：
```markdown
**前置条件**：
- [数据准备: 依赖用例 TC-ORDER-001] [验证: 依赖用例执行结果为通过]

**步骤**：
1. 在订单列表的搜索框输入 {引用:订单号}
2. 点击搜索到的订单，进入详情页（URL 含 order_id={引用:订单ID}）
```

**使用时机判定**：

```
下游用例是否需要精确引用上游用例产生的某个系统生成值？
├── 是（ID / 编号 / 系统生成文本） → 上游用例加 [输出:]，下游用例用 {引用:}
├── 是，但该值是步骤中手动输入的固定值 → 直接写在下游前置条件中，无需 [输出:]
└── 否（仅依赖执行顺序，不依赖具体值）→ 使用 [数据准备: 依赖用例] 即可，不加 [输出:]
```

**命名规则**：
- 变量名使用语义化名称（`订单号` / `项目ID`），不使用通用名（`id` / `value`）
- 全局唯一，不与 `test-data-preparation` 生成的 data-prep-report 中的变量重名（命名冲突时 data-prep-report 优先）

**与其他注解的区别**：

| 注解 | 作用时机 | 作用 |
|------|---------|------|
| `[验证: ...]` | 用例执行**前** | AI 确认前置数据已就绪 |
| `[输出: ...]` | 用例执行**后** | AI 从页面提取值写入运行时变量表，供下游用例使用 |
| `{引用:变量名}` | 用例执行**中** | 替换为运行时变量表中的实际值 |

**Step B 审查要点**：`[数据准备: 依赖用例]` 的下游用例中如有 `{引用:变量名}`，检查对应上游用例是否已声明 `[输出: 变量名 = ...]`；上游用例有 `[输出:]` 声明但下游用例未使用 `{引用:}` 时不报错，属于正常（可选引用）。

**Mode D 专项生成规则**：

- 编号格式：

- 每个预期结果标注 L1/L2/L3，确保 Happy Path 部分 L1 占比 60-70%；关键异常核心断言全部标注 L1（拦截/错误提示是核心业务结果）
- 参照 `references/smoke-test-standard.md` §七 生成**执行顺序**（Happy Path 先于关键异常核心执行）
- 备注字段标注关联的 PRJ 用例编号
- 套件末尾输出「开发自测说明」：标注本套件可作为开发提测前自测依据，关键异常核心用例须逐条执行并截图

**Mode C 环境标记与安全适配**：
- 对每条回归用例逐条判断 ENV 标签（按 `references/quality-checklist.md` §七 ENV 决策树）
- 对 `[ENV: PROD]` 的写操作用例追加回滚步骤
- 新功能线上验证用例使用 `PROD-[MODULE]-[序号]` 编号

**用例可维护性原则（所有模式）**：

| 脆弱写法（禁止） | 可维护写法（要求）|
|----------------|----------------|
| 点击"第三个菜单项" | 点击"用户管理"菜单项 |
| 查找包含"2024-01-15"的行 | 查找标题为"{测试单号}"的记录 |
| 验证标题为"用户管理 - 系统名称 v2.1" | 验证页面包含"用户管理" |
| 等待 3 秒后检查结果 | 等待加载指示器消失后检查 |
| 前置条件：系统中存在名为"测试用户A"的账号 | 前置条件：`[数据准备: API]` 创建测试用户，`[验证: 用户列表中可查询到]` |

核心原则：
- 用**语义角色**描述元素（按钮文字、输入框标签），不用位置序号或 DOM 结构
- 用**业务数据标识**（名称、编号、动态创建的唯一值）定位记录，不写死固定数据名
- 验证**业务意图**（记录已创建、状态已变更），而非与产品文案完全精确匹配
- 前置数据通过 `[数据准备: ...]` 策略动态创建，不依赖环境静态预置
