# 测试用例颗粒度审查指南（Mode F 流程规范）

> **适用场景**：对已有测试用例执行颗粒度审查与优化
> **配套规则文件**：`granularity-standard.md`（v4.0，14 条硬规则 + 10 条软指标）
> **版本**：v1.0
> **创建日期**：2026-04-16

---

## §1 适用场景与触发条件

**触发词**："优化用例颗粒度" / "调整用例粒度" / "用例太粗/太细" / "按颗粒度标准重构" / "颗粒度优化"

**输入**：
- 已有用例文件路径（Markdown 格式的测试用例套件）
- `granularity-standard.md`（自动加载，无需用户指定）

**输出**：
- 审查报告（过粗/过细/合理统计 + 逐条建议）
- 优化后的用例文件（用户确认后执行）
- 变更摘要（合并 N 条 / 拆分 N 条 / 覆盖度变化）

---

## §2 审查流程

### Phase 1：加载与解析

1. 加载 `granularity-standard.md`（硬规则 R1~R14 + 软指标 S1~S10）
2. 加载本文件（合并触发器 M1~M5 + 评分表）
3. 读取待优化用例文件，解析为结构化列表：
   - 每条用例提取：编号、名称、步骤数、预期结果数（验证点数）、前置条件、所属模块
4. 统计用例总数、各优先级分布、各模块分布

### Phase 2：逐条扫描

对每条用例依次执行以下 3 项检查：

#### 检查 A：失败归因测试（S4）

> 假设这条用例失败了，能写出一句话具体的 bug 标题吗？

- 能 → 标记 `✅ 合理`
- 需要写"A 或 B 或 C 可能有问题" → 标记 `⚠️ 过粗`，记录需要拆分的方向

#### 检查 B：硬规则违反扫描（R1~R14）

逐条对照硬规则，检查用例是否：
- 包含多个用户决策但未拆分（违反 R1）
- 混合了对立面场景（违反 R2）
- 同一规则多个维度合在一条（违反 R3）
- 正/反状态切换合在一条（违反 R4）
- 跨系统效果与操作本身合在一条（违反 R5）
- 权限正/反合在一条（违反 R6）
- 多状态条件操作未按状态组拆分（违反 R8）
- 审批流多结果合在一条（违反 R9）
- 只读展示未按卡片拆分（违反 R10）
- 检索项未按查询方式分组（违反 R11）
- 导入校验未按层级拆分（违反 R12）
- 消息通知未按渠道拆分（违反 R13）
- 定时任务未按触发/执行/结果拆分（违反 R14）

任一命中 → 标记 `🔴 过粗-R{N}违反`

#### 检查 C：过细检测（合并触发器 M1~M5）

检查当前用例是否可与相邻用例合并（见 §3 合并触发器）。

命中任一 → 标记 `🔵 过细-M{N}触发`

### Phase 3：输出审查报告 + 用户确认

**报告格式**：

```markdown
# 颗粒度审查报告

## 总体评估

| 维度 | 数值 |
|------|------|
| 用例总数 | {N} |
| 合理 | {N} ({%}) |
| 过粗（需拆分） | {N} ({%}) |
| 过细（需合并） | {N} ({%}) |
| 颗粒度得分 | {N}/100 |

## 过粗用例（需拆分）

| 用例编号 | 用例名称 | 违反规则 | 建议拆分方式 | 预计拆为 N 条 |
|---------|---------|---------|------------|-------------|
| ... | ... | R11 | 按查询方式拆为模糊/精确/日期 3 条 | 3 |

## 过细用例（需合并）

| 用例编号组 | 用例名称 | 合并触发器 | 建议合并为 | 合并后验证点数 |
|-----------|---------|----------|----------|-------------|
| 015+016 | 排序验证 + 分页验证 | M3 | 1 条"通用列表交互" | 6 |

## 优化后预估

| 维度 | 优化前 | 优化后 | 变化 |
|------|--------|--------|------|
| 用例总数 | {N} | {M} | +{拆分数} -{合并数} |
| 功能覆盖点 | {N} | {N} | 不变（守恒） |
```

→ **用户确认**后进入 Phase 4。用户可以：
- 全部接受
- 选择性接受（指定哪些拆分/合并执行，哪些保持不变）
- 拒绝（终止优化）

### Phase 4：执行优化

按用户确认的方案执行：

**拆分操作**：
1. 保留原用例编号作为组标识
2. 拆分后的子用例编号：`{原编号}a`、`{原编号}b`...
3. 每条拆分后的用例保留原前置条件，步骤和预期结果按拆分规则重写
4. 原用例删除，替换为拆分后的多条

**合并操作**：
1. 保留编号较小的用例编号
2. 合并后的用例名称取较大范围的描述
3. 步骤合并：相同步骤保留一份，不同步骤按顺序串联
4. 预期结果合并：所有验证点集中到一条
5. 被合并的用例编号标注"已合并至 {目标编号}"

**输出**：
- 优化后的用例文件（覆盖原文件，优化前自动备份为 `{原文件名}.bak.md`）
- 变更摘要

### Phase 5：覆盖度守恒检查

**对比维度**：

| 对比项 | 检查方法 |
|--------|---------|
| 功能点覆盖 | 提取优化前所有预期结果中的功能点 → 确认优化后全部存在 |
| 模块覆盖 | 优化前的模块列表 = 优化后的模块列表 |
| 优先级分布 | P0 数量不减少（拆分只增不减，合并不影响 P0） |

**输出**：

```markdown
## 覆盖度守恒检查

| 检查项 | 优化前 | 优化后 | 结论 |
|--------|--------|--------|------|
| 功能覆盖点 | {N} | {N} | ✅ 守恒 |
| P0 用例数 | {N} | {M} | ✅ 不减少 |
| 模块数 | {N} | {N} | ✅ 一致 |
| 丢失的验证点 | — | 0 | ✅ 无丢失 |
```

若发现覆盖度不守恒 → 标记 `🔴 覆盖度风险`，列出丢失的功能点，建议补回。

## §3 合并触发器

以下 5 种情况建议**合并**为一条用例：

### M1 步骤相同仅验证点不同

两条用例的操作步骤完全相同，仅最后一步预期结果不同 → 合并为一条，预期结果并列。

```
❌ 改前：
  用例-A: 打开作品详情 → 检查标题显示
  用例-B: 打开作品详情 → 检查副标题显示

✅ 改后：
  用例-AB: 打开作品详情 → 检查标题和副标题展示（R7 同一操作多观察点）
```

### M2 同一弹窗/卡片的不同字段

两条用例分别验证同一 UI 区域（弹窗/卡片/表单区块）的不同字段 → 合并为"该区域字段完整性验证"。

```
❌ 改前：
  用例-A: 基本信息卡片-验证CP名称展示
  用例-B: 基本信息卡片-验证运营方案展示
  用例-C: 基本信息卡片-验证垂类展示

✅ 改后：
  用例-ABC: 基本信息卡片字段完整性（CP名称、运营方案、垂类、别名、入库链接）
```

### M3 同一列表的通用交互

多条用例分别验证同一列表页的排序、分页、空状态等通用交互 → 按 S8 合并。

```
❌ 改前：
  用例-A: 列表默认按创建时间倒序
  用例-B: 列表分页-切换30/50/100条
  用例-C: 列表分页-跳转到指定页

✅ 改后：
  用例-A: 列表默认排序验证（创建时间倒序）       ← 保留独立（排序规则有业务含义）
  用例-BC: 分页器功能验证（行数切换+页码跳转+总数） ← 合并（R7 同一组件）
```

### M4 同查询方式的不同字段

多条用例分别验证同一查询方式（如全模糊）的不同字段 → 按 R11 合并。

```
❌ 改前：
  用例-A: 频道名称模糊查询
  用例-B: 作品名称模糊查询
  用例-C: 分销商名称模糊查询

✅ 改后：
  用例-ABC: 文本模糊查询验证（频道名称 + 作品名称 + 分销商名称）
```

### M5 各仅 1 个验证点且同一页面

两条用例各仅 1 个验证点，位于同一页面，不需要重新导航 → 按 R2 补充裁定 A 合并。

```
❌ 改前：
  用例-A: 特殊运作要求-有数据时显示内容
  用例-B: 特殊运作要求-无数据时显示"无"

✅ 改后：
  用例-AB: 特殊运作要求展示（有数据→显示内容，无数据→显示"无"）
```

## §5 颗粒度评分表

### 评分规则

| 类型 | 单项分值 | 违反扣分 |
|------|---------|---------|
| 硬规则 R1~R14 | 每条 5 分 | 违反 = -5 |
| 软指标 S1~S10 | 每条 2 分 | warn = -1, fail = -2 |
| 合并触发器 M1~M5 | 每条 1 分 | 命中 = -1 |

**满分**：14×5 + 10×2 + 5×1 = **95 分**

### 评级标准

| 得分 | 评级 | 建议 |
|------|------|------|
| 85~95 | 🟢 优秀 | 无需优化 |
| 70~84 | 🟡 良好 | 建议优化过粗用例 |
| 55~69 | 🟠 一般 | 应当优化 |
| < 55 | 🔴 较差 | 必须优化 |
