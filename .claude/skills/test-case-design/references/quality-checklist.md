## 〇、设计前置规则

### 功能重要性判定规则

> Phase 2 分析 PRD 时，对每个功能点先做重要性分级，再决定用例生成深度。**此标识决定用例的覆盖宽度，优先级判定在此基础上进行。**

| 标识 | 判定条件（满足任一即可） | 用例生成策略 |
|------|----------------------|------------|
| **Core（核心）** | ① 缺失则系统无使用价值（核心价值链）<br>② 多个功能模块共同依赖的前置操作<br>③ 涉及资金/不可逆状态变更的操作 | 必须有 P0；全优先级覆盖（P0-P3） |
| **Major（重要）** | ① 主流程的组成部分（Core 功能的直接延伸）<br>② CRUD 中的 Update / Delete 操作<br>③ 关键业务规则验证（权限、状态流转） | 从 P1 开始；覆盖 P1-P2，P3 按需 |
| **Minor（辅助）** | ① 独立的辅助功能，无下游模块依赖<br>② 纯展示类、配置类、个人偏好设置 | 从 P2 开始；资源紧张时可跳过 |

**判定示例**：

| 功能点 | 标识 | 判定理由 |
|-------|------|---------|
| 用户登录 | Core | 所有功能的前置依赖，缺失则系统不可用 |
| 创建订单 | Core | 系统核心价值链入口 |
| 编辑订单 | Major | 主流程组成部分，有 CRUD Update |
| 查询订单列表 | Major | 核心功能的直接延伸 |
| 修改个人头像 | Minor | 辅助功能，无下游依赖 |
| 导出报表 | Minor | 辅助功能，核心业务不依赖此功能 |

> **注意**：同一个功能点内部，不同操作可以有不同标识。例如「认证管理」模块：「提交认证」= Core，「查看认证进度」= Major，「下载认证证书」= Minor。

### 跨边界用例触发规则

> Phase 2 识别跨边界数据流时，按以下规则判断用例类型和模板。

**触发条件判定**：

```
模块 A 的操作产生数据，其他地方也消费/展示该数据？
│
├── 否 → 单模块用例（模板一），CRUD 链依赖用 **依赖用例** 字段声明
│
└── 是
    ├── 两个模块是否在同一产品内（相同域名 / 同一登录态可访问）？
    │   ├── 是 → 跨模块 E2E 用例（模板六），编号 PRJ-E2E-[序号]
    │   └── 否 → 跨系统用例（模板二），需构建系统注册表
    │
    └── 数据同步是否异步（消息队列 / 定时任务）？
        ├── 是 → 在目标模块验证步骤前插入 `等待` 步骤，L2 断言超时记录警告
        └── 否 → 正常同步验证，L1 断言
```

**跨模块 E2E 用例优先级**：

| 源功能重要性 | 目标功能重要性 | 优先级 |
|------------|-------------|-------|
| Core | Core / Major | P1 |
| Core | Minor | P2 |
| Major | Core / Major | P1 |
| Major | Minor | P2 |
| Minor | 任意 | P2 / 可跳过 |

**验收要求**：跨模块 E2E 用例必须同时包含源模块写入断言（`{记录为:}`）和目标模块读取断言（`{引用:}`），缺少任一端的 L1 断言视为不合格。

### 用例优先级判定规则

> 生成用例时，按以下判定标准为每条用例分配优先级。**以功能重要性为输入，判定顺序从上往下：满足 P0 条件则标 P0，否则判断 P1，以此类推。**

### 判定标准

| 优先级 | 判定条件（满足任一即可） | 典型示例 |
|-------|----------------------|---------|
| **P0** | ① 该功能不可用会导致用户**无法完成核心业务目标**（核心链路断裂）<br>② 多个功能模块共同依赖的前置操作（如登录、认证、创建主体记录） | 用户登录、创建订单、提交认证申请 |
| **P1** | ① 核心功能的**完整主路径**覆盖（含分支主路径）<br>② CRUD 四个操作（Create / Read / Update / Delete）的正向流程<br>③ 关键业务规则的正向验证（如权限隔离、状态流转） | 编辑订单信息、查询订单列表、删除草稿 |
| **P2** | ① 各输入项的**无效等价类**（错误格式、超出范围）<br>② 各输入项的**边界值**（最小值/最大值及其±1）<br>③ 异常流程的错误提示验证 | 手机号格式错误、输入超长文本、必填项为空提交 |
| **P3** | ① 多条件**组合场景**（非独立输入项的交叉组合）<br>② 极端边界细节（如 Unicode 字符、特殊空白符）<br>③ 低频使用路径或纯展示类验证 | 同时触发两个校验规则、输入 Emoji 字符 |

### 数量约束

| 优先级 | 每模块数量上限 | 说明 |
|-------|--------------|------|
| P0 | **1-3 条** | 超出时按业务重要度裁剪；Mode D 冒烟套件全部为 P0 |
| P1 | 无上限 | 确保主路径完整，不遗漏 CRUD 操作 |
| P2 | 无上限 | 按输入项数量线性增长，可合并同类异常 |
| P3 | 按需 | 优先级最低，资源紧张时可跳过 |

### P0 与冒烟测试的关系

- **Mode A 中的 P0 用例**：内嵌在完整用例套件中，标记优先级 P0，用于标识核心链路
- **Mode D 冒烟套件（suite_smoke.md）**：独立套件，全部为 P0，额外约束每条 < 2 分钟、套件 < 5 分钟
- 两者并不冲突：Mode A 的 P0 用例可作为 Mode D 冒烟套件的**来源候选**，但冒烟套件需额外经过时间约束筛选

## 一、通用检查项（§8.1）

- [ ] 是否覆盖了正常、异常、边界场景
- [ ] 每个用例是否可独立执行（或显式声明了依赖关系）
- [ ] 前置条件是否明确具体
- [ ] 预期结果是否具体且可判断（非模糊描述）
- [ ] 命名是否符合规范：`PRJ-MODULE-SEQ` 或 `PRJ-MODULE-S-SEQ`（补充）或 `PROD-MODULE-SEQ`（线上）
- [ ] 测试数据是否在用例中明确列出
- [ ] 是否考虑了不同用户角色和权限场景

## 二、AI 执行专项检查项（§8.2）

### 验证与断言

- [ ] **验证层级是否标注**（每条预期结果必须标注 L1/L2/L3）
- [ ] **断言类型是否正确标注**（`[断言: 视觉]` / `[断言: API]` / `[断言: DB-Query]`）
- [ ] **涉及精确数值（金额/积分/计数）是否升级为确定性断言**（`[断言: API]` 或 `[断言: DB-Query]`）
- [ ] **API 断言是否包含必要字段**：接口地址、预期响应、比对字段
- [ ] **DB 断言是否使用只读 SELECT 且有 WHERE + LIMIT**
- [ ] **是否遵循"操作-验证配对"原则**（每个操作后有明确的预期结果）
- [ ] **涉及表单提交是否包含反向验证**（成功后检查无错误提示）
- [ ] **检索类用例是否验证数据一致性**（不仅验证无报错，还验证结果数据符合条件）
- [ ] **检索类用例是否验证数量完整性**（前置条件已知匹配数量时，预期结果应写出确定的总条数）

### 前置条件

- [ ] **前置条件是否标注类型**（`[登录态]` / `[页面定位]` / `[数据存在]` / `[系统配置]` / `[外部依赖]`）
- [ ] **涉及"数据已存在"的前置条件是否标注数据准备策略**（UI自建 / 依赖用例 / 环境预置 / API / 静态数据池）
- [ ] **标注 `[数据准备: API]` 的用例是否引用了技术文档路径和接口地址**
- [ ] **标注 `[数据准备: 静态数据池]` 的用例是否引用了数据池文档和数据键**
- [ ] **涉及"数据已存在"的前置条件是否标注验证特征**（`[验证: ...]` 标签）
- [ ] **搜索/筛选类用例的前置条件是否明确匹配数据的精确数量**（如"3 条含关键词的记录"，而非"存在相关数据"）

### 测试步骤

- [ ] **是否使用标准动作词**（点击/输入/清空/选择/勾选/取消勾选/上传/拖拽/滚动/悬停/导航/等待/刷新/按键）
- [ ] **元素描述是否能唯一定位**（禁止"点击按钮"等无具体文本的描述）
- [ ] **动态内容是否使用标记语法**（`{时间戳}` / `{非空}` / `{格式:PATTERN}` / `{包含:关键词}` 等）
- [ ] **条件步骤是否使用 `[条件]` 前缀**

### 人工介入

- [ ] **人工介入标识是否使用三级分级**（Tier-1 可Mock / Tier-2 必须人工 / Tier-3 专用工具）
- [ ] **Tier-1 是否标注了 Mock 配置**（验证码固定值、万能短信码等）
- [ ] **Tier-2 是否标注了介入原因和介入时机**
- [ ] **Tier-3 是否标注了工具路径和调用参数**

### 跨系统与依赖

- [ ] **跨系统用例是否标注系统标识**（每步标注 `[SYS-X]`，文档头部声明系统注册表）
- [ ] **存在用例依赖时是否显式声明**（`**依赖用例**` 字段，且不超过 2 条依赖）

### 数据清理与可重复执行

- [ ] **创建数据的用例是否包含清理步骤或唯一标识策略**（确保可重复执行）
- [ ] **有依赖链的清理步骤是否按逆序编写**（先删子数据，后删父数据）
- [ ] **唯一性数据字段是否使用 `{时间戳}` 动态标记**（禁止硬编码日期值）

### 线上回归（Mode C）

- [ ] **每条用例是否标注执行环境标签**（`[ENV: TEST]` / `[ENV: PROD]` / `[ENV: BOTH]`）
- [ ] **无 `[ENV: TEST]` 用例被误标为 `[ENV: PROD]`**
- [ ] **G.3.5.3 禁止场景均未标记 `[ENV: PROD]`**（资金操作、不可逆状态变更、系统配置变更等）
- [ ] **所有 `[ENV: PROD]` 写操作用例包含回滚步骤**
- [ ] **线上用例数据使用 `_AITEST_` 标记**

## 四、预期结果禁用词表（附录 D）

以下词汇出现在预期结果中时必须改写：

| 禁用词 | 问题 | 改写示例 |
|--------|------|---------|
| `页面正常显示` | AI 无法定义"正常" | 页面标题显示"XXX"，表单包含"字段A""字段B"两个输入框（L1） |
| `数据正确 / 数据正常` | 不知道"正确"的数据长什么样 | 列表第一行显示公司名称"XX有限公司"，状态列显示"审核中"（L1） |
| `提交成功` | 缺少成功的具体表征 | 页面顶部出现绿色 toast 提示"提交成功"（L1） |
| `无异常 / 无报错` | 无法穷举"无异常"的范围 | 页面无错误弹窗、无红色错误提示、控制台无 Error 级别日志（L1） |
| `加载成功 / 加载完成` | 不知道"完成"的判定条件 | 骨架屏消失，列表显示至少 1 条数据，底部分页器显示总条数（L1） |
| `正确跳转` | 不知道目标页面是什么 | 浏览器地址栏 URL 变为 `/dashboard`，页面标题显示"工作台"（L1） |
| `样式正确 / 显示正常` | 纯主观判断 | 按钮文字为"提交"，按钮可点击（非 disabled 状态）（L1） |
| `功能正常` | 毫无判断依据 | 点击"导出"按钮后，浏览器触发文件下载，文件名包含当前日期（L1） |
| `符合预期 / 与预期一致` | 循环定义 | 输入框中的值为"张三"，与测试数据表中第 1 行"姓名"字段一致（L1） |
| `弹窗弹出 / 弹窗消失` | 缺少弹窗内容描述 | 弹出确认对话框，标题为"确认删除"，内容包含"此操作不可恢复"（L1） |

**自检公式**：能否回答"AI 在页面上看到【什么元素/文本/状态】就判定通过"？
- 能明确回答 → 合格
- 回答不出或需要"看情况" → 必须改写

## 五、前置条件禁用词表（附录 E）

以下写法出现在前置条件中时必须改写：

| 禁用写法 | 问题 | 正确写法 |
|---------|------|---------|
| `已登录系统` | 未指定账号和角色 | 使用管理员账号 admin/Test1234 登录管理端（URL: https://test.example.com/admin） |
| `已有测试数据` | 不知道什么数据、在哪里 | [数据存在][数据准备: UI自建] 系统中已存在一条认证记录：公司名称="测试公司_{时间戳}"，状态="待审核" |
| `进入XX页面` | 未给出导航路径 | 导航到「系统管理 → 团队管理」页面（URL: /admin/team-manage） |
| `环境正常` | AI 无法判断"正常" | 测试环境 https://test.example.com 首页可访问，登录功能正常 |
| `数据库中已有XX记录` | AI 无法操作数据库 | 通过管理端「数据管理→新增」页面创建一条记录 |
| `完成XX操作` | 未说明具体步骤 | 已完成企业认证提交：进入「认证中心」→ 填写公司名称 → 点击「提交」→ 看到"提交成功"提示 |

**前置条件自检公式**：
1. AI 读到这条前置条件后，能否在不问任何人的情况下自主达成？
2. 如果要求"数据已存在"，AI 是否知道：数据长什么样？数据在哪个页面能看到？数据如何创建？
3. 是否标注了 `[验证: ...]` 标签，告诉 AI 执行前如何确认数据仍然就绪？

## 七、ENV 标签安全适配决策树（附录 G.3.5）

为用例标注 `[ENV: PROD]` 前，按以下决策树逐条判断：

```
用例是否包含 CUD（Create/Update/Delete）操作？
│
├── 否（纯查询/展示/导航）
│   └── 标记 [ENV: BOTH]
│
└── 是
    ├── 是否同时满足以下三个条件？
    │   ① 操作可幂等（重复执行结果一致）
    │   ② 用例包含回滚/清理步骤
    │   ③ 使用测试标记数据（如 _AITEST_ 后缀）
    │
    ├── 三个条件全部满足 → 标记 [ENV: PROD]，追加回滚步骤
    └── 任一条件不满足 → 标记 [ENV: TEST]
```

**绝对禁止标记 `[ENV: PROD]` 的场景**：

| 禁止场景 | 原因 |
|---------|------|
| 修改/删除真实用户数据 | 影响线上用户 |
| 资金操作（充值、扣费、退款、转账） | 财务风险 |
| 不可逆状态变更（审核通过、合同签署、订单完结） | 无法回滚 |
| 系统配置变更（开关、权限、策略） | 影响所有用户 |
| 批量数据操作（批量导入、批量删除） | 影响范围不可控 |
