# 人工校验清单

> **用途**：AI 完成用例生成后，测试工程师逐项检查，确保用例质量符合执行标准。  
> **校验人**：测试工程师  
> **校验时机**：AI 输出用例后、用例正式录入/执行前  
> **预计耗时**：每套用例 15-30 分钟

---

## 使用说明

- 逐项检查，在 `[ ]` 中填 `x` 表示通过，填 `-` 表示不适用，填 `!` 表示有问题需修改
- 发现问题时，在对应项下方简要记录问题描述，反馈给 AI 进行二次优化
- 所有 `[ ]` 项必须处理完毕（无遗留 `[ ]`），才可提交用例

---

## 第一关：场景覆盖检查

> 对照业务约束文档，确认 AI 没有遗漏重要场景。

```
[ ] 1. 业务约束文档中的每条核心业务规则，都至少有 1 条用例覆盖
[ ] 2. 每个已知异常场景都有对应的异常用例
[ ] 3. 核心正向流程（Happy Path）有完整的 P0/P1 用例
[ ] 4. 关键输入字段的无效值（空值、格式错误、超长）都有覆盖
[ ] 5. 依赖条件章节中提到的前置数据状态，都在用例前置条件中有体现
```

## 第二关：逻辑正确性检查

> 判断 AI 对业务规则的理解是否准确，有没有"幻觉"生成。

```
[ ] 6. 用例中的字段名称与 PRD / 页面实际名称一致（无自造名称）
[ ] 7. 用例中的操作按钮名称与 PRD / 原型一致
[ ] 8. 预期结果中的提示语/状态文案与 PRD 描述一致（或合理推断）
[ ] 9. 业务流程顺序正确，无颠倒或跳步
[ ] 10. 异常场景的预期结果符合实际业务逻辑（不是 AI 自行假设的错误提示）
[ ] 11. 用例中没有覆盖"明确不测的范围"中列出的场景
```

## 第三关：可执行性检查

> 判断 AI 执行这批用例时能否顺利跑下去，不会卡住。

```
[ ] 12. 每条用例的前置条件都能实际达到（不存在"系统中有100条记录"这类难以满足的条件）
[ ] 13. 测试步骤中的每个操作都是 AI 可完成的动作（点击、输入、上传、等待）
[ ] 14. 测试数据是真实可用的（非占位符如 "xxx" / "测试数据"，且格式符合字段约束）
[ ] 15. 预期结果是可判断的（有明确的 UI 元素、文案、状态可验证，无"系统正常运行"这类模糊描述）
[ ] 16. 有数据依赖的用例已标注"依赖用例"字段，且依赖链没有循环
[ ] 17. 跨模块操作的用例已标注所属系统
```

```
（填写哪条用例、哪个步骤有问题）
```

## 第四关：优先级与数量合理性检查

> 判断用例集的整体结构是否合理，避免执行时资源错配。

```
[ ] 18. P0 用例数量合理（每模块 1-3 条，仅覆盖核心正向路径）
[ ] 19. 没有把异常/边界场景标为 P0（P0 应只走 Happy Path）
[ ] 20. 总用例数量与功能复杂度匹配（简单模块 10 条以内，中等模块 20-40 条）
[ ] 21. 各优先级分布合理（P0 < P1 < P2，P0 过多说明粒度太细）
```

## 第五关：用例文档规范检查

> 快速检查格式规范，确保后续录入和维护不出问题。

```
[ ] 22. 用例编号格式正确（PRJ-[MODULE]-[序号]，无重复编号）
[ ] 23. 每条用例都有"关联需求"字段，且能对应到 PRD 章节
[ ] 24. 冒烟用例使用 SMOKE- 前缀，与 PRJ- 用例区分

```
