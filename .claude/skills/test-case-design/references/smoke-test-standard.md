# 冒烟测试设计标准

> 本文件是 test-case-design skill Mode D 的知识库，由 SKILL.md 在 Mode D Phase 2 分析阶段加载。
> **替换此文件即可改变冒烟测试选取标准和设计规则，无需修改 SKILL.md 指令逻辑。**

---

## 一、冒烟测试定位与核心约束

### 定位

冒烟测试（Smoke Test）是对系统核心功能的**快速健康检查**，目的是在每次部署或构建后，以最短时间验证系统是否处于**基本可用**状态。

| 属性 | 冒烟测试（L5 / P0） | E2E 测试（L4 / P1） |
|------|-------------------|-------------------|
| 目标 | 核心链路是否畅通 | 完整用户流程正确性 |
| 覆盖范围 | 仅 Happy Path | 正常 / 异常 / 边界 |
| 单条用例时长 | **< 2 分钟** | 可达 5-10 分钟 |
| 套件总时长 | **< 5 分钟** | 可达 30-60 分钟 |
| 失败影响 | **阻塞部署** | 记录缺陷，不阻塞 |

### 核心约束（硬性限制）

| 约束项 | 限制值 | 说明 |
|-------|-------|------|
| 单条用例**执行**时间 | **< 2 分钟** | 不含数据准备 |
| 冒烟套件**执行**时间 | **< 5 分钟** | 不含数据准备 |
| 数据准备时间（API 调用） | **< 3 分钟** | 全套件共用一次数据准备 |
| **全流程**时间上限 | **< 8 分钟** | 数据准备 + 执行 + 清理 |
| 每个模块用例数上限 | **3-5 条** | Happy Path + 关键异常核心合计 |
| L1 强验证占比 | **60-70%** | Happy Path 部分 |

### 异步操作处理规则

| 场景 | 处理方式 |
|------|---------|
| 用例包含异步等待（导入、审批、MQ 同步等） | 设置超时上限（默认 60 秒），超时不计为冒烟失败 |
| 超时触发 | 标记为 `⚠️ 环境性能异常`，不阻塞部署，但生成告警通知 |
| 业务逻辑失败（如异步任务 status=失败） | 正常计为冒烟失败，阻塞部署 |

> 区分"业务失败"和"环境超时"：前者说明代码有问题，后者说明环境有问题。冒烟只拦截前者。

## 二、选取标准

### 必须纳入冒烟（Include）

| 类别 | 判定标准 |
|------|---------|
| 核心价值链 | 系统最核心业务流程，缺失则系统无使用价值 |
| 资金/交易操作 | 涉及金额变动、充值、支付等不可逆操作 |
| 认证/状态变更 | 用户身份或系统状态的关键转换点 |
| 跨模块数据流入口 | 模块 A 的操作直接影响模块 B 的数据 |
| 用户准入流程 | 新用户从注册到可正常使用的完整路径 |
| **纯后端核心逻辑** | MQ 消费、定时任务、外部 API 等无 UI 入口但属于核心链路的功能，标注 `[冒烟: API]`，由 api-test-execution 执行 |

### 不应纳入冒烟（Exclude）

| 类别 | 应归入的层级 |
|------|------------|
| 单字段格式校验 | E2E / P1 |
| 边界值测试 | E2E / P2 |
| UI 样式/布局验证 | E2E / P2 |
| 异常/容错场景 | E2E / P2 |
| 权限矩阵验证 | E2E / P1 |
| 需要人工介入的流程 | 标注 Tier-2 人工介入 |
| 性能基准测试 | 独立性能测试 |

### 选取决策树

分两步执行：第一步判定**是否纳入**，第二步判定**执行方式**。

**第一步：是否纳入冒烟？**

```
该功能是否属于系统核心价值链？
│
├─ 否 → 不纳入冒烟
│
└─ 是 → 该功能不可用时，系统是否完全不可使用？
      │
      ├─ 是 → 必须纳入冒烟（P0）
      │
      └─ 否 → 该功能涉及资金/状态不可逆变更？
            │
            ├─ 是 → 纳入冒烟（P0）
            │
            └─ 否 → 该功能是跨模块数据流的入口？
                  │
                  ├─ 是 → 纳入冒烟（P0）
                  │
                  └─ 否 → 降级为 E2E（P1）
```

**第二步：纳入后，确定执行方式**

```
该功能是否有 UI 操作入口？
├─ 是 → browser-use 执行（默认）
└─ 否 → 标注 [冒烟: API]，由 api-test-execution 执行
```

> `[冒烟: API]` 用例不经过 browser-use，而是由 `api-test-execution` 技能通过 HTTP 请求直接执行。编号格式不变（`SMOKE-{MODULE}-{SEQ}`），在用例头部标注 `[冒烟: API]`。

## 三、设计原则

### 原则 1：Happy Path Only

冒烟测试**只走正常流程**，不测试异常分支：

- ✅ 冒烟：登录 → 填写正确表单 → 提交 → 验证成功状态变更
- ❌ 不属于冒烟：输入无效信用代码 → 提交 → 验证报错（→ 归入 E2E P2）

### 原则 2：状态变更优于 UI 细节

| 验证目标 | 冒烟（推荐） | E2E（详细） |
|---------|------------|-----------|
| 认证结果 | 侧边栏状态变为"企业认证已通过" | 状态文字 + 颜色 + 图标 + 详情弹窗 |
| 充值结果 | 积分余额数值变化 | 余额 + 明细记录 + 流水号 + 时间戳 |
| 登录结果 | 跳转首页 + 用户名显示 | URL + 用户名 + 头像 + 菜单权限 |

### 原则 3：步骤可合并

连续的同类操作可合并描述，减少步骤数：

- ✅ 冒烟："填写企业认证表单全部字段（公司名称、信用代码、法人姓名、身份证号、手机号）"
- ❌ 过细："步骤3填写公司名称 → 步骤4验证已填 → 步骤5填写信用代码 → ..."

### 原则 4：容错处理

AI 执行冒烟测试时具备适度容错能力：

| 策略 | 说明 |
|------|------|
| 弹窗自动关闭 | 首次登录的引导弹窗等，遇到则关闭后继续 |
| 页面刷新重试 | 状态未及时同步时刷新页面后重新检查 |
| 等待页面稳定 | 操作后等待加载指示器消失再判断 |

## 四、验证层级分布

| 验证级别 | 在冒烟中的占比 | 说明 |
|---------|--------------|------|
| **L1 强验证** | **60-70%** | 核心业务结果，不通过即用例失败 |
| **L2 软验证** | **20-30%** | 辅助验证，不通过记录警告但不阻断 |
| **L3 观察性** | **≤ 10%** | 仅截图记录，不做判断 |

### 禁止的验证方式

| 禁止方式 | 原因 |
|---------|------|
| 跳过未出现的预期元素 | 导致假通过 |
| 不等待直接判断 | 页面未加载完即判断 |
| 纯 UI 样式验证 | 不属于冒烟范畴 |
| 超过 5 个字段的逐字段校验 | 粒度过细，合并为状态验证 |

## 五、编号规范

| 编号格式 | 适用场景 |
|---------|---------|
| `SMOKE-{MODULE}-{SEQ}` | 模块级冒烟用例 |
| `SMOKE-E2E-{SEQ}` | 跨模块端到端冒烟用例 |

- `MODULE`：模块缩写，与 PRJ 模块前缀保持一致（如 CERT / TEAM / CREDIT）
- `SEQ`：三位数字序号（001, 002...）
- 冒烟用例使用独立 `SMOKE-` 前缀，**不占用** `PRJ-` 编号序列
- 在冒烟用例的**备注**字段标注关联的 `PRJ-` 编号

---

## 六、失败阻塞策略

| 触发条件 | 动作 |
|---------|------|
| 任一冒烟用例 L1 验证失败 | **阻塞部署**，通知相关人员 |
| 全部冒烟用例通过 | 生成报告，继续部署/执行后续 E2E |

失败报告必须包含：失败用例编号、失败步骤、期望 vs 实际、执行环境、截图。

---

## 七、执行顺序

冒烟测试按**业务依赖关系**排序：

```
1. 登录冒烟（前置依赖）
2. 企业认证冒烟（依赖登录）
3. 积分中心冒烟（依赖认证）
4. 团队管理冒烟（依赖认证）
5. 跨模块 E2E 冒烟（依赖以上全部）
```

---

## 八、维护规则

| 触发事件 | 必须执行的动作 |
|---------|--------------|
| 新增业务模块 | 评估核心链路，新增 `SMOKE-{MODULE}-001` |
| 核心流程变更 | 同步更新对应冒烟用例的步骤和预期结果 |
| 页面改版 | 更新用例中涉及的操作描述 |
| **代码层变更**（system-function-analysis 更新知识文档） | 当知识文档的 §校验、§状态、§跨系统 章节有变更时，审查冒烟套件中对应模块的用例是否仍然有效 |

| 审查周期 | 审查内容 |
|---------|---------|
| 每月 | 检查冒烟用例是否与当前业务流程一致 |
| 每季度 | 全面评估冒烟套件：用例增减、执行时间 |
| 版本发布前 | 确认冒烟用例覆盖新版本核心功能 |

## 九、冒烟用例设计检查清单

```
☐ 编号格式为 SMOKE-{MODULE}-{SEQ}（Happy Path）或 SMOKE-{MODULE}-EX-{SEQ}（关键异常核心）或 SMOKE-E2E-{SEQ}
☐ 优先级标记为 P0
☐ Happy Path 仅包含正向流程，无异常分支
☐ 关键异常核心仅限三类：必填字段为空、权限拦截、幂等/重复操作拦截（每模块 ≤ 3 条，每条 ≤ 3 步）
☐ Happy Path 部分 L1 强验证占比 60-70%；关键异常核心断言全为 L1
☐ 单条用例预计执行时间 < 2 分钟
☐ 步骤按阶段合理划分（连续同类步骤已合并）
☐ 容错策略考虑完善（弹窗/刷新/等待）
☐ 异步操作设置超时上限（默认 60 秒），超时标记为环境性能异常（非冒烟失败）
☐ 纯后端核心逻辑已标注 [冒烟: API]，由 api-test-execution 执行
☐ 关联的 PRJ 用例编号已标注在备注中
☐ 套件总用例数每模块 3-5 条
☐ 全流程时间 < 8 分钟（数据准备 < 3 分钟 + 执行 < 5 分钟）
```
