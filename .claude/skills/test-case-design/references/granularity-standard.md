> **适用范围**：Phase 0 选择"人工执行"或"混合"格式时，Phase 3 Step A 生成用例时加载

> - 141 份历史 XMind 用例（2024Q4~2026Q1，覆盖 CRM/AMS/JLB/Fi/运营重构等系统）— 场景频率验证

## 一、硬规则（必须拆分）

以下 14 种情况**必须**拆为独立用例，不可合并：

### R1 弹窗/确认链按用户决策拆分

用户在确认弹窗中有 N 个选择 → 拆成 N+1 条（触发 + 每个选择各一条）

```
示例（CRM EDIT-009/010/011）：
  009 修改后关闭 → 触发未保存提示     ← 触发条件
  010 未保存提示点确认 → 弹窗关闭      ← 决策 A
  011 未保存提示点取消 → 留在弹窗      ← 决策 B
```

**R1 扩展 A：提交→校验→结果链路**

提交操作触发校验后有"通过/失败"两个分支 → 各一条。草稿保存和正式提交是不同操作（校验规则不同）→ 各一条。

```
示例（CRM 延长授权-补充页操作）：
  提交 → 校验通过 → 发起审批                ← 一条
  提交 → 校验失败（必填项未填） → 提示错误    ← 一条
  保存草稿 → 校验通过 → 保存成功             ← 另一条（不同校验规则）
```

**R1 扩展 B：批量操作前置校验**

批量操作有"未勾选→提示"和"已勾选→执行"两个分支 → 各一条。

```
示例（JLB 作品管理-批量分配）：
  未勾选点【批量分配】→ 提示"请至少勾选一条数据"  ← 一条
  已勾选点【批量分配】→ 出现分配弹窗               ← 一条
```

### R2 等价类对立面独立成条

"有 X → 行为 A" 和 "无 X → 行为 B" 分开，即使测同一功能

```
示例（CRM EDIT-009 vs EDIT-022）：
  009 有修改后关闭 → 弹确认提示
  022 无修改直接关闭 → 不弹确认提示

示例（CRM 延长授权-补充草稿）：
  没有延长授权的补充草稿 → 空白表单
  已有延长授权的补充草稿 → 回显已存在草稿内容
```

**R2 补充裁定 A：R2 与 S5 冲突时**

当两个对立分支各仅 1 个验证点且在同一页面（不需要重新导航）→ **允许合并**为一条，但需在用例步骤中明确标注两个场景。

```
示例（AMS DETAIL-015 特殊运作要求）：
  步骤：先打开有数据的作品 → 检查 → 关闭 → 再打开无数据的作品 → 检查
  预期：有数据时显示内容（1 个验证点），无数据时显示"无"（1 个验证点）
  → 合并合理：两个分支各 1 个验证点，拆开太碎，且在同一页面
```

**R2 补充裁定 B：Tab 间数据流转**

操作后数据从 Tab A 消失 + 出现在 Tab B → 属于同一流程的两个观察点，按 R7 **不拆**（一条用例同时验证消失和出现）。但 Tab A 和 Tab B 各自的列表字段完整性/检索项 → 各一条。

```
示例（JLB 视频登记-未登记→已登记）：
  点【登记】→ 未登记 Tab 该条消失 + 已登记 Tab 出现该条  ← 一条（R7 同一操作）
  未登记 Tab 列表字段完整性验证                          ← 另一条
  已登记 Tab 列表字段完整性验证                          ← 另一条
```

### R3 同规则不同维度独立成条

逻辑相同但作用对象不同 → 每个对象一条

```
示例（CRM EDIT-013 vs EDIT-014）：
  013 字幕语种可发/禁发互斥
  014 配音语种可发/禁发互斥

示例（CRM 延长授权-审批通过下游）：
  审批通过 + 模板类型=标准签约 → 发起电子签约
  审批通过 + 模板类型=自定义签约 → 上传签约文件
  审批通过 + 模板类型=纸质签约 → 线下签约流程
```

**R3 补充：导出多场景按维度拆分**

同一导出功能的不同触发条件 → 各一条。上限验证 → 独立一条（边界值）。

```
示例（JLB 作品管理-导出）：
  根据检索条件导出    ← 一条（R3 维度 A）
  根据勾选导出        ← 一条（R3 维度 B）
  未勾选未检索全量导出  ← 一条（R3 维度 C）
  超过 1W 上限        ← 一条（边界值）
```

### R4 正向/反向状态切换独立成条

A→B 和 B→A 是不同场景，预期结果不同

```
示例（CRM EDIT-004 vs EDIT-005）：
  004 可发布 → 不可发布（弹确认，清空配置）
  005 不可发布 → 可发布（展示配置区域，填默认值）
```

### R5 跨系统效果从功能验证中独立

操作本身成功 = 一条，跨系统同步效果 = 另一条

```
示例（CRM EDIT-001 vs EDIT-020）：
  001 编辑并重新交接成功（CRM 内验证）
  020 编辑已交接触发 AMS 同步（跨系统验证）

示例（CRM 延长授权-审批流）：
  提交延长授权 → CRM 状态变为待审核（CRM 内验证）
  审批通过 → 审批结果回传 CRM（跨系统验证）

示例（JLB 作品管理-内容分配）：
  AMS 创建分配单成功（AMS 内验证）
  分配后剧老板作品管理列表出现数据（跨系统验证）
```

### R6 权限正向/反向独立成条

有权限 → 可见/可操作 = 一条；无权限 → 不可见 = 另一条

```
示例（CRM EDIT-017 vs EDIT-018）：
  017 有权限时编辑按钮显示
  018 无权限时编辑按钮隐藏

示例（JLB 视频登记-权限控制）：
  有【导入登记】权限 → 按钮可见可用
  无【导入登记】权限 → 按钮不可见
```

### R7 同一操作结果的多个观察角度 → 不拆

验证点多但关注点单一时，保持一条

```
示例（CRM EDIT-015）：
  切换不可发布再切回 → 8 个验证点（首发时间重置、频道限制重置、字幕清空、配音清空...）
  → 全部是"重置规则"这一件事的不同字段观察 → 不拆

示例（AMS DETAIL-006）：
  基本信息卡片 → 5 个验证点（CP名称、运营方案、入库链接、垂类、别名）
  → 全部是"基本信息字段回显"这一件事 → 不拆

示例（CRM 延长授权-审批表单字段）：
  审批表单展示 → 多个字段（合约编号、频道信息、CP信息...）
  → 全部是"审批表单字段完整性"这一件事 → 不拆

示例（JLB 作品管理-列表字段）：
  运营中 Tab 列表 → 8 个字段（作品名称、垂类、运作套餐、可运作语种、运营人员...）
  → 全部是"列表字段完整性"这一件事 → 不拆
```

### R8 多状态条件操作按「状态组 × 操作」拆分

同一操作在不同状态组合下的出现条件不同时，每个状态组独立成条。但同一状态组下的多个操作可合并验证（R7 适用）。

```
示例（CRM 延长授权-补充记录操作按钮）：
  状态组 A（待提交/审核已撤回/审核不通过/已拒签/已失效）：
    → 编辑、删除按钮可见                    ← 一条（同状态组多操作，R7 合并）
  状态组 B（标准签约+待签约 / 自定义签约+待签约）：
    → 催签、作废按钮可见                    ← 另一条
  状态组 C（已生效 / 待审核）：
    → 编辑、删除、催签、作废均不可见          ← 另一条（反向验证）

⚠️ 不要拆成 N个状态 × M个操作 = N×M 条，那样太碎。
   按"出现条件相同的状态归为一组"拆分。
```

### R9 审批流按「审批结果 × 下游效果」拆分

审批通过/拒绝/撤回各一条（下游效果不同）。同一审批结果下不同分支（如模板类型）按 R3 拆分。

```
示例（CRM 延长授权-审批状态流转）：
  审批通过 + 标准签约 → 发起电子签约        ← 一条
  审批通过 + 自定义签约 → 上传签约文件       ← 一条（R3 不同维度）
  审批通过 + 纸质签约 → 线下签约流程         ← 一条（R3 不同维度）
  审批拒绝 → 状态回传 CRM                  ← 一条
  审批撤回 → 状态回传 CRM                  ← 一条
```

### R10 只读展示模块按 UI 卡片/区域拆分

0 决策的只读展示模块（纯观察），按 UI 卡片或信息区域拆分，每张卡片一条用例。同一卡片的条件显隐按 R2 拆分。

```
示例（AMS 作品详情抽屉）：
  DETAIL-006 基本信息卡片字段展示           ← 一个卡片一条
  DETAIL-007 合约信息卡片展示               ← 一个卡片一条
  DETAIL-009 发布配置卡片（可发布状态）       ← 条件显隐 R2
  DETAIL-010 发布配置卡片（不可发布状态）     ← 条件显隐 R2
  DETAIL-012 语种配置卡片（可发布时展示）     ← 条件显隐 R2
  DETAIL-013 语种配置卡片（不可发布时隐藏）   ← 条件显隐 R2
```

### R11 检索/筛选按查询方式分组

列表页的多个检索项，按**查询方式**（非字段数量）分组，每组一条。同查询方式的多个字段合并为一条（R7 适用）。"交叉查询"和"Tab 切换保留检索条件"各独立一条。

```
示例（JLB 视频登记-未登记 Tab 检索项）：
  字段：频道名称（全模糊）、作品名称（全模糊）
  字段：频道ID（精确）、视频ID（精确）
  字段：发布日期（日期范围选择器）

  拆法：
  文本模糊查询验证（频道名称 + 作品名称）           ← 一条（R7 同方式合并）
  精确匹配查询验证（频道ID + 视频ID）               ← 一条（R7 同方式合并）
  日期范围查询验证（发布日期）                       ← 一条
  多条件交叉查询验证                                ← 一条
  切换 Tab 后检索条件保留验证                        ← 一条

  ⚠️ 不要按 N 个筛选字段拆 N 条，那样太碎。
     不要把所有筛选项合成 1 条，S4 失败归因会模糊。
```

### R12 导入校验按层级拆分

导入功能的校验链路按**层级**拆分，每层一条。不按每个字段的每种错误拆 N×M 条。

```
示例（JLB 作品管理-批量导入）：
  Layer 1 文件级校验：格式(.xls/.xlsx) + 大小(≤5M) + 行数(≤1000)   ← 一条
  Layer 2 行级字段校验：必填(*作品名、*CP名称) + 格式(链接https)      ← 一条
  Layer 3 业务级校验：作品名是否存在重复 + 是否在合约内               ← 一条
  导入成功正向流程                                                  ← 一条

示例（JLB 视频登记-导入登记）：
  Layer 1 文件级校验：格式 + 大小(≤5M) + 行数(≤500)                ← 一条
  Layer 2 行级+业务级校验（若无法区分层级，合并为一条）               ← 一条
  导入成功 → 数据出现在已登记 Tab                                   ← 一条
```

### R13 消息/通知按「渠道 × 触发条件」拆分

同一业务事件通过多渠道通知时，每个渠道独立一条。同一渠道的不同触发条件（如到期天数阈值）按 R3 各一条。用户对通知的响应（同意/拒绝/忽略）按 R1 各一条。

```
示例（CRM 自动续约-续签通知，437 个人类用例节点）：
  渠道拆分：
  短信通知发送成功                          ← 一条
  小程序弹窗通知展示                        ← 一条
  公众号服务号通知推送                       ← 一条
  钉钉消息通知（发给客服）                    ← 一条

  触发条件拆分（同一渠道不同阈值，R3）：
  合约到期前 30 天触发首次通知               ← 一条
  合约到期前 23 天触发二次通知               ← 一条
  合约到期前 16 天触发三次通知               ← 一条

  用户响应拆分（R1 决策分支）：
  CP 在小程序同意续签 → 发起自动续签          ← 一条
  CP 在小程序拒绝续签 → 钉钉通知客服          ← 一条

  ⚠️ 不要按 N 渠道 × M 触发条件 × K 响应 = N×M×K 条。
     渠道验证和触发条件验证可独立执行，响应验证选一个渠道深测即可。
```

### R14 定时任务按「触发 + 执行 + 结果验证」拆分

后台定时任务/调度任务拆三部分，每部分至少一条。手动触发与自动触发各一条。

```
示例（CRM 合约状态定时更新，154 个人类用例节点）：
  ① 触发验证：
  自动触发 — 每日 00:00 执行                    ← 一条
  手动触发 — 管理员手动执行任务                    ← 一条（若支持）

  ② 正常执行结果：
  待生效合约（起始日期 ≤ 今天）→ 状态更新为生效中    ← 一条
  生效中合约（结束日期 < 今天）→ 状态更新为已过期    ← 一条（R3 不同维度）

  ③ 异常/边界场景：
  无满足条件的合约 → 任务正常结束，无数据变更         ← 一条
  任务执行失败 → 标记失败状态，支持重试              ← 一条

示例（JLB 视频登记-定时抓取视频）：
  ① 自动触发（每日 9 点）+ 同步时间更新             ← 一条
  ② 正常执行 → 未登记 Tab 出现新视频               ← 一条
  ③ 单频道手动重新抓取                             ← 一条
```

## 二、软指标（拆到什么程度合适）

### S1 执行时间上界：≤ 5 分钟/条

超过 5 分钟说明步骤太多或关注点太多，应拆分。

### S2 验证点下界：≥ 1 个有意义的验证

"按钮存在"单独成条太碎，除非是权限验证（R6）或状态条件验证（R8）。

### S3 验证点密度：3~8 个/条

| 验证点数 | 评价 | 处理 |
|---------|------|------|
| 1~2 | 偏细 | 考虑与相邻用例合并（同页面同操作的） |
| 3~8 | 合理 | 人能在一次操作流程中自然检查完 |
| 9~15 | 偏粗但可接受 | 仅限 R7 场景（同一操作的多角度观察） |
| >15 | 必须拆分 | 违反了某条硬规则 |

**验证点计数规则**：一个独立的预期结果编号 = 1 个验证点。同一个预期结果中检查多个字段（如"包含 CP 名称、运营方案、入库链接"）仍算 1 个验证点。

### S4 失败归因测试

> 假设这条用例失败了，你报 bug 时标题怎么写？

- 能写出**一句话**具体标题 → 颗粒度合理
- 需要写"A 或 B 或 C 可能有问题" → 太粗，按 R1~R14 拆

### S5 导航成本测试

> 拆出来的两条用例，是否需要重新导航到同一页面？

- 需要重新导航 → 考虑合并（导航成本高于拆分收益）
- 已在同一页面/弹窗中 → 保持独立

### S6 技术风险驱动用例

非业务规则但由前端/后端技术实现风险驱动的用例，独立成条。

```
示例（AMS DETAIL-023）：
  连续打开不同作品详情抽屉 → 验证数据刷新不残留
  → 不属于任何业务规则，但 SPA 组件复用是常见 bug 来源 → 独立成条

常见技术风险场景：
  - SPA 数据残留（组件复用不刷新）
  - 并发操作竞态（同时提交）
  - 大数据量渲染（列表超长）
  - 浏览器兼容性
```

### S7 状态×操作矩阵密度控制

当状态数 × 操作数 > 10 时，按以下策略控制用例数量：

| 策略 | 说明 |
|------|------|
| 归组 | 出现条件相同的状态归为一组（R8），减少组合数 |
| 分级 | 高频状态组合 P0 必测，低频组合 P2 或合并验证 |
| 采样 | 同组内选 1 个代表状态执行，备注覆盖范围 |

### S8 通用列表交互合并

每个列表页都有的通用交互（分页器、排序、空状态），不逐功能拆分：

| 交互项 | 拆法 | 说明 |
|--------|------|------|
| 分页器 | 合为**一条** | 行数切换（30/50/100）+ 页码跳转 + 前往X页 + 共N条 → R7 同一组件多观察点 |
| 默认排序 | **一条** | 验证列表默认排序规则（如"创建时间倒序"） |
| 空状态 | **一条** | 无数据时展示空状态提示 |
| 列表字段完整性 | **一条** | 所有列头 + 字段展示 → R7 合并 |

```
示例（JLB 作品管理-运营中 Tab）：
  列表字段完整性（8 个字段）              ← 一条（R7）
  分页器功能验证（行数/页码/跳转/总数）    ← 一条（R7）
  默认排序（未分配在前 + 交接时间倒序）    ← 一条
  空状态展示                             ← 一条

  ⚠️ 不要为每个列表页重复写分页器用例。
     若多个 Tab 共用同一分页器组件，只测一次 + 备注"其他 Tab 同理"。
```

### S9 统计看板按数据维度拆分

统计/看板/分析类页面，每个独立统计指标或图表一条（R7 同一看板多指标不拆）。筛选条件影响统计值 → 独立一条。统计值点击跳转明细列表 → 独立一条。

```
示例（CRM 数据统计看板，2,898 个人类用例节点）：
  线索转化漏斗图展示正确                         ← 一条（R7 多指标同一图表）
  转化率各阶段数值计算正确                        ← 一条
  全局筛选项切换后统计值联动刷新                    ← 一条
  统计栏数字点击跳转到筛选后列表                    ← 一条

  ⚠️ 不要按"每个筛选条件 × 每个指标"拆 N×M 条。
     筛选联动验证选 1~2 个代表性组合即可。
```

### S10 数据订正验证按影响范围拆分

一次性数据订正/迁移/历史数据处理，每种订正规则一条。订正前后对比验证合为一条（R7 同一操作多字段观察）。回滚/补偿验证独立一条。

```
示例（AMS_V1.2.2 数据订正，206 个人类用例节点）：
  规则1：签约频道/作品的 CP类型和签约主体从 CRM 获取  ← 一条
  规则2：频道/主页 CMS名称"与CMS关联=否"订正为"CMS外" ← 一条
  规则3：频道/主页 创建人有值则订正，没有则显示"-"      ← 一条
  订正后列表数据正确性抽检                           ← 一条（R7 多字段观察）

  ⚠️ 数据订正通常是一次性操作，用例执行后即失效。
     标注"[一次性]"，不纳入回归用例库。
```

## 三、判定公式

```
这条用例中，用户需要做几次"选择/决策"？

├── 0 次（纯观察）
│   ├── 是否为只读展示模块？
│   │   ├── 是 → 按 UI 卡片/区域拆分（R10）
│   │   └── 否 → ✅ 合理（如回显验证、列表展示）
│   └── ✅ 合理
│
├── 1 次 → ✅ 颗粒度合理
│
└── ≥ 2 次 → ⚠️ 检查硬规则
    ├── 是否为"提交→校验→结果"链路？
    │   ├── 是 → 按 R1 扩展 A 拆：校验通过/失败各一条
    │   └── 否 → 继续检查
    ├── 是否为审批流多结果？
    │   ├── 是 → 按 R9 拆：通过/拒绝/撤回各一条
    │   └── 否 → 继续检查
    ├── 是否为"检索→操作→验证结果"的列表场景？
    │   ├── 检索部分 → 按查询方式分组（R11）
    │   ├── 操作部分 → 按 R1/R8 拆
    │   └── 导入部分 → 按校验层级拆（R12）
    ├── 是否为消息/通知多渠道场景？
    │   ├── 是 → 按渠道拆 + 触发条件按 R3 + 用户响应按 R1（R13）
    │   └── 否 → 继续检查
    ├── 是否为定时任务/后台调度？
    │   ├── 是 → 按触发+执行+结果三段拆（R14）
    │   └── 否 → 继续检查
    └── 两次决策是否有因果关系？
        ├── 是（如"先修改再保存"是同一流程） → ✅ 不拆
        └── 否（如"先验证筛选再验证清空"是两件独立事） → 检查 R1~R14
            ├── 命中任一硬规则 → 🔴 必须拆
            └── 未命中 → 用 S4 失败归因测试最终判定
```

## 四、AI 执行格式不适用本标准

本标准仅适用于"人工执行"和"混合"格式。AI 执行格式的颗粒度由以下机制控制：
- `quality-checklist.md` §三 覆盖度检查
- `case-template.md` 模板结构（每条用例一个独立验证场景）
- Phase 3 Step B 内联审查
