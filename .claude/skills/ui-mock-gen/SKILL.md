# 界面 Mock 数据化搭建技能

> **设计模式**：Inversion（收集范围）→ Pipeline（环境准备 → 需求分析 → Mock 设计 → 实现 → 验证），各步骤间设硬门控

## 输入/输出契约

### 输入
| 输入项 | 来源 | 缺失时处理 |
|-------|------|----------|
| PRD 文档 | 用户指定路径 | Fatal：向用户索取 |
| 原型文件 | 用户指定路径 | Fatal：向用户索取 |
| 用户故事文档 | req-gen 生成的文档 | Fatal：向用户索取 |
| 后端接口文档 | 用户指定路径（如有） | Warn：标记接口待确认 |
| 参考页面代码 | 可选 | Warn：跳过代码参考 |

### 输出
| 输出产物 | 存放路径 |
|---------|---------|
| 页面组件代码 | `src/views/{模块名}/` |
| Mock 数据 | `src/views/{模块名}/mock/data.js` |
| Mock 接口层 | `src/views/{模块名}/service.js` |
| 组件样式 | `src/views/{模块名}/components/` |

```

- [ ] Phase 3: Mock 数据设计
- [ ] Phase 4: 界面实现
- [ ] Phase 5: 验证与交付
```

### Phase 1: 环境准备与规则加载

1. **读取项目规则**：加载项目编码规范和组件使用规范
2. **读取最佳实践**：加载 `references/ui-mock-best-practices.md`
3. **读取需求文档**：PRD 文档、原型文件、用户故事文档

**⚠️ 硬门控：DO NOT proceed to Phase 2 until:**
- [ ] 项目规则已加载
- [ ] 最佳实践文档已加载
- [ ] 需求文档已读取

---

### Phase 2: 需求分析与界面拆解

1. **读取用户故事文档**：提取 Quest、接口对接表、组件选型表、验收标准
2. **原型图分析**：布局、组件结构、视觉层次、交互逻辑
3. **PRD 文档分析**：核心模型、功能点、交互流程、业务逻辑
4. **现有代码分析**（关键）：
   - **必须** 检查目标页面/模块是否已存在代码
   - 搜索项目目录下相关页面文件
   - 对比原型图与现有代码，识别差异点

5. **功能点分类**：

   | 类别 | 说明 |
   |------|------|
   | 🆕 新增 | 原代码不存在，需全新开发 |
   | ✏️ 修改 | 原代码存在，需修改/扩展 |
   | ♻️ 复用 | 原代码存在，可直接复用 |
   | 🗑️ 删除 | 原代码存在，需删除 |

6. **接口映射**：
   - 从用户故事文档提取接口对接表
   - 将接口映射到对应功能点
   - 标记接口状态（✅ 已有 / ⚠️ 需 Mock）

7. **不确定点澄清**：遇到不确定的字段、交互、业务规则时，**立即停止并询问用户**

**⚠️ 硬门控：DO NOT proceed to Phase 3 until:**
- [ ] 用户故事文档已完整读取
- [ ] 功能点分类完成
- [ ] 接口映射完成
- [ ] 用户确认分析结果

---

### Phase 3: Mock 数据设计

**⚠️ 核心原则：Mock 数据必须集中管理，便于后续联调时删除**

1. **service.js 文件结构设计**：
   ```javascript
   // src/views/{模块名}/service.js
   
   // Mock 数据（联调时删除）
   const MOCK_XXX = [...];
   
   // Mock 开关（联调时改为 false）
   const USE_MOCK = true;
   
   // 模拟延迟
   const delay = (ms) => new Promise(resolve => setTimeout(resolve, ms));
   
   // 接口函数
   export async function getXxx(params) {
     if (USE_MOCK) {
       await delay(500);
       return { code: 200, data: MOCK_XXX };
     }
     // return request.get('/api/xxx', { params });
   }
   ```

2. **Mock 数据命名规范**：
   - ✅ **必须** 使用 `MOCK_` 前缀 + 大写下划线
   - 示例: `MOCK_USER_LIST`, `MOCK_ORDER_DETAIL`
   - ❌ **禁止** 使用 `data`, `mock`, `list` 等模糊命名

3. **Mock 数据规划**：
   - 目录结构: `src/views/{模块名}/mock/data.js`
   - 列表数据: ≥ 20 条（用于测试分页）
   - 枚举值: 覆盖所有可能状态
   - 边界场景: 空数据、超长文本、特殊字符

4. **接口层规范**：
   - ✅ **必须** 使用 `service.js` 统一封装接口
   - ✅ **必须** 提供 `USE_MOCK` 开关
   - ✅ **必须** 模拟网络延迟 (setTimeout)
   - ❌ **禁止** 在组件中直接写 Mock 逻辑
   - ❌ **禁止** 分散 Mock 数据到各个组件

5. **联调切换方案**：
   - Step 1: 修改 `const USE_MOCK = false`
   - Step 2: 取消注释真实接口代码
   - Step 3: 删除 Mock 数据和 delay 函数
   - Step 4: 删除 `mock/` 目录（可选）

6. **字段命名对齐**（关键）：
   - **必须** 从用户故事文档的"接口对接表"中提取字段名
   - **必须** 与后端接口文档中的字段名**完全一致**
   - **禁止** 自行发明字段名或使用不同命名风格
   - 如接口文档未明确，**必须向用户确认**，禁止自行推断
   
   示例：
   ```
   后端接口返回: { name: "张三", deptId: 1 }
   ✅ Mock 数据: { name: "张三", deptId: 1 }
   ❌ Mock 数据: { ClanmeName: "张三", departmentId: 1 }
   ```

**📋 计划输出点**（`--plan-only` 模式在此结束）

**⚠️ 硬门控：DO NOT proceed to Phase 4 until user confirms the mock design.**

### Phase 4: 界面实现

**⚠️ 前提条件：检查是否有对应的用户故事文档**

1. **组件选择（关键原则）**：
   - **必须优先使用项目已封装公共组件**，禁止直接使用底层 UI 库原生组件
   - 实现前先搜索 `src/components/` 目录确认是否有现成组件
   - 不确定时搜索项目内类似功能实现

2. **页面结构搭建**：按原型图创建文件结构和基本布局
3. **组件开发**：按原型实现各功能组件，使用 mock 数据填充
4. **表单校验**：使用项目标准校验方案，提交按钮必须始终可点击（通过校验提示错误而非禁用）
5. **样式实现**：严格对照原型图，颜色值必须使用原型图定义

6. **变量命名规范**：
   - **Mock 数据**: `MOCK_XXX_XXX` (前缀 + 大写下划线)
   - **接口函数**: 动词开头 + 驼峰 (如 `getUserList`, `createOrder`)
   - **组件变量**: 驼峰命名 (如 `userList`, `loading`, `formData`)
   - **组件名**: PascalCase (如 `UserTable`, `OrderForm`)
   - **文件名**: kebab-case (如 `user-list.vue`, `order-form.vue`)

**⚠️ 硬门控：DO NOT proceed to Phase 5 until:**
- [ ] 所有功能点已实现
- [ ] Mock 数据已集成到 service.js
- [ ] 变量命名符合规范
- [ ] 用户确认代码实现完成

---

### Phase 5: 验证与交付

**⚠️ 硬门控：DO NOT deliver until:**
- [ ] 所有原型图功能已实现
- [ ] 空状态/错误状态/加载状态已处理
- [ ] 使用项目封装组件而非原生组件
- [ ] 搜索表单使用项目标准搜索组件
- [ ] 校验使用项目标准校验方案
- [ ] 提交按钮不禁用
- [ ] 无框架不支持的语法
- [ ] 无重复代码
- [ ] 变量命名符合规范
- [ ] 与原型图视觉一致
- [ ] 颜色/间距/字体符合规范
- [ ] Mock 数据字段名与后端接口一致

## 核心约束

1. **Mock 数据集中管理**：所有 Mock 数据必须存放在 `mock/` 目录或 `service.js` 中，禁止分散在组件中
2. **Mock 数据命名规范**：必须使用 `MOCK_` 前缀 + 大写下划线 (如 `MOCK_USER_LIST`)
3. **接口层统一封装**：必须使用 `service.js` 统一管理接口，提供 `USE_MOCK` 开关
4. **严格遵循原型**：界面布局、样式、交互必须严格遵循原型图
5. **组件优先复用**：先搜索项目已有组件，再考虑新建
6. **变量命名规范**：
   - Mock 数据: `MOCK_XXX_XXX`
   - 接口函数: 动词开头 + 驼峰
   - 组件变量: 驼峰命名
   - 组件名: PascalCase
   - 文件名: kebab-case
7. **及时澄清疑问**：不确定的地方立即停止并询问用户
8. **联调友好**：Mock 数据必须便于后续删除，不影响真实接口对接
9. **字段名一致性**：Mock 数据字段名必须与后端接口文档完全一致，禁止自行修改字段名，避免联调时字段不匹配
