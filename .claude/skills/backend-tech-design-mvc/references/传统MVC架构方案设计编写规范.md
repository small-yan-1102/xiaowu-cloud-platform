# 传统 MVC 架构方案设计编写规范

本规范用于指导和检查传统 MVC（Controller-Service-DAO）三层架构下的技术设计文档编写质量。所有设计文档在完成后必须通过本规范的全部检查项。

> 本规范适用于多文档架构体系下的所有文档，包括：
> - `01-系统架构总纲.md`
> - `02-{业务域}-详细设计.md`（每个业务域一份）
> - `03-基础设施与公共能力设计.md`
> - `04-数据库设计总纲.md`
> - `05-非功能性设计.md`
> - `{版本号}-{迭代名称}-迭代变更总纲.md`（增量迭代时使用）
> - `API接口设计文档模版.md`
>
> 文档体系索引见 [00-系统方案文档索引.md](./00-系统方案文档索引.md)

---

## 零、多文档架构编写原则

### 0.1 文档间引用必须使用相对链接

**检查标准**：跨文档引用时必须使用 Markdown 相对链接，不允许使用绝对路径或纯文字引用。

**合格示例**：
```
详见 [03-基础设施与公共能力设计.md](./03-基础设施与公共能力设计.md) 第二章 缓存架构
```

**不合格示例**：
```
详见基础设施文档的缓存部分
```

### 0.2 业务域文档必须自包含

**检查标准**：每个 `02-{业务域}-详细设计.md` 文档必须能独立阅读理解该域的设计。域内的功能模块、状态流转、数据字典、表结构必须在该文档内完整描述，不允许拆散到其他域文档中。对公共能力（认证、缓存、MQ 等）的引用通过链接指向 `03-基础设施` 文档。

### 0.3 避免信息重复

**检查标准**：同一设计信息只在一处详细描述，其他文档通过引用关联。

| 信息类型 | 详细描述位置 | 其他文档引用方式 |
| --- | --- | --- |
| 业务域接口逻辑 | `02-{域}-详细设计.md` | 架构总纲中只列域概述 |
| 认证授权架构 | `03-基础设施.md` | 域文档中引用链接 |
| 表结构详情 | `02-{域}-详细设计.md` | 数据库总纲中列表格索引 |
| 多数据源路由 | `01-架构总纲.md` / `04-数据库总纲.md` | 域文档标注所属数据源 |

### 0.4 全量描述与增量迭代使用同一套模版

**检查标准**：模版中标注了 `增量迭代时填写` 的章节，在全量现状描述时可不填写（如迭代背景、估分、变更SQL等）。标注 `条件必选` 的章节，只在满足条件时填写。未标注的章节为必填项。

## 一、通用编写原则

### 1.1 业务逻辑必须按步骤编号展开

**检查标准**：每个功能模块接口的业务逻辑部分，必须使用有序编号（1. 2. 3. ...）逐步描述执行流程，不允许使用段落式自然语言描述。

**合格示例**：
```
1. 校验入参 orderId 不为空
2. 调用 OrderMapper.selectById(orderId) 查询订单记录
3. 判断订单是否存在
   1. 不存在：抛异常："订单不存在，orderId: {orderId}"
4. 判断订单状态
   1. 是 CANCELLED：抛异常："订单已取消，无法操作"
   2. 否：继续
5. 更新订单状态为 CONFIRMED
6. 调用 OrderMapper.updateById(order)
7. 返回结果
```

**不合格示例**：
```
先校验参数，然后查询订单，如果订单状态不对就报错，否则更新状态并保存。
```

### 1.2 条件分支必须列出所有分支路径及对应处理

**检查标准**：业务逻辑中出现"判断"、"如果"、"是否"等条件分支时，必须使用缩进列表明确列出所有分支路径（包括正常路径和异常路径），不允许只写正常路径而省略异常路径。

**合格示例**：
```
3. 判断用户是否存在
   1. 是：继续执行步骤 4
   2. 否：抛异常："用户不存在，userId: {userId}"
```

**不合格示例**：
```
3. 查询用户，如果存在则继续处理
```

### 1.3 涉及数据库/外部服务的步骤必须标明参数来源

**检查标准**：当业务逻辑步骤中涉及数据库查询（Mapper）、外部服务调用（Manager）、缓存读写等操作时，必须在步骤描述中标明调用的参数来源（来自入参、来自上一步的返回值、来自配置等），不允许只写"查询订单"而不说明用什么条件查询。

**合格示例**：
```
2. 根据入参 orderId 调用 OrderMapper.selectById(orderId) 查询订单记录
```

**不合格示例**：
```
2. 查询订单信息
```

### 1.4 异常处理必须写明具体异常类型和错误提示信息

**检查标准**：所有异常分支必须写明：(1) 具体的异常类型（如 `BusinessException`、`IllegalArgumentException`）；(2) 明确的错误提示信息（用引号包裹）。不允许只写"报错"或"抛异常"而不写具体内容。

**合格示例**（三元格式：异常类型 + 错误消息 + HTTP 状态码）：
```
抛出 BusinessException(code=STOCK_INSUFFICIENT, msg="库存不足，当前库存: {currentStock}，请求数量: {requestQty}") → HTTP 400
```

**不合格示例**：
```
如果库存不足就报错
抛出 BusinessException，信息："库存不足"  ← 缺少 code 和 HTTP 状态码
```

## 二、MVC 架构专项原则

### 2.1 Controller 层只做请求转发和响应封装

**检查标准**：功能模块设计（二 章节）中描述的业务逻辑步骤不应出现在 Controller 层。Controller 的职责仅限于：接收 HTTP 请求、参数校验（通过注解）、调用 Service 方法、封装响应格式。如果设计文档中的业务逻辑涉及 Controller 层的处理，说明分层不当。

**合格描述**：
- 业务逻辑步骤全部在 Service 层的方法中描述
- Controller 仅负责接收入参 DTO、调用 `OrderService.createOrder(dto)`、返回 VO

**不合格描述**：
- 业务逻辑步骤中出现"Controller 层进行业务判断"
- Controller 中包含数据转换、状态校验等业务逻辑

### 2.2 Service 层包含完整的业务逻辑编排和事务控制

**检查标准**：功能模块的业务逻辑步骤必须完整地在 Service 层描述，包括：参数校验、业务规则判断、数据持久化调用、外部服务调用（通过 Manager）。事务控制（`@Transactional`）必须在 Service 层标注，并明确事务边界。

**合格描述**：
- 业务逻辑步骤中明确了哪些步骤在 `@Transactional` 事务内
- Service 方法的入参和返回值在设计中完整列出
- Service 通过 Mapper 访问数据库，通过 Manager 访问外部依赖

**不合格描述**：
- 业务逻辑步骤不完整，省略了中间的校验或判断步骤
- 未标注事务边界
- Service 直接调用 FeignClient 而非通过 Manager

### 2.3 外部依赖访问必须通过 Manager 层隔离

**检查标准**：防腐层设计（九 章节）中，所有外部依赖访问（Feign 服务调用、Redis 读写、ES 查询、第三方 HTTP API 等）必须通过 Manager 接口 + ManagerImpl 实现类封装。Service 层不允许直接注入或调用 FeignClient、RedisTemplate、RestTemplate 等外部客户端。

**合格描述**：
- 有独立的 Manager 接口定义
- ManagerImpl 中封装了外部调用的参数转换、异常处理、重试逻辑
- Service 层通过注入 Manager 接口来访问外部依赖
- 防腐层设计中每个方法包含完整的执行逻辑（5 步：参数校验 → 模型转换 → 调用外部 → 结果判断 → 返回）

**不合格描述**：
- Service 中直接注入了 `@Autowired FeignClient`
- 没有 Manager 层，外部调用散落在 Service 中
- Manager 方法只是简单透传，没有参数转换和异常处理

### 2.4 DAO/Mapper 层只负责数据持久化

**检查标准**：数据库设计（七 章节）中定义的表结构和索引应与 Mapper 层的职责一致。Mapper/DAO 层只负责单表或简单关联的 CRUD 操作和 SQL 映射，不允许在 Mapper 层包含业务逻辑判断。复杂的跨表查询如需在 SQL 层完成，应在索引设计中有对应的优化说明。

**合格描述**：
- Mapper 方法与数据库表一一对应
- 复杂查询通过 MyBatis XML 的 SQL 映射实现，有对应的索引设计说明
- Mapper 不包含任何业务判断逻辑

**不合格描述**：
- Mapper 中包含业务状态判断的 SQL（应在 Service 层做判断后再调用 Mapper）
- 有复杂的多表关联查询但索引设计部分没有对应的优化说明

### 2.5 事务型接口和复杂查询必须提供业务流程图

**检查标准**：功能模块详细设计（二 章节）中，以下两类接口必须在"业务流程图"部分提供 Mermaid flowchart 图：(1) 事务型接口（涉及数据写入的 POST/PUT/DELETE 操作）；(2) 复杂查询（涉及多个条件分支或多表关联的查询）。如果确实不涉及，须在"业务流程图"部分标注"无"。

**合格描述**：
- 每个 POST/PUT/DELETE 接口都有对应的 Mermaid flowchart 流程图
- 流程图与业务逻辑步骤编号一一对应
- 不涉及的接口明确标注"业务流程图：无"

**不合格描述**：
- 事务型接口缺少流程图，"业务流程图"部分留空
- 流程图与业务逻辑步骤不对应

### 2.6 入参和返回值必须完整列出字段说明

**检查标准**：功能模块详细设计（二 章节）中，每个接口的"入参"和"返回"部分必须使用 JSON 格式完整列出所有字段，每个字段必须包含：字段名、类型、注释说明。不允许省略字段或只列出部分字段。

**合格描述**：
```json
{
  "orderId": "Long // 订单ID",
  "productName": "String // 商品名称",
  "quantity": "Integer // 数量",
  "price": "BigDecimal // 单价"
}
```

**不合格描述**：
```json
{
  "orderId": "订单ID",
  "其他字段": "略"
}
```

### 2.7 实现检查清单必须覆盖所有代码变更层

**检查标准**：功能模块详细设计（二 章节）中，每个接口末尾的「实现检查清单」必须完整覆盖该接口涉及的所有代码变更层（Controller / Service / Mapper / DTO / VO / Entity / SQL / 事务 / 并发控制 / 幂等 / MQ / 缓存）。不涉及的层标注「无」，不允许省略。其中「事务」行必须声明步骤范围（事务内哪些步骤 / 事务外哪些步骤），「并发控制」和「幂等」行必须给出具体策略（见规范 2.9）。

**合格描述**：
```
- [ ] Controller: `POST /api/v1/order/create` → `OrderController.create()`
- [ ] Service: `IOrderService.createOrder(OrderCreateDTO)` → 返回 `Long`
- [ ] Mapper: `OrderMapper.insert(Order)` → MyBatis Plus 自带
- [ ] DTO: `OrderCreateDTO` (新增类)
- [ ] VO: 无
- [ ] Entity: `Order` 无变更
- [ ] SQL: 无
- [ ] 事务: `@Transactional` 标注在 `createOrder()`；事务内：步骤 1~4，事务外：步骤 5（发MQ）
- [ ] 并发控制: 乐观锁: `version` 字段 CAS更新
- [ ] 幂等: 唯一索引: `uk_user_product` (user_id, product_id)
- [ ] MQ: 发送 Topic=`ORDER_TOPIC` Tag=`order-created`
- [ ] 缓存: 无
```

**不合格描述**：
```
- [ ] Controller: OrderController
- [ ] Service: OrderService
（缺少 Mapper/DTO/VO/Entity/SQL/事务/并发控制/幂等/MQ/缓存 各层）
- [ ] 事务: 有事务  ← 缺少步骤范围声明
```

### 2.9 并发控制与幂等策略必须在实现检查清单中显式声明

**检查标准**：实现检查清单中「并发控制」和「幂等」两行不允许省略，不涉及时必须标注「无」。涉及的接口必须给出具体策略，不允许只写「有」而不描述策略：
- **并发控制**：乐观锁需标注字段名（如 `version`）；分布式锁需标注锁 key 格式（可含占位符）和 TTL
- **幂等**：唯一索引需标注索引名和约束字段；Redis 防重需标注 key 格式和 TTL；幂等 Token 需说明 Token 的来源和存储位置

**合格描述**：
```
- [ ] 并发控制: 乐观锁: `version` 字段 CAS更新（MyBatisPlus 自带）
- [ ] 幂等: 唯一索引: `uk_order_no` (order_no)

- [ ] 并发控制: 分布式锁: key=`lock:order:submit:{userId}` TTL=10s
- [ ] 幂等: Redis防重: key=`idempotent:order:{clientToken}` TTL=60s
```

**不合格描述**：
```
- [ ] 并发控制: 有         ← 没有具体策略
- [ ] 并发控制: （缺少此行）← 不允许省略
```

### 2.8 数据模型定义必须包含完整的类级规格

**检查标准**：业务域文档（八 章节「数据模型定义」）中，每个 DTO/VO/Request/Query 类必须包含：类名、包路径、模型类型、用途说明、引用接口编号。字段表必须包含：字段名、Java 类型、必填标识、校验注解、字段说明。不允许只在接口的入参/返回 JSON 中描述而缺少类级聚合定义。

**合格描述**：

| 项目 | 内容 |
| --- | --- |
| **类名** | `OrderCreateDTO` |
| **包路径** | `domain.dto` |
| **模型类型** | DTO |
| **用途** | 创建订单请求体 |
| **引用接口** | 2.1.3, 2.1.5 |

| 字段名 | Java 类型 | 必填 | 校验注解 | 说明 |
| --- | --- | --- | --- | --- |
| `productName` | `String` | Y | `@NotBlank @Size(max=100)` | 商品名称 |
| `quantity` | `Integer` | Y | `@NotNull @Min(1)` | 数量 |

**不合格描述**：
- 只在各接口的入参 JSON 中散落描述，没有聚合的类定义
- 缺少校验注解列
- 缺少引用接口编号（无法追溯哪些接口使用了这个 DTO）

## 三、多文档架构专项原则

### 3.1 架构总纲必须覆盖业务域全景

**检查标准**：`01-系统架构总纲.md` 中的「业务域全景」章节必须列出所有业务域，包括：域名称、域类型（核心域/支撑域/通用域）、核心职责、域间协作关系。不允许遗漏业务域或缺少域间关系描述。

**合格描述**：
- 业务域清单表格完整，每个域都有对应的详细设计文档链接
- 域间协作关系表说明了调用方向和协作方式（同步调用/MQ异步/事件通知）

**不合格描述**：
- 只列出了部分业务域，遗漏了支撑域或通用域
- 没有域间协作关系描述

### 3.2 外部集成必须区分集成方式并描述容错策略

**检查标准**：`01-系统架构总纲.md` 中的「外部系统集成拓扑」章节必须按同步/异步/SDK 三类分别列出所有外部集成，每个集成必须包含：通信协议、认证方式、超时配置、降级策略。

**合格描述**：
- 同步集成列出超时和降级策略
- 异步集成列出 Topic/Tag 和幂等策略
- SDK 集成列出版本和关键配置

**不合格描述**：
- 所有外部系统混在一个表格中，没有区分集成方式
- 缺少超时和降级策略

### 3.3 基础设施文档必须描述完整的技术架构而非仅配置项

**检查标准**：`03-基础设施与公共能力设计.md` 中的每个章节必须先描述架构设计（为什么这样设计、整体结构是什么），再列出具体配置项。不允许只有配置表格而没有架构说明。

**合格描述**：
- 缓存章节先有多层缓存架构图，再列 Key 设计表
- 认证章节先有认证流程图，再列配置项

**不合格描述**：
- 缓存章节只有 Redis Key 表格，没有说明为什么要多层缓存
- 认证章节只列了 Token 配置，没有 SSO 流程说明

### 3.4 数据库总纲必须包含数据源路由规则

**检查标准**：`04-数据库设计总纲.md` 在多数据源场景下必须包含：数据源拓扑图、各库的表分布、路由规则（默认路由/注解路由）、跨库限制说明。单数据源场景下可在该章节标注「不涉及」。

### 3.5 批量处理和跨服务编排必须独立描述

**检查标准**：`02-{域}-详细设计.md` 中，以下两类场景不能用普通接口模版描述，必须使用对应的专项章节：
1. **批量处理**（Excel 导入导出、批量审批等）：必须描述分批策略、错误处理、事务边界
2. **跨服务编排**（涉及多 Service 协作的业务流程）：必须提供时序图、标注事务策略和失败补偿

### 3.6 非功能性设计必须有量化指标

**检查标准**：`05-非功能性设计.md` 中的容量规划、性能优化、监控告警等章节必须包含量化指标（如 QPS、响应时间、数据量、告警阈值），不允许使用模糊描述（如"高性能"、"大数据量"）。

**合格描述**：
```
峰值 QPS: 500，P99 响应时间 < 200ms，日增数据量 10万条
```

**不合格描述**：
```
系统需要支持高并发和大数据量
```

## 四、检查清单

> 文档评审时按以下清单逐项检查

### 4.1 架构总纲检查清单

| # | 检查项 | 对应章节 | 通过标准 |
| --- | --- | --- | --- |
| 1 | 业务域划分完整 | 二、业务域全景 | 所有域均已列出，有域间关系 |
| 2 | 技术栈版本明确 | 三、技术栈总览 | 所有核心依赖标注版本号 |
| 3 | 架构图包含横切层 | 四、系统架构设计 | 拦截器/切面/过滤器已标注 |
| 4 | 数据源拓扑清晰 | 五、多数据源架构 | 每个库的用途和路由规则明确 |
