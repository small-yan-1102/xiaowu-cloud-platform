# INVEST 原则 - 需求拆解标准

本文件定义了用户故事(Quest)拆解时必须遵循的 INVEST 原则。

## INVEST 原则详解

### I - Independent (独立的)

**原则**: 每个 Quest 应该尽可能独立于其他 Quest

**检查清单**:
- [ ] Quest 之间没有强依赖关系
- [ ] 可以单独开发和测试
- [ ] 不依赖其他 Quest 的完成

**反例**:
```
❌ Quest 1: 创建用户
❌ Quest 2: 编辑用户 (依赖 Quest 1)
❌ Quest 3: 删除用户 (依赖 Quest 1)
```

**正例**:
```
✅ Quest 1: 用户列表管理 (包含创建、编辑、删除)
✅ Quest 2: 用户权限管理 (独立功能模块)
```

---

### N - Negotiable (可协商的)

**原则**: 用户故事是需求的占位符,细节应在开发过程中协商确定

**检查清单**:
- [ ] 不过度规定实现细节
- [ ] 保留技术实现灵活性
- [ ] 关注"做什么"而非"怎么做"

**反例**:
```
❌ 使用 Element Plus 的 el-table 组件,配置 pagination 属性为 { currentPage: 1, pageSize: 20 }
```

**正例**:
```
✅ 用户列表支持分页展示,默认每页 20 条
```

---

### V - Valuable (有价值的)

**原则**: 每个 Quest 必须为用户或业务交付可衡量的价值

**检查清单**:
- [ ] 能回答"这个功能解决了什么问题?"
- [ ] 价值可以从用户角度描述
- [ ] 不是纯技术重构(除非有明确业务价值)

**反例**:
```
❌ 重构 API 调用层 (没有业务价值说明)
```

**正例**:
```
✅ 优化接口调用策略,页面加载时间从 3s 降低到 1s
```

---

### E - Estimable (可估算的)

**原则**: 团队应该能够估算实现该 Quest 所需的工作量

**检查清单**:
- [ ] 需求描述足够清晰,可以估算
- [ ] 没有过多的未知技术风险
- [ ] 团队具备所需技术能力

**无法估算的情况**:
- 涉及未验证的新技术
- 依赖第三方服务但接口未确定
- 需求描述模糊,范围不清晰

**处理方式**: 拆分为更小的研究性质 Quest,或补充需求细节

---

### S - Small (足够小的)

**原则**: Quest 应该足够小,能在一个迭代内完成

**检查清单**:
- [ ] 单个 Quest 工作量 ≤ 3 天
- [ ] 可以在一个迭代内完成开发、测试、验收
- [ ] 如果需要 5+ 天,考虑拆分

**拆分策略**:
- 按 CRUD 操作拆分(列表/详情/编辑)
- 按用户角色拆分(管理员/普通用户)
- 按业务流程拆分(第一步/第二步/第三步)
- 按 Happy Path 和 Edge Case 拆分

**反例**:
```
❌ Quest 1: 完整的用户管理系统 (预计 10 天)
```

**正例**:
```
✅ Quest 1: 用户列表展示与搜索 (2 天)
✅ Quest 2: 用户详情查看 (1 天)
✅ Quest 3: 用户信息编辑 (2 天)
✅ Quest 4: 用户权限管理 (2 天)
```

---

### T - Testable (可测试的)

**原则**: 必须有明确的验收标准,可以验证是否完成

**检查清单**:
- [ ] 有具体的验收标准(使用 ✓ 列表)
- [ ] 验收标准可量化、可验证
- [ ] 包含正常流程和边界情况

**反例**:
```
❌ 验收标准: 用户体验良好 (无法验证)
```

**正例**:
```
✅ 验收标准:
  ✓ 用户列表加载时间 < 1s
  ✓ 搜索关键词为空时显示全部数据
  ✓ 搜索关键词超过 50 字符时显示错误提示
  ✓ 接口失败时显示友好错误提示
```

---

## 需求拆解检查清单

在拆解需求时,逐条检查:

### 结构检查
- [ ] 每个 Quest 都有明确的业务价值
- [ ] Quest 之间依赖关系清晰
- [ ] 单个 Quest 工作量适中(≤ 3 天)
- [ ] 验收标准具体可验证

### 内容检查
- [ ] 产品原文严格引用 PRD,不修改
- [ ] 需求解读站在前端视角
- [ ] 接口对接信息基于后端文档
- [ ] 组件选型基于项目实际

### 完整性检查
- [ ] 覆盖 PRD 中的所有功能点
- [ ] 包含正常流程和异常流程
- [ ] 包含边界条件和错误处理
- [ ] 验收标准覆盖所有关键场景

## 常见拆解模式

### 模式 1: 按 CRUD 拆解

```
Quest 1: {资源}列表展示与搜索
Quest 2: {资源}详情查看
Quest 3: {资源}创建
Quest 4: {资源}编辑
Quest 5: {资源}删除
```

### 模式 2: 按用户角色拆解

```
Quest 1: 管理员 - {功能}管理
Quest 2: 普通用户 - {功能}查看
Quest 3: 审核人员 - {功能}审批
```

### 模式 3: 按业务流程拆解

```
Quest 1: {流程}第一步 - 数据录入
Quest 2: {流程}第二步 - 数据校验
Quest 3: {流程}第三步 - 提交审核
Quest 4: {流程}第四步 - 审核结果展示
```

### 模式 4: 按 Happy Path / Edge Case 拆解

```
Quest 1: {功能}正常流程
Quest 2: {功能}异常处理 - 网络错误
Quest 3: {功能}异常处理 - 数据校验失败
Quest 4: {功能}边界条件 - 空状态/大数据量
```

---

## 使用建议

1. **优先保证 Testable**: 如果无法写出明确的验收标准,说明需求还不够清晰
2. **Small 是相对的**: 根据团队能力和迭代周期调整,通常 ≤ 3 天
3. **Independent 是目标**: 完全独立很难做到,尽量减少依赖即可
4. **Negotiable 不意味着模糊**: 业务规则必须明确,技术实现可以灵活
