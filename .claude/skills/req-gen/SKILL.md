# 需求全案自动化生成技能

> **设计模式**：Inversion（收集需求范围）→ Generator（基于模板生成用户故事文档）

## 输入/输出契约

### 输入
| 输入项 | 来源 | 缺失时处理 |
|-------|------|----------|
| PRD 文档 | 用户指定路径 | Fatal：向用户索取 |
| 原型文件 | 用户指定路径 | Warn：仅基于 PRD 分析 |
| 后端接口文档 | 用户指定路径或后端方案目录 | Warn：标记接口待确认,跳过接口对接分析 |
| 参考页面代码 | 可选,`ref_page` 参数 | Warn：跳过代码参考 |
| 用户故事模板 | `assets/user-story-template.md` | Fatal：模板缺失 |
| 版本号 | 用户指定,默认 `V1.0.0` | 使用默认值 |

### 输出
| 输出产物 | 存放路径 |
|---------|---------|
| 用户故事文档 | `requirements/user-stories/{version}/{version}.md` |

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 范围收集（Inversion）
- [ ] Phase 1: 需求分析
- [ ] Phase 2: 接口文档分析
- [ ] Phase 3: 组件与方法匹配
- [ ] Phase 4: 生成用户故事文档（Generator）
- [ ] Phase 5: 自检与交付
```

### Phase 0: 范围收集（Inversion 模式）

**⚠️ 硬门控：DO NOT proceed to Phase 1 until:**
- [ ] PRD 文档路径已确认
- [ ] 原型文件路径已确认（如有）
- [ ] 后端接口文档路径已确认（如有）
- [ ] 版本号已确认
- [ ] 需求覆盖范围已确认
- [ ] 用户确认所有输入信息无误

使用 AskUserQuestion 收集：
- PRD 文档和原型路径
- 后端接口文档路径（如有）
- 版本号
- 是否有参考页面代码
- 需求覆盖的功能范围

---

### Phase 1: 需求分析

1. 如果提供了参考页面代码（`ref_page`），先读取分析其结构
2. 读取 PRD 文档，**逐条分析需求内容，不能遗漏或曲解**
3. 提取业务核心模型（实体属性）
4. 分析页面设计与交互流程
5. 按功能模块拆分为多个职责单一的 Quest（遵循 INVEST 原则，详见 `references/invest-principles.md`）

**⚠️ 硬门控：DO NOT proceed to Phase 2 until:**
- [ ] 所有需求条目已逐条分析
- [ ] Quest 拆解符合 INVEST 原则
- [ ] 用户确认 Quest 拆解方案

---

### Phase 2: 接口文档分析

**如果提供了后端接口文档：**

1. 读取后端接口文档（OpenAPI/Swagger/Markdown 等格式）
2. 提取接口列表：路径、请求方法、请求参数、响应结构
3. 将接口映射到前端功能模块
4. 识别接口缺失或不匹配的情况，标记为 ⚠️ 需后端新增/调整

**如果没有接口文档：**
- 跳过本阶段
- 在用户故事中接口对接部分标记为"待确认"

**⚠️ 硬门控：DO NOT proceed to Phase 3 until interface analysis is complete.**

---

### Phase 3: 组件与方法匹配

对每个 Quest 的功能点进行组件和方法匹配，按以下优先级：

1. **优先匹配项目公共组件库**：搜索 `src/components/` 目录
2. **匹配项目 hooks/utils**：搜索 `src/hooks/`、`src/utils/` 目录
3. **匹配后端接口**：如有接口文档，优先使用已有接口
4. **无匹配时标记为需新建** ⚠️，并设计实现方案

> **💡 提示**: 组件分析方法详见 `development/frontend-impl-analysis/SKILL.md`

**⚠️ 硬门控：DO NOT proceed to Phase 4 until:**
- [ ] 所有 Quest 的组件匹配已完成
- [ ] 所有 Quest 的接口匹配已完成（如有接口文档）
- [ ] 需新建的组件/接口已明确标记

---

### Phase 4: 生成用户故事文档（Generator 模式）

基于 `assets/user-story-template.md` 模板生成文档，每个 Quest 必须包含：

1. **产品原文**：严格使用 PRD 中的原始内容
2. **原型图解读**：原型在需求中的作用、界面样式和布局
3. **需求解读**：站在前端角度的具体实现需求
4. **接口对接**（如有接口文档）：
   - 接口路径、请求方法
   - 前端传递参数
   - 响应数据处理
   - 接口状态（✅ 已有 / ⚠️ 需新增）
5. **前端设计**：
   - 布局描述
   - 交互描述
   - 数据流描述（接口数据如何流转到组件）
   - 错误处理（接口异常时的前端处理）
   - 校验规则
   - 修改位置

   **组件选型表**（每个 Quest 必须包含）：

   | 功能模块 | 组件 | 来源 | 说明 |
   |---------|------|------|------|
   | {功能} | `{组件名}` | `{路径}` ✅ | {使用方式} |
   | {功能} | `{组件名}` | ⚠️ 需新建 | 见下方设计说明 |

6. **验收标准**：使用 `✓` 列表，覆盖功能完整性、接口对接和用户体验

---

### Phase 5: 自检与交付

1. 验证文档结构完整性
2. 验证每个 Quest 都有接口对接表（如有接口文档）、组件选型表和验收标准
3. 验证接口信息与后端文档一致
4. 输出生成的文件路径

---

## 约束

- 产品原文必须严格引用 PRD 原始内容，不修改或简化
- 需求拆解遵循 INVEST 原则，颗粒度适中
- 组件选型必须基于项目实际存在的组件，来源标记准确
- 接口对接信息必须基于后端接口文档，不主观臆造接口
- 接口缺失时明确标记 ⚠️，并在验收标准中注明依赖
- 前端参数设计需与后端接口文档保持一致
- 不添加主观推测，不确定时向用户询问
