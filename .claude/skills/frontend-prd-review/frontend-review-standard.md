# 前端 PRD 审阅规范

> **目标**：为前端开发工程师提供 PRD 审阅的标准化参照基准，确保审阅维度完整、问题分类准确、建议可操作
> **核心原则**：从"能否确定前端实现方案"的角度审视每个需求——如果一个功能的 PRD 描述不足以让前端确定组件设计、交互逻辑、状态管理或数据展示方案，那就是需要提出的问题

---

## 一、审阅维度总览

前端审阅 PRD 应覆盖以下 8 个核心维度，每个维度对应审阅报告中的一个章节。此外还有 1 个补充章节（SG 技术实现建议）用于记录优化建议：

```
┌── 核心审阅维度（8 个）──────────────────────────────┐
│  UI  界面实现可行性   ← 设计稿/原型能不能还原、复杂度多高  │
│  IX  交互逻辑完整性   ← 交互流程、反馈、状态变化是否完整     │
│  CM  组件与状态设计   ← 能不能合理拆组件、状态怎么管理       │
│  PF  前端性能风险     ← 渲染量、首屏加载、大数据展示能不能撑住│
│  BL  前端业务逻辑     ← 表单校验、条件展示、计算逻辑是否完整  │
│  AM  需求歧义         ← 不同理解是否导致不同前端方案          │
│  CP  完整性缺失       ← 空状态、加载态、异常态、边界是否缺失  │
│  NF  非功能需求       ← 响应式/兼容性/无障碍/国际化是否明确   │
├── 补充章节（1 个）──────────────────────────────────┤
│  SG  技术实现建议     ← 降低实现复杂度或提升体验的建议        │
└──────────────────────────────────────────────────┘
```

---

## 二、各维度审阅检查清单

### UI - 界面实现可行性检查

对 PRD 和原型中的每个页面/功能，检查其 UI 实现可行性：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 布局可行性 | 页面布局是否可用标准 CSS 布局方案（Flex/Grid）实现？是否有非常规布局要求？ | UI 系列 |
| 视觉还原度 | 原型中的视觉效果（渐变、阴影、动画）是否有明确规格？是否依赖特定素材（图标/字体/图片）？ | UI 系列 |
| 响应式要求 | 不同屏幕尺寸下的布局变化是否定义？断点规则是否明确？ | UI/NF 系列 |
| 动画效果 | PRD 中描述的动画/过渡效果是否有具体参数（时长、缓动曲线）？ | UI 系列 |
| 数据驱动展示 | 页面中的数据展示区域在不同数据量下的表现是否定义（0 条/1 条/满页/超多）？ | UI/CP 系列 |
| 图表/可视化 | 是否涉及图表、仪表盘等可视化需求？图表类型和交互是否明确？ | UI 系列 |
| 富文本/编辑器 | 是否涉及富文本编辑？支持的格式和功能范围是否明确？ | UI 系列 |

### IX - 交互逻辑完整性检查

对 PRD 中描述的每个用户交互，检查其完整性：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 操作反馈 | 用户操作后的反馈方式是否明确（Toast/弹窗/页面跳转/内联提示）？成功和失败的反馈是否都定义？ | IX 系列 |
| 表单交互 | 表单的提交方式、校验时机（实时/失焦/提交时）、校验失败展示方式是否定义？ | IX 系列 |
| 确认流程 | 危险操作（删除/提交/审批）是否需要二次确认？确认弹窗的文案和按钮是否定义？ | IX 系列 |
| 页面跳转 | 操作完成后的页面跳转目标是否明确？是否需要保留/重置页面状态？ | IX 系列 |
| 加载行为 | 数据加载时的用户体验是否定义（骨架屏/Loading/占位符）？ | IX/CP 系列 |
| 拖拽/排序 | 涉及拖拽操作时，拖拽范围、放置规则、排序保存时机是否明确？ | IX 系列 |
| 键盘/快捷键 | 是否需要支持键盘操作？Tab 顺序和快捷键是否定义？ | IX/NF 系列 |
| 多步骤流程 | 多步骤表单/向导的步骤是否明确？步骤间的前进/后退/跳过规则是否定义？ | IX 系列 |
| 实时更新 | 数据是否需要实时更新展示（WebSocket/轮询/SSE）？更新频率和动画是否定义？ | IX/PF 系列 |

### CM - 组件与状态设计检查

对 PRD 中的功能评估其组件和状态管理可行性：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 数据展示结构 | 列表/表格/卡片等数据展示区域的字段、格式、排列方式是否明确？ | CM 系列 |
| 跨页面状态 | 多个页面共享的数据（如当前用户信息、全局配置）是否识别？ | CM 系列 |
| 表单状态 | 复杂表单的联动规则（字段 A 变化影响字段 B 的可选项）是否完整？ | CM 系列 |
| 缓存策略 | 页面切换时数据是否需要保留（如列表页筛选条件）？返回上一页时是否需要保持状态？ | CM 系列 |
| 权限驱动 UI | 不同角色/权限下页面元素的显隐/禁用规则是否明确？ | CM/BL 系列 |

### PF - 前端性能风险检查

对每个功能评估其前端性能风险等级：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 列表数据量 | 列表/表格的预期数据量是否明确？是否需要虚拟滚动/无限滚动？ | PF 系列 |
| 首屏加载 | 首屏需要加载的数据量和接口数量是否可控？是否需要懒加载/预加载？ | PF 系列 |
| 大文件处理 | 文件上传/预览的大小限制是否定义？是否需要分片上传/在线预览？ | PF 系列 |
| 图片资源 | 图片的尺寸、格式、压缩策略是否明确？是否需要图片懒加载？ | PF 系列 |
| 实时计算 | 前端是否需要执行大量实时计算（如数据汇总、图表渲染）？ | PF 系列 |
| 搜索/筛选 | 搜索输入的防抖/节流策略？前端筛选还是后端筛选？ | PF 系列 |
| 动画性能 | 复杂动画（如拖拽排序、数据可视化动画）是否可能引发性能问题？ | PF 系列 |

### BL - 前端业务逻辑检查

对 PRD 中需要前端处理的业务逻辑进行完整性审查：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| **表单校验规则矩阵** | **（强制检查）** 对每个表单，是否逐字段定义了完整的校验规则？须使用校验矩阵逐字段检查 | BL 系列 |
| 条件展示 | 页面元素的条件显隐逻辑是否明确？（如满足条件 A 时展示区域 B） | BL 系列 |
| 计算逻辑 | 前端需要执行的计算（如合计金额、倒计时、进度百分比）是否有公式和精度要求？ | BL 系列 |
| **数据展示规则** | **（强制检查）** 涉及格式化展示的字段是否逐一定义了展示规则？须使用格式化规则清单检查 | BL 系列 |
| 联动规则 | 字段间的联动关系（如省市区级联、类型切换表单变化）是否完整？ | BL 系列 |
| 状态流转展示 | 业务实体状态变化在前端的展示方式（颜色/标签/操作按钮变化）是否定义？ | BL 系列 |
| 权限控制 | 按钮级/字段级权限控制规则是否明确？不同权限下页面差异是否列全？ | BL/CM 系列 |
| **业务规则清单化** | PRD 中提到的每条业务规则是否都明确了：触发条件、判断逻辑、通过/不通过的前端表现？ | BL 系列 |

**表单校验规则矩阵**（对每个表单必须逐字段填充以下信息，PRD 中缺失的项为审阅发现）：

| 字段名 | 是否必填 | 格式校验 | 长度限制 | 取值范围 | 自定义校验 | 校验时机 | 校验失败提示 |
|--------|---------|---------|---------|---------|----------|---------|------------|
| {示例} | 是/否 | 邮箱/手机/身份证/URL/自定义正则 | 最小-最大字符数 | 数值范围/枚举值 | 与其他字段的关联校验 | 实时/失焦/提交时 | 具体提示文案 |

**数据格式化规则清单**（对涉及格式化展示的字段逐一检查）：

| 数据类型 | 检查项 | 示例 |
|---------|--------|------|
| 金额/货币 | 小数位数、千分位、货币符号、负数展示方式 | ¥1,234.56 vs 1234.56元 |
| 日期/时间 | 日期格式、时区处理、相对时间（如"3分钟前"）的切换阈值 | 2025-03-26 vs 03/26/2025 vs "刚刚" |
| 数字 | 千分位、小数精度、大数缩写（如"1.2万"）、百分比精度 | 12,345 vs 1.2万 |
| 手机号/身份证 | 脱敏规则（如中间四位用*号替代） | 138****1234 |
| 文本 | 超长截断规则（字符数/行数）、截断后的展示（省略号/tooltip/展开） | "这是一段很长的..." |
| 枚举/状态 | 枚举值的展示文案、颜色、图标 | "待审核"=橙色、"已通过"=绿色 |
| 文件大小 | 单位自动转换规则 | 1024KB → 1MB |
| 空值 | null/undefined/空字符串的展示方式 | "--" vs "暂无" vs 不展示 |

### AM - 需求歧义检查

对 PRD 中的描述逐项检查是否存在理解歧义：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| 方案差异放大 | 模糊描述是否可能导致不同的前端实现方案？方案差异大则为阻塞级歧义 | AM 系列 |
| 术语一致性 | PRD 中的业务术语与原型中的标签文案是否一致？是否存在同一概念多种叫法？ | AM 系列 |
| 交互方式歧义 | PRD 描述的交互方式是否有多种理解？（如"弹窗"是 Modal 还是 Drawer？"列表"是 Table 还是 Card？） | AM 系列 |
| 隐含条件 | PRD 中是否存在未显式表述但前端需要知道的条件？ | AM/BL 系列 |

### CP - 完整性缺失检查

对 PRD 中的每个功能检查是否存在关键信息遗漏：

| 检查项 | 检查要点 | 未通过时归入 |
|--------|---------|------------|
| **页面全状态覆盖** | **（强制检查）** 对每个页面/组件，是否定义了以下全部 8 种状态的展示方式？缺失任一状态视为完整性缺失 | CP 系列 |
| 空状态 | 列表/数据区域无数据时的展示方式是否定义？（空状态图/文案/引导操作） | CP 系列 |
| 加载状态 | 数据加载中的展示方式是否定义？（骨架屏/Spinner/进度条） | CP 系列 |
| 错误状态 | 接口失败、网络异常时的展示和恢复方式是否定义？ | CP 系列 |
| 部分加载状态 | 数据部分加载成功/部分失败时的展示方式是否定义？ | CP 系列 |
| 禁用/无权限状态 | 无权限访问时的展示方式是否定义？（403 页/跳转/隐藏入口/禁用按钮） | CP 系列 |
| 满载状态 | 数据量达到上限时的展示方式是否定义？（如列表达到分页上限） | CP 系列 |
| **异常处理场景** | **（强制检查）** 对每个涉及接口调用的功能，是否定义了以下异常场景的前端处理方式？ | CP 系列 |
| 边界条件 | 文本超长截断/换行规则、数值溢出展示、列表满页等边界是否说明？ | CP 系列 |
| 操作提示文案 | 成功/失败/警告等提示的具体文案是否定义？ | CP 系列 |
| PRD-原型一致性 | PRD 文字描述是否与原型页面展示一致？原型中有的内容 PRD 是否都有对应描述？ | CP 系列 |
| 多端适配 | 不同终端（PC/平板/手机）下的展示差异是否说明？ | CP/NF 系列 |
| 浏览器兼容 | 需要支持的浏览器版本和特性是否明确？ | CP/NF 系列 |

**页面全状态矩阵**（每个页面/核心组件必须逐一检查以下 8 种状态）：

| 状态 | 定义 | 前端需要知道的 |
|------|------|-------------|
| 初始状态 | 首次进入页面，尚未请求数据 | 是否展示默认值？是否自动触发数据加载？ |
| 加载状态 | 数据请求中 | 骨架屏/Spinner/进度条？用户是否可以取消？ |
| 正常状态 | 数据加载成功，正常展示 | 数据展示字段、格式、排列方式 |
| 空状态 | 查询无结果/列表为空 | 空状态图/文案？是否有引导操作（如"去创建"）？ |
| 错误状态 | 接口失败/网络异常 | 错误提示方式？是否有重试按钮？重试的交互？ |
| 部分状态 | 数据部分加载成功 | 成功部分正常展示？失败部分如何标记？ |
| 禁用状态 | 无权限/条件不满足 | 整页不可见 vs 部分元素禁用？禁用时的视觉样式？ |
| 满载状态 | 数据量达到上限 | 是否提示"已达上限"？是否禁用新增操作？ |

**异常处理场景清单**（对每个涉及接口调用的功能必须逐一检查）：

| 异常场景 | 前端需要知道的 |
|---------|-------------|
| 网络断开/超时 | 提示方式？是否自动重试？重试次数？ |
| 接口返回 4xx | 不同错误码的前端展示差异？（400 参数错误 vs 403 无权限 vs 404 不存在） |
| 接口返回 5xx | 用户看到什么？是否有"稍后重试"提示？ |
| 接口返回数据格式异常 | 缺少必要字段时如何展示？是否有兜底展示？ |
| 并发冲突 | 多人同时编辑时用户看到什么提示？ |
| Token 过期/登录态失效 | 是否跳转登录页？当前表单数据是否保留？ |
| 操作部分成功 | 批量操作中部分成功部分失败时如何展示结果？ |

## 三、问题严重度分级标准

审阅发现的问题应按严重度分级，便于产品经理优先处理：

| 严重度 | 判定标准 | 对前端的影响 |
|--------|---------|------------|
| **阻塞级** | 不解决则无法确定前端实现方案，技术设计无法启动 | 组件无法拆分、交互无法设计、页面无法搭建 |
| **重要级** | 不解决则前端方案存在重大风险，可能导致大范围返工 | 方案选型有误、性能不达标、交互体验差 |
| **一般级** | 不解决则实现细节不确定，但不影响整体方案 | 个别组件行为不确定、边界样式不明 |
| **建议级** | 可以按前端理解实现，但建议产品确认以避免后期争议 | 前端有合理假设，但不确定是否符合产品意图 |

**分级原则**：
- 问题类型不等于严重度——UI 系列问题可以是阻塞级也可以是一般级
- 严重度取决于"不解决对前端实现方案的影响程度"
- 在审阅报告中，每个问题应在描述中体现其严重度

## 四、审阅思维方法

### 1. 用户体验追问

对 PRD 中的每个功能，沿着用户操作路径逐步追问：

```
PRD 说"做什么" → 追问"用户在哪个页面操作"
                → 追问"操作前页面是什么状态"
                → 追问"操作过程中的反馈是什么"
                → 追问"操作成功后页面如何变化"
                → 追问"操作失败后用户看到什么"
                → 追问"用户能否撤销/重试"
```

如果任一环节答不上来，就是需要提给产品的问题。

### 2. 方案差异放大

遇到 PRD 中的模糊描述时，刻意构造两种不同的前端方案，评估方案差异：

- PRD 说"展示详情" → 方案 A：跳转详情页（简单）vs 方案 B：侧滑抽屉（复杂但体验好）→ 方案差异大 → 阻塞级歧义
- PRD 说"支持筛选" → 方案 A：前端本地筛选 vs 方案 B：后端接口筛选 → 取决于数据量 → 需确认量级
- PRD 说"实时更新" → 方案 A：轮询 vs 方案 B：WebSocket → 取决于实时性要求 → 需确认

**核心：如果不同理解导致不同的组件结构或技术选型，就必须澄清。**

### 3. 状态穷举法

对每个页面/组件，穷举其所有可能的展示状态：

- **初始状态**：首次进入页面，无数据/有默认值
- **加载状态**：数据加载中
- **正常状态**：有数据，正常展示
- **空状态**：查询无结果、列表为空
- **错误状态**：接口失败、网络异常
- **部分状态**：数据部分加载成功
- **禁用状态**：无权限、条件不满足
- **满载状态**：数据量达到上限

PRD 中未定义的状态就是完整性缺失。

### 4. 奥卡姆剃刀

与后端审阅相同，审阅报告本身也要遵循简约原则：

- 不提"凑数"问题——每个问题都应有实际前端实现影响
- 同一根因不拆成多个问题——合并为一个问题
- 问题描述直击要点——产品经理应能在 10 秒内理解

## 五、常见反模式（PRD 中的前端雷区）

| PRD 中的写法 | 前端风险 | 应追问的问题 |
|-------------|---------|------------|
| "展示列表" | 不清楚展示哪些字段、什么布局 | 展示哪些字段？表格还是卡片？支持排序筛选吗？ |
| "弹窗提示" | 不清楚弹窗类型和内容 | Modal 还是 Toast？弹窗内容和按钮是什么？关闭后回到哪里？ |
| "根据权限展示" | 无法确定页面元素显隐规则 | 哪些元素受权限控制？无权限时隐藏还是禁用？ |
| "支持搜索" | 不清楚搜索范围和方式 | 前端筛选还是后端搜索？搜索哪些字段？实时还是按回车？ |
| "上传文件" | 不清楚上传约束和体验 | 支持什么格式？大小限制？支持批量？上传中可以取消吗？ |
| "展示图表" | 不清楚图表类型和交互 | 什么图表类型？是否可交互（缩放/下钻/筛选）？数据量多大？ |
| "适配移动端" | 不清楚适配策略 | 响应式还是单独移动端页面？断点是多少？哪些功能在移动端隐藏？ |
| "实时更新" | 轮询/WebSocket/SSE 方案差异巨大 | 更新频率？多少用户同时在线？更新时有动画吗？ |
| "支持拖拽" | 不清楚拖拽范围和行为 | 拖拽什么元素？放置到哪里？排序保存时机？ |
| "表单校验" | 不清楚校验时机和规则 | 每个字段的校验规则？实时校验还是提交时校验？校验失败的提示方式？ |
| "给出提示" | 不清楚提示方式和文案 | Toast/Alert/内联提示？具体文案是什么？展示多久？ |
| "页面跳转" | 不清楚跳转方式和状态 | 路由跳转还是新开窗口？返回时保留之前的状态吗？ |
| "展示状态" | 不清楚状态展示方式 | 用什么颜色/标签/图标？不同状态下操作按钮如何变化？ |
| "多选操作" | 不清楚多选交互细节 | 全选逻辑（当前页/所有页）？选中后的操作栏在哪里？ |

## 六、审阅报告编号规则

| 前缀 | 含义 | 示例 |
|------|------|------|
| UI-{序号} | 界面实现可行性问题 | UI-1：仪表盘页面的图表类型和交互方式未定义 |
| IX-{序号} | 交互逻辑完整性问题 | IX-1：删除操作是否需要二次确认未说明 |
| CM-{序号} | 组件与状态设计问题 | CM-1：订单列表与详情页之间的数据共享方式未明确 |
| PF-{序号} | 前端性能风险 | PF-1：商品列表预期数据量未说明，无法确定虚拟滚动需求 |
| BL-{序号} | 前端业务逻辑问题 | BL-1：优惠金额计算公式和精度要求未定义 |
| AM-{序号} | 需求歧义 | AM-1："详情展示"可理解为跳转新页面或侧滑抽屉 |
| CP-{序号} | 完整性缺失 | CP-1：订单列表的空状态展示方式未定义 |
| NF-{序号} | 非功能需求 | NF-1：需要支持的浏览器最低版本未定义 |
| SG-{序号} | 技术实现建议 | SG-1：建议将复杂表单拆分为多步骤向导以降低实现复杂度 |
