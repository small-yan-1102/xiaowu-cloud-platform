# 代码规约自检

根据团队编码规约，对代码进行提交前逐项检查，输出分级问题报告。

---

## 附属资源文件

| 文件 | 路径 | 加载时机 |
|------|------|---------|
| 检查项清单（含规范文档引用） | `references/checklist.md` | Step 2 确定检查范围后必须加载 |
| L1 核心规范（权威） | `references/CODING_STANDARDS.md` | 检查所有 Java 代码时加载 |
| L2 代码规范（权威） | `references/CODING_STANDARDS_CODE.md` | 检查 Service/业务逻辑时追加 |
| L3 数据规范（权威） | `references/CODING_STANDARDS_DATA.md` | 检查 Repository/数据操作时追加 |
| L4 架构规范（权威） | `references/CODING_STANDARDS_ARCH.md` | 检查 Controller/领域模型时追加 |

---

## 核心工作流

### Step 1：确定检查范围

```
用户指定文件/目录  → 检查指定范围
用户说"检查我的修改" → 执行 git diff --name-only 获取变更文件列表
未指定范围        → 询问用户：检查哪些文件或目录？
```

> ⛔ **DO NOT** 开始读取代码或加载检查清单，直到明确检查范围。

### Step 2：加载检查清单

读取 `references/checklist.md`，根据文件类型按需使用对应检查章节：

| 代码类型 | 使用清单章节 |
|---------|-----------|
| 所有 Java 文件 | **L1 强制红线**（必查） |
| Service / 业务逻辑类 | + L2 代码层 |
| Repository / DAO / 数据操作类 | + L3 数据层 |
| Controller / API 接口类 | + L4 架构层 |
| 领域模型 / Entity / 聚合根 | + L4 架构层 |

### Step 3：逐文件逐项检查

对每个文件，按清单逐项扫描，记录每个问题的：
- 文件名和行号
- 违反的规约条目
- 问题代码片段
- 具体修正建议

> ⛔ **DO NOT** 在所有文件扫描完成前输出报告。

### Step 4：输出分级报告

按以下格式输出，严重问题优先展示：

```
========================================
         代码规约自检报告
========================================

📁 检查范围：[文件/目录]
📅 检查时间：[yyyy-MM-dd HH:mm]
📊 检查文件数：N 个

----------------------------------------
🚨 严重问题（必须修复，影响提交）
----------------------------------------

[1] [规约条目名称]
    📍 位置：[文件名]:[行号]
    ❌ 问题代码：
       [代码片段]
    ✅ 修正建议：
       [修正代码]

----------------------------------------
⚠️  警告问题（建议修复）
----------------------------------------

[1] [规约条目名称]
    📍 位置：[文件名]:[行号]
    💡 建议：[改进说明]

----------------------------------------
✅ 检查通过项 / 优秀实践
----------------------------------------

- [文件名]：[值得表扬的实践]

========================================
📈 检查统计
----------------------------------------
严重问题：X 个
警告问题：Y 个
通过项：  Z 项
========================================
```

**严重问题为 0 时**，在统计栏追加：
```
🎉 代码符合团队规约，可以提交！
```
