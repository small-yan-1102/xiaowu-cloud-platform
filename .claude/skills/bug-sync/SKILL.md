# 缺陷同步技能

> **版本**: v1.0
> **更新日期**: 2026-04-08
> **适用模式**: AI 读取测试执行报告 → 从失败用例提取缺陷信息 → 生成结构化缺陷记录 → 输出缺陷清单文件

---

## 附属资源文件

| 文件 | 路径 | 说明 |
|------|------|------|
| 缺陷报告模板 | `assets/bug-report-template.md` | 单条缺陷记录模板 + 缺陷清单模板 + CSV 导入格式，Phase 3 输出时加载 |
| 云效工具库 | `tools/yunxiao/` | 云效 API 客户端和缺陷创建脚本，Phase 3b 使用 |

---

## 核心工作流

```
Task Progress:
- [ ] Phase 0: 定向收集（Inversion）
- [ ] Phase 1: 加载执行报告
- [ ] Phase 2: 生成缺陷记录
- [ ] Phase 3: 输出缺陷清单
- [ ] Phase 3b: [可选] 提交到云效 Projex
```

---

### Phase 0: 定向收集

**目标**：在任何操作开始前，确认执行报告来源和缺陷输出格式；若选择云效输出，引导完成配置。

向用户提出以下问题，**等待用户回答后再进行任何操作**：

1. **执行报告路径**：要处理的执行报告文件是什么？
   - UI 执行报告默认：`test_reports/test_report_*.md`（取最新一份）
   - 接口执行报告默认：`test_reports/api_report_*.md`（取最新一份）
   - 可同时指定多份（取并集，去重）
2. **输出格式**：
   - `Markdown`（默认）：生成缺陷清单 Markdown 文件，供人工逐条提交到 Bug 系统
   - `CSV`：生成 CSV 文件，可批量导入 TAPD / Jira 等系统
   - `两者都要`
   - `云效 Projex`：自动调用云效 API 创建缺陷工作项
   - `Markdown + 云效`：生成本地报告，同时提交到云效
3. **缺陷优先级映射**：失败用例的优先级如何映射到缺陷严重等级？（默认规则：见 Phase 2 §优先级映射表）

> ⛔ **DO NOT** 读取任何文件，直到用户回答上述问题。

将用户的回答记录为**定向上下文**，在后续所有 Phase 中使用。

**决策点**：
- 用户已回答或表示"按默认来" → 使用默认路径和 Markdown 格式，进入 Phase 1
- 用户选择「云效 Projex」或「Markdown + 云效」→ 进入 **Phase 0b: 云效配置引导**，完成后再进入 Phase 1

---

### Phase 0b: 云效配置引导

> 仅当用户选择「云效 Projex」或「Markdown + 云效」输出格式时执行。

**目标**：检查并完成 `tools/yunxiao/config.yaml` 配置，无需用户提前准备。

**步骤**：

1. **读取现有配置**：
   - 尝试读取 `tools/yunxiao/config.yaml`
   - 文件不存在 → 以 `tools/yunxiao/config.example.yaml` 为基础模板，在内存中初始化空配置

2. **处理一次性字段（PAT、organization_id）**：
   - 若 `yunxiao.personal_access_token` 和 `yunxiao.organization_id` 已填写（非空）→ **静默复用，不再提示用户**
   - 若任一为空 → 逐一提示用户填写，说明获取方式：
     - **PAT**：云效 → 右上角头像 → 个人设置 → 个人访问令牌 → 新建（勾选「测试管理」+「项目协作」读写权限），格式 `pt-xxxx...`
     - **organization_id**：云效控制台 → 设置 → 组织信息 → 组织 ID

3. **每次必须确认的字段（project_id、sprint_id）**：
   - 无论配置文件是否存在，**始终向用户展示当前值并要求确认或更新**：

   ```
   当前云效配置：
   - 项目 ID（project_id）：[当前值 或 未配置]
   - 当前迭代 ID（sprint_id）：[当前值 或 未配置]

   项目或迭代有变更时请直接输入新值，无变更请回复「确认」。
   ```

   - 用户输入新值 → 更新；用户回复「确认」→ 沿用当前值
   - 两个字段均为**必填**，当前值为空时必须填写，不可跳过

4. **自动获取剩余配置项**（PAT 和 project_id 就绪后执行）：
   - **`projex.workitem_type_id`**：`GET /oapi/v1/projex/organizations/{orgId}/projects/{projectId}/workitemTypes?category=Bug`，取名称为「缺陷」的条目的 `id`
   - **`projex.severity_option_id`**：`GET /oapi/v1/projex/organizations/{orgId}/projects/{projectId}/workitemTypes/{typeId}/fields`，取 `id=seriousLevel` 字段中「严重」等级的 option `id`
   - **`projex.priority_option_id`**：同上接口，取 `id=priority` 字段中「高」优先级的 option `id`
   - API 调用失败 → 提示用户手动填写并附接口说明，不阻塞整体流程

5. **将所有值写入 `tools/yunxiao/config.yaml`**

6. **输出配置摘要，等待用户最终确认后进入 Phase 1**：

   ```
   ✅ 云效配置已就绪：
   - PAT：pt-At5z***（已配置）
   - 组织 ID：649e8985...
   - 项目 ID：fa0d97a5...
   - 当前迭代 ID：d604c75c...
   - 缺陷类型 ID：37da3a07...（自动获取）
   - 严重程度选项 ID：ba01f2c6...（自动获取）
   - 优先级选项 ID：c2427c00...（自动获取）

   确认无误后开始处理执行报告。
   ```

> ⛔ **DO NOT** 在 project_id 或 sprint_id 未经用户确认时进入 Phase 1。

### Phase 1: 加载执行报告

**目标**：读取执行报告，提取所有失败/阻塞用例的详细信息。

**步骤**：

1. **检查缺陷报告模板**：
   - 检查 `assets/bug-report-template.md` 是否存在
   - 不存在 → **Fatal**：终止执行，提示「缺陷报告模板文件缺失」

2. **读取执行报告文件**（按 Phase 0 用户指定路径）：
   - 文件不存在 → **Fatal**：终止执行，提示「执行报告文件不存在」
   - 多份报告 → 逐一读取，合并失败用例列表（按用例编号去重）

3. **提取失败/阻塞用例**：
   - 从执行报告中识别结果为 ❌ **确认失败** 或 🚫 **阻塞** 的用例
   - 对每条失败/阻塞用例，提取：
     - 用例编号、用例名称、所属模块
     - 失败步骤（操作描述）
     - 期望结果
     - 实际结果（截图路径 / 响应体）
     - 执行环境、执行时间
     - 用例优先级（P0/P1/P2/P3）
   - 阻塞用例单独标注阻塞原因（前置条件失败/数据缺失/环境问题）

4. **识别 Flaky 用例**（分级处理）：
   - 从执行报告中识别结果为 ⚡ **通过（含 Flaky）** 的用例
   - 对每条 Flaky 用例，提取：用例编号、所属模块、Flaky 步骤编号、重试次数

   **Flaky 升级检查**（执行报告「Flaky 警告」章节中若有 🔺 标记时触发）：
   - 识别报告中标注了「🔺 需关注」的 Flaky 用例（本迭代累计 ≥ 2 次）
   - 对这些用例**创建稳定性缺陷**：
     - 缺陷类型：`稳定性（Flaky）`
     - 严重等级：`Major`
     - 优先级：`中`
     - 标题格式：`[Flaky] {用例名称} — 本迭代已抖动 N 次`
     - 描述包含：每次 Flaky 的步骤编号、现象摘要、所在报告文件
   - 未达升级阈值的 Flaky 用例**仍不创建缺陷**，在 Phase 3 中单独输出「Flaky 追踪清单」

5. **输出提取摘要**：
   - 共发现 N 条确认失败用例、M 条阻塞用例、K 条 Flaky 用例
   - 按模块分布统计

> ⛔ **DO NOT** 进入 Phase 2，直到所有执行报告均已解析完毕。

### Phase 2: 生成缺陷记录

**目标**：为每条失败用例生成结构化缺陷记录。

**步骤**：

#### 优先级映射表（默认规则）

| 用例优先级 | 缺陷严重等级 | 缺陷优先级 | 说明 |
|-----------|-----------|-----------|------|
| P0 | 致命（Fatal） | 高 | 核心功能/冒烟失败 |
| P1 | 严重（Critical） | 高 | 主流程/鉴权/关键错误码 |
| P2 | 一般（Major） | 中 | 异常处理/边界值 |
| P3 | 轻微（Minor） | 低 | 组合参数/极端值 |

> 用户在 Phase 0 可覆盖此映射规则。

#### 缺陷编号规则

- 格式：`BUG-[MODULE]-[序号]`（序号按模块内递增，从 001 开始）
- 示例：`BUG-ORDER-001`、`BUG-PAYMENT-003`
- 阻塞用例标注：`BUG-ORDER-001 [阻塞]`

#### 对每条失败用例，生成以下字段

1. **缺陷标题**：`[{模块名}] {用例名称}` — 简洁、可搜索
2. **关联测试用例**：`{用例编号}`
3. **严重等级**：按优先级映射表确定
4. **缺陷优先级**：按优先级映射表确定
5. **所属模块**：从用例所属模块提取
6. **复现环境**：从执行报告中提取（Base URL / 执行时间）
7. **复现步骤**：从用例步骤中提取，以「1. 2. 3.」格式逐步列出
8. **期望结果**：原用例预期结果（L1 断言部分）
9. **实际结果**：执行报告中记录的实际值 / 截图描述
10. **附件**：截图文件路径（如有）/ API 响应体摘要（接口用例）
11. **缺陷类型**：
    - 功能缺陷（L1 断言失败）
    - 环境阻塞（前置条件失败、数据缺失）
    - 性能问题（超时）

**阻塞用例的特殊处理**：
- 若阻塞原因为「环境问题」或「测试数据缺失」→ 缺陷类型标注为**环境阻塞**，不创建功能缺陷
- 若阻塞原因为「依赖用例失败」→ 不单独创建缺陷，在关联的失败用例缺陷记录中注明「导致 {N} 条用例跳过」

> ⛔ **DO NOT** 进入 Phase 3，直到所有失败/阻塞用例均已生成缺陷记录。

### Phase 3: 输出缺陷清单

**目标**：按 `assets/bug-report-template.md` 中的模板，生成缺陷清单文件。

**步骤**：

1. **读取缺陷报告模板**：加载 `assets/bug-report-template.md`

2. **确定输出路径**：

   | 输出格式 | 文件路径 |
   |---------|---------|
   | Markdown | `bug_reports/bug-report-YYYYMMDD_HHmmss.md` |
   | CSV | `bug_reports/bug-import-YYYYMMDD_HHmmss.csv` |

3. **Markdown 输出**：
   - 缺陷汇总统计（总数 / 致命 / 严重 / 一般 / 轻微 / 环境阻塞）
   - 按模块分组列出每条缺陷记录（参照模板一）
   - 已知重复项标注「可能重复，请核查」
   - **Flaky 追踪清单**（独立章节，不计入缺陷统计）：按模块列出所有 Flaky 用例，注明重试次数，提示测试工程师评估是否需要修复（等待策略/断言方式/数据准备）

4. **CSV 输出**：
   - 按 `assets/bug-report-template.md` §CSV 格式 输出
   - 字段与常见 Bug 系统导入字段对齐（标题 / 严重等级 / 优先级 / 模块 / 复现步骤 / 期望结果 / 实际结果 / 附件 / 关联用例）

5. **向用户展示**：
   - 输出文件路径
   - 缺陷汇总摘要（总数 / 按严重等级分布）
   - 高严重等级（致命/严重）缺陷标题列表

---

### Phase 3b: [可选] 提交缺陷到云效 Projex

> 仅当用户在 Phase 0 选择了「云效 Projex」或「Markdown + 云效」输出格式时执行此阶段。

**前置条件**：
- `tools/yunxiao/config.yaml` 已配置 yunxiao 认证 + projex 缺陷配置
- 配置方法参见 `tools/yunxiao/README.md` 和 `tools/yunxiao/config.example.yaml`

**步骤**：

1. **校验云效配置**：
   - 读取 `tools/yunxiao/config.yaml`
   - 校验 projex 必填项：`project_id`、`workitem_type_id`、`severity_option_id`、`priority_option_id`
   - 配置不完整 → 提示用户参照 `tools/yunxiao/config.example.yaml` 补充，回退到仅 Markdown 输出

2. **调用缺陷创建工具**：
   - 工具路径：`tools/yunxiao/scripts/create_defect.py`
   - **单条模式**：对 Phase 2 生成的每条缺陷记录，传入 `--case-id`、`--title`、`--steps`、`--expected`、`--actual`、`--environment` 参数
   - **批量模式**：如果 Phase 3 已生成 Markdown 缺陷报告，可使用 `--from-report` 参数直接传入报告文件
   - 建议先用 `--dry-run` 验证再正式提交

3. **汇总提交结果**：
   - 输出成功数、失败数
   - 对于失败项，输出错误信息并建议手动提交

4. **增强输出**：
   - 在 Phase 3 生成的 Markdown 缺陷报告末尾，追加「云效提交状态」汇总表
   - 记录每条缺陷的云效工作项 ID

> ⛔ **DO NOT** 在配置校验失败时终止整个流程，应回退到仅本地文件输出。

## 验收标准

### 提取完整性

- [ ] 执行报告中所有「❌ 确认失败」结果均已生成缺陷记录
- [ ] 所有「阻塞」结果已按类型处理（环境阻塞 / 依赖失败跳过）
- [ ] 每条缺陷记录包含完整的复现步骤
- [ ] ⚡ Flaky 用例已单独列出，未混入普通缺陷清单
- [ ] 本迭代累计 ≥ 2 次 Flaky 的用例已创建稳定性缺陷（Major，类型标注"Flaky"）
- [ ] 稳定性缺陷包含每次抖动的步骤编号和现象描述

### 字段准确性

- [ ] 缺陷严重等级与用例优先级映射关系正确
- [ ] 期望结果与实际结果有明确对比（非模糊描述）
- [ ] 接口用例缺陷包含请求/响应摘要
- [ ] UI 用例缺陷包含截图文件路径（如有）

### 输出完整性

- [ ] Markdown 文件已写入正确路径
- [ ] CSV 文件（如选择）格式符合模板规范
- [ ] 缺陷汇总统计数字与实际记录数一致
