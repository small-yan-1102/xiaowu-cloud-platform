# Bug 修复技能

> **设计模式**：Pipeline — 信息收集 → 根因分析 → 最小修复 → 验证 → 审查，严格顺序不可跳步

## 核心工作流

```
Task Progress:
- [ ] Step 1: 信息收集
- [ ] Step 2: 根因定位
- [ ] Step 3: 最小化修复
- [ ] Step 4: 验证
- [ ] Step 5: 修复报告 + 代码审查
```

---

### Step 1: 信息收集

**⚠️ 硬门控：DO NOT proceed to Step 2 until all information is collected.**

在动手前，确认以下信息：

- [ ] 复现路径：哪个页面 / 操作路径 / 触发条件？
- [ ] 预期行为 vs 实际行为
- [ ] 控制台是否有报错（Error / Warning）
- [ ] 问题是必现还是偶现？

如信息不足，使用 AskUserQuestion 向用户收集。

---

### Step 2: 根因定位

按以下优先级排查：

1. **控制台报错** → 直接定位报错文件和行号
2. **网络请求** → 检查请求参数、响应状态、返回数据
3. **响应式数据** → 检查数据绑定是否被正确更新
4. **组件通信** → 检查 Props / Events / 双向绑定传递是否正确
5. **逻辑分支** → 检查条件判断、异步时序、边界值

**⚠️ 硬门控：DO NOT proceed to Step 3 until root cause is identified and documented.**

输出根因分析结论（一句话），如无法确定根因 → 向用户说明排查结果并请求更多信息。

---

### Step 3: 最小化修复

- **只改必须改的**：禁止借修 bug 重构逻辑或调整封装
- **不改样式**：除非 bug 本身是样式问题
- **不改 API**：除非接口调用本身有误
- **不改未涉及的文件**：修复范围严格限定在问题文件内

---

### Step 4: 验证

- [ ] 原问题已消失
- [ ] 相关功能未受影响（回归检查）
- [ ] 无新增控制台报错

**⚠️ 硬门控：DO NOT proceed to Step 5 until verification is complete.**

---

### Step 5: 修复报告 + 代码审查

**修复报告格式**：

```markdown
## Bug 修复报告

**问题描述**: 简述现象

**根因**: 一句话说明根本原因

**修复方案**: 说明改了什么、为什么这样改

**影响范围**: 涉及哪些文件 / 组件

**验证方式**: 如何验证修复有效
```

**修复后置动作**：
1. 自动执行 `code-review` 技能，对所有修改过的文件进行代码审查
2. 审查重点关注修复引入的代码是否符合项目规范
3. 如审查发现 Critical 问题，立即修复后再次审查，直至通过

---

## 错误处理

| 场景 | 处理方式 |
|------|---------|
| 无法复现 bug | 要求用户提供更详细的复现步骤或截图 |
| 根因涉及后端 | 标注为后端问题，输出前端侧的排查结论 |
| 修复可能引入副作用 | 向用户说明风险，确认后再执行修复 |
