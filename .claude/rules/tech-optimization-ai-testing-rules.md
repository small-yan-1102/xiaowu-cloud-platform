# 技术优化项目 AI 测试准入规则

> 适用场景：开发提交的技术优化/重构项目，无产品需求文档，由开发自行发起。
> 本规则定义了开展 AI 直接执行测试所需的完整素材链、职责分工和准入标准。
> 本规则为通用规则，适用于所有技术优化项目，不绑定具体业务。

---

## 一、素材链总览

```
开发交付                    测试产出
────────                    ────────
① 技术设计文档       ──┐
② 变更确认表         ──┤──→  ⑤ 测试用例文档
③ 测试补充文档(填写) ──┤──→  ⑥ AI 测试 Skill 实例
④ 源代码与部署信息   ──┘
```

①②③④ 是开发必须交付的输入，缺任何一项则测试不具备准入条件。
⑤⑥ 是测试工程师基于输入产出的测试资产。

---

## 二、开发交付物清单

### ① 技术设计文档

开发自行编写，描述改造的完整技术方案。必须包含以下内容：

| 必含章节 | 内容要求 |
|---------|---------|
| 改造背景 | 当前痛点是什么，为什么要改 |
| 现有流程 | 改造前的完整业务流程（含代码调用链） |
| 改造后流程 | 改造后的完整业务流程（含代码调用链） |
| 新增/变更数据模型 | 新增的数据库表 DDL、实体定义、状态流转（如有） |
| 变更概述 | 新旧流程的对比表 |
| 代码变更清单 | 新增文件列表、修改文件列表、不变文件列表及兼容性分析 |
| 影响分析 | 对现有功能的影响评估（逐项列出：有影响/无影响/行为变更） |

### ② 变更确认表

开发基于设计文档整理，明确列出所有行为变更，作为测试判定的唯一依据。格式：

| 行为项 | 改造前 | 改造后 | 是否预期变更 |
|--------|--------|--------|-------------|
| （具体行为描述） | （原行为） | （新行为） | 是/否 |

**要求：**
- 涵盖设计文档影响分析中所有"行为变更"和"无影响"项
- "无影响"项也必须列出，作为回归测试依据
- 开发确认签字，表示"以上为预期行为，偏离即为 Bug"
- 测试工程师准入检查时需逐条对照设计文档影响分析章节，确认无遗漏

### ③ 测试补充文档（开发填写）

由测试工程师提供模板，开发填写。补充设计文档中未覆盖但测试必须的信息。分为两部分：

**第一部分：技术细节补充（AI 直接执行测试 + 开发侧单元/集成测试共用）**

| 章节 | 内容 | 优先级 |
|------|------|--------|
| 现有表结构 | 改造涉及的所有表 DDL 或关键字段说明 | P0 |
| 配置对象数据模型 | 完整结构定义 + 示例数据 + 字段约束 | P0 |
| 已有方法签名 | 设计文档引用但未定义的方法的完整签名和行为说明 | P0 |
| 测试基础设施 | 测试框架、Mock 框架、已有测试模式和示例 | P0 |
| 错误处理规格 | 异常处理策略表：每个操作的异常类型、处理方式、是否告警 | P1 |
| 验收标准与边界条件 | 每个改造点的场景表（场景ID/前置条件/操作/预期结果） | P1 |
| 外部依赖接口契约 | 外部系统调用的方法签名、返回值、异常类型 | P2 |
| 并发与幂等性规格 | 并发场景清单、事务隔离级别、乐观锁机制 | P2 |

**第二部分：测试工程师执行测试所需**

| 章节 | 内容 | 优先级 | 适用条件 |
|------|------|--------|---------|
| 可观测的验证手段 | 按变更确认表逐项提供：每个行为项对应的日志关键字、验证 SQL 或其他可观测验证方式 | P0 | 所有项目 |
| 测试数据构造指南 | 各场景的造数脚本或操作步骤 | P0 | 所有项目 |
| 异步任务/定时任务触发方式 | 管理台地址、手动触发步骤、参数格式 | P0 | 涉及定时任务/异步任务时 |
| 存量数据现状 | 生产数据量级、已知数据质量问题、测试环境数据来源 | P1 | 涉及数据迁移时 |
| 新旧流程切换策略 | 功能开关、部署顺序、灰度策略、回滚方案 | P1 | 涉及流程变更时 |
| 异常降级业务预期 | 各异常场景的业务影响、恢复方式、最大容忍时长 | P1 | 涉及关键业务路径时 |
| 核心业务计算示例 | 复杂计算逻辑的手算验证示例 | P1 | 涉及复杂业务计算时 |

### ④ 源代码与部署信息

开发提测时提供代码和部署所需信息：

**代码交付：**
- Codeup 代码仓库克隆地址（HTTPS 或 SSH）
- 功能分支名称（如 `feature/xxx`）
- 变更文件清单（新增、修改、删除的文件列表）
- 已有的单元测试和集成测试（及覆盖率报告）

> **AI 源码集成：** AI 在测试启动时自动将应用仓库克隆到本地工作目录，切换到功能分支，通过 `git diff master...feature/xxx` 获取本次改造的完整变更范围。AI 基于变更 diff 和源码进行变更范围确认、测试结果分析和用例补充。

**部署清单（测试工程师据此部署测试环境）：**

| 项目 | 内容要求 |
|------|---------|
| 服务清单 | 需部署的服务名称 + 对应分支或版本号 |
| 部署顺序 | 多服务时的部署先后顺序（如有依赖） |
| 配置变更 | 新增或修改的配置项及其值（如有） |

## 三、测试职责分工

### 开发负责（提测前完成）

- 单元测试：每个方法的逻辑正确性
- 集成测试：数据访问层正确性（SQL、Repository）
- 提供单元/集成测试覆盖率报告

### 测试工程师负责

- 流程测试：端到端业务流程验证（核心工作量）
- 迁移测试：数据迁移正确性、幂等性、一致性（如涉及迁移）
- 回归测试：验证"不变"的功能确实不变
- 并发测试：可选，开发已提供覆盖率报告时可跳过
- 性能测试：可选，数据量级不大时可跳过

## 四、准入检查清单

测试工程师在开始测试前，逐项检查：

```
必须项（缺任何一项则打回）:
  □ 技术设计文档已提交，包含全部必含章节
  □ 变更确认表已提交，开发已确认，且逐条覆盖设计文档影响分析中的全部条目
  □ 测试补充文档 P0 项已全部填写（验证手段已按变更确认表逐项提供）
  □ 源代码已提交至 Codeup 代码仓库，仓库克隆地址和功能分支名称已提供
  □ 部署清单已提交（服务名称 + 版本号，如有部署顺序或配置变更需一并说明）
  □ 测试环境已按部署清单完成部署，基本功能可访问
  □ 数据库变更脚本已提供（如涉及表结构变更）
  □ 新增的异步任务/定时任务已在测试环境可触发（如涉及）

建议项（缺失不阻塞，但影响测试覆盖度）:
  □ 测试补充文档 P1 项已填写
  □ 单元/集成测试覆盖率报告已提供
  □ 测试补充文档 P2 项已填写
```

## 五、AI 测试 Skill 规范

每个技术优化项目应基于通用 Skill 模板创建项目专属的 AI 测试 Skill 实例。

Skill 实例必须定义以下工作模式：

| 模式 | 用途 | 输入 | 输出 |
|------|------|------|------|
| 测试数据构造 | 生成指定用例的前置数据 | 用例 ID + 补充文档 | 可执行的数据脚本 |
| 验证脚本生成 | 生成检查点查询 | 用例 ID + 表结构 | 带预期结果的验证脚本 |
| 业务计算校验 | 手算业务逻辑验证系统输出 | 配置参数 + 当前状态 | 分步计算过程 + 结论 |
| 用例补充 | 发现遗漏场景时补充 | 场景描述 | 标准格式用例 |
| 结果分析 | 实际结果与预期不符时定位 | 现象 + 日志 + 源码 | 原因分析 + 代码定位 + 排查建议 |
| 变更范围验证 | 验证代码变更与设计文档一致性 | 设计文档变更清单 + 源码 | 匹配/超出/遗漏报告 |
| 迁移评估 | 迁移前风险检查 | 数据量级 + 迁移方案 | 风险评估 + 校验脚本 |

> 注意：不是所有项目都需要全部 7 种模式。"业务计算校验"仅在涉及复杂计算逻辑时需要，"迁移评估"仅在涉及数据迁移时需要。

**通用约束（所有 Skill 实例必须遵守）：**
- 不修改业务代码，只生成测试辅助产物
- 补充文档中未填写的项，告知测试工程师需要开发补充，不猜测
- 生成的数据变更脚本必须有条件限制，禁止全表操作
- 测试数据使用可识别前缀（如 `TEST_`），便于清理
- 时间字段使用相对表达式，避免硬编码
- 标注操作之间的依赖关系

通用 Skill 模板文件：`rules/templates/tech-optimization-ai-test-skill-template.md`

## 六、测试用例组织规范

测试用例按阶段组织，固定结构如下：

```
阶段一: 迁移/存量数据处理（如涉及）
  → 前置校验 → 正向执行 → 幂等验证 → 一致性校验

阶段二: 核心流程变更验证
  → 逐个改造点验证 → 端到端全流程

阶段三: 关联流程变更验证（如涉及）
  → 逐个改造点验证

阶段四: 回归测试
  → 变更确认表中"不变"项逐一验证

[可选] 阶段五: 并发测试
  → 条件: 开发未提供覆盖率报告

[可选] 阶段六: 性能测试
  → 条件: 数据量级大或有性能敏感操作
```

每个用例必须包含：

| 字段 | 说明 |
|------|------|
| 用例 ID | 全局唯一，格式 `TC-{阶段缩写}-{序号}` |
| 优先级 | P0（必须通过）/ P1（应通过）/ P2（建议） |
| 前置条件 | 需要什么数据状态、什么环境配置 |
| 操作步骤 | 编号步骤，每步包含具体操作和检查点 |
| 检查点 | 具体的验证脚本或日志搜索指令 |
| 预期结果 | 明确的预期值，不用模糊描述 |

## 七、测试退出准则

测试工程师在以下条件全部满足时，可判定测试通过、允许上线：

```
必须满足（任一不满足则不可上线）:
  □ P0 用例全部通过（通过率 = 100%）
  □ P1 用例通过率 ≥ 90%，未通过的 P1 已有明确的风险评估和处置方案
  □ 变更确认表中所有"预期变更"项均已验证且结果符合预期
  □ 变更确认表中所有"无影响"项均已回归验证，行为未发生变化
  □ 所有 B 级（阻塞性）Bug 已修复并复测通过
  □ 遗留问题清单已整理，相关方已确认风险可接受

建议满足（不阻塞上线，但需记录）:
  □ P2 用例已执行
  □ 性能测试/并发测试已执行（如适用）
```

**遗留问题处理：**
- 未修复的非阻塞性 Bug 需记录到遗留问题清单
- 每条遗留问题需注明：影响范围、风险等级、计划修复版本
- 遗留问题清单需开发和测试双方确认

## 八、适用边界

本规则适用于：
- 开发发起的技术优化/重构
- 无产品需求文档
- 涉及数据模型变更、流程变更、定时任务变更、性能优化等

本规则不适用于：
- 有完整需求文档的功能开发（按需求文档驱动测试）
- 纯代码重构无行为变更（开发自测即可）
- 紧急线上 hotfix（走 hotfix 流程）
