# 一人全栈 + AI 协作工作指南

> 面向团队中采用"一个人 + AI 同时承担前后端开发"模式的工程师。
> 技术栈：Spring Boot + Vue/React。

---

## 第一部分：工作模式与总原则

### 什么是"一人全栈 + AI"

一个开发工程师，借助 AI 工具同时完成前端和后端的编码工作，独立交付完整功能。

核心变化不是"你要学会所有技术栈"，而是：
- **你负责**：需求理解、架构决策、业务判断、质量兜底
- **AI 负责**：代码生成、模式套用、重复实现、文档输出

### 总原则

```text
每次提问尽量同时说清楚三件事：
1. 让 AI 站在哪个视角（后端/前端/全栈/审阅者）
2. 基于哪些资料（PRD、设计文档、代码路径）
3. 输出什么结果（代码、报告、问题清单）

如果信息不足以支撑安全推进，请先列出问题，不要自行猜测业务规则、字段含义、状态流转或默认处理方式。
```

### 时间分配参考

| 环节 | 占比 | 说明 |
|------|:----:|------|
| 需求理解、梳理依赖、准备上下文 | 30% | 最关键的投入，决定 AI 产出质量 |
| 编写提示词、与 AI 交互 | 10% | 用本手册的模板可以压缩这部分 |
| 审查 AI 产出、修正问题 | 20% | 不能省，AI 会犯错 |
| 联调、集成、处理 AI 搞不定的部分 | 20% | 一人全栈的核心挑战 |
| 自测验证 | 20% | 没有别人帮你测，必须自己闭环 |

## 第二部分：开发顺序选择

一人全栈时，先做什么直接影响效率。以下是三种常见策略：

| 策略 | 适合场景 | 做法 |
|------|---------|------|
| **后端先行** | 业务逻辑重、前端主要是展示和表单 | 后端编码 → 自验接口 → 前端编码 → 联调 |
| **前端先行** | 交互复杂、需要先验证 UX 再定接口 | 前端 Mock 页面 → 确认交互 → 反推接口设计 → 后端编码 → 联调 |
| **接口先行** | 前后端逻辑都重，需要先锁定契约 | 设计接口文档 → 前后端并行（后端实现 + 前端 Mock 对接）→ 联调 |

**默认推荐：后端先行。** 大多数业务系统（CRUD + 审批流 + 报表）后端逻辑是核心，前端围绕接口展示。后端先写完、自验通过后，前端可以直接对接真实接口，省掉 Mock 和替换的成本。

## 第三部分：会话管理策略

### 前后端是否用同一个会话？

| 场景 | 建议 | 原因 |
|------|------|------|
| 简单接口 + 对应页面（< 3 个接口） | 同一会话 | AI 能看到刚写的后端代码，前端对接更准 |
| 复杂模块（> 5 个接口或复杂业务逻辑） | 分开会话 | 避免后端上下文太长，污染前端实现思路 |
| 后端已写完并自验通过 | 新开会话写前端 | 后端已稳定，前端会话只需关注页面和交互 |

### 什么时候新开会话？

```text
继续当前会话：
- 需求和设计未变化
- 当前会话不长
- 本次只推进小范围实现

建议新开会话：
- 已从后端切换到前端（且后端已自验通过）
- 当前会话已较长（超过 10 轮有效对话）
- 需要重新聚焦某个模块或页面

必须新开会话：
- PRD、设计文档、接口文档已变更
- PM 刚补充了关键业务规则
- 当前会话已混入旧版本结论或错误代码
```

### 新会话恢复上下文

```text
这是一个新会话。请先读取以下资料，再继续当前任务：

- 当前 PRD：[路径]
- 当前设计文档：[路径]
- 当前接口文档：[路径]
- 后端代码：[路径]
- 前端代码：[路径]

当前任务进度：
[说明已完成的部分和接下来要做的部分]

要求：先基于上述资料恢复上下文，确认你的理解后再继续推进。
```

**使用场景**：从前端页面实现和交互闭环角度补充审阅。

**使用场景**：两份审阅报告合并后发给产品经理确认。

其中标注【阻塞】的问题请优先回复，后续技术设计和编码依赖这些信息。

### 阶段 4：PM 回复后的处理

根据 PM 的回复方式，选择对应处理：

**场景 A：PM 只在报告中补了回复，没改 PRD**

```text
产品经理已在审阅报告中补充了回复，请读取报告中的"产品回复"内容，整合到你的理解中，并基于更新后的结论继续推进后续工作。
审阅报告路径：[路径]
```

**场景 B：PM 小幅修改了 PRD**

```text
PM 更新了 PRD，新版本路径如下：[路径]
本次改动主要是补充细节，没有新增核心功能，也没有调整主流程。
请针对改动部分进行增量审阅，不需要重新做全量审阅，并将结果补充到原审阅结论中。
```

**场景 C：PM 大幅修改了 PRD**

```text
PM 更新了 PRD，新版本路径如下：[路径]
本次改动涉及新增模块或核心流程调整，请重新执行完整审阅，覆盖之前的审阅产物，并输出新的需求理解总结、审阅报告和待确认问题清单。
```

### 阶段 6：后端编码

**使用场景**：设计文档确认后，开始后端代码实现。

**指导说明**：
- 新会话编码：提供完整的资料路径
- 同会话继续：可省略资料路径，引用前面的上下文
- 单模块实现：明确指定本次只做哪个模块/接口

**新会话编码模板**（完整版）：

```text
请基于已确认的设计文档和接口文档，实现本次迭代范围内的后端代码。

输入资料：
- 详细设计文档：[路径]
- API 接口文档：[路径]
- 后端工程目录：[路径]

本次目标：[整体实现 / 具体模块名 / 具体接口名]

要求：
- 遵循现有 Spring Boot 项目风格和分层方式（Controller → Service → Mapper）
- 只实现本次迭代范围内的内容，不要修改无关代码
- 优先复用现有实现模式（分页结构、返回包装、异常处理）
- 如果信息不足或文档冲突，请先列出问题，不要自行猜测业务规则

正式修改代码前，请先说明：
- 你理解的本次实现范围
- 预计会修改或新增的主要文件
```

**同会话继续模板**（简版）：

```text
基于前面已确认的需求、接口设计和技术方案，继续实现后端代码。
本次目标：[具体模块名 / 接口名]
要求：遵循现有 Spring Boot 项目风格，只实现本次范围内的内容，不要修改无关代码；发现问题先指出，不要自行猜测。
```

### 阶段 7：后端自验（新增）

**使用场景**：后端编码完成后、开始前端编码前，验证后端接口的正确性。

**指导说明**：
这是一人全栈模式下最容易跳过、也最容易踩坑的环节。后端自验通过后再写前端，能避免大量联调返工。

**自验内容包括**：
1. 接口能否正常启动和调用
2. 接口的请求参数和返回结构是否与设计文档一致
3. 核心业务逻辑是否正确（增删改查、状态流转、校验规则）

```text
后端编码已完成，请在开始前端编码前，对本次实现的接口进行自验：

- API 接口文档：[路径]
- 后端代码：[路径]

请逐个检查已实现的接口：
1. 请求路径、方法（GET/POST/PUT/DELETE）是否与文档一致
2. 请求参数（字段名、类型、必填项）是否与文档一致
3. 返回结构（字段名、类型、嵌套结构、分页包装）是否与文档一致
4. 枚举值、状态码是否与文档定义匹配
5. 异常场景（参数校验失败、数据不存在、权限不足）的返回是否合理

如果发现实现与文档不一致，请明确列出差异项，并说明是文档需要更新还是代码需要修改。
```

---

### 阶段 8：前端编码

**使用场景**：后端自验通过后，开始前端页面实现。

**指导说明**：
- 以后端 API 接口文档为主要依据
- 如果接口文档信息不足，优先核对相关后端接口实现（Controller、请求/返回对象、枚举），不要全量扫描后端项目
- 如果接口文档与后端已实现代码不一致，明确指出并暂停，不要自行选择一版继续

**新会话编码模板**（完整版）：

```text
请基于 PRD、UI 原型和后端 API 接口文档，实现本次迭代范围内的前端页面与交互。

输入资料：
- PRD 文档：[路径]
- UI 原型图：[路径]
- 前端代码：[路径]
- 后端 API 接口文档：[路径]

本次目标：[整体实现 / 具体页面名]

要求：
- 以后端 API 接口文档作为主要依据
- 如果接口文档信息不足，请只核对当前页面实际依赖的后端接口实现，不要全量扫描整个后端项目
- 遵循现有 Vue/React 项目的组件结构、状态管理和请求方式
- 只实现本次迭代范围内的页面和交互，不要修改无关代码
- 如果 PRD、原型、接口文档之间有冲突，先列出问题，不要自行猜测

正式修改代码前，请先梳理当前页面的功能点及对应接口清单，确认后再开始实现。
```

**同会话继续模板**（简版）：

```text
基于前面已确认的 PRD、原型和接口设计，继续实现前端代码。
本次目标：[具体页面名]
以接口文档为主，必要时再核对相关后端实现；只做本次范围内的页面和交互，不要修改无关代码，遇到冲突先指出。
```

### 阶段 9：前端核对后端的规则

**使用场景**：前端编码过程中需要查看后端实现细节时。

**指导说明**：
按三层递进查看，不要一开始就全量扫描后端代码：

```
第一层：接口文档 → 能解决大部分问题
第二层：相关后端文件（Controller、DTO、枚举）→ 文档不够时看这些
第三层：搜索后端仓库 → 只有前两层都找不到时才用
```

**接口文档与代码不一致时**：

```text
发现接口文档与后端实际代码不一致，请列出所有差异项。
[如已确认以代码为准]：以后端实际实现的接口结构为准继续前端编码，并标注文档需要同步更新的地方。
[如未确认]：暂停前端编码，等待确认后再继续。
```

### 阶段 10：前后端自联调（新增）

**使用场景**：前后端都写完后，验证端到端流程是否跑通。

**指导说明**：
一人全栈最容易出的问题就是"后端觉得没问题，前端也觉得没问题，一联调就对不上"。自联调是交付前的最后一道关。

```text
前后端编码已完成，请帮我检查前后端的接口对接是否一致：

- 后端接口代码：[路径]
- 前端接口调用代码：[路径]
- API 接口文档：[路径]

请逐个接口检查以下内容：
1. 前端调用的请求路径和方法是否与后端 Controller 定义一致
2. 前端发送的请求参数（字段名、类型、格式）是否与后端接收参数匹配
3. 前端解析的返回结构是否与后端实际返回结构一致（特别注意嵌套层级和分页包装）
4. 枚举值的前后端定义是否一致（如状态、类型等字段）
5. 错误处理：前端是否正确处理了后端可能返回的异常状态码

请输出：一致的接口列表 + 不一致的接口及具体差异。
```

### 阶段 11：编码异常处理

**使用场景**：编码过程中遇到问题需要暂停或回退。

**场景 A：PRD 或设计文档有问题**

```text
发现当前需求或设计文档存在问题，已不足以支撑继续实现。
请记录问题点、影响范围和当前阻塞原因，暂停当前实现，等待文档更新后再继续。
不要在当前会话中基于未确认的规则继续编码。
```

**场景 B：自己的后端实现需要修改（前端写到一半发现）**

```text
前端编码过程中发现后端接口 [接口名] 的设计/实现需要调整：
- 问题描述：[具体问题]
- 需要改什么：[具体修改内容]

请先修改后端接口代码，确认修改后的接口结构，然后继续当前前端页面的实现。
修改后端时注意：只改必要的部分，不要顺手重构其他接口。
```

**场景 C：自己的前端实现需要推翻重来**

```text
当前页面 [页面名] 的实现方案有问题，需要换一种方式重新实现：
- 问题描述：[为什么当前方案不行]
- 新方案思路：[大致方向]

请基于新方案重新实现该页面，保留可复用的部分（API 调用、工具方法），替换需要变更的部分（组件结构、交互逻辑）。
```

### 阶段 12：调试排错（新增）

**使用场景**：代码报错或功能不符合预期时。

**后端报错**：

```text
后端代码运行时报错，请帮我排查：

错误信息：
[粘贴完整的错误日志/堆栈信息]

相关代码文件：[路径]

请分析错误原因，给出修复方案。如果需要更多上下文，请告诉我需要看哪些文件。
```

**前端报错**：

```text
前端页面出现问题，请帮我排查：

问题描述：[页面表现 / 控制台报错信息]
相关组件文件：[路径]
浏览器控制台错误：[如有]
网络请求异常：[如有，贴请求和响应]

请分析原因并给出修复方案。
```

**功能不符合预期（非报错）**：

```text
[功能名] 的行为与预期不符：
- 预期行为：[应该是什么样]
- 实际行为：[现在是什么样]
- 相关代码：[路径]

请排查逻辑问题并修复。如果涉及业务规则理解偏差，请先指出你对规则的理解，确认后再修复。
```

## 第五部分：完成标准清单

### 后端编码完成标准

- [ ] 所有接口已按设计文档实现
- [ ] 接口的请求参数和返回结构与文档一致（阶段 7 自验已通过）
- [ ] 核心业务逻辑有基本的参数校验和异常处理
- [ ] 代码能正常编译，无编译错误
- [ ] 新增/修改的代码不影响已有功能（没有改动无关代码）

### 前端编码完成标准

- [ ] 所有页面已按原型和 PRD 实现
- [ ] 页面能正常渲染，无控制台报错
- [ ] 核心交互流程可走通（按钮、表单、跳转、弹窗）
- [ ] 列表分页、搜索筛选、状态展示与接口对齐
- [ ] 新增/修改的代码不影响已有页面

### 联调完成标准

- [ ] 所有接口的前后端调用一致（阶段 10 自联调已通过）
- [ ] 核心业务流程端到端可走通（创建 → 编辑 → 查询 → 删除 等）
- [ ] 异常场景前端有合理提示（网络错误、参数校验失败、无权限等）
- [ ] 无硬编码的 Mock 数据残留

### 整体交付标准

- [ ] 功能完整：PRD 中要求的功能点全部实现
- [ ] 自测通过：核心路径手动走过一遍无问题
- [ ] 代码干净：无调试用的 console.log、无注释掉的旧代码、无临时 hack

### 后端审阅

```text
请使用 backend-prd-review，从后端实现可行性和业务规则完整性视角审阅以下资料，并输出需求理解总结、审阅报告和待确认问题清单。
```

### 前端审阅

```text
请使用 frontend-prd-review，从前端页面实现和交互闭环视角审阅以下资料，并输出前端需求理解总结、审阅报告和待确认问题清单。
```

### 后端编码

```text
请基于已确认的设计文档和接口文档，实现本次迭代范围内的后端代码；遵循现有 Spring Boot 项目风格和分层，不要修改无关代码，信息不足时先列问题。
```

### 后端自验

```text
后端编码已完成，请逐个对比接口文档和实际代码，检查路径、参数、返回结构、枚举值是否一致，列出所有差异。
```

### 前端编码

```text
请基于 PRD、原型和后端接口文档，实现本次迭代范围内的前端页面与交互；以接口文档为主，必要时再核对相关后端实现，不要修改无关代码。
```

### 前后端联调检查

```text
前后端编码已完成，请逐个接口检查前端调用与后端实现是否一致（路径、参数、返回结构、枚举值），列出不一致的地方。
```

### 调试排错

```text
[功能名] 有问题：预期 [X]，实际 [Y]。相关代码：[路径]。请排查原因并修复，涉及业务规则理解偏差时先确认再改。
```
