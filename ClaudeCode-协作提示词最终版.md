# ClaudeCode 协作提示词最终版

这份文档用于在与 ClaudeCode 协作时，快速复制对应阶段的高质量提示词。

---

## 总原则

```text
每次提问尽量同时说清楚三件事：
1. 让 AI 站在哪个视角
2. 基于哪些资料
3. 输出什么结果

如果信息不足以支撑安全推进，请先列出问题，不要自行猜测业务规则、字段含义、状态流转或默认处理方式。
```

---

## 1. 后端 PRD 审阅

```text
请使用 backend-prd-review，从后端实现可行性和业务规则完整性视角审阅以下资料：

- PRD 文档：[路径]
- UI 原型图：[路径]
- 后端代码：[路径]
- 前端代码：[路径]

请输出：
- 需求理解总结
- 审阅报告
- 需要产品确认的问题清单

如果信息不足，请标记为问题项，不要自行猜测业务规则。
```

---

## 2. 前端 PRD 审阅

```text
请使用 frontend-prd-review，从前端页面实现和交互闭环视角审阅以下资料：

- PRD 文档：[路径]
- UI 原型图：[路径]
- 前端代码：[路径]

请重点关注：
- 页面和交互流程是否完整
- 原型与 PRD 是否一致
- 前端展示、操作、状态切换所需字段是否明确
- 是否存在前端无法直接实现或需要产品/后端进一步确认的问题

请输出：
- 前端需求理解总结
- 前端审阅报告
- 需要确认的问题清单
```

---

## 3. 审阅报告发给 PM

```text
这是 AI 对本次 PRD 的审阅报告。报告中列出了一些需要确认的问题，请在“产品回复”列填写答案。
其中标注【阻塞】的问题请优先回复，我们后续技术设计和开发会依赖这些信息。
```

---

## 4. PM 只补回复，未改 PRD

```text
产品经理已经在审阅报告中补充了回复，请读取报告中的“产品回复”内容，整合到你的理解中，并基于更新后的结论继续推进后续技术设计。
审阅报告路径：[路径]
```

---

## 5. PM 小幅修改 PRD，做增量审阅

```text
PM 更新了 PRD，新版本路径如下：
[路径]

本次改动主要是补充细节，没有新增核心功能，也没有调整主流程。
请针对改动部分进行增量审阅，不需要重新做全量审阅，并将结果补充到原审阅结论中。
```

---

## 6. PM 大幅修改 PRD，重新审阅

```text
PM 更新了 PRD，新版本路径如下：
[路径]

本次改动涉及新增模块或核心流程调整，请重新执行完整审阅，覆盖之前的审阅产物，并输出新的需求理解总结、审阅报告和待确认问题清单。
```

## 7. 后端技术设计

```text
请使用 backend-tech-design-mvc，基于已确认的 PRD 和现有后端代码，从当前项目既有架构兼容性的视角完成本次迭代的技术设计。

输入资料：
- 已确认的 PRD：[路径]
- 后端代码：[路径]

请输出：
- 迭代变更总纲
- 业务域设计文档
- API 接口文档
- 待确认的技术问题清单

要求：
- 优先遵循现有项目分层、命名和工程模式
- 如果业务规则仍不明确，请标记为待确认项，不要自行补充
```

---

## 8. 后端编码，新会话版

```text
请基于已确认的设计文档和接口文档，实现本次迭代范围内的后端代码。

输入资料：
- 详细设计文档：[路径]
- API 接口文档：[路径]
- 后端工程目录：[路径]

要求：
- 遵循现有 Spring Boot 项目风格和分层方式
- 只实现本次迭代范围内的内容，不要修改无关代码
- 优先复用现有实现模式
- 如果信息不足或文档冲突，请先列出问题，不要自行猜测业务规则

正式修改代码前，请先说明：
- 你理解的本次实现范围
- 预计会修改或新增的主要文件
```

## 9. 后端编码，同会话版

```text
基于前面已经确认的需求、接口设计和技术方案，继续实现本次迭代相关的后端代码。
要求：遵循现有 Spring Boot 项目风格，只实现本次迭代范围内的内容，不要修改无关代码；如果发现设计信息不足或有冲突，先指出问题，不要自行猜测业务规则。
```

## 10. 后端单模块/单接口实现

```text
请读取以下资料，并实现本次后端开发任务：

- 详细设计文档：[路径]
- API 接口文档：[路径]
- 后端工程目录：[路径]

本次目标：
[具体模块名 / 接口名]

要求：
- 仅实现本目标范围内的代码
- 保持与现有项目分层、命名、异常处理、分页和返回结构一致
- 如果文档信息不足，请先列出阻塞问题，不要猜测实现
```

---

## 11. 前端编码，新会话版

```text
请基于 PRD、UI 原型、前端代码和后端 API 接口设计文档，实现本次迭代范围内的前端页面与交互。

输入资料：
- PRD 文档：[路径]
- UI 原型图：[路径]
- 前端代码：[路径]
- 后端 API 接口设计文档：[路径]

要求：
- 以后端 API 接口设计文档作为主要依据
- 如果接口文档信息不足，请只核对当前页面实际依赖的后端接口实现
- 不要一开始全量扫描整个后端项目
- 只实现本次迭代范围内的页面和交互，不要修改无关代码
- 如果 PRD、原型、接口文档之间有冲突，先列出问题，不要自行猜测
```

---

## 12. 前端编码，同会话版

```text
基于前面已经确认的 PRD、原型和接口设计，继续实现本次迭代相关的前端代码。
以后端 API 接口设计文档为主，必要时再核对相关后端实现；只做本次迭代范围内的页面和交互，不要修改无关代码，遇到冲突先指出问题。
```

---

## 13. 前端单页面实现前，先梳理接口清单

```text
请先基于 PRD、原型和接口文档，梳理当前页面的功能点及对应接口清单；确认后再开始实现页面。
```

## 14. 前端单页面实现

```text
请读取以下资料，然后实现 [页面名] 页面：

- UI 原型对应页面：[路径或截图]
- 前端代码：[路径]
- 后端 API 接口设计文档：[路径]
- 参考现有页面：[可选路径]

要求：
- 严格按原型和接口文档实现页面展示与交互
- 如果接口文档信息不足，再核对当前页面相关的后端接口实现
- 保持与现有前端项目风格一致
- 不要修改与本页面无关的代码
- 如发现字段、状态或交互规则不明确，先指出问题
```

---

## 15. 前端核对后端代码的标准说法

```text
以后端 API 接口设计文档为主要依据。
如果文档信息不足，请优先核对与当前页面相关的后端接口实现文件，包括 Controller、请求/返回对象、相关枚举和分页结果结构；不要一开始全量扫描整个后端项目，只有在无法定位相关实现时再搜索后端仓库。
```

---

## 16. 如果后端接口设计文档与已实现代码不一致

```text
如果接口设计文档与后端已实现代码不一致，请明确指出不一致之处。
如已确认后端代码为最终版本，则以后端实际实现的接口结构为准；否则先暂停并列出待确认问题，不要自行选择其中一版继续实现。
```

---

## 17. 编码中发现 PRD/设计有问题

```text
发现当前需求或设计文档存在问题，已不足以支撑继续实现。
请先记录问题点、影响范围和当前阻塞原因，暂停当前实现，等待 PRD 或设计文档更新后再继续，不要在当前会话中直接基于未确认规则继续编码。
```

```text
当前任务进度：
[贴进度]

```

## 19. 会话切换速查规则

```text
继续当前会话：
- 需求和设计未变化
- 当前会话不长
- 本次只推进小范围实现

建议新开会话：
- 已从讨论切换到正式编码
- 当前会话已较长
- 需要重新聚焦某个模块或页面

必须新开会话：
- PRD、设计文档、接口文档已变更
- PM 刚补充了关键规则
- 当前会话已混入旧版本结论
```
