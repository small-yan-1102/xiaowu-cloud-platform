## 1. 需求概述

### 1.1 背景与目标
**背景：**
当前AMS系统中，【YT频道】的"当下运营类型"与"结算用运营类型"耦合在同一字段（主表 `pageAttribute`）。当总编室人员提前维护下月运营类型（如：4月20日将频道改为5月起归属分销商）时，月底定时任务快照会抓取到新值，导致当月结算数据取错（4月本应按代运营结算，却取到了分销商运营）。

**目标：**
通过产品机制优化，在运营类型发生变更时，系统自动识别并提供将"原运营类型"锁定为当月结算数据的能力，从根源上阻断提前维护对当月快照的污染，确保结算数据的准确性。

### 1.2 用户角色与权限
| 角色 | 权限说明 |
|------|----------|
| **总编室人员** | 负责在AMS系统中维护YT频道的运营类型（单条设置或批量导入） |
| **系统定时任务** | 每月底自动对未发生变更的频道进行运营类型快照归档 |

### 1.3 功能模块总览
本次优化涉及以下两个核心入口的改造：
1. **【设置】弹窗**：单条维护运营类型时，增加变更二次确认机制。
2. **【Excel批量导入】**：批量导入时，系统自动比对变更并静默保护当月历史记录。

### 1.4 需求依赖与变更影响分析
| 依赖系统/模块 | 依赖类型 | 变更影响说明 |
|--------------|----------|--------------|
| **结算系统** | 数据 | 结算系统依赖 `ams_operation_type_history` 表中的按月运营类型数据。本次改造通过提前锁定当月数据，提升了结算数据准确性，不改变结算系统取数接口。 |
| **定时任务** | 状态 | 月底定时任务（`OperationTypeArchiveJob`）依赖历史记录表的状态。由于定时任务使用的是 `INSERT IGNORE`，本次改造提前写入的人工记录不会被定时任务覆盖，两者完美兼容。 |
| **下游运营系统** | 数据 | 主表 `pageAttribute` 变更时会即时推送下游系统。本次改造保持主表更新逻辑不变，对下游无影响。 |

## 2. 功能详述

### F-01 【设置】弹窗-运营类型变更二次确认

#### ① 业务场景与价值
**场景描述**：总编室人员在【YT频道】列表点击"设置"，修改某个频道的运营类型并保存。
**业务价值**：在单条修改场景下，通过系统主动询问，引导用户将修改前的原运营类型锁定为当月的结算数据，防止提前维护导致当月结算错误。

#### ② 业务规则定义
**前置条件**：用户在"设置"弹窗中修改了【运营类型】字段。

**规则 1：变更检测与拦截**
- **触发条件**：用户点击【保存】按钮。
- **处理逻辑**：系统比对用户选择的【新运营类型】与该频道主表当前的【原运营类型】。
- **输出结果**：
  - 如果两者一致：按现有逻辑直接保存。
  - 如果两者不一致：拦截保存动作，弹出二次确认对话框。

**规则 2：二次确认对话框交互**
- **提示文案**："运营类型已从【{原运营类型名称}】变更为【{新运营类型名称}】，是否将【{原运营类型名称}】写入本月（{当前自然月，如：2026-04}）的结算运营类型历史记录？"
- **操作选项**：提供【是】和【否】两个按钮。

**规则 3：确认后的数据处理**
- **如果用户选择【是】**：
  1. 更新主表 `pageAttribute` 为新值。
  2. 将新值推送给下游系统。
  3. **将【原运营类型】写入当前自然月的历史记录表中**（锁定当月结算数据）。
- **如果用户选择【否】**：
  1. 更新主表 `pageAttribute` 为新值。
  2. 将新值推送给下游系统。
  3. **不写入历史记录表**（当月结算数据由月底定时任务自动快照新值）。

**枚举值定义（运营类型）**：
| 枚举值 | 页面显示名称 |
|--------|-------------|
| SELF | 自运营 |
| DYY_ZZ | 制作代运营 |
| YSDYY | 影视代运营 |
| DYY | 代运营 |
| YCDYY | 原创代运营 |
| ZYY_FHBXDQSZGJ | 自运营-符合变现地区所在国家 |
| ZYY_BFHBXDQSZGJ | 自运营-不符合变现地区所在国家 |
| AQY | 爱奇艺 |
| JSWS | 江苏卫视 |
| FSXYY | 分销商运营 |
| CIDZY | CID/RM专用 |
| YCZZ | 原创自制 |
| NONE | 无主页属性 |

#### ③ 验收标准 (AC)
- **AC1（未变更）**：当用户未修改运营类型点击保存时，系统应直接保存，不弹出二次确认框。
- **AC2（发生变更-选是）**：当用户将运营类型从"代运营"改为"分销商运营"并保存时，系统应弹出二次确认框；用户点击"是"后，主表更新为"分销商运营"，当月历史记录表中应新增一条该频道当月为"代运营"的记录。
- **AC3（发生变更-选否）**：当用户将运营类型从"代运营"改为"分销商运营"并保存时，系统应弹出二次确认框；用户点击"否"后，主表更新为"分销商运营"，当月历史记录表中不应新增记录。
- **AC4（文案变量替换）**：二次确认框中的{原运营类型名称}、{新运营类型名称}和{当前自然月}变量应正确替换为实际值（如："运营类型已从【代运营】变更为【分销商运营】，是否将【代运营】写入本月（2026-04）的结算运营类型历史记录？"）。

### F-02 【Excel批量导入】运营类型自动保护

#### ① 业务场景与价值
**场景描述**：总编室人员通过Excel模板批量导入更新多个频道的运营类型。
**业务价值**：在批量维护场景下，系统自动接管当月数据的保护工作，总编室无需改变现有的操作习惯（模板不变），即可实现"修改当下、锁定当月"的效果，降低人工操作成本。

#### ② 业务规则定义
**前置条件**：用户上传符合现有格式的Excel导入模板（包含【运营类型】和【运营类型修改月份】列）。

**规则 1：变更自动检测与静默保护**
- **触发条件**：系统解析导入的每一行数据。
- **处理逻辑**：
  1. 系统比对Excel中填写的【运营类型】与该频道主表当前的【原运营类型】。
  2. **如果发生变更**：
     - 更新主表为Excel中的新值。
     - **系统自动将【原运营类型】写入当前自然月的历史记录表中**（静默锁定当月结算数据）。
  3. **如果未发生变更**：
     - 仅更新主表（或跳过），不写当月历史记录。

**规则 2：【运营类型修改月份】列的处理逻辑**
- 该列保持现有逻辑不变，定位为"纯纠错/追溯历史月份"使用。
- **如果不填**：仅执行上述规则1的自动保护逻辑。
- **如果填了**：在执行规则1的基础上，将Excel中的【运营类型】写入该指定月份的历史记录中。

#### ③ 验收标准 (AC)
- **AC1（导入变更-不填修改月份）**：当导入的Excel中将某频道的运营类型从"代运营"改为"分销商运营"，且未填写"运营类型修改月份"时，系统应将主表更新为"分销商运营"，并自动在当月历史记录中新增一条该频道为"代运营"的记录。
- **AC2（导入未变更）**：当导入的Excel中的运营类型与主表当前值一致时，系统不应在当月历史记录中自动新增记录。
- **AC3（导入变更-填写历史修改月份）**：当导入的Excel中将某频道的运营类型从"代运营"改为"分销商运营"，且"运营类型修改月份"填写了"2026-03"时，系统应将主表更新为"分销商运营"，自动在当月（4月）历史记录中写入"代运营"，并在3月历史记录中写入"分销商运营"。

#### ⑤ 操作路径定义
`【YT频道】列表页` → `点击【导入】` → `选择【导入运营类型】` → `上传Excel文件` → `系统后台解析并执行自动保护逻辑` → `提示导入结果（成功/失败条数）`

## 3. 非功能需求

### 3.1 性能需求
- 页面响应：设置弹窗保存并弹出二次确认框的响应时间不超过1秒。

### 3.2 安全需求
- 操作审计：所有通过设置弹窗和导入修改运营类型的操作，需记录详细的操作日志（操作人、操作时间、修改前值、修改后值）。

### 3.3 兼容性需求
- 浏览器兼容：支持Chrome、Firefox、Edge最新版本。

### 3.4 国际化/多语言
- 不适用。当前系统仅支持中文环境。
