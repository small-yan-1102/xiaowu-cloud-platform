### 2.4 数据来源说明

剧老板的视频下架数据**全部来自 AMS 通过 RocketMQ 同步**，不能在剧老板端直接创建。完整数据同步链路：

```
AMS 创建任务单（分销商来源）
  → 审核通过
    → dispatcher.video_takedown_queue (queue_status=pending)
      → YouTube API 执行（视频删除/私享）
        → dispatcher.video_takedown_queue (queue_status=completed)
          → 回写 AMS 任务单状态 → 已完成
            → RocketMQ（distributionTopic + TAKEDOWN_PROGRESS Tag）
              → 剧老板 silverdawn_distribution.video_takedown_record
```

> **分发层数据库**：`dispatcher @ 172.16.24.61:3306`（user=xiaowu_db，password=}C7n%7Wklq6P）
> YouTube 后台视频下架成功后，可在 `dispatcher.video_takedown_queue` 查询 `queue_status` 确认执行状态。

**关键规则**：
- 仅「已完成」状态的任务单数据会同步到剧老板
- 仅同步有处理结果（成功/失败）的视频，处理中/待处理不展示
- 同步SLA：任务完成后30分钟内
- 仅「分销商来源」的任务单才会同步（运营团队来源不同步）

## 5. 数据产出计划

### 5.1 前置：AMS视频下架套件执行

JLB所有数据依赖AMS视频下架套件（suite_ams_video_takedown.md）的执行产出。按 ams_video_takedown_data_config.md §7 的13轮执行顺序：

- **Round 2** VT-006（分销商来源任务单，HELLO BEAR）→ 走完流程后MQ同步 → JLB有数据
- **Round 5-7** 更多任务单执行完成 → JLB数据量增加

### 5.2 需要在AMS端特别构造的数据

为满足JLB用例的搜索/筛选测试需求，在AMS端创建任务单时应注意：

| JLB用例需求 | AMS构造要求 | 建议操作 |
|------------|-----------|---------|
| 作品名含"测试" | 选择名称含"测试"的作品 | 从HELLO BEAR的510个作品中搜索含"测试"的 |
| 频道名含"Drama" | 选择频道名含"Drama"的频道 | 需确认HELLO BEAR的频道中有名称含"Drama"的 |
| 下架原因"分销商解约" | 创建任务单时选择"分销商解约"原因 | 至少1条选"分销商解约"，另外选其他原因 |
| 同一作品3个频道 | 同一作品在3个不同频道分别创建任务单 | 需找到绑定>=3个频道的作品 |
| >=35条记录（分页） | 创建足够多的任务单 | P3优先级，可适当降低要求 |
| 含成功+失败视频 | 任务单执行后自然产出 | 如全部成功无失败，需构造无效视频ID |
| 全部失败视频 | 所有视频ID无效/已被删除 | 故意使用不存在的视频ID |

### 5.3 数据回填时机

AMS视频下架核心流程执行完毕且MQ同步后，需要：
1. 用 {JLB_DIST_USER} 登录剧老板，记录实际的频道ID、视频ID
2. 回填 §4.1 中的占位符（{JLB_KNOWN_CHANNEL_ID} 等）
3. 确认数据量是否满足各用例需求
