## 二、问题清单

### 问题 #1：Ant Design Dropdown 下拉菜单交互阻塞

| 项目 | 内容 |
|------|------|
| **严重程度** | 高（导致首轮全部 12 条用例阻塞） |
| **影响用例** | AMS-VTD-001/002、AMS-TER-001 及其全部下游用例 |
| **问题描述** | AMS 系统的「创建任务单」和「创建解约单」按钮为 Ant Design Dropdown+Button 组合组件，点击后弹出二级菜单（"按作品创建"/"按视频ID创建"）。browser-use MCP 无法稳定完成"点击按钮 → 等待菜单渲染 → 点击菜单项"的交互序列。菜单项在悬停/点击触发后才动态渲染到 DOM 中，浏览器代理的元素定位和点击时序控制不足。 |
| **重试记录** | 共尝试 3 次，复现率 100% |
| **解决方式** | **绕过 UI，改用 API 直接调用**。通过 SSO API 获取 token 后，直接调用后端接口创建测试数据：<br>- `POST /video-takedown/task/create-by-composition` 创建视频下架任务单<br>- `POST /video-takedown/task/{taskId}/audit` 审核通过<br>- `POST /compositionTerminate/create` 创建解约单 |
| **经验教训** | 1. Ant Design Dropdown 的延迟渲染机制是 AI 浏览器自动化的已知短板<br>2. 冒烟测试应提前准备 API 备选路径，不应完全依赖 UI 操作<br>3. 前端应在下拉菜单项上添加 `data-testid` 属性，提升定位稳定性 |

### 问题 #2：SPA 导航不稳定

| 项目 | 内容 |
|------|------|
| **严重程度** | 中 |
| **影响用例** | AMS-VTD-005（列表页 Tab 切换与搜索） |
| **问题描述** | 浏览器代理进入 AMS 系统后，通过左侧菜单导航到视频下架页面成功率不稳定；直接 URL 导航在 SPA 应用中返回 404。 |
| **解决方式** | 二轮执行时，利用已存在的浏览器会话状态，配合 API 数据查询和 DB 验证，绕开纯依赖导航的步骤。最终通过存量数据直接在已打开的页面上操作完成验证。 |
| **经验教训** | SPA 应用不能直接通过 URL 跳转到子路由；应先进入主页再通过菜单导航。 |

### 问题 #3：BUG-002 误判 — 使用了错误的测试数据

| 项目 | 内容 |
|------|------|
| **严重程度** | 中（导致一次无效的 BUG 提报） |
| **影响用例** | AMS-VTD-006（编辑审核拒绝任务单并重新提交） |
| **问题描述** | 初测时使用 V2603300002（CREATE_FAILED 状态）作为测试数据执行编辑重提交操作，提交成功但状态仍为 CREATE_FAILED，被判定为 BUG。实际原因是 V2603300002 底层视频已被 V2603300001（VIDEO_DELETE）删除，后端无法正确处理已删除视频的重提交。 |
| **解决方式** | 使用正确状态的数据 V2603300003（REVIEW_REJECTED）重测，编辑重提交后状态正确流转为 PENDING_REVIEW。BUG-002 关闭。 |
| **经验教训** | 1. 测试数据选择需检查底层数据完整性，不能仅看表面状态<br>2. CREATE_FAILED 状态的任务单底层数据可能不可用，不应作为编辑重提交的测试对象<br>3. 遇到失败时，应先排查测试数据是否满足前置条件 |

### 问题 #5：ENV-001 误判 — is_linked 字段错误归因

### 问题 #10：直接修改 COMPLETED 任务的 team_id 未触发 MQ

| 项目 | 内容 |
|------|------|
| **严重程度** | 高（这是最难排查的问题） |
| **影响用例** | JLB-VM-001、JLB-VM-002、AMS-E2E-001 |
| **问题描述** | 第一次尝试：直接在 DB 中为已 COMPLETED 的 V2603310003 设置 `task_source=DISTRIBUTOR` 和 `team_id`，等待 30 秒后 AMS 定时任务日志仍然报"无待同步的任务单"。MQ 未发送。 |
| **根因分析** | 通过分析 AMS 服务器日志（`/alidata/webroot/silverdawn-ams-server/all.log`，98MB）发现：`VideoTakedownProgressSchedule` 定时任务（每 10 秒执行）查询的是 `WHERE status = 'PENDING_PROCESS'` 而非 COMPLETED。MQ 消息在 **PENDING_PROCESS → COMPLETED 状态转换时** 发送，不是通过轮询 COMPLETED 任务触发。 |
| **解决方式** | **状态回滚 + 重审方案**：<br>1. 将 V2603310003 状态回滚至 PENDING_REVIEW（保留 team_id）<br>2. API 审核通过 → 状态变为 PENDING_PROCESS<br>3. 定时任务检测到 PENDING_PROCESS → 查询 Dispatcher 进度 → 更新为 COMPLETED<br>4. 检测到 team_id 存在 → 发送 MQ（TAKEDOWN_PROGRESS）<br>5. 剧老板消费 MQ → 插入 3 条 video_takedown_record |
| **MQ 证据** | <br>AMS 发送: `[MQ发送成功] Topic=Topic_Distribution_Test, msgId=...06E`<br>JLB 消费: `接收MQ同步消息, tag:TAKEDOWN_PROGRESS, messageId:...06E`<br>JLB 插入: `[下架记录] 批量插入 total=3, saved=3, skipped=0` |
| **经验教训** | 1. **理解业务架构再操作**：MQ 触发机制与数据轮询不同，需理解状态机转换逻辑<br>2. **服务器日志是最终真相**：定时任务的 SQL 查询条件只能从日志中确认<br>3. **状态回滚+重走流程** 比直接修改数据更可靠 |

| 项目 | 内容 |
|------|------|
| **严重程度** | 高（测试质量标准问题） |
| **问题描述** | 用户指出 JLB-VM-001 等用例标记为 BLOCKED 时，结论是基于推理（"无 DISTRIBUTOR+COMPLETED 任务 → MQ 不会同步"），但没有实际的 MQ 日志或失败凭证支撑。用户要求"用例执行结果需要有实际的数据、MQ消费支撑"。 |

| **经验教训** | 1. **BLOCKED 结论也需要证据**：不能仅靠推理判断，需实际查看日志和数据库确认<br>2. **测试执行需追求"有凭有据"**：每个结论都应有数据、日志、截图支撑<br>3. **阻塞用例应尽力构造数据解除阻塞**，而非简单标记为 BLOCKED |

## 三、关键架构发现

在解决问题 #10 的过程中，发现了 AMS 视频下架系统的核心 MQ 同步架构：

```
任务创建 (CREATING)
  → 异步验证 → PENDING_REVIEW
    → 审核通过 → PENDING_PROCESS
      → VideoTakedownProgressSchedule (每10秒轮询)
        → 检测 status=PENDING_PROCESS 的任务
          → 调用 Dispatcher API 查询执行进度
            → 更新视频明细状态
              → 全部完成 → status=COMPLETED
                → 检查 team_id 是否存在
                  → 若存在 → 发送 MQ (Topic=Topic_Distribution_Test, Tag=TAKEDOWN_PROGRESS)
                    → 剧老板消费 → 插入 video_takedown_record
```

**关键结论**：
- MQ 消息在 **PENDING_PROCESS → COMPLETED** 状态转换时触发，不是轮询 COMPLETED 任务
- 只有 `team_id` 不为空的任务才会发送 MQ
- 定时任务间隔 10 秒，延迟可控
- MQ 使用阿里云 ONS (RocketMQ)，Guava Retryer 重试 3 次，失败记录入 `mq_messages_fail` 表
