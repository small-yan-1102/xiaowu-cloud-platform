### 整体评估说明

本轮为技术文档更新后的复审。第1轮审阅发现的2个阻塞性问题（抽屉宽度矛盾、修改记录加载方式矛盾）**仍未解决**。

本轮复审新发现 **2个严重阻塞性问题**，涉及 CRM 发布配置的数据结构设计在文档间存在根本性矛盾：

1. **CRM 详细设计与 CRM API 设计的数据结构不一致**：详细设计将发布配置放在 `channelList[]` 内（按频道），API 设计将发布配置放在独立的 `compositionList[]` 中（按作品）。这是两种完全不同的数据架构，直接影响数据库表设计、MQ 消息体结构和 AMS 消费逻辑。
2. **CRM MQ 消息体输出与 AMS MQ 消费输入不匹配**：CRM 发送的消息以 channelId 为核心（无 compositionId），AMS 期望接收的消息以 compositionId 为核心（无 channelId），两个系统之间的消息契约无法衔接。

**B=4（含2个共享根因），核心问题为 CRM 侧数据结构设计矛盾**。按判定标准 B>3 趋向"需重新设计"，但考虑到：(1) B1/B2 为规格确认类问题（非设计缺陷），(2) B3/B4 共享一个根因且可能是文档更新不同步导致的版本冲突，综合评估为 **"需补充后进入"** —— 但 **B3/B4 必须在测试设计前完成对齐**，否则 AI 无法判断 CRM API 的正确请求结构。

> **开发回复更新（2026-03-31）**：4 个阻塞性问题已全部获得开发回复，状态从 **4 Open → 4 已确认**：
> - **B1**：确认 900px（与 PRD 一致，技术文档需更新）
> - **B2**：确认使用分页方式（与技术文档一致，PRD 需更新）
> - **B3**：**架构方案重大变更** — CRM→AMS 数据同步从 MQ 异步消息推送改为复用现有同步接口 `DomesticChannelController.syn()`（同步调用），CRM/AMS 两侧 VO 各新增 6 字段
> - **B4**：随 B3 解决（改用同步接口后，MQ 消息体不匹配问题不再存在）
>
> **测试设计影响**：B3 的架构变更将 CRM→AMS 同步从 MQ 异步改为同步 API 调用，SYNC 测试套件需从验证 MQ 异步消费改为验证同步 API 响应；MQ 相关等待策略（A-1-S1/A-4-S2）和可靠性风险（NR-1）的前提条件发生变化。

### 7.1 问题现状：两套互斥的数据架构

```
方案A（CRM详细设计采用）—— 发布配置在 channelList 内

│ }                                                        │
└─────────────────────────────────────────────────────────┘

方案B（CRM API设计采用）—— 发布配置在独立的 compositionList 内
┌─────────────────────────────────────────────────────────┐
```

### 7.2 各文档的立场

| 文档 | 采用方案 | 关键证据 |
|------|---------|---------|
| CRM详细设计 §2.1.1 入参 | 方案A（channelList） | `channelList[].internalPublish` |
| CRM详细设计 §7.1 数据库 | 方案A（channelList） | 发布配置字段在 delivery_receipt 表上（per-channel） |
| CRM详细设计 §8.2 DTO | 方案A（channelList） | DeliveryReceiptDto 新增字段在 channelList 子对象中 |
| CRM详细设计 §10.1 MQ | 方案A（channelList） | MQ消息含 channelId，不含 compositionId |
| **CRM API设计 §2.1.1 请求体** | **方案B（compositionList）** | **独立的 compositionList[] 数组** |
| CRM API设计 §2.1.2 响应 | 方案B（compositionList） | 响应含独立的 compositionList[] |
| CRM API设计 §B DTO汇总 | **方案A（channelList）** | DeliveryPrChannelVo 新增发布配置字段 |
| AMS详细设计 §2.5.1 MQ消费 | 方案B（compositionList） | 期望消息含 compositionId |
| **PRD §补充说明.4** | **方案B（compositionList）** | "以作品为单位，交接组概念不影响同步" |

### 7.3 影响链路

```
CRM前端 → CRM后端(API) → CRM数据库 → MQ消息体 → AMS消费 → AMS数据库
  ?          A+B冲突        A方案        A方案       B方案      B方案

全链路上存在方案A/B的交叉，无法形成闭环。
```

### 7.4 建议对齐方向

基于PRD §补充说明.4"以作品为单位"的明确定义，**建议采用方案B（compositionList）**。需要修改的文档/设计：

1. CRM详细设计 §2.1.1：入参从 channelList 内移到独立的 compositionList
2. CRM详细设计 §7.1/§7.2：数据库设计需增加作品-发布配置的存储方案（可能需要独立表或JSON字段）
3. CRM详细设计 §8.2：DTO定义更新
4. CRM详细设计 §10.1：MQ消息体增加 compositionId、operator 字段
5. CRM API设计 §B：DeliveryPrChannelVo 移除发布配置字段，确保与 compositionList 方案一致
6. CRM详细设计 §2.1.2：返回结构增加独立的 compositionList
