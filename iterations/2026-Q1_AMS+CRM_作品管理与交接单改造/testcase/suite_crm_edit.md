# CRM 交接单编辑模块测试用例

**执行范围**：CRM 交接单编辑功能（F03-E），包含数据回显、保存并重新交接、联动规则、关闭确认、语种互斥验证、重置规则验证、编辑权限控制、首发时间选项验证、并发编辑风险验证
**预估用例数**：23 条
**关联需求版本**：PRD V1.2 第四节 F03-E
**预估执行时间**：约 45 分钟
**适用范围约束**：本期交接单改造仅支持线索类型=作品类（PGC），不支持频道类（UGC）。所有用例的测试数据须使用 PGC 类型的交接单/作品。依赖创建套件（suite_crm_create.md）的用例自动继承 PGC 数据约束。

### ZWGL-EDIT-001 编辑已交接状态交接单并重新交接成功

**优先级**：P0
**用例类型**：冒烟测试
**所属模块**：CRM 交接单编辑
**关联需求**：PRD V1.2 F03-E §E.5
**依赖用例**：ZWGL-CREATE-001（需先执行且通过，产出数据：一条状态为"已交接"的交接单）

**前置条件**：
1. [登录态][SYS-CRM] 使用账号 15057199668 / 验证码 1111 通过 SSO 登录，进入 CRM 系统
2. [页面定位] 导航到 CRM「交接单管理」列表页
3. [数据存在][数据准备: 依赖用例 ZWGL-CREATE-001] 列表中存在一条状态为"已交接"的交接单记录

**测试步骤**：
1. 在交接单列表中找到状态为"已交接"的交接单记录（由 ZWGL-CREATE-001 创建）
2. 点击该记录的「编辑」按钮
3. 在编辑弹窗中修改「频道发布限制」为另一个选项（如从"无限制"改为"仅CP指定频道"）
4. 点击「保存并重新交接」按钮

**预期结果**：
1. 编辑弹窗出现，标题为"编辑交接单"（L1）
2. 弹窗中已有数据回显（作品、发布配置等字段已填充）（L1）
3. 底部按钮显示为「保存并重新交接」（因交接单状态为已交接）（L1）
4. 修改频道发布限制后，值已更新显示（L1）
5. 点击「保存并重新交接」后弹窗关闭（L1）
6. 页面顶部出现 Toast 提示 {包含:编辑成功，已重新交接}（L1）
7. 交接单列表刷新，该记录状态仍为"已交接"（L1）
8. 页面无错误弹窗、无红色错误提示（L1 反向验证）

**是否需要人工介入**：否
**备注**：重新交接成功同时触发 CRM→AMS 数据同步。

### ZWGL-EDIT-021 并发编辑同一交接单验证数据覆盖风险

**优先级**：P2
**用例类型**：异常测试
**所属模块**：CRM 交接单编辑
**关联需求**：PRD V1.2 F03-E §E.5
**关联技术文档**：tech-doc-review-report.md §2.3 C-7（CRM详设 §2.1.1 先删后插模式）
**依赖用例**：ZWGL-CREATE-001（需先执行且通过，产出数据：一条已交接状态的可发布交接单）

**前置条件**：
1. [登录态][SYS-CRM] 使用账号 15057199668 / 验证码 1111 通过 SSO 登录，进入 CRM 系统
2. [页面定位] 在浏览器 Tab1 和 Tab2 分别打开 CRM「交接单管理」列表页

**测试步骤**：
1. [Tab1] 点击目标交接单的「编辑」按钮，打开编辑弹窗
2. [Tab2] 点击同一条交接单的「编辑」按钮，打开编辑弹窗
3. [Tab1] 修改「频道发布限制」从"无限制"改为"仅CP指定频道"
4. [Tab1] 点击「保存并重新交接」按钮
5. [Tab1] 确认 Toast 提示 {包含:编辑成功，已重新交接}
6. [Tab2] 修改「字幕可发语种」：增加语种 B（不修改频道发布限制）
7. [Tab2] 点击「保存并重新交接」按钮
8. [Tab2] 观察是否成功提交或提示冲突
9. [任意Tab] 刷新列表页，点击该交接单的「编辑」按钮查看最终数据

**预期结果**：
1. [Tab1] 编辑并重新交接成功（L1）
2. [Tab2] 提交时，系统有以下两种可能的合理表现之一（L1）：
   - 方案A：提交成功（无冲突检测），Tab2 的修改覆盖 Tab1 的全部修改 → 最终数据中频道发布限制回到"无限制"（Tab1 的修改丢失）
   - 方案B：提交失败，提示"数据已被修改，请刷新后重试"或类似冲突提示
3. [验证最终数据] 打开编辑弹窗查看：（L1）
   - 如果方案A（覆盖）：频道发布限制="无限制"（Tab1修改丢失），字幕可发语种包含语种 B（Tab2修改生效）
   - 如果方案B（拒绝）：频道发布限制="仅CP指定频道"（Tab1修改保留），字幕可发语种不含语种 B（Tab2修改未生效）
4. 无论哪种方案，系统不应出现数据部分丢失或脏数据（如频道发布限制和语种配置同时丢失）（L1）

**是否需要人工介入**：否
**备注**：技术文档审阅发现 CRM 交接单更新采用"先删后插"模式（CRM详设 §2.1.1 步骤4：先删除原有记录再批量插入新记录），两个用户同时编辑时存在数据覆盖风险。本用例旨在确认并发编辑的实际行为（是覆盖还是冲突检测），记录实际表现供团队评估是否需要增加乐观锁机制。此用例的通过标准为"行为一致且无脏数据"，而非要求必须有冲突检测。
